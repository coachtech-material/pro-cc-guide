## 導入: 開発現場で起きている変化

プログラミングスクール COACHTECH の選抜試験を経て、フリーランスエージェントに所属しているあなた。これから企業に紹介され、Laravel を使った実務開発に携わっていくことになります。この教材では、あなたのように COACHTECH から実務に踏み出すエンジニアを「Pro生」と呼びます。

Pro生として実務に入ると、スクールの課題とは異なる壁にぶつかります。コードベースは数万行、数十万行の規模になり、仕様書を読み解きながら既存のコードに手を入れる必要があります。「どこから読めばいいかわからない」「この変更が他の機能に影響しないか不安」。こうした悩みは、ジュニアエンジニアなら誰もが経験するものです。

一方、開発の現場では AI コーディングツールの導入が急速に進んでいます。GitHub Copilot、Cursor、そして Claude Code。これらのツールは、コード補完にとどまらず、バグの原因調査、リファクタリングの提案、テストコードの生成まで幅広く支援してくれます。

重要なのは、AI ツールの普及が採用市場にも影響を及ぼしているという事実です。一人のエンジニアが AI を活用して高い生産性を発揮できるようになった結果、企業はより少ない人数で開発を回せるようになりました。Stack Overflow の調査記事「[AI vs Gen Z: How AI has changed the career pathway for junior developers](https://stackoverflow.blog/2025/12/26/ai-vs-gen-z/)」によると、エンジニアリングリーダーの 54% が AI コパイロットの導入を理由にジュニアの採用数を減らす計画だと回答しています。採用のハードルは上がり、ジュニアエンジニアには以前よりも高いバリューが求められるようになっています。

つまり、AI コーディングツールを使いこなせるかどうかは、もはや「便利かどうか」の問題ではありません。実務で成果を出すための前提条件になりつつあります。

### 🧠 先輩エンジニアはこう考える

> 自分がジュニアの頃は「AI なんてまだ先の話」と思っていました。ところが、現場で AI コーディングツールが当たり前になった途端、コードを書くスピードだけでなく、設計判断の質まで変わり始めました。
>
> たとえば、慣れないライブラリの使い方を調べるのに以前は半日かかっていたのが、Claude Code に「このプロジェクトで○○を実装するならどう書く？」と聞くだけで、既存コードの規約に沿った実装案が返ってくる。もちろん、その提案をそのまま採用していいかを判断するのは自分です。でも「ゼロから調べる時間」が圧倒的に減ったことで、設計を考える時間やレビューの質に回せる余裕が生まれました。
>
> ジュニアのうちから AI ツールを正しく使えると、「コードを書ける人」ではなく「質の高いソフトウェアを届けられる人」として評価されやすくなります。逆に言えば、AI ツールを使えない状態で現場に入ると、周囲とのスピード差に苦しむことになるかもしれません。

## Claude Code とは何か

Claude Code は、Anthropic が開発した AI コーディングツールです。ターミナル（コマンドライン）で動作し、あなたのプロジェクト全体を読み解いたうえで、ファイルの編集、コマンドの実行、Git 操作などを自律的に行います。

ChatGPT のようなチャット型 AI と比較すると、Claude Code の最大の特徴は **エージェント型** であることです。チャット型 AI はあなたが貼り付けたコードに対して回答を返しますが、Claude Code はプロジェクトのディレクトリ構造を探索し、必要なファイルを自分で読み、コードを生成し、テストを実行し、エラーがあれば修正する、という一連の作業を自律的に進めます。

```
あなたの指示
    ↓
Claude Code がプロジェクトを探索
    ↓
関連ファイルを読み込み、変更方針を決定
    ↓
コードを生成・編集
    ↓
テスト実行・エラー修正
    ↓
結果を報告
```

Claude Code は以下のような場面で力を発揮します。

- **バグ修正**: エラーメッセージを渡すだけで、原因を特定し修正案を提示する
- **機能開発**: 自然言語で要件を伝えると、複数ファイルにまたがる実装を行う
- **リファクタリング**: 既存コードの改善点を指摘し、安全に書き換える
- **コードリーディング**: 大規模なコードベースの構造や処理の流れを解説する
- **Git 操作**: コミットメッセージの生成、ブランチ作成、PR の作成を行う

Claude Code はターミナルだけでなく、VS Code や JetBrains などの IDE 内、デスクトップアプリ、さらに Web ブラウザからも利用できます。ただし、この教材ではターミナルでの使用を基本とします。ターミナル操作は Claude Code のすべての機能にアクセスできる最も基本的な方法であり、開発環境に依存しないためです。

## なぜ「Claude Code」なのか

AI コーディングツールは Claude Code だけではありません。GitHub Copilot や Cursor など、選択肢は複数あります。そのなかで Claude Code を選ぶ理由を整理します。

**エージェント型であること**

補完型ツール（GitHub Copilot など）は、あなたが書いているコードの「次の行」を予測して提案します。便利ですが、作業の主体はあなた自身です。一方、Claude Code はタスク全体を委任できます。「認証機能のテストを書いて、実行して、失敗したら修正して」という指示を出せば、Claude Code が一連の作業を自律的に完了させます。

**コードベース全体を理解すること**

Claude Code はプロジェクト内のファイルを自由に探索できます。特定のファイルを指定しなくても、関連するモデル、コントローラ、テスト、設定ファイルを横断的に読み取ったうえで回答します。これは、大規模なコードベースを扱う実務において特に重要です。

**ターミナルネイティブであること**

Claude Code はターミナルで動作するため、既存の開発ワークフロー（Git、Docker、テストコマンド等）とシームレスに連携できます。新しいエディタや IDE を覚える必要がなく、普段の開発環境にそのまま組み込めます。

> 📝 AI コーディングツールの分野は急速に進化しています。ツール間の機能差は今後変わる可能性があります。この教材で重視するのは、特定のツールに詳しくなることではなく、AI コーディングツールを正しく活用するための「考え方」を身につけることです。Claude Code はそのための最適な題材です。

## AI との協働に求められる思考力

ここまでで、Claude Code がエージェント型の AI コーディングツールであり、プロジェクト全体を理解して自律的にタスクを遂行することを学びました。ここで一つ、AI コーディングツール全般について向き合っておきたいテーマがあります。

### 「AI を使うと考えなくなる」は本当か

AI コーディングツールに対して、「AI に頼ると自分で考えなくなる」「思考力が衰える」という批判を耳にすることがあります。コードを AI に書かせるだけなら、確かにそう見えるかもしれません。

しかし、AI を活用した開発の実態を見ると、その逆です。

ThoughtWorks のエンジニア Nik Malykhin 氏は、AI アシスト開発について「AI を使うとき、私たちはより**多く** 考える必要がある。少なくではない（we need to think *more* when assisted by AI, not less）」と述べています（ThoughtWorks「[Do Developers Need to Think Less with AI?](https://www.thoughtworks.com/en-us/insights/blog/generative-ai/do-developers-need-think-less-ai)」）。曖昧な指示を出せば曖昧な結果が返り、手戻りが増える。むしろ AI を効果的に使うには、問題を明確に言語化し、構造的に分解する力が求められるというのです。

Kent Beck 氏（テスト駆動開発の提唱者）も、AI との協働を「1時間あたりの重要な判断が増え、退屈な定型作業が減る（more consequential programming decisions per hour, fewer boring vanilla decisions）」と表現しています（Kent Beck「[Augmented Coding: Beyond the Vibes](https://tidyfirst.substack.com/p/augmented-coding-beyond-the-vibes)」）。AI が定型作業を引き受けるからこそ、人間はより高度な判断に集中することになります。

つまり、AI は「考えなくていい」ツールではなく、**「考える領域が変わる」** ツールです。コードの1行1行を書く作業から解放される代わりに、「何を作るべきか」「どう分解すべきか」「この出力は正しいか」という、より抽象的で本質的な思考が求められるようになります。

### 具体と抽象を行き来する

では、AI との協働で求められる「思考」とは具体的に何でしょうか。その核にあるのが、**具体と抽象の行き来** です。

Google Chrome チームの Addy Osmani 氏は、AI を活用する開発者に求められる姿勢を次のように説明しています。「AI は私がより抽象的なレイヤー（設計、インターフェース、アーキテクチャ）に集中することを可能にし、定型コードは AI が生み出す（the AI lets me operate at a higher level of abstraction ... while it churns out the boilerplate）」（Addy Osmani「[My LLM Coding Workflow Going Into 2026](https://addyosmani.com/blog/ai-coding-workflow/)」）。しかし同時に、抽象的なレイヤーで的確に考えるためには、具体的なコードやシステムへの深い理解が前提になるとも指摘しています。

この関係をピラミッドで表すと、次のようになります。

```
        あなたが考える
        ┌──────────┐
        │   抽象    │  「受講管理機能を作りたい」
        │ (ゴール)  │
        └────┬─────┘
             │ 分解（漏れなく・重複なく構造化する）
             ▼
    ┌────────────────────┐
    │     受講登録        │
    │     進捗管理        │  「3つの領域に分かれる。
    │     完了処理        │    それぞれにモデル、ルーティング、権限がある」
    └────────┬───────────┘
             │ 分解（さらに具体化する）
             ▼
┌──────────────────────────────┐
│  Enrollment モデル / テーブル   │
│  EnrollmentController@store  │  AI が実装する
│  LessonProgress モデル        │
│  Policy / Migration / Test   │
└──────────────┬───────────────┘
               │
               ▼
          検証 ──→ 次のサイクルへ（抽象に戻る）
```

ピラミッドの頂点にある抽象的なゴールを、あなたが漏れなく・重複なく分解して中間層を作り、AI がそれを具体的なコードとして実現する。そしてこのサイクルは一度で終わりません。AI が生成した具体的なコードを検証し、新たな課題を見つけ、再び抽象的なレベルで方針を立て直す。この **具体と抽象の行き来を高速に繰り返す** ことが、AI との協働の本質です。

ここで特に重要なのが、層と層をつなぐ **「分解」** のプロセスです。ゴールを具体的な作業に落とし込むとき、漏れなく・重複なく分解できているかどうかで、AI の出力品質が大きく変わります。

分解が甘いと何が起きるのか。たとえば「受講管理機能を作って」とだけ指示した場合、AI は「受講管理」という言葉から想像できる範囲で実装を進めます。あなたが想定していなかった管理画面を作り始めるかもしれないし、逆に進捗率の計算ロジックが丸ごと抜け落ちるかもしれません。AI は指示されていないことを「察する」わけではなく、指示の範囲を自分なりに解釈して動くのです。

Gene Kim 氏と Steve Yegge 氏は、AI との協働における鍵を次のように述べています。「すべてのタスクを、可能な限り小さなステップに分解せよ（decompose and subdivide every task into the smallest steps you can）」（Gene Kim & Steve Yegge「[The Three Developer Loops](https://itrevolution.com/articles/the-three-developer-loops-a-new-framework-for-ai-assisted-coding/)」）。大きなタスクを一度に AI に投げると、動くが理解しづらいコードが生まれる。小さく分解して一つずつ実装させることで、各ステップの出力を確認しながら進められるというわけです。

### Laravel で考えてみよう

具体的な例で見てみましょう。Laravel で「受講管理機能を作りたい」というゴールがあるとします。ピラミッドの各層に対応させて、3パターンの指示の出し方を比較します。

**抽象のまま丸投げした場合（ピラミッドの頂点だけ）:**

```
> 受講管理機能を作って。生徒がコースに登録して、レッスンを進めて、完了できるようにして。
```

この指示では、AI はひとまとめに実装を進めます。結果として以下のような問題が起きやすくなります。

- 受講登録時に「定員チェック」や「重複登録の防止」が漏れる
- 進捗管理で「どのレッスンまで完了したか」の粒度が曖昧になる
- 完了条件（全レッスン完了？ テスト合格？）が未定義のまま実装される
- 権限設計（生徒は自分の受講のみ閲覧、管理者は全件閲覧）が考慮されない

**1段階分解して指示した場合（ピラミッドの中間層まで）:**

```
> 受講管理機能を作ります。以下の3つの領域に分けて、順番に実装してください。
>
> 1. 受講登録: 生徒がコース詳細画面から受講登録する。重複登録は防止する
> 2. 進捗管理: レッスンごとに完了/未完了を記録し、コース全体の進捗率を計算する
> 3. 完了処理: 全レッスン完了時にコースを完了状態にする。生徒は自分の受講のみ操作可能とする
>
> まず受講登録から始めてください。
```

頂点のゴールを3つの領域に分解し、それぞれの要件を明示しています。AI はこの構造に沿って「受講登録」から順に実装を進められます。定員チェック、完了条件、権限設計といった重要な要素も指示に含まれているため、漏れにくくなります。

このように、1段階分解するだけでも AI の出力は大きく改善します。さらに、ここから各領域をより具体的なタスクに分解していくこともできます。

**さらに具体化した場合（ピラミッドの下層まで）:**

```
受講管理機能
├── 受講登録
│   ├── 生徒がコース詳細画面から受講登録する
│   ├── 重複登録を防止する
│   ├── Enrollment モデルと enrollments テーブルを作成する
│   └── EnrollmentController@store でバリデーションと登録処理を行う
│
├── 進捗管理
│   ├── レッスンごとに完了/未完了を記録する
│   ├── LessonProgress モデルで進捗を管理する
│   └── コース全体の進捗率を計算するメソッドを Enrollment に追加する
│
└── 完了処理
    ├── 全レッスン完了時にコースを完了状態にする
    ├── 完了日時を記録する
    └── Policy で「自分の受講のみ操作可能」を制約する
```

ここまで分解するには、Laravel の Eloquent リレーション、Controller の責務、Policy による認可といった**具体的な知識** が必要です。具体を知っているからこそ、適切な粒度で分解できるのです。

> 💡 実務では、いきなり最下層まで分解する必要はありません。まず1段階分解して AI に渡し、出力を見てから次の分解に進むというサイクルが自然です。この教材でも、この段階的な進め方を実践していきます。

> 💡 この「分解」の技術は、プログラミングに限った話ではありません。バーバラ・ミント氏がコンサルティングの世界で提唱した「ピラミッド原則（The Pyramid Principle）」では、伝えたい結論をまず提示し、それを支える根拠を漏れなく・重複なく構造化する方法が説かれています。AI への指示も、相手に伝わる構造で組み立てるという点で本質は同じです。

### AI の進化と抽象度の変化

ここで一つ補足しておきたいのが、AI の進化に伴って「どの抽象度で指示を出せるか」が変わり続けているという点です。

ほんの数年前まで、AI にコードを書かせるには非常に具体的な指示が必要でした。「この関数にこの引数を渡して、この条件分岐で処理して」というレベルです。しかし現在の Claude Code のようなエージェント型ツールは、「認証機能を追加して」という抽象的な指示からでも、プロジェクトの構成を読み取り、適切なファイルに適切なコードを生成できます。

これは素晴らしい進化ですが、同時に注意が必要です。**AI が受け取れる抽象度が上がるほど、人間の側に求められる構造化の力も高まります。** なぜなら、抽象度の高い指示はそれだけ解釈の幅が広く、AI が「あなたの意図と違う方向」に進むリスクも大きくなるからです。

```
AI の進化と抽象度の変遷:

  以前:  具体的な指示 ──→ 具体的な実装
         （関数単位）      （1ファイル）

  現在:  抽象的な指示 ──→ 複数ファイルにまたがる実装
         （機能単位）      （Model, Controller, Migration, Test ...）

  未来:  より抽象的な指示 ──→ システム全体にまたがる実装
         （プロダクト単位）    （アーキテクチャ設計、インフラ構成を含む）

  人間に求められる力:

  以前:  コードを書く力（具体→具体）

  現在:  ゴールを構造化して伝える力（抽象→分解→具体の検証）

  未来:  ビジネス要件を技術要件に翻訳し、AI の設計判断を評価する力
```

AI が進化して「何でもやってくれる」ように見えるほど、あなたが「何をやってほしいか」を明確に構造化する力が重要になる。これは逆説的ですが、この教材を通じて実感してもらえるはずです。

この流れが意味するのは、AI の進化によって**人間が不要になるのではなく、人間が勝負する層が上がっていく** ということです。以前はコードの書き方で差がついていたものが、今は分解と検証の力で差がつく。将来は、ビジネスの課題を正しく捉えて技術的な方針に落とし込む力がさらに重要になるでしょう。具体と抽象を行き来する力は、この変化のどの段階でも土台になります。

### 🧠 先輩エンジニアはこう考える

> 「AI を使うと考えなくなる」という意見、自分は正反対だと思っています。AI を使い始めてから、自分の思考がどれだけ曖昧だったかに気づかされました。
>
> 以前は「なんとなくこういう機能が必要」という状態でコードを書き始めて、書きながら考えていた。それでも自分で書いているから、細部は手を動かしながら調整できた。でも AI に任せるとなると、「なんとなく」では通用しない。何が必要で、何が不要で、どこまでをスコープにするか。それを事前に言語化しないと、AI は見当違いの方向に走ってしまう。
>
> つまり、AI を使うほど「具体を理解して、抽象的に構造化する」力が鍛えられるんです。これは AI がなくなっても残る力で、設計書を書くとき、チームメンバーに説明するとき、要件を整理するとき、すべてに通じます。AI との協働は、エンジニアとしての思考力そのものを磨くトレーニングでもあると思っています。

## この教材で養う3つの能力

Claude Code は強力なツールですが、ただ使うだけでは実務で成果は出せません。AI が生成したコードをそのまま本番環境にデプロイして障害を起こしたり、ツールの制約を理解せずに非効率な使い方を続けたりするリスクがあります。

この教材では、Claude Code を実務で正しく活用するために、3つの能力を段階的に養います。これらは、先ほど述べた「具体と抽象の行き来」を支える能力でもあります。

### 1. 使いこなす力

Claude Code の機能、設定、制約を把握し、目的に応じて的確に活用する力です。

Claude Code は多機能なツールです。プロンプトの書き方、CLAUDE.md による指示の与え方、MCP（Model Context Protocol）を使った外部ツール連携など、活用の幅は広いです。しかし、機能を知らなければ使えません。この能力は、Claude Code に「何ができて、何ができないか」を正確に把握し、場面に応じて最適な使い方を選択する力を指します。

ピラミッドで言えば、**抽象から分解への橋渡し** を担う力です。ゴールを構造化して AI に伝えるとき、Claude Code のどの機能をどう使えば最も効果的に伝わるかを判断します。

### 2. 見極める力

AI が生成したコードを、正しさ・品質・安全性の3つの観点で検証し、責任を持って採用する力です。

Claude Code が生成するコードは、多くの場合そのまま動作します。しかし「動くこと」と「正しいこと」は違います。要件を満たしているか（正しさ）、プロジェクトのコーディング規約に沿っているか（品質）、セキュリティ上の問題がないか（安全性）。この3つの観点で検証できなければ、AI の生成物に責任を持つことはできません。

ピラミッドで言えば、**具体の検証** を担う力です。AI が生成した具体的なコードを、抽象的な要件や基準と照らし合わせて判断します。

この教材では、コードが生成されるたびに「見極めチェック」を実施します。最初はチェックリストに沿って確認し、慣れてきたら自然と3つの観点で検証できるようになることを目指します。

```
          ┌──────────┐
          │   抽象    │
          └────┬─────┘
               │
  使いこなす力  │ 分解（漏れなく・重複なく構造化して AI に伝える）
               │
       ┌───────┼───────┐
       ▼       ▼       ▼
   ┌──────┐┌──────┐┌──────┐
   │具体  ││具体  ││具体  │
   └──┬───┘└──┬───┘└──┬───┘
      │       │       │
  見極める力   │ 検証（正しさ・品質・安全性で判断する）
      │       │       │
      ▼       ▼       ▼
   次のサイクルへ（抽象に戻る）
```

使いこなす力が抽象を分解して AI に伝え、見極める力が AI の出力した具体を検証する。この2つの能力が、具体と抽象の行き来を駆動します。

### 3. 学び続ける力

ツールや技術の進化を追い、自ら試し、実務に適応し続ける力です。

AI コーディングツールは急速に進化しています。Claude Code も頻繁にアップデートされ、新しい機能が追加されます。この教材で学んだ知識が半年後にはそのまま通用しない可能性もあります。だからこそ、公式ドキュメントを読む習慣、新機能を自分で試す姿勢、変化に適応する柔軟性が重要です。

先ほど「AI の進化で抽象度が上がり続ける」と述べた通り、**ピラミッドそのものが変化し続けます。** この力があれば、ピラミッドの形が変わっても、使いこなす力と見極める力をアップデートし続けられます。

この3つの能力は相互に支え合っています。使いこなす力がなければ良い出力は得られず、見極める力がなければ出力を信頼できず、学び続ける力がなければ使いこなす力も見極める力も陳腐化します。

## ✨ まとめ

- AI コーディングツールの普及により、ジュニアエンジニアにも高い生産性が求められる時代になっている
- Claude Code はエージェント型の AI コーディングツールで、プロジェクト全体を理解したうえで自律的にコーディング作業を行う
- AI との協働は「考えなくなる」のではなく、具体と抽象を行き来する思考力がより求められる。ゴールを漏れなく構造化し、AI の出力を検証するサイクルを回すことが本質
- AI の進化で指示の抽象度は上がり続けるが、それに伴い構造化の力の重要性も高まる
- この教材では「使いこなす力」「見極める力」「学び続ける力」の3つの能力を養い、具体と抽象の行き来を支える土台を作る
