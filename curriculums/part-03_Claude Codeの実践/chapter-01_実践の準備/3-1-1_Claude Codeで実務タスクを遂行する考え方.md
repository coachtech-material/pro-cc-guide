# 3-1-1 Claude Code で実務タスクを遂行する考え方

## 🎯 このセクションで学ぶこと

- Part 2（機能の習得）から Part 3（実務での活用）への考え方の転換を理解する
- AI が担う 70% と人間が担う 30% の役割分担を理解する
- AI エージェントの活用がコードレビューのプロセスであることを理解する
- AI 生成コードの検証が必要な理由（脆弱性・設計判断・説明責任）を理解する

まず Part 2 と Part 3 の違いを確認し、次に AI と人間の役割分担を理解し、最後に Part 3 全体の見通しを把握します。

---

## 導入: 機能を知ることと、実務で使いこなすことは違う

COACHTECH の選抜試験を経て実務に携わる Pro生 のあなたは、Part 2 で Claude Code の主要機能を一つずつ学びました。CLAUDE.md の書き方、権限設定、プロンプト設計、Plan Mode、Skills、Hooks、MCP、Sub-agents。自分のプロジェクトやスターターキットを使って、各機能を個別に体験してきました。

しかし、実務で求められるのは「機能を知っていること」ではなく「機能を使って成果を出すこと」です。

実務では、バグ報告を受け取って原因を特定し、修正し、テストで検証し、PR を作成してレビューを受ける。機能追加の要件を読み解き、既存の設計パターンに合わせて実装し、チームに共有する。こうした一連のタスクの中で、Claude Code の機能をどう組み合わせ、どう判断するかが問われます。

Part 3 では、この「実務で使いこなす」力を鍛えます。

Part 3 のメインは、実務で日常的に発生する3つのタスクを Claude Code と協働して遂行することです。

- **バグ修正**: 報告を受けて原因を特定し、修正し、テストで検証する
- **機能開発**: 要件を理解し、既存設計に合わせて実装し、品質を確認する
- **リファクタリング**: 既存コードの構造やパフォーマンスを改善し、動作が変わらないことを保証する

この3つは、ジュニアエンジニアとしてプロジェクトに参画したときに最初に任される典型的なタスクです。提供プロジェクト「CourseHub」を題材に、これらのタスクを一気通貫で実践します。

### 🧠 先輩エンジニアはこう考える

> 振り返ってみると、ツールの機能を一通り覚えた段階と、実務で使いこなせる段階には大きなギャップがある。自分も Claude Code を使い始めた頃、機能は知っているのに思ったほど生産性が上がらなかった時期があった。何が足りなかったかと言うと、「Claude が出力したコードを自分で判断する力」だった。ツールを使いこなすだけでなく、出力を見極める目を持つこと。Part 3 ではそこを徹底的に鍛える。

## AI がルーティンの 70% を処理し、人間の価値は残り 30% に集中する

Google Chrome チームのエンジニアリングリーダーである Addy Osmani 氏は、記事「[The 70% Problem: Hard truths about AI-assisted coding](https://addyo.substack.com/p/the-70-problem-hard-truths-about)」で、AI を活用したコーディングについて「70% の問題」という考え方を提唱しています。

AI はタスクの約 70% を処理できます。定型的なコードの生成、既知のパターンの適用、ボイラープレートの作成など、ルーティンワークの大部分を高速にこなします。しかし残りの 30%、つまりエッジケースの処理、アーキテクチャ上の判断、セキュリティの考慮、ビジネスロジックの正確な反映には、人間の判断が必要です。

```
┌─────────────────────────────────────────────────────────┐
│                  タスク全体（100%）                       │
│                                                         │
│  ┌─────────────────────────────────┐  ┌──────────────┐  │
│  │      AI が処理する 70%          │  │ 人間の 30%   │  │
│  │                                 │  │              │  │
│  │  ・定型コードの生成             │  │ ・設計判断   │  │
│  │  ・既知パターンの適用           │  │ ・エッジケース│  │
│  │  ・ボイラープレート作成         │  │ ・セキュリティ│  │
│  │  ・リファクタリングの実行       │  │ ・ビジネス   │  │
│  │  ・テストコードの生成           │  │   ロジック   │  │
│  │                                 │  │ ・説明責任   │  │
│  └─────────────────────────────────┘  └──────────────┘  │
│                                                         │
│  ← Claude Code が加速 →              ← あなたの価値 →   │
└─────────────────────────────────────────────────────────┘
```

この 30% こそ、シニアエンジニアとジュニアエンジニアの差が最も現れる領域です。Fastly の 2025 年調査「[Senior Developers Ship 2.5x More AI Code Than Juniors](https://www.fastly.com/blog/senior-developers-ship-more-ai-code)」によると、シニア開発者（経験 10 年以上）の 32% が「出荷コードの半分以上は AI 生成」と回答しており、ジュニア（経験 2 年以下）の 13% と比べて約 2.5 倍の差があります。この差は「AI を使う速度」ではなく「AI の出力に対する問題検出能力」にあります。シニアエンジニアは AI が見落とすエッジケースや設計上の問題を素早く見抜き、修正できるのです。

ここで重要なのは、**あなたがジュニアだから 30% ができないということではない** ということです。30% の判断力は、意識的に鍛えれば身につきます。Part 3 では、この 30% の判断を繰り返し実践することで、AI と協働して成果を出す力を養います。

## AI エージェントの活用はコードレビューのプロセスである

GitHub のエンジニア Sean Goedecke 氏は、記事「[If you are good at code review, you will be good at using AI agents](https://www.seangoedecke.com/ai-agents-and-code-review/)」で、「コードレビューが得意な人は、AI エージェントの活用も得意になる」と述べています。

考えてみると、AI エージェントが生成したコードに対してあなたがやるべきことは、チームメンバーのプルリクエストをレビューするときとほぼ同じです。

- **コードの意図を理解する**: 何をしようとしているのか、なぜそのアプローチなのか
- **正しさを検証する**: ロジックにバグはないか、エッジケースは考慮されているか
- **設計との整合性を確認する**: プロジェクトの既存パターンやコーディング規約に従っているか
- **セキュリティを確認する**: 脆弱性はないか、入力のバリデーションは適切か

Part 2 で学んだ見極めチェック（正しさ・品質・安全性）は、まさにこのコードレビューの観点を構造化したものです。

```
┌──────────────────────────────────────────────────┐
│         Claude Code との協働ワークフロー           │
│                                                  │
│  ① 指示     あなた → Claude Code                 │
│     │       タスクの要件・コンテキストを伝える     │
│     ↓                                            │
│  ② 生成     Claude Code → コード                 │
│     │       コードを読み、編集し、テストを実行     │
│     ↓                                            │
│  ③ レビュー  あなた → コード                      │
│     │       正しさ・品質・安全性を検証             │
│     ↓                                            │
│  ④ 判断     あなた → 採用 or 修正指示             │
│             問題があれば Claude Code に戻す        │
└──────────────────────────────────────────────────┘
```

Claude Code の公式ドキュメント「Best Practices」でも、タスクを遂行する際のワークフローとして以下の4フェーズが推奨されています。

1. **探索（Explore）**: Plan Mode でコードを読み、現状を把握する
2. **計画（Plan）**: 実装計画を立て、アプローチを決める
3. **実装（Implement）**: Normal Mode に切り替えて実装する。テストを書いて実行し、失敗を修正する
4. **コミット（Commit）**: 変更をコミットし、PR を作成する

ここで注目すべきは、**検証が独立したフェーズではなく、実装フェーズに組み込まれている** ということです。公式ドキュメントでも「Give Claude a way to verify its work（Claude に自分の仕事を検証する手段を与えよ）」が最も効果の高いプラクティスとして最初に挙げられています。テストを書き、実行し、結果を確認する。この検証サイクルが実装と一体になっているのです。

Part 3 では、この検証の部分に特に注目します。あなたの判断が最も求められるのは **③ レビュー** と **④ 判断** のフェーズです。Claude Code が生成したコードをそのまま受け入れるのではなく、自分の目で検証し、責任を持って採用する。この姿勢が、AI と協働する上での基本です。

## AI 生成コードの検証が必要な3つの理由

「Claude Code は優秀なのだから、生成されたコードをそのまま使えばいいのでは？」と思うかもしれません。しかし、検証なしに AI 生成コードを採用することには、実務上のリスクがあります。

### 理由 1: セキュリティの脆弱性

セキュリティ企業 Veracode の 2025 年調査「[GenAI Code Security Report](https://www.veracode.com/resources/analyst-reports/2025-genai-code-security-report/)」によると、**AI が生成したコードの約 45% にセキュリティ上の脆弱性が含まれている** ことが報告されています。これは人間が書いたコードと比較して約 2.7 倍のリスクです。

特に脆弱性が発生しやすい領域は以下の通りです。

- **クロスサイトスクリプティング（XSS）**: AI が生成したコードの 86% が XSS 対策に失敗
- **インジェクション攻撃**: SQL インジェクションやコマンドインジェクションへの防御が不十分
- **認証・認可の不備**: アクセス制御のロジックが不完全

AI は「動くコード」を生成することには長けていますが、「安全なコード」を生成することは別の問題です。Part 3 のバグ修正（Chapter 3-3）では、認可の漏れという実際に起こりうるセキュリティバグを題材に、安全性の見極めを重点的に実践します。

### 理由 2: 設計判断の妥当性

AI はプロジェクト固有のコンテキストを完全に理解しているわけではありません。CLAUDE.md や rules でプロジェクトのルールを伝えても、以下のような判断は人間が行う必要があります。

- **既存の設計パターンとの整合性**: プロジェクト内で Service クラスを使っているのに、Claude Code が Controller に全ロジックを書いてしまうことがある
- **要件の解釈**: 「受講生がコースを受講できる」という要件に対して、どこまでのバリデーションが必要かは、ビジネス要件の理解がなければ判断できない
- **トレードオフの選択**: パフォーマンスと可読性、シンプルさと拡張性のバランスは、プロジェクトの状況によって正解が変わる

Part 3 の機能開発（Chapter 3-4）では、既存設計との整合性を判断する「品質」の見極めを重点的に実践します。

### 理由 3: 説明責任

チーム開発において、コードをコミットしてプルリクエストを出すのはあなたです。レビューで「このコードはなぜこの設計にしたのか？」と聞かれたとき、「Claude Code が生成したのでわかりません」では通用しません。

AI が生成したコードであっても、あなたがコミットした時点で、それはあなたのコードです。コードの意図を理解し、なぜそのアプローチを選んだのかを説明できる状態で採用する必要があります。

Part 3 のチーム共有（Chapter 3-6）では、AI 生成コードに対する説明責任を実践します。

## Part 3 の全体像

Part 3 は6つの Chapter で構成されています。中核となるのは **Chapter 3-3〜3-5 の3つの実務タスク**（バグ修正・機能開発・リファクタリング）です。Chapter 3-1〜3-2 はその準備として環境とコード理解を整え、Chapter 3-6 は成果をチームに届ける仕上げです。

```
           準備                     実務タスク（メイン）                 仕上げ
  ┌──────────────────┐  ┌──────────────────────────────────────┐  ┌──────────┐
  │ 3-1       3-2    │  │ 3-3         3-4         3-5         │  │ 3-6      │
  │ 実践の →  コードを│→│ バグを   →  機能を   →  コードを    │→│ チームに │
  │ 準備      理解する│  │ 修正する    開発する    改善する    │  │ 共有する │
  │                  │  │                                      │  │          │
  │ 環境構築         │  │ 原因特定    要件理解    対象特定    │  │ PR 作成  │
  │ 設定整備         │  │ 修正実装    設計・実装  テスト確認  │  │ 環境整備 │
  │ 全体像把握       │  │ テスト検証  品質検証    改善実行    │  │          │
  └──────────────────┘  └──────────────────────────────────────┘  └──────────┘

                        ┌──────────────────────────────────────┐
                        │     見極めチェックの重点              │
                        │  3-3: 安全性（認可の漏れ）           │
                        │  3-4: 品質（既存設計との整合性）     │
                        │  3-5: 正しさ（動作不変の保証）       │
                        └──────────────────────────────────────┘
```

3つの実務タスクそれぞれで、見極めチェック（正しさ・品質・安全性）の重点を変えて実践します。どの Chapter でも3観点すべてを確認しますが、特に注力する観点が異なることで、検証の引き出しが増えていきます。

### 提供プロジェクト: CourseHub

Part 3 では、**CourseHub** というオンライン学習プラットフォームを題材に使います。コーチがコースを作成し、受講生が受講・進捗管理するシステムです。技術スタックは Laravel 10 + Sail + MySQL + Blade + Tailwind CSS で、あなたが COACHTECH で学んだ技術と同じ構成です。

> ※ CourseHub は教材用に構築した簡易的な Web アプリケーションです。実務の規模感とは異なりますが、実務で遭遇する問題（ドキュメントのズレ、コーディング規約と実態の乖離、バグ、設計の不統一など）を意図的に盛り込んでいます。

想定シナリオは、初期開発チーム（2〜3名）が退職し、あなたがジュニアエンジニアとしてプロジェクトに参画するという状況です。1年ほど運用されたプロジェクトなので、コードスタイルのバラつきやテストカバレッジのムラなど、実務でよくある「リアルな状態」があります。

### 各 Chapter で使う Part 2 の機能

Part 3 では、Part 2 で学んだ機能をタスクの中で実践的に活用します。

| Chapter | 中心的に使う機能 |
|---|---|
| 3-1 実践の準備 | CLAUDE.md / rules / skills / settings.json |
| 3-2 コードを理解する | Sub-agents, MCP, `/compact`, `/context`, `/btw` |
| 3-3 バグを修正する | プロンプト設計, テスト, `/rewind` |
| 3-4 機能を開発する | Plan Mode, セッション管理, Hooks, `/simplify` |
| 3-5 コードを改善する | Worktree, Plan Mode, `/cost`, `/context` |
| 3-6 チームに共有する | Git, PR・レビュー, GitHub Actions, Skills, 権限管理 |

Part 2 で学んだ機能が、実務タスクの中でどう活きるかを体感してください。

## ✨ まとめ

- AI はタスクの約 70% を処理できるが、残りの 30%（設計判断・エッジケース・セキュリティ）には人間の判断が必要
- AI エージェントの活用はコードレビューのプロセスと同じ。生成コードを正しさ・品質・安全性の観点で検証する
- AI 生成コードには脆弱性リスク・設計判断の誤り・説明責任の問題がある。検証なしの採用は実務では通用しない
- Part 3 のメインは3つの実務タスク（バグ修正・機能開発・リファクタリング）。提供プロジェクト CourseHub を題材に、準備から成果共有まで一気通貫で実践する
