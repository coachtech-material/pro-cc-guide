## 導入: 既存プロジェクトに参画する「初日」

実務で新しいプロジェクトに参画したとき、最初にやるのは開発環境のセットアップです。リポジトリをクローンして、README に書かれた手順に従って環境を構築する。これが初日のタスクです。

しかし、実務のプロジェクトでは README が最新の状態とは限りません。開発が進む中でコマンドが変わったのに README が更新されていない、環境変数の設定例が古い、そもそも手順が抜けている。こうした「ドキュメントと実態のズレ」は、現場では日常茶飯事です。

このセクションでは、CourseHub のセットアップを通じて、まさにこの体験をします。

### 🧠 先輩エンジニアはこう考える

> プロジェクトに入って最初にやることは README を読むこと。でも、README を100% 信じてはいけないということも、現場で学んだ。README は「参考情報」くらいの気持ちで読んで、実際にやってみてうまくいかなかったら、コードや設定ファイルを見て正しい手順を突き止める。Claude Code はこの「突き止める」作業をめちゃくちゃ加速してくれる。

## 🏃 CourseHub をセットアップする

3-1-1 で紹介した CourseHub を実際にセットアップします。技術スタック（Laravel 10 + Sail + MySQL）やユーザーロール（admin / coach / student）の詳細は 3-1-1 の「提供プロジェクト: CourseHub」を参照してください。

ここからは、前任チームが退職したプロジェクトにジュニアエンジニアとして参画した初日の想定で進めます。

### Step 1: リポジトリをクローンする

ターミナルで、プロジェクトを配置するディレクトリに移動し、CourseHub をクローンします。

```bash
git clone https://github.com/coachtech-material/pro-cc-coursehub.git
cd pro-cc-coursehub
```

### Step 2: README を読む

クローンしたら、まず README.md を読みます。これは実務でも最初にやることです。

README には以下のようなセットアップ手順が書かれています（一部抜粋）。

<!-- README.md より抜粋 -->
```
1. リポジトリをクローン
2. .env ファイルを作成: cp .env.example .env
3. .env の設定例:
   DB_HOST=127.0.0.1
   DB_USERNAME=sail
   DB_PASSWORD=password
4. Docker を起動: docker compose up -d
5. マイグレーション・シーディング: php artisan migrate --seed
6. アプリケーションキーの生成: php artisan key:generate
```

一見すると、COACHTECH の教材で慣れた手順と似ています。しかし、いくつか違和感を感じるかもしれません。`docker compose up -d` は知っているけれど、Sail プロジェクトなら `./vendor/bin/sail up -d` では？`php artisan` をホストマシンから直接実行できるのか？

まずは README の手順通りに試してみましょう。

### Step 3: README の手順に従う（エラーに遭遇する）

README の手順に従って進めます。

```bash
cp .env.example .env
```

`.env` ファイルが作成されます。README に従って `DB_HOST=127.0.0.1`、`DB_USERNAME=sail`、`DB_PASSWORD=password` に設定します。

次に Docker を起動しようとします。

```bash
docker compose up -d
```

ここで、以下のようなエラーが発生します。

> ⚠️ **よくあるエラー**: Docker の起動に失敗する
>
> ```
> failed to solve: failed to read dockerfile: open Dockerfile: no such file or directory
> ```
>
> または、以下のようなエラーが表示されることもあります。
>
> ```
> service "laravel.test" depends on undefined service "mysql": invalid compose project
> ```
>
> **原因**: クローンしたリポジトリには `vendor/` ディレクトリが含まれていません（`.gitignore` で除外されているため）。`compose.yaml` の `laravel.test` サービスは `./vendor/laravel/sail/runtimes/8.2` にある Dockerfile を参照していますが、このファイルがまだ存在しません。
>
> **つまり**: Composer で依存パッケージをインストールするステップが README に書かれていないのです。

README にはこのステップが記載されていません。ここで手が止まります。

> 💡 これは実務でよくある場面です。README が不完全で、書かれている通りにやっても動かない。こんなとき、以前なら先輩エンジニアに聞くか、自分で `compose.yaml` や `Dockerfile` を読んで解決策を探すしかありませんでした。

### Step 4: Claude Code に助けを求める

プロジェクトのディレクトリで Claude Code を起動します。

```bash
claude
```

README の手順でうまくいかなかったことを伝え、正しいセットアップ手順を聞きます。

```
> README に従ってセットアップしようとしたが、docker compose up -d で Dockerfile が見つからないエラーが出た。このプロジェクトの正しいセットアップ手順を教えてほしい
```

Claude Code は `compose.yaml`、`.env.example`、`README.md` などのプロジェクトファイルを読み取り、以下のような問題点を指摘してくれるはずです。

- **`vendor/` が存在しない**: クローン直後は Composer の依存パッケージがインストールされていないため、まず `composer install` を Docker 経由で実行する必要がある
- **`DB_HOST` の設定が間違っている**: README は `127.0.0.1` を指定しているが、Sail（Docker）環境ではコンテナ間通信にサービス名 `mysql` を使う必要がある
- **コマンドが Sail 経由でない**: `php artisan` をホストマシンから直接実行するのではなく、`./vendor/bin/sail artisan` を使うべき

Claude Code が提案する正しい手順は、おおむね以下のような流れになります（あなたの環境では出力が異なる場合があります）。

### Step 5: 正しい手順でセットアップする

Claude Code の提案に従って、正しい手順でセットアップを進めます。

**1. Composer で依存パッケージをインストール**

`vendor/` ディレクトリがないため、Docker コンテナを使って `composer install` を実行します。COACHTECH の教材でプロジェクトを作成したときと同じアプローチです。

```bash
docker run --rm \
    -u "$(id -u):$(id -g)" \
    -v "$(pwd):/var/www/html" \
    -w /var/www/html \
    laravelsail/php82-composer:latest \
    composer install
```

このコマンドは、Sail 用の PHP 8.2 + Composer が入った Docker イメージを使って、現在のディレクトリに `vendor/` をインストールします。

**2. `.env` の `DB_HOST` を修正する**

`.env` ファイルの `DB_HOST` を `mysql` に変更します。

```
DB_HOST=mysql
```

`127.0.0.1` は「自分自身」を指すアドレスです。Docker 環境では、Laravel のアプリケーションコンテナ（`laravel.test`）と MySQL コンテナ（`mysql`）は別々のコンテナとして動作しています。アプリケーションコンテナから MySQL コンテナに接続するには、Docker のサービス名 `mysql` を使う必要があります。

**3. Sail で Docker を起動する**

```bash
./vendor/bin/sail up -d
```

> 📝 `docker compose up -d` でも起動はできますが、Sail プロジェクトでは `./vendor/bin/sail up -d` を使うのが標準です。Sail は `docker compose` のラッパーで、環境変数の設定やユーザー権限の処理を自動的に行ってくれます。

**4. アプリケーションキーを生成する**

```bash
./vendor/bin/sail artisan key:generate
```

**5. マイグレーションとシーディングを実行する**

```bash
./vendor/bin/sail artisan migrate --seed
```

**6. フロントエンドのビルド**

```bash
./vendor/bin/sail npm install
./vendor/bin/sail npm run dev
```

> 📝 `npm run dev` は Vite の開発サーバーを起動します。ターミナルを占有するため、別のターミナルタブを開いて作業を続けてください。`npm run build` で本番用にビルドすることもできます。

### テストアカウントでログインする

Seeder によって以下のテストアカウントが作成されています。ブラウザで `http://localhost/login` にアクセスし、ログインできることを確認してください。

| ロール | メールアドレス | パスワード |
|---|---|---|
| **admin**（管理者） | `admin@coursehub.com` | `password` |
| **coach**（コーチ） | `coach1@coursehub.com` | `password` |
| **student**（受講生） | `student@coursehub.com` | `password` |

> 📝 テストアカウントはすべて Seeder で作成されます。student アカウントは `student@coursehub.com` が固定で用意されています。以降の実践で受講生としてログインする場面が何度もあるため、このメールアドレスを控えておいてください。

> ⚠️ **よくあるエラー**: ページが表示されない場合
>
> ```
> ERR_CONNECTION_REFUSED
> ```
>
> **対処法**: `./vendor/bin/sail ps` でコンテナが起動しているか確認してください。起動していない場合は `./vendor/bin/sail up -d` を再実行します。MySQL の起動に時間がかかる場合があるため、数秒待ってからマイグレーションを再実行してみてください。

## README のズレから学んだこと

このセットアップで、README に3つのズレがあることがわかりました。

| README の記載 | 実際に正しい内容 | 原因 |
|---|---|---|
| `composer install` ステップなし | Docker 経由で `composer install` が必要 | README の記載漏れ |
| `DB_HOST=127.0.0.1` | `DB_HOST=mysql`（Docker サービス名） | Docker 環境と直接実行の混同 |
| `php artisan` | `./vendor/bin/sail artisan` | Sail コマンドへの記載更新漏れ |

このような「README と実態のズレ」は、実務では珍しいことではありません。開発が進む中でドキュメントの更新が後回しになったり、環境の変更が反映されなかったりします。

重要なのは、**ドキュメントを鵜呑みにせず、実際の環境やコードと照らし合わせて検証すること** です。これは AI 生成コードの検証と同じ姿勢です。Claude Code は、プロジェクトのファイルを読み取って正しい手順を特定してくれる強力なパートナーですが、最終的に「この手順で合っている」と判断するのはあなたです。

> 📝 このセクションの実践はセットアップと動作確認が目的であり、コード生成を伴わないため、🔍 見極めチェックは省略します。

## ✨ まとめ

- 既存プロジェクトへの参画初日は、まず README を読み、セットアップを試みる
- README が最新でないことは実務ではよくある。ドキュメントを鵜呑みにしない姿勢が重要
- Claude Code はプロジェクトのファイルを読み取り、正しいセットアップ手順を特定してくれる
- Sail プロジェクトでは `./vendor/bin/sail` 経由でコマンドを実行し、`DB_HOST` には Docker サービス名を指定する
