## 導入: 規約は書いてある。だが守られているとは限らない

前のセクションで CourseHub の環境を立ち上げ、README のセットアップ手順にズレがあることを体験しました。では、README に書かれたコーディング規約や設計方針はどうでしょうか。実際のコードはそれに従っているのでしょうか。

実務では、プロジェクトに途中参加したとき、まずドキュメントを読んでプロジェクトのルールを把握します。しかし、規約が書かれていても実際のコードが従っていないケースは珍しくありません。開発が進む中でルールが形骸化したり、メンバーによって解釈が異なったり、途中で方針が変わったのに既存コードの修正が追いつかなかったりします。

このセクションでは、README の規約と実コードを照合し、乖離を発見します。あわせて Claude Code の設定ファイルも確認・更新します。ここで「何に従うべきか」を明確にすることが、以降の Chapter でバグ修正・機能開発・リファクタリングを「規約に従って」遂行するための基盤になります。

### 🧠 先輩エンジニアはこう考える

> プロジェクトに入ったら、まず規約を読む。次にコードを読んで、規約通りになっているかを確認する。この2つが一致していれば安心して開発を進められるが、ズレがあるなら「どちらが正しいのか」を判断しなければならない。判断を先送りにすると、自分が書いたコードが規約違反なのか正しいのかわからないまま開発することになる。

> 📝 実務では、コーディング規約や設計方針は README や社内ドキュメントに整備されているのが一般的です。Claude Code の設定（CLAUDE.md、rules、skills）もチームの方針に沿って構築されていることが前提で、ジュニアエンジニアがゼロから作ることは多くありません。設定を編集する機会も稀で、チームの方針に従うのが原則です。ですが、この教材では Part 2 で学んだ知識の実践も兼ねて、既存の設定を確認・更新する体験をします。

CourseHub のプロジェクトディレクトリで作業を進めます。Claude Code を起動しておいてください。

```bash
cd pro-cc-coursehub
claude
```

### Task 1: README のコーディング規約・設計方針を読む

VS Code（またはお使いのエディタ）で `README.md` を開いてください。前のセクションではセットアップ手順を確認しましたが、README にはそれ以外にも重要なセクションがあります。

README の下部に「コーディング規約」と「設計方針」のセクションがあります。前任チームがプロジェクトのルールとして定めた内容です。

**コーディング規約** には、以下のようなルールが記載されています。

- **バリデーション**: Controller では Form Request を使用する
- **命名規則**: 変数名・メソッド名は camelCase、テーブル名・カラム名は snake_case
- **エラーハンドリング**: Blade 画面はリダイレクトでエラーを返し、フラッシュメッセージで通知する
- **テスト**: 新機能には Feature テストを書く。テストメソッド名は `test_` プレフィックス

**設計方針** には、以下の方針が記載されています。

- **Controller**: Controller は薄く保つ。ビジネスロジックが複雑な場合は Service クラスに切り出す
- **Service クラス**: 複数モデルにまたがる処理は Service クラスに集約する（例: EnrollmentService）
- **Policy**: リソースのアクセス制御は Policy で実装する
- **イベント**: 重要なドメインイベントは Event/Listener パターンで実装する

これらを頭に入れた上で、実際のコードがこのルールに従っているかを確認しましょう。

### Task 2: 規約と実コードの乖離を確認する

Claude Code に、README の規約に対するコードの準拠状況を調べてもらいます。

```
> README のコーディング規約と設計方針を読んで、実際のコードがそれに従っているかを確認してほしい。乖離がある箇所をリストアップして
```

Claude Code がプロジェクトのコードを調査すると、いくつかの乖離が見えてきます。あなたの環境での出力は異なりますが、おおむね以下のような発見があるはずです。

**コーディング規約との乖離:**

| 規約 | 実態 | 乖離 |
|---|---|---|
| Form Request を使用する | 一部の Controller は使用しているが、CoachCourseController は `$request->validate()` を直接使用（inline バリデーション） | 不統一 |
| 新機能には Feature テストを書く | Course のテストは存在するが、Enrollment・Quiz・Submission 等のテストが存在しない | 未実施の領域あり |

**設計方針との乖離:**

| 方針 | 実態 | 乖離 |
|---|---|---|
| Controller は薄く保つ | CoachCourseController の `store` メソッドが約 190 行。バリデーション、スラッグ生成、画像アップロード、タグ同期などすべてを1メソッドに詰め込んでいる | 明らかな違反 |
| Service クラスに切り出す | EnrollmentService は存在するが、それ以外の複雑な処理（コース作成等）は Service 化されていない | 適用が限定的 |
| Event/Listener パターン | CourseCompleted Event は実装されているが、他のドメインイベントは未整備 | 適用が限定的 |

> 💡 これらの乖離は、以降の Chapter で実際に取り組む題材になります。たとえば Fat Controller は Chapter 3-5（リファクタリング）で改善します。今の段階では「どこにどんな乖離があるか」を把握しておくことが重要です。

ここで気づいてほしいのは、**規約と実態のズレは「前任チームの怠慢」ではなく、実務のプロジェクトでは自然に発生するもの** だということです。開発の優先順位の中で、規約の徹底よりも機能リリースが優先されることは珍しくありません。大切なのは、ズレの存在を認識し、**新しく書くコードでは規約に従う** という方針を持つことです。

### Task 3: .claude/rules/ と README の整合性を確認する

次に、Claude Code の rules ファイルが README の規約と整合しているかを確認します。VS Code で `.claude/rules/coding.md` を開いてください。

`coding.md` の内容は、Form Request の使用、モデルのリレーション定義、命名規則、README のコーディング規約セクションへの参照で構成されています。

README の規約と照合すると、`.claude/rules/coding.md` は README のコーディング規約の**サブセット**（一部のみ抜粋したもの）であることがわかります。README にはエラーハンドリングやテストの規約がありますが、rules には反映されていません。また、README の設計方針（Controller を薄く保つ、Service クラスの使い方、Policy の方針）が rules にまったく含まれていません。

これは、Claude Code がコードを生成するとき、rules の内容だけに従い、README の設計方針を参照しない可能性があることを意味します。rules が不十分だと、たとえば Claude Code が Controller に全ロジックを書いてしまうかもしれません。

Claude Code に rules の更新を依頼します。

```
> .claude/rules/coding.md を更新したい。以下の方針で改善してほしい:
> - README のコーディング規約と設計方針のうち、Claude Code がコードを生成するときに従うべきルールを追加する
> - 現状のコードとの乖離がある項目（Form Request の不統一、Fat Controller）には、「新規コードでは規約に従う」旨を明記する
> - 簡潔さを保つ（詳細は README を参照する構成は維持）
```

あわせて `git.md` も確認しましょう。「コミットメッセージは英語で記述する」と書かれていますが、実際のコミット履歴を見ると日本語と英語が混在しているはずです。チームの実態に合わせて方針を明確化してください。

### Task 4: CLAUDE.md を確認・更新する

VS Code で `CLAUDE.md` を開いてください。前任チームが最低限の情報だけ書いた状態です。

**発見すべき問題点:**

| 項目 | 問題 |
|---|---|
| 開発環境のコマンド | `docker compose up -d` や `php artisan` と記載されているが、Sail 環境では `./vendor/bin/sail` を使う |
| 技術スタックの詳細 | PHP バージョン（8.2）、MySQL バージョン（8.0）の記載がない |
| コース構造 | CourseHub の中核であるコース構造（Course > Chapter > Lesson > Quiz）の説明がない |
| テスト情報 | テスト実行コマンドが `php artisan test`（Sail でない） |

> 📝 Part 2 で学んだように、CLAUDE.md はプロジェクトに参加した新メンバーへの「歩き方ガイド」です。この CLAUDE.md を渡されたとして、プロジェクトの概要が十分に伝わるでしょうか？

Claude Code に以下の方針で更新を依頼します。

```
> CLAUDE.md を更新したい。以下の方針で改善案を提案してほしい:
> - すべてのコマンドを Sail 経由に修正（./vendor/bin/sail）
> - PHP 8.2、MySQL 8.0 を技術スタックに明記
> - コース構造（Course > Chapter > Lesson > Quiz）の概要を追加
> - テスト実行コマンドを Sail 経由に修正
> - コーディング規約と設計方針は README に記載があるので、そちらを参照する形にする
```

生成された内容を確認し、以下の点をチェックしてください。

- コマンドがすべて `./vendor/bin/sail` 経由になっているか
- プロジェクトの実態（実際のディレクトリ構造やモデル）と一致しているか
- 簡潔で、必要な情報に絞られているか（Part 2 で学んだ「200 行以内」のガイドラインを思い出してください）

> ⚠️ Claude Code が生成する CLAUDE.md は、あなたの環境やプロジェクトの状態によって異なります。生成された内容をそのまま採用するのではなく、自分の理解と照らし合わせて判断してください。

### Task 5: .claude/skills/ を確認・更新する

```
> .claude/skills/ のファイル一覧と各ファイルの内容を確認して
```

CourseHub には2つの Skills が定義されています。それぞれに問題があります。

**test Skill の問題:**

test Skill は `php artisan test` と記載されていますが、Sail 環境では `./vendor/bin/sail artisan test` が正しいコマンドです。このまま `/test` を実行すると、ホストマシンで PHP を実行しようとしてエラーになります。

Claude Code に修正を依頼します。

```
> test Skill のコマンドを Sail 経由に修正してほしい
```

**lint Skill の問題:**

lint Skill は `./vendor/bin/php-cs-fixer fix --dry-run --diff` と記載されています。しかし、実際にこのコマンドを実行するとどうなるでしょうか？

```
> /lint を実行してみて
```

php-cs-fixer がインストールされていないため、エラーが発生します。

```
sh: ./vendor/bin/php-cs-fixer: No such file or directory
```

前任チームが php-cs-fixer の導入を計画したものの、途中で断念した残骸のようです。しかし、Laravel 10 には **Laravel Pint** というコードフォーマッターがプリインストールされています。Pint は php-cs-fixer をベースにした Laravel 専用のツールで、`./vendor/bin/pint` で実行できます。

Claude Code に Pint への切り替えを依頼します。

```
> lint Skill を php-cs-fixer から Laravel Pint に切り替えてほしい。Pint は ./vendor/bin/pint で実行できる。Sail 経由のコマンドにすること
```

更新後、`/lint` を再度実行して動作を確認しましょう。

### Task 6: .claude/settings.json を確認・更新する

```
> .claude/settings.json の内容を表示して
```

既存の settings.json には `php artisan`、`composer`、`npm` コマンドの allow ルールがあります。しかし、Sail 環境での開発に必要なコマンドが不足しています。

**不足しているコマンド:**

| コマンド | 用途 |
|---|---|
| `./vendor/bin/sail *` | Sail 経由のすべてのコマンド（artisan、npm、composer 等） |
| `git *` | バージョン管理（コミット、ブランチ操作等） |

**既存ルールの問題:**

| 既存の allow | 問題 |
|---|---|
| `Bash(php artisan *)` | Sail 環境ではホストの PHP を使うべきでない |
| `Bash(composer *)` | Sail 環境では `sail composer` を使うべき |
| `Bash(npm *)` | Sail 環境では `sail npm` を使うべき |

Claude Code に更新を依頼します。

```
> settings.json の permissions を更新したい。Sail 経由のコマンドと git コマンドを allow に追加してほしい。既存の php artisan / composer / npm のルールは Sail 経由に置き換える
```

> 📝 Part 2 で学んだように、`allow` ルールはプロジェクトスコープ（`.claude/settings.json`）に設定することで、チーム全員に適用されます。個人的な設定は `.claude/settings.local.json`（Git 管理外）に書きます。

更新後、Claude Code に Sail コマンドを実行させて、権限の確認なしで実行できることを確認しましょう。

```
> sail artisan --version を実行して
```

## 🔍 見極めチェック

> 🧠 先輩エンジニアの思考: 「規約と実態の乖離を見つけたら、『どちらが正しいか』ではなく『今後どうするか』を決めるのが大事。過去を責めても仕方ない。新しく書くコードの基準を明確にして、チームで共有できる状態にする。設定ファイルの更新もコード変更と同じで、間違った設定は Claude Code の出力品質を直接左右する。」

Claude Code が生成・更新した設定ファイル、および規約の確認結果について、以下の観点で確認してください。

- [ ] **正しさ**: 規約と実コードの乖離を正しく特定できたか。CLAUDE.md のコマンドがすべて Sail 経由になっているか。rules の内容がプロジェクトの方針と合っているか。skills のコマンドが正しく動作するか
- [ ] **品質**: CLAUDE.md が簡潔で必要な情報に絞られているか（200 行以内が目安）。rules の記述が具体的で、Claude Code が解釈しやすいか。README の規約と rules の役割分担が明確か（README に詳細、rules に簡潔なルール）
- [ ] **安全性**: settings.json の allow ルールが広すぎないか（例: `Bash(*)` のようなワイルドカードは危険）。deny ルールの追加が必要な操作はないか（例: 本番環境への接続コマンド）

> 🔑 この Section では特に「正しさ」に注目してください。ここで確認した規約と設定は、以降のすべての Chapter で Claude Code がコードを生成する際の基盤になります。

## ✨ まとめ

- 実務では、README に書かれた規約と実際のコードが乖離していることは珍しくない。プロジェクト参画時にこのズレを把握しておくことが、以降の開発品質を左右する
- 乖離を見つけたら「新規コードでは規約に従う」という方針を明確にする。過去のコードは今すぐ直す必要はない
- `.claude/rules/` は README の規約のサブセット。Claude Code がコード生成時に参照するため、設計方針を含めて十分な情報を持たせる
- CLAUDE.md はプロジェクトの実態（Sail 環境、コース構造等）に合わせて更新する
- skills は実際に動作するか確認する。動作しないコマンド（php-cs-fixer）は代替ツール（Pint）に切り替える
- settings.json は Sail 環境に合わせた権限設定にする。広すぎる allow ルールは避ける
