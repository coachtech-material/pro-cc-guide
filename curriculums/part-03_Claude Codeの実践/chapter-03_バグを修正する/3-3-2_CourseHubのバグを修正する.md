# 3-3-2 CourseHub のバグを修正する

## 🎯 このセクションで学ぶこと

- バグ修正の全体フロー（報告→再現→調査→修正→検証）を3つのバグで実践する
- バグの類型（データの正しさ・アクセス制御・機能不全）に応じた調査アプローチの違いを体験する
- 見極めチェック（正しさ・品質・安全性）の観点をバグの類型に応じて使い分ける

3つのバグを、3-3-1 で学んだ全体フロー（報告→再現→調査→修正→検証）に沿って順に修正します。

> 📝 このセクションで使う機能: プロンプト設計（[2-2-3 プロンプト設計](../../part-02_Claude%20Codeの基礎/chapter-02_基本を理解する/2-2-3_プロンプト設計.md) で学習）、`/rewind`（[2-2-2 コンテキストとセッション管理](../../part-02_Claude%20Codeの基礎/chapter-02_基本を理解する/2-2-2_コンテキストとセッション管理.md) で学習）、`/test` Skill（[3-1-3 プロジェクトの規約・設定を確認する](../chapter-01_実践の準備/3-1-3_プロジェクトの規約・設定を確認する.md) で更新）

## 🏃 実践 1: 進捗率の計算バグを修正する（データの正しさ）

最初のバグは「データの正しさ」の類型です。アプリケーションはエラーなく動作しますが、表示される値が正しくありません。

### バグ報告

あなたに以下のバグ報告が届きました。

> **報告**: マイコース画面の進捗率が正しくない。全レッスンを完了したのに進捗率が100%にならないコースがある。

まず、3-3-1 で学んだとおり、報告内容を整理しましょう。

| 項目 | 内容 |
|---|---|
| 症状 | 全レッスンを完了しても進捗率が100%にならない |
| 期待値 | 受講可能な全レッスンを完了したら100%になるべき |
| 再現条件 | まだ不明（これから確認する） |

### Step 1: 再現する

ブラウザでマイコース画面（`http://localhost/my-courses`）を確認します。受講生（student）アカウントでログインし、進捗率の表示を確認してください。

<!-- TODO: 画像追加 - マイコース画面の進捗率表示（100%にならないコースがある状態） -->

シーダーデータには、非公開レッスン（`is_published = false`）を含むコースを受講中の受講生が含まれています。そのコースの全公開レッスンを完了しても、進捗率が100%にならないことが確認できるはずです。

> 📝 再現条件を確認するために、データベースの状態を直接確認するのも有効です。phpMyAdmin（`http://localhost:8080`）で `lessons` テーブルの `is_published` カラムを確認すると、一部のレッスンが `0`（非公開）になっていることがわかります。

### Step 2: 調査する

再現できたら、Claude Code に調査を依頼します。ここで重要なのが、[2-2-3 プロンプト設計](../../part-02_Claude%20Codeの基礎/chapter-02_基本を理解する/2-2-3_プロンプト設計.md) で学んだ「症状・再現条件・期待値を明確に伝える」プロンプト設計です。

```
> マイコース画面（/my-courses）の進捗率表示にバグがあります。
>
> 症状: 全レッスンを完了しても進捗率が100%にならないコースがある
> 再現条件: 非公開レッスン（is_published = false）が含まれるコース
> 期待値: 受講生がアクセスできる公開レッスンだけで進捗率を計算し、全完了時に100%になる
>
> まず関連するコードを読んで、原因を特定してください。修正はまだしないでください。
```

最後の「修正はまだしないでください」が重要です。調査と修正を分けることで、原因を理解してから修正に入ることができます。

Claude Code は、マイコース画面に関連するコードを辿って調査します。典型的には次のような流れです。

1. ルーティング（`routes/web.php`）で `/my-courses` のコントローラを特定する
2. `MyCourseController@index` を読み、ビューに渡すデータを確認する
3. ビュー（`resources/views/my-courses/index.blade.php`）で進捗率の表示方法を確認する
4. `Course` モデルの `getProgressRate()` メソッドを読み、計算ロジックを調査する

Claude Code は、以下のような原因を報告するはずです。

> `getProgressRate()` メソッドで、総レッスン数の取得時に `is_published` のフィルタがありません。非公開レッスンも総数に含まれるため、公開レッスンだけを完了しても100%に到達しません。

**ここで、Claude Code の報告を実コードで検証してください。** [3-2-1 コードリーディングの方法論](../chapter-02_既存コードを理解する/3-2-1_コードリーディングの方法論.md) で学んだ「Claude の説明を検証する」ステップです。

`app/Models/Course.php` の `getProgressRate()` メソッドを確認しましょう。

```php
// app/Models/Course.php

public function getProgressRate($userId): int
{
    $totalLessons = $this->chapters()->withCount('lessons')->get()
        ->sum('lessons_count');
    // ↑ ここが原因: 全レッスン（非公開含む）をカウントしている

    $completedLessons = LessonProgress::where('user_id', $userId)
        ->whereIn('lesson_id', $this->getAllLessonIds())
        ->where('status', 'completed')
        ->count();

    return $totalLessons > 0 ? (int) round($completedLessons / $totalLessons * 100) : 0;
}
```

同じファイル内の `getAllLessonIds()` メソッドも確認してください。

```php
// app/Models/Course.php

public function getAllLessonIds(): array
{
    return Lesson::whereIn('chapter_id', $this->chapters()->pluck('id'))
        ->where('is_published', true)
        ->pluck('id')
        ->toArray();
}
```

ここに問題の本質があります。`getAllLessonIds()` は `is_published = true` でフィルタして公開レッスンだけを返しているのに、`getProgressRate()` の総数は `withCount('lessons')` で全レッスンをカウントしています。分子（完了数）は公開レッスンだけ、分母（総数）は全レッスン。これでは100%に到達しません。

> ⚠️ **あなたの環境では Claude Code の出力が異なる場合があります。** Claude Code が原因を正しく特定できたかどうかを、実際のコードを読んで確認してください。Claude Code の報告が正しければ次に進み、不正確であれば追加の質問で掘り下げてください。

### Step 3: 修正する

原因が特定できたので、修正を依頼します。

```
> getProgressRate() の総レッスン数の計算で、公開レッスンだけをカウントするように修正してください。
> getAllLessonIds() が既に公開レッスンをフィルタしているので、同じ条件で統一してください。
```

Claude Code は、`getProgressRate()` メソッドの総レッスン数の取得部分を修正します。あなたの環境では異なるコードが生成されますが、修正の方向性は同じはずです。

<details>
<summary>生成されたコードの例を確認する（クリックで展開）</summary>

```php
// app/Models/Course.php

public function getProgressRate($userId): int
{
    $totalLessons = $this->chapters()
        ->with(['lessons' => fn ($q) => $q->where('is_published', true)])
        ->get()
        ->flatMap->lessons
        ->count();

    $completedLessons = LessonProgress::where('user_id', $userId)
        ->whereIn('lesson_id', $this->getAllLessonIds())
        ->where('status', 'completed')
        ->count();

    return $totalLessons > 0 ? (int) round($completedLessons / $totalLessons * 100) : 0;
}
```

</details>

修正のポイントは、`withCount('lessons')` を `with(['lessons' => ...])` に変更し、`is_published = true` のフィルタを追加していることです。これで分母と分子の条件が一致します。

### Step 4: 検証する

修正が正しいことを確認します。まず、ブラウザで再現手順を再度実行し、進捗率が正しく表示されることを目で確認してください。

次に、テストを書いて修正を証明します。Claude Code に依頼しましょう。

```
> getProgressRate() の修正をテストで検証したいです。
> 以下のケースをテストしてください:
> 1. 非公開レッスンがあるコースで、全公開レッスンを完了したら100%になること
> 2. 一部だけ完了した場合の進捗率が正しいこと
> 3. レッスンが0件のコースで0%になること（0除算にならないこと）
>
> テストは tests/Feature/ に作成してください。
```

テストが生成されたら、実行して確認します。

```
> テストを実行してください
```

> 📝 テストの実行には、[3-1-3](../chapter-01_実践の準備/3-1-3_プロジェクトの規約・設定を確認する.md) で更新した `/test` Skill が使われます。Sail 環境では `./vendor/bin/sail artisan test` で実行されます。

テストが全て通ったことを確認したら、`/rewind` の使い方を確認しておきましょう。


### 🔍 見極めチェック（進捗率バグ）

> 🧠 先輩エンジニアの思考: 「データの計算バグは、修正自体は簡単なことが多い。でも重要なのは、修正が他のデータに影響しないかを確認すること。進捗率を使っている画面は他にないか？ 進捗率に基づく別のロジック（コース完了判定など）はないか？ 修正の波及効果まで見るのがプロの検証だ」

- [ ] **正しさ**: 非公開レッスンが総数から除外されているか
- [ ] **正しさ**: 全公開レッスン完了時に100%になるか
- [ ] **正しさ**: レッスンが0件の場合に0除算にならないか
- [ ] **正しさ**: `getProgressRate()` を使用している他の画面（コーチのダッシュボードなど）でも正しく表示されるか
- [ ] **品質**: 分母と分子で同じフィルタ条件（`is_published = true`）を使用しているか
- [ ] **品質**: テストが主要なケース（100%、一部完了、0件）をカバーしているか
- [ ] **安全性**: 該当なし（このバグはデータ表示の問題であり、セキュリティに直接影響しません）

> 🔑 この実践では特に「正しさ」に注目してください。データの計算バグでは、修正後の値が仕様どおりかを境界値（0%、100%、中間値）で確認することが重要です。

## 🏃 実践 2: 認可バグを修正する（アクセス制御）

2つ目のバグは「アクセス制御」の類型です。本来アクセスできないはずのリソースにアクセスできてしまう、セキュリティに関わるバグです。

### バグ報告

以下のバグ報告が届きました。

> **報告**: 下書き（draft）のコースが受講生から見えてしまっている。URL を直接入力すると、まだ公開していないコースの詳細ページが表示される。

報告内容を整理します。

| 項目 | 内容 |
|---|---|
| 症状 | 下書き状態のコースに受講生がアクセスできる |
| 期待値 | 受講生は公開済み（published）のコースのみ閲覧できるべき |
| 再現条件 | 下書きコースの URL（`/courses/{id}`）を直接入力 |

### Step 1: 再現する

ブラウザで実際に確認しましょう。まず、phpMyAdmin（`http://localhost:8080`）で `courses` テーブルを確認し、`status` が `draft` のコースの `id` をメモしてください。

受講生（student）アカウントでログインした状態で、`http://localhost/courses/{そのid}` にアクセスします。

<!-- TODO: 画像追加 - 下書きコースの詳細ページが表示されてしまっている画面 -->

本来なら403エラー（権限なし）が返されるべきところ、コースの詳細が表示されてしまいます。これはセキュリティの問題です。

> ⚠️ **アクセス制御のバグは「正常に見える」のが厄介です。** コース一覧（`/courses`）には下書きコースが表示されないため、通常の操作では気づきません。しかし、URL を直接入力する方法（Forced Browsing、強制ブラウジング）で回避できてしまう状態です。

### Step 2: 調査する

Claude Code に調査を依頼します。アクセス制御のバグなので、3-3-1 で学んだとおり「ルーティング → ミドルウェア → Policy」の順に認可の流れを確認するよう指示します。

```
> コース詳細ページ（/courses/{course}）に認可のバグがあります。
>
> 症状: 受講生（student）が下書き（draft）コースのURLに直接アクセスすると、コース詳細が表示される
> 期待値: 受講生は公開済み（published）のコースのみ閲覧できる。下書き・アーカイブは403を返す
> 再現手順: student でログイン → /courses/{draft状態のコースID} にアクセス
>
> ルーティング → ミドルウェア → Policy の順に認可の流れを確認し、どこに問題があるか特定してください。
> 修正はまだしないでください。
```

Claude Code は認可の流れを調査し、`CoursePolicy` の `view` メソッドに問題があることを報告するはずです。

**Claude Code の報告を検証しましょう。** `app/Policies/CoursePolicy.php` を直接確認します。

```php
// app/Policies/CoursePolicy.php

public function view(User $user, Course $course): bool
{
    if ($user->role === 'admin' || $course->user_id === $user->id) {
        return true;
    }

    return true;
    // ↑ 問題: どのユーザーでも無条件に true を返している
    // 本来: return $course->status === 'published';
}
```

問題は明確です。`view` メソッドの最終行が無条件に `true` を返しているため、admin でもコースの作成者でもないユーザー（つまり受講生）が、どのコースでも閲覧できてしまいます。本来は `$course->status === 'published'` を返すべきです。

> 📝 この種のバグは、実装時のコピー&ペーストミスや、仮実装（とりあえず `return true`）を本実装に戻し忘れたケースでよく発生します。コードだけ見ると1行の問題ですが、セキュリティへの影響は大きいため、アクセス制御のコードは特に注意深くレビューする必要があります。

### Step 3: 修正する

原因が特定できたので、修正を依頼します。

```
> CoursePolicy の view メソッドを修正してください。
> admin またはコースの作成者（coach）は全てのコースを閲覧でき、
> それ以外のユーザー（student）は公開済み（published）コースのみ閲覧できるようにしてください。
```

<details>
<summary>生成されたコードの例を確認する（クリックで展開）</summary>

```php
// app/Policies/CoursePolicy.php

public function view(User $user, Course $course): bool
{
    if ($user->role === 'admin' || $course->user_id === $user->id) {
        return true;
    }

    return $course->status === 'published';
}
```

</details>

修正は最小限で、最終行の `return true;` を `return $course->status === 'published';` に変更するだけです。

ここで、修正が `.claude/rules/coding.md` の規約に従っているかも確認しましょう。[3-1-3 プロジェクトの規約・設定を確認する](../chapter-01_実践の準備/3-1-3_プロジェクトの規約・設定を確認する.md) で確認したとおり、rules には「リソースのアクセス制御は Policy で実装する」「Controller で `$this->authorize()` を使用する」と記載されています。今回の修正は Policy 内の変更なので、この規約に沿っています。もし Claude Code が Policy ではなくコントローラに直接 `if` 文を追加する修正を提案した場合は、規約に従って Policy での修正を指示し直してください。

### Step 4: 検証する

アクセス制御のバグは、セキュリティに直接関わるため、テストによる検証が特に重要です。Claude Code にテストの作成を依頼します。

```
> CoursePolicy の view メソッドのテストを書いてください。
> 以下のケースを網羅してください:
> 1. admin は下書きコースを閲覧できる
> 2. コースの作成者（coach）は自分の下書きコースを閲覧できる
> 3. student は公開済みコースを閲覧できる
> 4. student は下書きコースを閲覧できない（403が返る）
> 5. student はアーカイブ済みコースを閲覧できない（403が返る）
>
> テストを作成して実行してください。
```

テストケースを指定しているのは、**アクセス制御のテストは全ロール × 全ステータスの組み合わせを網羅する必要がある** ためです。Claude Code にテストケースの判断を丸投げすると、重要な組み合わせが漏れる可能性があります。

テストが全て通ったら、手動でも確認しましょう。先ほどの再現手順を繰り返し、今度は403エラーが返されることを確認します。

### `scopePublished` の未活用を確認する（品質の観点）

ここで、もう1つ確認しておくべきポイントがあります。Claude Code に聞いてみましょう。

```
> Course モデルに scopePublished スコープが定義されていますが、
> コース一覧の取得で実際に使われていますか？確認してください。
```

Claude Code は、`app/Http/Controllers/CourseController.php` の `index` メソッドを調査します。

```php
// app/Http/Controllers/CourseController.php（index メソッドの一部）

$courses = Course::where('status', 'published')->paginate(12);
```

`scopePublished()` が `Course` モデルに定義されているにもかかわらず、コントローラでは直接 `where('status', 'published')` と書いています。

**これはバグではありません。** 動作は正しく、コース一覧には公開済みのコースだけが表示されます。しかし、**品質の観点** では改善の余地があります。

| 観点 | 現状 | 改善後 |
|---|---|---|
| コードの重複 | 複数箇所で `where('status', 'published')` を直接書く | `published()` スコープに統一 |
| 変更容易性 | 公開条件が変わったら全箇所を修正する必要がある | スコープ1箇所を修正すれば全体に反映 |
| 可読性 | 条件の意味を読み取る必要がある | `published()` でメソッド名から意図が明確 |

> 💡 このような「動くけど改善できるポイント」を見つけられるようになることも、見極める力の一部です。ただし、**バグ修正のタスクではバグ修正に集中し、リファクタリングは別のタスクとして行う** のが原則です。「改善できるポイント」はメモに残しておき、[Chapter 3-5「コードを改善する」](../chapter-05_コードを改善する/3-5-0_Chapter%205の地図.md) のリファクタリングで扱います。

### Step 5: コミットする

```

```

### 🔍 見極めチェック（認可バグ）

> 🧠 先輩エンジニアの思考: 「認可のバグ修正は、テストの網羅性が命だ。"直した"だけでは不十分で、"全ロール × 全ステータスで正しく制御されている"ことをテストで証明する。1ケースでも漏れがあると、そこがセキュリティホールになる。自分はアクセス制御のレビューでは、必ず全ロールのマトリクスを頭の中で作ってからテストを確認している」

- [ ] **正しさ**: admin はすべてのコースを閲覧できるか
- [ ] **正しさ**: coach は自分のコースをすべて閲覧できるか
- [ ] **正しさ**: student は公開済みコースのみ閲覧できるか
- [ ] **安全性**: student が下書きコースの URL を直接入力したとき403が返るか
- [ ] **安全性**: student がアーカイブ済みコースの URL を直接入力したとき403が返るか
- [ ] **安全性**: テストが全ロール × 全ステータスの組み合わせを網羅しているか
- [ ] **品質**: `scopePublished()` が定義されているのに使われていない箇所を認識したか（修正は別タスク）

> 🔑 この実践では特に「安全性」に注目してください。アクセス制御のバグはセキュリティの脆弱性です。「正常系で動く」だけでなく、「異常系（URL 直打ち、権限のないロール）で正しく拒否される」ことの検証が必要です。

## 🏃 実践 3: 小テスト送信の500エラーを修正する（機能不全）

3つ目のバグは「機能不全」の類型です。特定の操作でアプリケーションがクラッシュし、500エラーが返される問題です。

### バグ報告

以下のバグ報告が届きました。

報告内容を整理します。

| 項目 | 内容 |
|---|---|
| 症状 | 小テスト送信時に500エラー |
| 期待値 | 未回答の問題があっても正常にスコアを計算し、結果を表示する |
| 再現条件 | 一部の問題を未回答のまま送信する |

### Step 1: 再現する

受講生（student）アカウントでログインし、小テストがあるレッスンに移動します。小テスト画面（`/courses/{course}/quizzes/{quiz}`）で、**一部の問題を未回答のまま送信** してください。

送信すると、500エラーが表示されます。

機能不全の類型では、3-3-1 で学んだとおり **エラーログが最大の手がかり** です。エラーの詳細を確認しましょう。

Claude Code にエラーログの確認を依頼します。

```
> storage/logs/laravel.log の直近のエラーログを確認してください
```

このエラーメッセージが調査の起点になります。

### Step 2: 調査する

エラーメッセージをもとに、Claude Code に調査を依頼します。

```
> 小テスト送信時に500エラーが発生します。
>
> エラーログ:
> Trying to access array offset on value of type null
> at app/Http/Controllers/QuizController.php:29
>
> 症状: 一部の問題を未回答のまま小テストを送信すると500エラー
> 期待値: 未回答の問題は不正解として扱い、スコアを計算して結果画面を表示する
>
> エラーの原因を特定してください。加えて、他にも例外が発生しうるケースがないか確認してください。
> 修正はまだしないでください。
```

「他にも例外が発生しうるケースがないか確認してください」と付け加えているのがポイントです。機能不全のバグでは、報告された症状の裏に別の問題が隠れていることがあります。

Claude Code は `QuizController@submit` メソッドを調査し、2つの問題を報告するはずです。

**報告を実コードで検証しましょう。**

```php
// app/Http/Controllers/QuizController.php（submit メソッド）

public function submit(Request $request, Course $course, Quiz $quiz)
{
    $answers = $request->input('answers', []);

    $correctCount = 0;
    foreach ($quiz->questions as $question) {
        $userAnswer = collect($answers)->firstWhere('question_id', $question->id);
        $selectedOption = Option::find($userAnswer['option_id']);
        // ↑ 問題1: $userAnswer が null の場合、null['option_id'] でエラー
        if ($selectedOption && $selectedOption->is_correct) {
            $correctCount++;
        }
    }

    $score = (int) round($correctCount / $quiz->questions->count() * 100);
    // ↑ 問題2: questions が0件の場合、0除算エラー

    $submission = Submission::create([
        'user_id' => auth()->id(),
        'quiz_id' => $quiz->id,
        'score' => $score,
        'answers' => $answers,
        'submitted_at' => now(),
    ]);

    return redirect()->route('courses.quizzes.result', [$course, $quiz]);
}
```

2つの問題が確認できます。

**問題 1: 未回答の問題で null アクセス**

`$userAnswer = collect($answers)->firstWhere('question_id', $question->id)` は、その問題に対する回答がない場合 `null` を返します。次の行で `$userAnswer['option_id']` にアクセスすると、`null` に対する配列アクセスとなり、`Trying to access array offset on value of type null` エラーが発生します。

**問題 2: 問題が0件の Quiz で0除算**

`$quiz->questions->count()` が0の場合、`$correctCount / 0` で `DivisionByZeroError` が発生します。シーダーデータには問題が0件の小テストが含まれているため、この問題も実際に発生しえます。

> 📝 機能不全のバグでは、報告された症状（問題 1）だけでなく、同じコードに潜む別の問題（問題 2）も見つかることがあります。Claude Code に「他にも問題がないか」と確認を促すことで、見落としを防げます。

### Step 3: 修正する

2つの問題を修正します。

```
> QuizController の submit メソッドを修正してください。
>
> 1. 未回答の問題がある場合は不正解として扱う（null チェックを追加）
> 2. 問題が0件の Quiz ではスコアを0にする（0除算を防ぐ）
>
> 既存のコードスタイルを維持し、最小限の変更にしてください。
```

<details>
<summary>生成されたコードの例を確認する（クリックで展開）</summary>

```php
// app/Http/Controllers/QuizController.php（submit メソッド）

public function submit(Request $request, Course $course, Quiz $quiz)
{
    $answers = $request->input('answers', []);

    $correctCount = 0;
    foreach ($quiz->questions as $question) {
        $userAnswer = collect($answers)->firstWhere('question_id', $question->id);
        if ($userAnswer) {
            $selectedOption = Option::find($userAnswer['option_id']);
            if ($selectedOption && $selectedOption->is_correct) {
                $correctCount++;
            }
        }
    }

    $questionCount = $quiz->questions->count();
    $score = $questionCount > 0
        ? (int) round($correctCount / $questionCount * 100)
        : 0;

    $submission = Submission::create([
        'user_id' => auth()->id(),
        'quiz_id' => $quiz->id,
        'score' => $score,
        'answers' => $answers,
        'submitted_at' => now(),

    return redirect()->route('courses.quizzes.result', [$course, $quiz]);
}
```

</details>

修正のポイントは2つです。

1. `if ($userAnswer)` で null チェックを追加し、未回答の場合は `Option::find()` を呼ばない（不正解として扱われる）
2. `$questionCount > 0` で0除算を防ぎ、問題が0件の場合はスコアを0にする

### 🔍 見極めチェック（500エラー）

> 🧠 先輩エンジニアの思考: 「500エラーの修正は、"エラーが出なくなった"で終わりにしがちだ。でも大事なのは、エラーを出さないだけでなく、正しい結果を返しているかの確認だ。null を握りつぶしてエラーを消しただけで、計算結果がおかしくなっていたら意味がない。修正後の"正常動作"が仕様どおりかまで見るのが検証のゴールだ」

- [ ] **正しさ**: 未回答の問題が不正解として扱われ、スコアが正しく計算されるか
- [ ] **正しさ**: 全問正解・全問不正解・一部正解のスコアがそれぞれ正しいか
- [ ] **正しさ**: 問題が0件の Quiz でスコアが0になるか（エラーにならないか）
- [ ] **品質**: null チェックの追加が最小限の変更になっているか（既存ロジックの構造を不必要に変えていないか）
- [ ] **品質**: テストが正常系・異常系の主要なケースをカバーしているか
- [ ] **安全性**: 該当なし（この修正は入力値のハンドリングの問題であり、認可やデータ漏洩には影響しません）

> 🔑 この実践では特に「正しさ」に注目してください。機能不全のバグでは、「エラーが出なくなった」だけでなく「正しい結果を返すようになった」ことまで検証する必要があります。

## 3つのバグ修正を振り返る

3つのバグを修正しました。それぞれの修正プロセスを比較してみましょう。

| | 実践 1: 進捗率バグ | 実践 2: 認可バグ | 実践 3: 500エラー |
|---|---|---|---|
| 類型 | データの正しさ | アクセス制御 | 機能不全 |
| 調査の起点 | データの流れを追跡 | 認可の流れを確認 | エラーログを確認 |
| 修正の本質 | クエリ条件の追加 | 戻り値の修正 | null チェックと0除算ガード |
| 見極めの重点 | 正しさ | 安全性 | 正しさ |
| テストの焦点 | 境界値（0%, 100%, 中間値） | 全ロール × 全ステータス | 正常系 + 例外ケース |

3つに共通しているのは、**報告→再現→調査→修正→検証** の5フェーズを一貫して踏んでいることです。バグの類型が変わっても、このフローは変わりません。変わるのは「調査の起点」と「見極めの重点」です。

### 🧠 先輩エンジニアはこう考える

> 3つのバグを通して気づいたかもしれないが、Claude Code はバグの原因特定と修正コードの生成をかなり正確にやってくれる。でも「テストケースの設計」と「修正の妥当性判断」は人間が主導した方がよい。特にテストケースは、"何をテストすべきか"を自分で考えてから Claude Code に指示しないと、ハッピーパス（正常系）のテストだけが生成されて、肝心の異常系が漏れることがある。バグ修正で最も価値があるのは、修正コードを書くことではなく、修正の妥当性を判断し、テストで証明することだ。それが「見極める力」の本質であり、70-30 モデルの 30% にあたる部分だ。

## ✨ まとめ

- バグの類型（データの正しさ・アクセス制御・機能不全）によって **調査の起点** が異なる。データの流れ、認可の流れ、エラーログ、とそれぞれの起点から調査を始めることで効率的に原因を特定できる
- Claude Code への調査指示は **症状・再現手順・期待値** を明確に伝え、「修正はまだしない」と分離することで、原因を理解してから修正に入れる
- アクセス制御のバグは **セキュリティの脆弱性** に直結する。全ロール × 全ステータスの網羅的なテストで安全性を証明する
- 機能不全のバグでは、報告された症状だけでなく **同じコードに潜む別の問題** も確認する
- 見極めの3観点（正しさ・品質・安全性）は、バグの類型に応じて **重点を変えて** 使い分ける
- テストケースは Claude Code に丸投げせず、**何をテストすべきかを自分で設計** してから指示する
