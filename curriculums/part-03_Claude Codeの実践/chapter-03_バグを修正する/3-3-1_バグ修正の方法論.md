# 3-3-1 バグ修正の方法論

## 🎯 このセクションで学ぶこと

- バグ修正の全体フロー（報告→再現→調査→修正→検証）の各フェーズで何をすべきかを理解する
- Claude Code への調査指示の組み立て方（症状・再現手順・期待値の伝え方）を理解する
- バグの3類型（データの正しさ・アクセス制御・機能不全）ごとの調査アプローチの違いを理解する

まずバグ報告を受けたときの心構えを確認し、次にバグ修正の全体フローを学び、最後にバグの種類に応じた調査アプローチの違いを押さえます。

---

## 導入: バグ報告を受けたら何から始めるか

実務でバグ報告を受けたとき、どう動くべきでしょうか。

「マイコースの進捗率がおかしいらしい」「本番で500エラーが出ている」。こうした報告は、たいてい曖昧です。報告者はユーザーやプロダクトマネージャーであることが多く、技術的な原因まで特定して報告してくれるとは限りません。

曖昧な報告を受けて、いきなりコードを修正し始めるとどうなるでしょうか。原因の特定が不十分なまま修正すると、表面的な症状は消えても根本原因が残り、別の形で再発します。あるいは、修正したつもりが別の機能を壊してしまうこともあります。

バグ修正は「コードを直す作業」ではなく、「原因を突き止め、正しく直し、直ったことを証明する」プロセスです。このプロセスを体系的に進められるかどうかが、ジュニアとミドル以上のエンジニアの大きな違いの1つです。

### 🧠 先輩エンジニアはこう考える

> バグ修正は、実務で最も頻繁に発生するタスクの1つだ。自分がジュニアの頃は「バグ報告 → すぐコードを探す → とりあえず直す」と動いていたが、それだと修正漏れや二次障害が起きやすい。先輩に「まず再現して、原因を特定してから直せ」と教わってから、修正の精度が格段に上がった。Claude Code と協働する場合も同じで、いきなり「このバグを直して」と投げるのではなく、まず調査を依頼して原因を理解してから修正に入る方が、結果的に速い。

## バグ修正の全体フロー

バグ修正は、次の5つのフェーズで進めます。

```
報告 → 再現 → 調査 → 修正 → 検証
 │      │      │      │      │
 │      │      │      │      └─ テストで修正を証明する
 │      │      │      └─ 原因に対して最小限の変更を加える
 │      │      └─ 関連コードを読み、原因を特定する
 │      └─ 自分の環境で症状を確認する
 └─ 報告内容を正確に把握する
```

一見シンプルですが、各フェーズにはそれぞれ重要なポイントがあります。順に見ていきましょう。

### フェーズ 1: 報告を正確に把握する

バグ報告を受けたら、まず以下の3つを整理します。

| 整理すべき情報 | 説明 | 例 |
|---|---|---|
| **症状** | 何が起きているか（事実） | 「マイコースの進捗率が100%にならない」 |
| **期待値** | 本来どうあるべきか | 「全レッスンを完了したら100%になるべき」 |
| **再現条件** | どういう状況で発生するか | 「非公開レッスンがあるコースで発生する」 |

報告が曖昧な場合は、報告者に確認を取ります。「進捗率がおかしい」だけでは、「0%のまま動かない」のか「100%にならない」のか「マイナスになる」のかで、原因がまったく異なります。

> 💡 **実務での報告形式**: 実務では、報告は GitHub Issue や Slack メッセージ、口頭などさまざまな形で届きます。どんな形式でも「症状・期待値・再現条件」の3つを自分で整理する習慣をつけてください。

### フェーズ 2: 再現する

報告された症状を、自分の開発環境で再現します。再現できないバグは修正できません。

再現が重要な理由は2つあります。

1. **原因の手がかりを得る**: 再現時のエラーメッセージ、画面の状態、ログの内容が調査の起点になります
2. **修正後の検証基準になる**: 「修正前はこう壊れていた」が明確なら、「修正後は正しく動く」を確認できます

再現できない場合は、報告者に追加情報を求めるか、ログやデータベースの状態から手がかりを探します。

> ⚠️ **再現をスキップしない**: 「コードを読めば原因がわかりそうだ」と思っても、再現を省略しないでください。コードの読み違いや思い込みによる誤修正を防ぐためには、実際に症状を目で見ることが最も確実です。

### フェーズ 3: 調査する（原因を特定する）

再現できたら、なぜその症状が起きているかを調査します。ここが最も時間をかけるべきフェーズです。

調査の基本は、[3-2-1 コードリーディングの方法論](../chapter-02_既存コードを理解する/3-2-1_コードリーディングの方法論.md) で学んだアプローチの応用です。バグ修正の場合は、次の流れで進めます。

1. **関連コードを読む**: 症状が発生している画面やエンドポイントから、該当するController、モデル、ビューを特定する
2. **原因の仮説を立てる**: 再現時の症状とコードの構造から「ここが原因ではないか」という仮説を立てる
3. **処理フローを追って仮説を検証する**: データがどこで作られ、どう加工され、どう表示されるかを追跡し、仮説が正しいか確認する
4. **原因を絞り込む**: 「期待値と実際の結果が乖離するポイント」を特定する

ステップ 2 の **仮説を立てる** が重要です。仮説がないまま闇雲にコードを読むと、関連のない箇所まで調査して時間を浪費します。報告の内容と再現時の症状から、まず「どの処理が怪しいか」を推測し、その仮説を起点に調査を進めましょう。

例えば「進捗率が100%にならない」というバグなら、「進捗率の計算ロジックに問題があるのでは？」という仮説を立て、計算ロジックから調査を始めます。「500エラーが出る」なら、エラーメッセージから「この行の null アクセスが原因では？」と仮説を立てます。仮説が外れたら、別の仮説を立てて再調査します。

> 📝 調査のフェーズは、3-2 で学んだコードリーディングの実践です。バグ修正に限らず、機能開発やリファクタリングでも「まず関連コードを読む」ことが起点になります。

### フェーズ 4: 修正する

原因が特定できたら、修正を加えます。修正で心がけるべきは **最小限の変更** です。

- 原因に対して直接的な修正を加える
- 「ついでにリファクタリング」はしない（バグ修正とリファクタリングを混ぜると、修正の影響範囲が広がり、レビューが難しくなります）
- 既存の設計パターンやコーディング規約に従う（[3-1-3 プロジェクトの規約・設定を確認する](../chapter-01_実践の準備/3-1-3_プロジェクトの規約・設定を確認する.md) で確認した規約を思い出してください）

### フェーズ 5: 検証する

修正が正しいことを確認します。検証は3段階で行います。

1. **手動検証**: 再現手順をもう一度実行し、症状が解消されたことを確認する
2. **テストによる検証**: 修正内容をテストコードで保証する（回帰テスト）
3. **影響範囲の確認**: 修正が他の機能を壊していないことを確認する

テストは、修正前に書くのが理想です。まず「このバグを再現するテスト」を書いて失敗させ、修正後に成功に変わることを確認する。この流れなら、テスト自体がバグの再現と修正の証明を兼ねます。

3つ目の **影響範囲の確認** は見落とされがちですが重要です。具体的には次の方法で進めます。

- **修正したメソッドの呼び出し元を確認する**: Claude Code に「このメソッドを使っている箇所を全て見つけて」と依頼し、修正が他の画面や処理に影響しないか確認する
- **既存のテストスイートを実行する**: 修正に関連するテストだけでなく、プロジェクト全体のテストを実行して既存機能が壊れていないことを確認する
- **関連する画面を手動で確認する**: 修正した処理を利用している別の画面（例: 進捗率を表示するコーチのダッシュボード）が正しく動作するか目で確認する

> 🔑 **テストはバグ修正の証明書**: 「直した」と口頭で伝えるだけでなく、テストで修正を証明できることが実務では求められます。テストがあれば、将来同じバグが再発したときに自動で検出できます。

## Claude Code と進めるバグ修正

ここまで見てきた5フェーズを、Claude Code とどう進めるかを整理します。

### Claude Code への指示の組み立て方

Claude Code にバグ調査を依頼するとき、最も重要なのは **症状・再現手順・期待値** の3つを伝えることです。[3-1-1 Claude Code で実務タスクを遂行する考え方](../chapter-01_実践の準備/3-1-1_Claude%20Codeで実務タスクを遂行する考え方.md) で学んだ「協働ワークフロー」の第1フェーズ（指示）にあたります。

例として、次の2つの指示を比べてみてください。

**曖昧な指示**:

```
> 進捗率のバグを直して
```

**具体的な指示**:

```
> マイコース画面（/my-courses）の進捗率表示にバグがあります。
>
> 症状: 全レッスンを完了しても進捗率が100%にならないコースがある
> 再現条件: 非公開レッスン（is_published = false）が含まれるコース
> 期待値: 受講生が受講可能な公開レッスンだけで進捗率を計算し、全完了時に100%になる
>
> まず関連するコードを読んで、原因を特定してください。
```

曖昧な指示では、Claude Code は何が「バグ」なのかを推測する必要があります。推測が正しい保証はありません。一方、具体的な指示では、症状と再現条件が明確なため、Claude Code は原因の特定に集中できます。

> 💡 Anthropic の公式ドキュメント（[Common Workflows](https://code.claude.com/docs/ja/common-workflows)）でも、バグ修正のワークフローとして「エラーメッセージや再現手順を Claude Code に共有し、修正を依頼する」方法が推奨されています。

### 各フェーズでの Claude Code の活用

| フェーズ | Claude Code の活用 | 人間が判断すること |
|---|---|---|
| 報告 | - | 症状・期待値・再現条件を整理する |
| 再現 | 再現手順のコマンドを実行させる | 症状が再現したか目で確認する |
| 調査 | 関連コードの読み取り、原因の仮説提示 | 仮説が妥当か判断する |
| 修正 | 修正コードの生成 | 修正が適切か（最小限か、規約に従っているか）を検証する |
| 検証 | テストの生成・実行 | テストが正しいケースを網羅しているか確認する |

重要なのは、**Claude Code に調査と修正を任せたとしても、「原因が正しいか」「修正が適切か」の判断は人間が行う** ことです。これは [3-1-1](../chapter-01_実践の準備/3-1-1_Claude%20Codeで実務タスクを遂行する考え方.md) で学んだ「70-30 モデル」の 30% にあたる部分です。

### `/rewind` によるセーフティネット

Claude Code の checkpointing 機能（`/rewind`）は、バグ修正で特に心強いセーフティネットです。

Claude Code はあなたのプロンプトのたびに自動でチェックポイントを作成します。修正が意図と異なっていた場合や、修正中に別の問題が発生した場合は、`/rewind` でその時点の状態に戻ることができます。

```
チェックポイント1     チェックポイント2     チェックポイント3
（修正前）     →     （修正後）      →     （テスト追加後）
     │                    │                    │
     └── /rewind で ──────┘                    │
         いつでも戻れる                          │
                          └── /rewind で ──────┘
                              いつでも戻れる
```

> ⚠️ `/rewind` は Claude Code のファイル編集ツールによる変更のみを追跡します。Bash コマンドで直接ファイルを操作した変更（`rm`、`mv` など）は追跡されません。また、`/rewind` はローカルの作業を巻き戻す機能であり、Git の代わりにはなりません。修正が確定したら、Git でコミットして永続的な記録を残してください。

## バグの3類型と調査アプローチ

バグにはさまざまな種類がありますが、実務で遭遇するバグの多くは **影響領域** によって3つに分類できます。この Chapter では、それぞれ1つずつ実践します。

### 1. データの正しさ

**特徴**: 表示されるデータや計算結果が正しくない。アプリケーションはエラーなく動作するが、出力が期待値と異なる。

| 項目 | 内容 |
|---|---|
| 典型例 | 集計値のズレ、フィルタ条件の漏れ、小数点の丸め誤差 |
| 調査の起点 | データの流れを追う（DB → モデル → Controller → ビュー） |
| 見極めの重点 | **正しさ**: 計算ロジックが仕様どおりか、境界値で正しく動くか |
| 発見の難しさ | エラーが出ないため、テストやユーザー報告がないと気づきにくい |

**調査アプローチ**: 「期待値と実際の結果が乖離するポイント」をデータの流れに沿って探します。DB のデータは正しいか、クエリの結果は正しいか、計算ロジックは正しいか、と上流から順に確認します。

### 2. アクセス制御

**特徴**: 本来アクセスできないはずのリソースにアクセスできてしまう、またはアクセスできるべきリソースにアクセスできない。

| 項目 | 内容 |
|---|---|
| 典型例 | 認可（Policy）の条件漏れ、ミドルウェアの設定漏れ、URL 直接アクセスでの迂回 |
| 調査の起点 | ルーティング → ミドルウェア → Policy の順に認可の流れを確認する |
| 見極めの重点 | **安全性**: 不正アクセスの経路が残っていないか、全ロールで正しく制御されているか |
| 発見の難しさ | 正常系の操作では気づかない。意図的に異常な操作（URL 直打ち等）を試す必要がある |

**調査アプローチ**: アクセス制御のバグは、「正常に動いているように見える」点が厄介です。調査では、認可の仕組み全体（ミドルウェア → Policy → Scope）を確認し、「誰が」「何に」「どの操作を」できるかを網羅的にチェックします。

> 🔑 **アクセス制御のバグはセキュリティの問題**: アクセス制御のバグは、機能的なバグとは異なり、セキュリティの脆弱性に直結します。修正後は特に念入りにテストを書き、全ロールでの動作を検証してください。

### 3. 機能不全

**特徴**: 特定の操作でアプリケーションがエラーを返す、またはクラッシュする。500エラー、例外、画面が真っ白になる、などの明確な症状が出る。

| 項目 | 内容 |
|---|---|
| 典型例 | null アクセス、0除算、未定義メソッドの呼び出し、外部 API のタイムアウト |
| 調査の起点 | エラーメッセージ・スタックトレースから発生箇所を特定する |
| 見極めの重点 | **正しさ**: 例外ケース（null、0件、空配列など）を正しくハンドリングしているか |
| 発見の難しさ | エラーが明確なため発見自体は容易。ただし根本原因が別の箇所にある場合がある |

**調査アプローチ**: エラーメッセージとスタックトレースが最大の手がかりです。Laravel の場合、`storage/logs/laravel.log` にエラーの詳細が記録されます。エラーメッセージを Claude Code に共有し、「このエラーの原因を調査して」と依頼するのが効率的です。

> 💡 **エラーログの活用**: Laravel はデフォルトで `storage/logs/laravel.log` にエラーを記録します。500エラーが発生したら、まずこのログを確認してください。スタックトレースにはエラーが発生したファイルと行番号が記録されており、原因の特定に直結します。

### 3類型の比較

| 類型 | 症状 | 調査の起点 | 見極めの重点 |
|---|---|---|---|
| データの正しさ | 表示値が期待と異なる | データの流れ（DB → 表示） | 正しさ |
| アクセス制御 | 不正なアクセスが通る | 認可の流れ（ルート → Policy） | 安全性 |
| 機能不全 | エラー・クラッシュ | エラーメッセージ・ログ | 正しさ |

### 🧠 先輩エンジニアはこう考える

> バグを分類して考えるのは、調査の効率を上げるためだ。「データがおかしい」と言われたら計算ロジックを疑う。「権限がおかしい」と言われたら Policy やミドルウェアを疑う。「500エラーが出る」と言われたらエラーログを見る。出発点が違うだけで、やることは同じ（仮説を立てて検証する）だが、出発点を間違えると遠回りになる。自分も以前、データの問題だと思って計算ロジックを調査し続けた結果、実は権限の問題でそもそも表示されるべきでないデータが混入していた、ということがあった。バグの類型を意識するだけで、こうした遠回りを減らせる。

## ✨ まとめ

- バグ修正は **報告→再現→調査→修正→検証** の5フェーズで進める。再現を省略しない
- Claude Code への調査指示は **症状・再現手順・期待値** を明確に伝える。曖昧な指示では推測に頼ることになり、原因特定の精度が下がる
- バグは影響領域で **データの正しさ / アクセス制御 / 機能不全** の3類型に分類でき、それぞれ調査の起点と見極めの重点が異なる
- 修正は **最小限の変更** にとどめ、修正後は **テスト** で修正を証明する
- `/rewind` で作業を安全に巻き戻せるが、確定した修正は Git でコミットする
