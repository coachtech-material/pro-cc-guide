# 3-5-1 リファクタリングの方法論

> 動いているコードを「壊さずに良くする」。この Chapter では、リファクタリングの方法論を学び、CourseHub のコードを Claude Code と協働して改善します。

| セクション | 種類 | 内容 |
|---|---|---|
| 3-5-1 | 概念 | リファクタリングの全体フロー、改善対象の特定基準、テストによる動作保証、Claude Code の改善提案を判断する方法 |
| 3-5-2 | 混合 | CourseHub の2つの問題（Fat Controller・N+1 クエリ）を改善・検証する |

## 📖 この Chapter の進め方

まず 3-5-1 でリファクタリングの方法論を学びます。「動作を変えずにコードを良くする」ためのフローと、Claude Code の改善提案を判断する基準を理解してください。次の 3-5-2 では、CourseHub のコードを実際に改善します。1つ目は Fat Controller の責務分離（コード構造の改善）、2つ目は N+1 クエリの解消（パフォーマンスの改善）です。改善の対象が違えばアプローチも変わることを体験します。

## 導入: 「動いているコードに触る」不安

前の Chapter では、CourseHub に2つの機能を追加しました。機能開発は「新しい価値を加える」タスクで、完成すれば目に見える成果があります。

これから取り組むリファクタリングは、性質が異なります。外から見た動作は何も変わりません。ユーザーが気づくことはなく、新しい画面が増えるわけでもありません。それでも、プロの現場ではリファクタリングは日常的に行われます。なぜでしょうか。

CourseHub の `CoachCourseController@store` を思い出してください。3-4-2 の導入で「200 行超の Fat Controller」として触れました。このメソッドは動いています。コースの作成は正しく機能しています。しかし、このメソッドに機能を追加したり、バグを修正したりする場面を想像してください。バリデーション、スラッグ生成、画像アップロード、タグ同期、初期 Chapter 作成が全て1つのメソッドに詰め込まれている状態で、「画像アップロードの処理だけ修正したい」と思っても、200 行の中から該当箇所を見つけ、他の処理に影響がないか確認する必要があります。

リファクタリングは「今の動作を壊さずに、コードの内部構造を改善する」作業です。外からは何も変わらないけれど、次に変更を加えるときのコストが下がる。バグが入り込みにくくなる。新しいメンバーがコードを理解しやすくなる。こうした「将来の変更しやすさ」のために、今のコードを整理するのがリファクタリングです。

### 🧠 先輩エンジニアはこう考える

> 「動いてるコードに触りたくない」という気持ちはよくわかる。自分もジュニアの頃はそうだった。でも、動いてるけど読めないコードは、いずれ誰かが触らなければならない。そのとき、200 行のメソッドを読み解くのは本人かもしれないし、次に入るメンバーかもしれない。リファクタリングは「自分（または仲間）の未来の作業を楽にする」投資だと思っている。
>
> それに、テストがあれば「動作を変えていない」ことを証明できる。テストが通り続ける限り、安心してコードを整理できる。テストのないリファクタリングは怖いけど、テストのあるリファクタリングは意外と安全だ。

## リファクタリングの全体フロー

リファクタリングは、次の4つのフェーズで進めます。

```
対象特定 → テスト確認 → 改善 → テスト再実行
   │           │          │          │
   │           │          │          └─ テストで「動作が変わっていない」を証明する
   │           │          └─ 小さな単位で段階的に改善する
   │           └─ 既存のテストを実行し、現状の動作を確認する
   └─ 改善すべきコードを見つけ、何をどう改善するか決める
```

バグ修正のフロー（報告→再現→調査→修正→検証）と比較すると、大きな違いがあります。バグ修正は「壊れている」という明確な起点がありました。リファクタリングには「壊れている」がありません。動いているコードの中から、改善すべき箇所を自分で見つけるところから始まります。

### フェーズ 1: 対象を特定する

改善すべきコードを見つけるには、「コードスメル」と呼ばれるシグナルに注目します。コードスメルとは、プログラムが正しく動いているにもかかわらず、設計上の問題を示す兆候のことです。Martin Fowler 氏が著書「[Refactoring: Improving the Design of Existing Code](https://martinfowler.com/books/refactoring.html)」で体系化した概念で、リファクタリングの出発点として広く使われています。

実務で遭遇しやすいコードスメルを整理します。

| コードスメル | 兆候 | 改善の方向性 |
|---|---|---|
| **Fat Controller / Long Method** | 1つのメソッドが100行を超える。複数の責務を1つのメソッドに詰め込んでいる | 責務ごとにメソッドやクラスに分割する |
| **N+1 クエリ** | ループ内でデータベースクエリが実行される。一覧画面の表示が遅い | Eager Loading でクエリをまとめる |
| **重複コード** | 同じ処理が複数箇所にコピーされている | 共通メソッドやクラスに抽出する |
| **不統一なパターン** | 同じ種類の処理が場所によって違う書き方をされている | プロジェクトの規約に統一する |

> 💡 コードスメルは「絶対に直すべき問題」ではなく「改善の余地があるシグナル」です。全てのコードスメルを直す必要はありません。改善することで得られるメリット（可読性向上、パフォーマンス改善）と、改善にかかるコスト（時間、リスク）を比較して判断します。

コードスメルを見つける方法はいくつかあります。

- **自分でコードを読む**: 3-2 で学んだコードリーディングの実践。気になる箇所をメモしておく
- **Claude Code に分析を依頼する**: 「このファイルの改善点を分析して」と依頼すると、コードスメルを検出してくれる
- **ツールを活用する**: Laravel では `preventLazyLoading()` を設定すると、N+1 クエリが発生した時点で例外が発生し、問題箇所を特定できる

ただし、Claude Code にコードスメルの検出を依頼する際には注意が必要です。Claude Code は改善の余地がある箇所を網羅的にリストアップしがちです。「全部直しましょう」と提案されても、優先順位をつけるのはあなたの判断です。

改善対象を選ぶ基準として、以下を意識してください。

1. **頻繁に変更が入る箇所か**: めったに触らないコードより、よく変更する箇所を優先する
2. **他のタスクで触る箇所の近くにあるか**: バグ修正や機能追加で関連コードを読んでいるときにコードスメルに気づくことは多い。ただし、3-3-1 で学んだ通り、バグ修正とリファクタリングは同じコミットに混ぜないでください。気づいた問題はメモしておき、別のコミットや PR として着手するのが安全です
3. **チームで合意が取れるか**: 大規模なリファクタリングは影響範囲が広いため、チームで方針を合わせてから着手する

### フェーズ 2: テストを確認する

対象を決めたら、改善に着手する前にテストを実行します。

```
/test
```

このフェーズが重要な理由は、リファクタリングの定義そのものにあります。「動作を変えずにコードの内部構造を改善する」ためには、「動作が変わっていない」ことを証明する手段が必要です。テストがその手段です。

テストを実行して、2つのことを確認します。

1. **既存のテストが通るか**: リファクタリング前の時点で全テストが通っていれば、これが「現在の正しい動作」のベースラインになる
2. **対象のコードに対するテストがあるか**: テストがない場合、リファクタリング後に「動作が変わっていない」ことを証明できない

テストがない場合はどうするか。**まずテストを書いてからリファクタリングに着手します**。これは鉄則です。

```
テストなしでリファクタリング → 動作が変わったことに気づけない → バグを生む
テストを書いてからリファクタリング → テストが通れば動作は同じ → 安全に改善できる
```

テストを書く段階では、現在の動作をそのままテストに反映します。「本来こうあるべき」ではなく「今こう動いている」をテストにする点がポイントです。リファクタリングは動作を変えない作業なので、テストも現状の動作を基準にします。

> ⚠️ **よくある失敗**: テストを書かずに「ちょっとした整理だから大丈夫」とリファクタリングを始め、意図しない動作変更を見落とすケース。特にバリデーションルールの移動やクエリ条件の変更は、テストなしでは気づきにくいです。

### フェーズ 3: 小さな単位で段階的に改善する

改善は **小さなステップを積み重ねる** 形で進めます。

```
1つの変更 → テスト実行 → 通る → 次の変更 → テスト実行 → 通る → ...
```

たとえば Fat Controller の責務分離であれば、以下のように段階的に進めます。

1. バリデーションを Form Request に抽出する → テスト実行
2. スラッグ生成を private メソッドに抽出する → テスト実行
3. 画像アップロードを private メソッドに抽出する → テスト実行
4. タグ同期を private メソッドに抽出する → テスト実行

一度に全てを変更すると、テストが失敗したときにどの変更が原因かわかりません。小さな単位で進めれば、失敗の原因が直前の変更に限定されます。

Claude Code と協働する場合、Plan Mode を使って改善のステップを設計してから実行するのが効果的です。Claude Code に「一気に全部リファクタリングして」と依頼すると、大量の変更が一度に行われ、テストが失敗したときの原因特定が難しくなります。

```
> Plan Mode で CoachCourseController@store のリファクタリング計画を立ててください。
> 段階的に進めたいので、1ステップ1つの責務の抽出にしてください。
```

> 💡 Claude Code の Plan Mode は、リファクタリングの計画立案に適しています。どの責務をどの順序で抽出するか、各ステップでどのテストを実行するかを事前に設計できます。計画が固まったら Plan Mode を解除し、1ステップずつ実行します。

### フェーズ 4: テストで動作不変を証明する

各ステップの後と、全ての改善が完了した後にテストを実行します。

```
/test
```

テストが全て通れば、動作が変わっていないことが証明されました。リファクタリングの完了です。

テストが失敗した場合は、直前の変更を見直します。リファクタリングの過程で動作を変えてしまっている箇所があるはずです。よくある原因は以下の通りです。

| 失敗の原因 | 具体例 |
|---|---|
| バリデーションルールの変更 | Form Request に移す際にルールを微妙に変えてしまった |
| クエリ条件の変更 | メソッド抽出時に `where` 条件を漏らした |
| 戻り値の変更 | メソッドの戻り値の型や構造が変わった |
| 依存関係の漏れ | 抽出したメソッドに必要な変数を渡し忘れた |

> 🔑 **リファクタリングの鉄則**: テストが失敗したら、まず自分の変更を疑う。リファクタリングは「動作を変えない」ことが前提なので、テスト失敗 = 自分の変更が動作を変えてしまった、ということです。

## Claude Code の改善提案を判断する

Claude Code にリファクタリングを依頼すると、多くの場合、改善提案が返ってきます。この提案をそのまま受け入れるのではなく、「必要十分か」を判断する必要があります。

### よくある過剰な提案

Claude Code は「より良い設計」を目指す傾向があります。これ自体は悪いことではありませんが、現時点のプロジェクトにとって過剰な改善を提案することがあります。

| 依頼 | 過剰な提案の例 | 判断基準 |
|---|---|---|
| Fat Controller の整理 | Service クラスへの切り出し + Repository パターンの導入 + インターフェースの定義 | 今の課題は「1メソッドが長すぎる」こと。Form Request 抽出 + private メソッド分割で十分か |
| N+1 の解消 | Eager Loading + キャッシュ層の導入 + クエリビルダーの最適化 | 今の課題は「N+1 クエリ」。Eager Loading だけで解決するか |
| 重複コードの解消 | 共通メソッド抽出 + 抽象クラスの導入 + Trait の作成 | 今の課題は「同じコードが2箇所にある」こと。共通メソッドの抽出だけで十分か |

### 判断の3つの基準

Claude Code の改善提案を評価するときは、以下の3つの基準を使います。

**1. プロジェクトの規約に合っているか**

rules/coding.md やプロジェクトの設計方針に照らします。たとえば CourseHub の README には「ビジネスロジックが複雑な場合は Service クラスに切り出す」とあります。バリデーションの抽出は「複雑なビジネスロジック」ではないので、Form Request への抽出で十分であり、Service クラスまで導入する必要はありません。

**2. 既存のパターンと一貫しているか**

プロジェクト内の他のコードがどう書かれているかを参考にします。CourseHub では `LessonController` が Form Request を使い、`EnrollmentService` がビジネスロジックを担当するパターンが既にあります。このパターンに合わせるのが一貫性の観点では正しい判断です。

**3. 今の問題を解決するのに十分か**

リファクタリングの目的は「今あるコードスメルを解消する」ことです。将来の拡張性のために過度な抽象化を導入するのは、リファクタリングの範囲を超えています。「今の問題を最小限の変更で解決できるか」が判断の基本です。

### 🧠 先輩エンジニアはこう考える

> Claude Code に「リファクタリングして」と依頼すると、教科書的に理想的な設計を提案してくることがある。Repository パターン、インターフェース、DI コンテナの活用。どれも良い設計パターンだけど、「今このプロジェクトに必要か?」は別の問題だ。
>
> 自分が判断するときは「この変更で、次にこのコードを触る人が楽になるか?」を考える。200 行のメソッドを5つの private メソッドに分割するだけで、次の人は「画像アップロードの処理」をすぐに見つけられる。それで十分。Repository パターンの導入は、データアクセスが本当に複雑になってからでいい。

## 構造改善とパフォーマンス改善の違い

リファクタリングの改善対象は大きく2つに分かれます。**コード構造の改善** と **パフォーマンスの改善** です。どちらも「動作を変えない」点は共通ですが、アプローチが異なります。

### コード構造の改善

**目的**: コードの可読性・保守性を向上させる

**典型的な対象**: Fat Controller、重複コード、不統一なパターン

**アプローチ**:

```
1. 対象のコードを読み、責務を整理する
2. Plan Mode で分割の計画を立てる
3. 1つの責務ずつ抽出し、その都度テストを実行する
4. 全ステップ完了後にテストを再実行する
```

**検証のポイント**: テストが通ることに加え、改善後のコードが規約に沿っているか、既存パターンと一貫しているかを確認する

**具体例**: CourseHub の `CoachCourseController@store`（200 行超）は、バリデーション・スラッグ生成・画像アップロード・タグ同期・Chapter 作成など、複数の責務が1メソッドに詰め込まれています。Form Request の抽出と private メソッドへの分割で、各責務が独立した単位になります。

### パフォーマンスの改善

**目的**: 実行速度やリソース効率を向上させる

**典型的な対象**: N+1 クエリ、不要なクエリ、大量データの非効率な処理

**アプローチ**:

```
1. パフォーマンス問題を特定する（ログ、ツール、コードリーディング）
2. 改善前のクエリ数・実行時間を把握する
3. 改善を適用する
4. テストで動作不変を確認し、クエリ数・実行時間の変化を確認する
```

**検証のポイント**: テストが通ることに加え、クエリ数が実際に減っているか、画面の表示速度が改善されているかを確認する

**具体例**: CourseHub のコース一覧画面では、各コースのカテゴリ名・コーチ名・Chapter 数・受講者数を表示するために、コースの数だけクエリが実行されます（N+1 問題）。Eager Loading を追加すれば、クエリ数が N+1 から数クエリに減ります。

### 2つのアプローチの比較

| 観点 | コード構造の改善 | パフォーマンスの改善 |
|---|---|---|
| 目的 | 可読性・保守性の向上 | 実行速度・リソース効率の向上 |
| 効果の確認方法 | コードレビュー + テスト | クエリ数・実行時間の測定 + テスト |
| リスク | 動作変更のリスク（テストで検出可能） | 動作変更のリスク + クエリ結果の変化 |
| Claude Code の活用 | Plan Mode で分割計画、段階的に実行 | 問題箇所の特定と Eager Loading の提案 |
| 作業の独立性 | 1つのファイルを順に変更 | 異なるファイルを並列に変更可能 |

パフォーマンス改善の特徴は、異なる画面の N+1 問題を **並列に作業できる** 点です。コース一覧の N+1 と生徒一覧の N+1 は、関連するファイルが異なるため、同時に進められます。Claude Code の Worktree 機能を使えば、複数の改善を並行して進めることができます。

## ✨ まとめ

- リファクタリングは「動作を変えずにコードの内部構造を改善する」作業。ユーザーから見た動作は同じだが、将来の変更コストを下げる
- 全体フローは **対象特定 → テスト確認 → 改善 → テスト再実行** の4フェーズ。テストが動作不変の証明手段
- 改善対象は「コードスメル」で見つける。Fat Controller、N+1 クエリ、重複コード、不統一なパターンが代表例
- テストがない場合は **まずテストを書いてからリファクタリングに着手する** 。これは鉄則
- Claude Code の改善提案は「規約に合っているか」「既存パターンと一貫しているか」「今の問題を解決するのに十分か」の3基準で判断する
- コード構造の改善は Plan Mode で計画を立て段階的に進める。パフォーマンスの改善は Worktree で並列に進められる

次のセクションでは、CourseHub のコードを実際に改善します。Fat Controller の責務分離（コード構造）と N+1 クエリの解消（パフォーマンス）の2つに取り組み、3-5-1 で学んだフローを実践します。
