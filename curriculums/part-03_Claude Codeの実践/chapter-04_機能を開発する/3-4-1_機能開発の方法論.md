# 3-4-1 機能開発の方法論

> 機能開発は、バグ修正とは異なる判断力が求められるタスクです。この Chapter では、Claude Code と協働して既存機能の拡張と新規機能の構築を進める方法論を学び、CourseHub で実践します。

まず 3-4-1 で機能開発の方法論を学びます。バグ修正との違いを押さえ、要件理解から検証までの全体フローと、Claude Code の設計提案を判断する基準を理解してください。次の 3-4-2 では、CourseHub で2つの機能を実際に開発します。1つ目は既存機能の拡張（小テスト再受験）、2つ目は新規機能の構築（コースレビュー）です。スコープの違いによってアプローチがどう変わるかを体験します。

## 導入: バグ修正と機能開発の違い

前の Chapter では、Claude Code と協働してバグを修正しました。進捗率の計算ミス、認可の漏れ、500 エラー。いずれも「正しい動作がすでに定義されている」状態で、そこからの逸脱を検出し、修正するタスクでした。

機能開発は、これとは質的に異なります。

バグ修正では「どう動くべきか」が明確です。進捗率は公開レッスンのみで計算されるべき、非公開コースは student に見えるべきでない、未回答でも 500 エラーにならないべき。正解がはっきりしているからこそ、Claude Code に症状を伝えれば調査と修正を任せやすい構造でした。

一方、機能開発では「どう動くべきか」を自分で決める必要があります。小テストの再受験を許可するとして、何回まで? 合格後も再受験できる? 過去の受験履歴は見せる? これらは仕様の判断であり、コードの問題ではありません。Claude Code は実装の選択肢を提示してくれますが、「どの選択肢が正しいか」を決めるのはあなたです。

さらに、機能開発には **設計判断** が加わります。新しいコードをどこに配置するか、既存のパターンに従うかどうか、テーブル設計はどうするか。バグ修正では既存コードの構造を変えずに問題を直すのが基本でしたが、機能開発では構造そのものを追加・変更する判断が必要です。

| 観点 | バグ修正 | 機能開発 |
|---|---|---|
| ゴール | 既存の正しい動作に戻す | 新しい価値を加える |
| 正解の有無 | 明確（仕様からの逸脱を直す） | 不明確（仕様自体を決める） |
| コードの変更範囲 | 既存コードの修正（最小限） | 既存コードの修正 + 新規コードの追加 |
| 設計判断 | 基本的に不要（既存構造を維持） | 必須（配置・命名・パターンの選択） |
| Claude Code の役割 | 調査・修正の実行 | 設計提案 + 実装の実行 |
| 人間の判断 | 修正の正しさを検証 | 設計の妥当性を判断 + 実装の品質を検証 |

### 🧠 先輩エンジニアはこう考える

> 機能開発は自由度が高いぶん、判断を間違えると影響が大きい。バグ修正は「壊れた窓を直す」ようなもので、正解がはっきりしている。でも機能開発は「新しい部屋を増築する」ようなもの。既存の建物の構造を理解せずに増築すると、後から問題が出る。
>
> Claude Code に「この機能を作って」と丸投げすると、動くコードは出てくる。でもそのコードが既存の設計と合っているか、チームのコーディング規約に従っているか、将来の拡張を考慮しているかは、自分で判断しないといけない。特に既存プロジェクトでは、「動くけど設計が合っていない」コードが一番厄介だ。

## 機能開発の全体フロー

機能開発は、以下の4つのフェーズで進めます。

```
要件理解 → 設計 → 実装 → 検証
```

バグ修正のフロー（報告→再現→調査→修正→検証）と比較すると、**要件理解**と**設計**が新たに加わり、「調査」が「設計」に置き換わっています。バグ修正の「調査」は原因を突き止める作業でしたが、機能開発の「設計」はこれから作るものの構造を決める作業です。

```
バグ修正:   報告 → 再現 → 調査 → 修正 → 検証
                         ↑ 原因を特定する

機能開発:   要件理解 → 設計 → 実装 → 検証
                       ↑ 構造を決める
```

各フェーズで何をするか、Claude Code をどう活用するかを見ていきましょう。

### フェーズ 1: 要件理解

機能開発の最初のステップは、「何を作るのか」を正確に把握することです。

実務では、要件は GitHub Issue、Slack のメッセージ、ミーティングのメモなど、さまざまな形で届きます。重要なのは、実装に入る前に以下を明確にすることです。

- **誰が使うか**: どのロールのユーザーが対象か
- **何ができるようになるか**: 機能の目的と期待される動作
- **制約は何か**: 既存機能との関係、パフォーマンス要件、セキュリティ要件
- **完了条件は何か**: 何をもって「完成」とするか

> 💡 要件が曖昧なまま実装を始めると、途中で手戻りが発生します。Claude Code は与えられた要件に基づいて実装しますが、要件自体の妥当性は判断しません。「何を作るか」を決めるのは、あなたの責任です。

Claude Code は要件の整理を手伝ってくれます。要件を伝えた上で「考慮すべきエッジケースはあるか」「既存機能への影響は何か」と質問すると、見落としがちなポイントを洗い出してくれます。ただし、最終的にどの要件を採用するかの判断は自分で行います。

### フェーズ 2: 設計

要件が明確になったら、「どう作るか」を設計します。このフェーズが機能開発の核心です。

設計で決めることは主に3つあります。

1. **データ設計**: どのテーブル・カラムが必要か、既存モデルへの変更は何か
2. **処理設計**: どのクラスにどのロジックを配置するか、処理の流れはどうなるか
3. **画面設計**: ユーザーにどう見せるか、既存の UI パターンにどう合わせるか

ここで **Claude Code の Plan Mode** が力を発揮します。Plan Mode では Claude Code が読み取り専用で動作し、コードを変更せずに設計を検討できます。

```
> Plan Mode に切り替えて、以下の機能の設計を検討してください。
> [要件の説明]
> 既存の○○Controller と○○Model のパターンに合わせてください。
```

Plan Mode の使い方は [2-3-1 Plan Mode](../../part-02_Claude%20Codeの基礎/chapter-03_機能を使いこなす/2-3-1_Plan%20Mode.md) で学びました。ここではその知識を実務の機能開発に応用します。

> 📝 Plan Mode で作成した設計は、Ctrl+G でテキストエディタに開いて直接編集できます。Claude Code が提案した設計に「ここは既存の EnrollmentService のパターンに合わせてほしい」「この処理は Controller に書かずに Service に切り出してほしい」と注釈を加え、修正版を Claude Code に渡すことで、設計をブラッシュアップできます。

### フェーズ 3: 実装

設計が固まったら、実装に入ります。ここからは Normal Mode に切り替えて、Claude Code にコードを書いてもらいます。

実装で重要なのは、**一度に全てを作らない** ことです。大きな機能を一気に実装すると、問題が起きたときにどこが原因か特定しにくくなります。

```
要件全体を一気に実装する ← 避けるべき
   ↓
段階的に実装する ← 推奨
   │
   ├── Step 1: データベース（マイグレーション + モデル）
   │     → テスト or 動作確認
   │
   ├── Step 2: ロジック（Controller + Policy）
   │     → テスト or 動作確認
   │
   └── Step 3: 画面（Blade テンプレート）
         → テスト or 動作確認
```

各ステップで動作確認を挟むことで、問題を早期に発見できます。Claude Code への指示も「まずマイグレーションとモデルを作成して」「次に Controller を作成して」と段階的に出すのが効果的です。

> ⚠️ **よくある失敗**: 「この機能を全部作って」と一括で指示すると、Claude Code は動くコードを生成しますが、既存のパターンから逸脱したり、不要な複雑さを持ち込んだりする可能性が高くなります。段階的に進め、各ステップで確認する方が結果的に速く進みます。

### フェーズ 4: 検証

実装が完了したら、見極めチェックで品質を検証します。

バグ修正の Chapter では、バグの類型に応じて見極めの重点を変えました。機能開発では **品質（既存設計との整合性）** が重点です。

- **正しさ**: 要件通りに動作するか。テストで確認する
- **品質**: 既存のコーディング規約に従っているか。既存の設計パターンと整合しているか
- **安全性**: 認可の漏れはないか。入力値のバリデーションは適切か

> 🔑 機能開発の検証では「動くかどうか」だけでなく「既存のコードベースに馴染んでいるか」を確認します。Claude Code が生成したコードが、チームの他のメンバーが書いたコードと同じスタイル・パターンで書かれているかを確認してください。

## 着手前のコードリーディング: 既存パターンを読み取る

[3-2-1 コードリーディングの方法論](../chapter-02_既存コードを理解する/3-2-1_コードリーディングの方法論.md) では、プロジェクトに参画したときの全体理解としてコードリーディングを学びました。機能開発では、タスクに着手する前に **関連する既存コードを読む** ことが重要です。

なぜ着手前に読むのか。それは、**既存コードが「設計書」の役割を果たす** からです。

多くのプロジェクトでは、設計ドキュメントが不完全だったり、コードと乖離していたりします。CourseHub でも [3-1-3 プロジェクトの規約・設定を確認する](../chapter-01_実践の準備/3-1-3_プロジェクトの規約・設定を確認する.md) で確認したように、README のコーディング規約と実際のコードには乖離がありました。実際のコードがどう書かれているかが、事実上の「設計書」です。

### 何を読むか

機能開発の前に読むべきポイントは3つあります。

**1. 類似機能のコード**

これから作る機能に最も近い既存機能のコードを読みます。たとえば「コースレビュー機能」を作るなら、同じく受講生がコースに対して操作する「受講登録（Enrollment）」の実装が参考になります。

```
> EnrollmentController と EnrollmentService の実装を読んで、
> 受講登録の処理フローを説明してください
```

**2. コーディング規約の実態**

README や `.claude/rules/` に書かれた規約と、実際のコードの両方を確認します。

```
> 既存の Controller で Form Request が使われている箇所と
> インラインバリデーションが使われている箇所を調べてください
```

[3-1-3 プロジェクトの規約・設定を確認する](../chapter-01_実践の準備/3-1-3_プロジェクトの規約・設定を確認する.md) でこの乖離は把握済みです。規約と実態が異なる場合は、**規約に従う**（そのためにチームで規約を整備した）のが原則です。

**3. テストのパターン**

既存のテストがどう書かれているかも確認します。テストのスタイル（PHPUnit のアサーションの使い方、データのセットアップ方法等）を揃えることで、プロジェクト全体の一貫性を保てます。

```
> tests/Feature/ 配下の既存テストを確認して、
> テストのスタイルとパターンを教えてください
```

### 読んだ結果を指示に含める

既存パターンを把握したら、Claude Code への指示に含めます。「既存パターンに従って」と抽象的に伝えるだけでなく、具体的にどのファイルのパターンを参考にするかを明示すると、Claude Code はより正確に既存設計に揃えたコードを生成します。

```
> StoreLessonRequest のパターンに従って、レビュー投稿の
> Form Request を作成してください。authorize() でユーザーの
> 認可チェック、rules() でバリデーションルールを定義し、
> messages() で日本語エラーメッセージを設定してください
```

Addy Osmani 氏が [My LLM coding workflow going into 2026](https://addyosmani.com/blog/ai-coding-workflow/) で述べているように、コードベースに高品質な例があれば、AI はそのスタイルに従うコードを生成しやすくなります。既存パターンを指示に含めることは、AI の出力品質を高める最も効果的な方法の1つです。

## 要件をタスクに分解する

要件が大きいまま Claude Code に渡すと、一度に大量のコードが生成され、検証が困難になります。要件を実装可能な単位に分解することで、各ステップで確認しながら進められます。

### 分解の基準

タスクを分解する際の基準は以下の3つです。

**1. 独立して動作確認できる単位**

各タスクが完了した時点で、何かしらの動作確認ができる状態にします。「マイグレーションを作成する」「Controller を作成する」ではなく、「マイグレーションを実行してテーブルが作成されることを確認する」「エンドポイントにアクセスしてレスポンスを確認する」のように、確認可能な粒度にします。

**2. 依存関係に沿った順序**

データベース → モデル → Controller → View のように、依存関係の下流から上流に向かって実装します。View を先に作っても、表示するデータがなければ動作確認できません。

```
依存関係の方向:
  データベース（テーブル）
       ↓
  モデル（Eloquent）
       ↓
  Controller + Policy + Form Request
       ↓
  ルーティング
       ↓
  View（Blade テンプレート）
```

**3. 既存コードへの影響を最小化**

新規ファイルの作成と、既存ファイルの変更を分けます。新規ファイルは他に影響を与えませんが、既存ファイルの変更は既存機能を壊すリスクがあります。

### 分解の例

「コースレビュー機能」を例に、タスクを分解してみましょう。

| 順序 | タスク | 確認方法 |
|---|---|---|
| 1 | Review モデルとマイグレーション作成 | マイグレーション実行、テーブル確認 |
| 2 | ReviewPolicy 作成 | テストで認可ルールを確認 |
| 3 | StoreReviewRequest 作成 | テストでバリデーションを確認 |
| 4 | ReviewController 作成 + ルーティング追加 | エンドポイントへのアクセスを確認 |
| 5 | コース詳細画面にレビュー表示・投稿フォームを追加 | ブラウザで画面を確認 |
| 6 | テストを追加・実行 | 全テストが通ることを確認 |

Claude Code に分解を手伝ってもらうこともできます。

```
> 以下の機能を実装したいです。実装タスクを依存関係の順に分解してください。
> 各タスクには確認方法も含めてください。
>
> [要件の説明]
```

## AI の設計提案を判断する基準

Claude Code に機能開発を依頼すると、単にコードを書くだけでなく、設計の提案もしてくれます。「Review モデルを作る」と伝えれば、テーブル構造やリレーション、Controller の構成まで提案してきます。

ここが、[3-1-1 Claude Code で実務タスクを遂行する考え方](../chapter-01_実践の準備/3-1-1_Claude%20Codeで実務タスクを遂行する考え方.md) で学んだ 70-30 モデルの核心です。Claude Code がコードの 70% を生成してくれますが、残りの 30%（設計判断、品質検証、安全性確認）は人間が担います。特に設計判断は、機能開発における 30% の中で最も重要な部分です。

### 判断基準 1: 既存パターンとの整合性

最も重要な基準は、**既存のコードベースで使われているパターンと整合しているか** です。

たとえば、Claude Code が「コースレビュー機能には ReviewService を作成しましょう」と提案したとします。このとき確認すべきは:

- プロジェクトで Service クラスはどの程度使われているか?（CourseHub では EnrollmentService のみ）
- README の設計方針で Service クラスの使い方はどう定義されているか?（「複数モデルにまたがる処理は Service クラスに集約する」）
- レビュー投稿は「複数モデルにまたがる」複雑さがあるか?

この場合、レビュー投稿は Review モデルの作成だけで完結するシンプルな処理なので、Service クラスは不要かもしれません。Controller に直接書くか、Fat Controller になりそうなら Service を作るかは、処理の複雑さで判断します。

### 判断基準 2: コーディング規約との一致

[3-1-3 プロジェクトの規約・設定を確認する](../chapter-01_実践の準備/3-1-3_プロジェクトの規約・設定を確認する.md) で整備した `.claude/rules/coding.md` のルールに従っているかを確認します。

- バリデーションに Form Request を使っているか（インラインバリデーションではないか）
- 認可に Policy を使っているか
- 命名規則に従っているか

`.claude/rules/` に規約が書かれていれば、Claude Code はそれに従おうとします。ただし、100% 従うとは限りません。生成されたコードを確認し、規約からの逸脱があれば指摘してください。

### 判断基準 3: 変更の影響範囲

新しいコードが既存機能に影響を与えていないかを確認します。

- 既存のルーティングと衝突していないか
- 既存のモデルリレーションが壊れていないか
- 既存のテストが通るか

特に既存機能の拡張では、「新しいコードを追加したら既存のテストが失敗した」というケースが起きやすいです。各ステップで `/test` を実行し、既存テストが通ることを確認しながら進めてください。

### 🧠 先輩エンジニアはこう考える

> 自分がジュニアだった頃、「動けばOK」と思ってコードを書いていた。でもコードレビューで「なぜこのパターンにしたの?」と聞かれて答えられなかったことが何度もある。
>
> Claude Code が提案した設計をそのまま採用するのは簡単だ。でも「なぜこの設計なのか」をチームメイトに説明できないなら、それは理解していないということ。[3-1-1](../chapter-01_実践の準備/3-1-1_Claude%20Codeで実務タスクを遂行する考え方.md) で「AI エージェントの活用 = コードレビュー」という話をした。レビューでは設計の意図が問われる。Claude Code が「なぜこの設計にしたか」を説明してくれたとしても、自分の言葉で説明できるまで理解してから採用しよう。

## 拡張と新規構築のアプローチの違い

機能開発には大きく2つのパターンがあります。**既存機能の拡張** と **新規機能の構築** です。それぞれでアプローチが異なります。

### 既存機能の拡張

既存のモデル・Controller・View に変更を加えて機能を追加するパターンです。たとえば「小テストの再受験機能」は、既存の Quiz・Submission の仕組みを拡張します。

**特徴**:
- 既存コードの理解が前提（どこに何が書いてあるか把握していないと始められない）
- 既存の動作を壊さない制約がある
- テストが既にある場合は、変更後も既存テストが通ることが必須

**アプローチ**:

```
1. 既存コードを読む（関連する Model, Controller, View, Test）
      ↓
2. 変更箇所を特定する（何を追加・修正するか）
      ↓
3. 既存テストを実行して現状を確認する
      ↓
4. 変更を実装する
      ↓
5. 既存テスト + 新規テストを実行する
```

Claude Code への指示では、既存コードへの参照を明示します。

```
> 既存の QuizController@submit と QuizController@result を確認してください。
> 小テストの再受験機能を追加したいです。以下の要件です:
> - 合格済みの場合は再受験不可
> - 受験回数を Submission の件数で管理
> - 結果画面に受験履歴を表示
> 既存の処理フローを壊さないように拡張してください。
```

### 新規機能の構築

新しいモデル・Controller・View をゼロから作成するパターンです。たとえば「コースレビュー機能」は、Review モデルを新規作成し、関連する全てのファイルを新たに追加します。

**特徴**:
- 設計の自由度が高い（パターンを自分で決められる）
- 既存コードへの影響が比較的小さい（新規ファイルが中心）
- ただし、既存のパターンと整合しないコードを書くリスクがある

**アプローチ**:

```
1. 類似機能のコードを読む（参考パターンの把握）
      ↓
2. Plan Mode で設計する（モデル・Controller・View の構成）
      ↓
3. 段階的に実装する（データベース → モデル → Controller → View）
      ↓
4. 各ステップでテスト・動作確認する
      ↓
5. 全体を通して検証する
```

新規構築では、Plan Mode で設計を固めてから実装に入ることを推奨します。設計の自由度が高いぶん、最初の方向を間違えると手戻りが大きくなります。

```
> Plan Mode に切り替えてください。
> コースレビュー機能を設計したいです。
> EnrollmentController と EnrollmentService のパターンを参考に、
> Review モデルの設計を提案してください。
```

### 2つのアプローチの比較

| 観点 | 既存機能の拡張 | 新規機能の構築 |
|---|---|---|
| 事前に読むコード | 拡張対象の実装全体 | 類似機能のパターン |
| 設計フェーズ | 変更箇所の特定が中心 | Plan Mode で全体設計 |
| 実装の制約 | 既存の動作を壊さない | 既存パターンとの整合性 |
| テストの重点 | 既存テスト + 追加分 | 新規テスト中心 |
| リスク | 既存機能の破壊 | パターンの不整合 |

どちらのパターンでも共通するのは、**着手前にコードを読む** ことと、**段階的に実装・検証する** ことです。

## ✨ まとめ

- 機能開発はバグ修正と異なり、**設計判断** が求められる。「何を作るか」「どう作るか」を自分で決める必要がある
- 機能開発の全体フローは **要件理解→設計→実装→検証** の4フェーズ。特に「設計」フェーズが機能開発の核心であり、Plan Mode を活用する
- 着手前に **既存コードを読み、設計パターンとコーディング規約を把握する** ことが品質の鍵。既存コードが事実上の「設計書」
- 要件は **独立して動作確認できる単位** に分解し、依存関係の順に段階的に実装・検証する
- Claude Code の設計提案は、**既存パターンとの整合性**、**コーディング規約との一致**、**変更の影響範囲** の3基準で判断する
- 既存機能の拡張と新規機能の構築では、事前の読み込み範囲と設計フェーズの深さが異なる
