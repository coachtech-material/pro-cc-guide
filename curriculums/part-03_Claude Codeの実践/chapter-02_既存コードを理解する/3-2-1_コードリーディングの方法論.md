# 3-2-1 コードリーディングの方法論

## 🎯 このセクションで学ぶこと

- コードリーディングが難しいと感じる認知科学的な理由（記憶システム、チャンキング、認知負荷）を理解する
- 認知負荷を段階的に下げる5ステップ（探索→検索→転写→理解→増強）の方法論を理解する
- 各ステップで Claude Code をどう活用するか、そして Claude の説明をどう検証するかを理解する

まず「なぜコードが読めないのか」を認知科学の視点から理解し、次に具体的な5ステップの方法論を学び、最後に Claude の説明を検証する方法を理解します。

## 導入: なぜコードが「読めない」のか

[Chapter 3-1「実践の準備」](../chapter-01_実践の準備/3-1-0_Chapter%201の地図.md) では、CourseHub のセットアップと Claude Code の設定整備を行いました。README のズレに気づき、コーディング規約と実コードの乖離を確認し、Claude Code が適切に動作する環境を整えました。

次のステップは、「このプロジェクトがどう動いているのか」を理解することです。

実務で既存プロジェクトに参画すると、最初のタスクが与えられる前に「まずコードを読んで全体像を把握してください」と言われることがあります。あるいは、バグ報告を受け取って「これを直してください」と言われたとき、まず関連するコードを読んで何が起きているのかを理解する必要があります。

しかし、「コードを読む」のは簡単ではありません。自分が書いたコードでも時間が経てば読みにくくなりますし、他人が書いた大規模なプロジェクトならなおさらです。コードを読もうとして、途中で頭がいっぱいになって「何がわからないのかもわからない」状態になった経験はないでしょうか。

実は、これは能力不足ではありません。**脳の仕組み** として当然のことです。

### 🧠 先輩エンジニアはこう考える

> 新しいプロジェクトに入ったとき、最初の1〜2日は「コードを読む日」にしている。いきなりタスクに着手したくなる気持ちはわかるけれど、全体像を把握せずにコードを書き始めると、既存の設計パターンから外れたコードを書いてしまったり、すでに用意されているユーティリティを知らずに車輪の再発明をしたりする。最初に時間を投資してコードを読むことが、結果的に最も速い。
>
> 自分も駆け出しの頃はコードを読むのが本当に遅くて、「自分には向いていないんじゃないか」と思ったこともある。でも、認知科学の視点で言えば、あれは単にパターンの蓄積が足りなかっただけだった。読めば読むほどチャンキングが効くようになって、いつの間にか速くなっていた。焦る必要はない。
>
> Claude Code が登場してからは、このプロセスが劇的に変わった。以前は1〜2日かかっていた全体像の把握が、数時間で終わるようになった。ただし、Claude の説明を鵜呑みにするのは危険だ。Claude は「もっともらしいが不正確な説明」をすることがある。Claude に聞いた後、自分でコードを開いて確認する。この「聞く→確認する」のサイクルが、AI 時代のコードリーディングの基本形だと思っている。

## コードが読めない理由を認知科学で理解する

ここからは、「なぜコードが読めないのか」を認知科学の視点から掘り下げます。原因がわかれば、対策も見えてきます。

### コードを読むとき、脳の中で何が起きているか

フェリエンヌ・ヘルマンス著『プログラマー脳』（水野貴明訳、秀和システム、2023年）は、認知科学の知見をプログラミングに応用した書籍です。この書籍によると、コードを読むとき、脳では3つの記憶システムが連携して働いています。

| 記憶システム | たとえ | 役割 |
|---|---|---|
| **長期記憶** | 本棚 | 過去に学んだ知識や経験の蓄積。大量に保管できるが、取り出すには手がかりが必要 |
| **短期記憶** | 手のひらのメモ | 目の前のコードから読み取った情報を一時的に保持。容量は2〜6個程度で、30秒ほどで消える |
| **ワーキングメモリ** | 作業机 | 短期記憶の情報と長期記憶から引き出した知識を組み合わせて「考える」場所。机の広さには限りがある |

コードを読むとき、あなたの脳は以下の3つを同時にこなしています。

1. **短期記憶** に目の前のコードの情報を載せる（変数名、メソッド名、条件分岐）
2. **長期記憶** から関連する知識を引き出す（「この構文は `foreach` ループだ」「`belongsTo` は Eloquent のリレーションだ」）
3. **ワーキングメモリ** の上で両者を組み合わせて意味を理解する（「このループはユーザー一覧を回して、各ユーザーの受講状況を確認している」）

一度に載せられる情報量を超えると、脳は処理しきれなくなります。これが **認知負荷が高い** 状態です。「頭がいっぱいで読めない」という感覚の正体は、ワーキングメモリのオーバーフローなのです。

### チャンキング: ベテランが速い理由

では、ベテランエンジニアはなぜ大量のコードを素早く読めるのでしょうか。その鍵は **チャンキング** という脳の仕組みにあります。チャンキングとは、複数の情報をひとつの「かたまり（チャンク）」として扱うことです。

たとえば、以下の PHP コードを見てみましょう。

```php
foreach ($users as $user) {
    if ($user->isActive()) {
        $activeUsers[] = $user;
    }
}
```

**ベテランの場合**: このコードは「アクティブなユーザーを抽出する処理」というひとつのチャンクとして処理されます。ワーキングメモリに載るのは1つだけです。

**ジュニアの場合**: `foreach` の構文、`$users` が何の配列か、`as $user` で何が取り出されるか、`isActive()` メソッドの意味、`$activeUsers[]` への代入……と個別に解釈しようとします。ワーキングメモリに5つ以上の情報が載り、すぐにいっぱいになります。

ベテランが速いのは「頭が良いから」ではなく、**長期記憶に大量のコードパターンが蓄積されていて、チャンキングが効くから** です。つまり、ジュニアがコードを読むのが遅いのは能力の問題ではなく、経験の蓄積量の問題です。

> 📝 チャンキングは意識的に鍛えられます。Step 5（増強）で詳しく説明しますが、コードを読んだ後に「自分でも書いてみる」ことで、パターンが長期記憶に蓄積され、チャンキングが効くようになっていきます。

### 認知負荷が高くなるもうひとつの原因

コードが読めないとき、原因は大きく2つに分かれます。

1. **プログラミングの知識不足**: 言語の構文、フレームワークの使い方、デザインパターンがわからない
2. **ビジネスドメインの知識不足**: そのコードが解決しようとしている業務の意味がわからない

ジュニアエンジニアは、この両方を同時に理解しようとしがちです。「コードの構文も追いつかないのに、業務ロジックの意味も考えなきゃ……」となると、認知負荷は一気に跳ね上がります。

**対策は「同時にやらず、片方ずつ攻略する」** ことです。

- プログラミングの知識が足りないなら、まず言語やフレームワークのドキュメントで構文を確認する
- ビジネスドメインの知識が足りないなら、仕様書を読む、チームメンバーに業務の流れを聞く

この「分けて攻略する」考え方は、次に紹介する5ステップの設計にも反映されています。

## 認知負荷を下げるコードリーディングの5ステップ

認知負荷が高くなる原因がわかったところで、具体的な対策を見ていきましょう。ポイントは **一度に処理する情報量を制限し、段階的に認知負荷を上げていく** ことです。

以下の5ステップは、『プログラマー脳』の認知科学的知見に基づき、コードリーディングの認知負荷を段階的に下げるために設計されています。

```
Step 1        Step 2        Step 3        Step 4        Step 5
探索          検索          転写          理解          増強
───────────   ───────────   ───────────   ───────────   ───────────
全体像を      狙いを        読む計画を    自分の言葉    使える知識
把握する      絞る          立てる        で説明する    に変える

「森を見る」  「木を選ぶ」  「道を決める」「歩いて確認」「地図を描く」

認知負荷: 低 ─────────────────────────────────────────────→ 高
```

各ステップで Claude Code をどう活用できるかも合わせて解説します。

### Step 1: 探索（全体像を把握する）

**目的**: いきなりコードを読み始めるのではなく、まず「地図」を手に入れる。

プロジェクトに最初に触れるとき、まず「何をするシステムなのか」を把握します。このステップでは **コードの中身は読みません**。ファイル名とディレクトリ名だけで全体像をつかむことに集中します。

- **ディレクトリ構成**: フォルダの名前と構造から、プロジェクトの規模感とアーキテクチャを推測する
- **技術スタック**: `composer.json`、`package.json`、設定ファイルから使われているフレームワークやライブラリを確認する
- **ルーティング**: `routes/web.php` や `routes/api.php` を読んで、このアプリケーションが外部に提供するエンドポイントを把握する
- **ドキュメント**: README、CLAUDE.md、コメントなど、前任チームが残した情報を確認する

> 💡 ルーティング定義はプロジェクトの「目次」です。どんな画面があり、どの Controller が処理を担当するかが一覧できます。まずここを読むことで、コードの全体像がつかめます。

> ⚠️ **よくある失敗**: 「全部読まなきゃ」と思って最初からコードに潜ってしまうこと。探索ステップでは、コードの中身は読まなくて大丈夫です。

#### Claude Code の活用: Sub-agents で広く探索する

Part 2 で学んだ Sub-agents（[2-3-5 Sub-agents](../../part-02_Claude%20Codeの基礎/chapter-03_機能を使いこなす/2-3-5_Sub-agents.md) で学習）は、探索ステップで特に威力を発揮します。

Claude Code に「プロジェクトの全体像を教えて」と聞くと、Claude は内部で Sub-agents を起動し、複数のファイルを並行して探索します。メインのコンテキストウィンドウを消費せずに大量のファイルを読めるため、プロジェクトの全体像把握に最適です。

```
> CourseHub のディレクトリ構成、モデルの一覧とリレーション、ルーティング定義を調べて、
> プロジェクトの全体像を教えて
```

このような質問を投げると、Claude は Sub-agents を使って `app/Models/`、`routes/web.php`、各 Controller を探索し、結果をまとめて報告してくれます。

> 💡 Claude Code のドキュメント「[Best practices](https://code.claude.com/docs/en/best-practices)」では、コードベースの探索は Sub-agents に委ね、メインのコンテキストを実装作業のために確保することが推奨されています。コードリーディングは多くのファイルを読む作業なので、Sub-agents の活用が特に効果的です。

### Step 2: 検索（狙いを絞る）

**目的**: 「自分が理解すべき箇所」にフォーカスする。

全体像をつかんだら、次は自分のタスクに関連するコードに狙いを絞ります。プロジェクト全体を読む必要はありません。

- **エントリポイントを探す**: 自分のタスクに関連するルーティングや Controller を特定する
- **関連ファイルをリストアップする**: Controller から呼ばれる Service、Model、Policy などを追う
- **構造を理解する**: モデルのリレーション（`belongsTo`、`hasMany`）、認可の仕組み（Middleware、Policy）、設計パターン（Service クラス、Event/Listener）を確認する

> 💡 モデルのリレーション定義を読むと、ER 図を見なくてもデータ構造が把握できます。`Course` が `hasMany(Chapter)` を持ち、`Chapter` が `hasMany(Lesson)` を持つなら、コース→チャプター→レッスンの階層構造だとわかります。

> ⚠️ **よくある失敗**: 関連コードが芋づる式に出てきて「あれもこれも読まなきゃ」となるパターン。判断基準は「今の自分のタスクに直接関係するか?」の一点です。関係ないものは「後で見る」リストに入れて、今は無視して構いません。

#### Claude Code の活用: 質問の仕方を工夫する

Claude Code にコードについて質問するとき、聞き方によって回答の質が大きく変わります。

**効果的な質問の例:**

```
> EnrollmentController の store メソッドで受講登録の処理を追いたい。
> ルーティング定義から Controller、Service、Model、Event の呼び出しまで、
> 処理の流れを順番に説明して
```

この質問は「どのコードを」「どの粒度で」「どういう順番で」説明してほしいかが明確です。Claude は具体的なファイルパスとコード行を引用しながら説明してくれます。

**曖昧な質問の例:**

```
> このプロジェクトのコードを説明して
```

この質問は範囲が広すぎて、Claude はどこから説明すべきか判断できません。結果として、表面的で長い説明が返ってきます。

Claude Code のドキュメント「[Best practices](https://code.claude.com/docs/en/best-practices)」では、コードベースについて質問する際は「シニアエンジニアに聞くように」質問することが推奨されています。つまり、具体的な対象を指定し、何を知りたいのかを明確にするということです。

```
> この Enrollment の処理で、なぜ直接 Eloquent を使わず
> Service クラスに切り出しているのか？
> Controller に書いた場合と比べてどんなメリットがあるか？
```

「なぜこうなっているか」という質問は、Claude からコードの設計意図に関する洞察を引き出します。

#### Claude Code の活用: `/btw` で脱線質問をする

コードを追跡している途中で「これは何だろう?」という疑問が湧くことがあります。しかし、その疑問を深掘りすると、元のフロー追跡から脱線してしまいます。

Part 2 で学んだ `/btw`（[2-2-2 コンテキストとセッション管理](../../part-02_Claude%20Codeの基礎/chapter-02_基本を理解する/2-2-2_コンテキストとセッション管理.md) で学習）は、こうした脱線質問に最適です。`/btw` で質問すると、Claude は現在の作業コンテキストに影響を与えずに回答します。

```
> /btw Laravel の Event/Listener パターンは Observer パターンとどう違うの？
```

フロー追跡の本筋を中断せずに疑問を解消できます。認知科学の言葉で言えば、「今のワーキングメモリの内容を壊さずに、長期記憶を補充する」操作です。

### Step 3: 転写（読む計画を立てる）

**目的**: 何をどの順番で読むかを整理し、ワーキングメモリの負荷を外部に逃がす。

「転写」とは、頭の中にある情報を外部に書き出すことです。『プログラマー脳』では、ワーキングメモリの負荷を軽減するために「考えていることを書き出す」ことが推奨されています。紙やメモアプリに書き出すだけで、作業机が広くなったような効果が得られます。

- **読むべきファイル・クラス・メソッドをリストアップする**
- **処理の流れに沿った順番で並べる**
- **「ここまで理解できたらOK」というゴールを設定する**

例として、「受講登録の処理を理解する」場合のメモを示します。

```
【受講登録の処理を理解する】

読む順番:
1. routes/web.php の受講登録ルート（エントリポイント）
2. EnrollmentController の store メソッド（リクエスト処理）
3. EnrollmentService の enroll メソッド（ビジネスロジック）
4. Enrollment モデルのリレーション（データ構造）

ゴール: 受講登録ボタンを押してから DB に保存されるまでの流れを説明できる状態
```

> ⚠️ **よくある失敗**: 計画を立てずに目についたコードを片っ端から読んでしまうこと。途中で「あれ、今何を読んでいたんだっけ?」と見失うのは、ワーキングメモリがオーバーフローしたサインです。

#### Claude Code の活用: 読む順番を Claude に整理してもらう

検索ステップで関連ファイルを見つけた後、Claude にその読む順番を整理してもらうことができます。

```
> 受講登録の処理を理解したい。関連するファイルを処理の流れ順にリストアップして、
> それぞれ何を読めばいいか（メソッド名、行番号）を教えて
```

Claude が出力したリストを手元にメモしておけば、コードを読み進める際のガイドマップになります。

#### Claude Code の活用: コンテキスト管理を意識する

コードリーディングでは、多くのファイルを読み、Claude と何往復もやり取りします。コンテキストウィンドウが埋まると、Claude の回答品質が低下します。

- **`/context`** で現在のコンテキスト使用量を確認する
- 一つのテーマの調査が終わったら **`/compact`** でコンテキストを整理する
- 長い探索は Sub-agents に委ね、メインのコンテキストを節約する

`/context` と `/compact` は Part 2（[2-2-2 コンテキストとセッション管理](../../part-02_Claude%20Codeの基礎/chapter-02_基本を理解する/2-2-2_コンテキストとセッション管理.md) で学習）で学んだコンテキスト管理の実践です。コードリーディングは特にコンテキストを消費しやすい作業なので、計画的な管理が重要です。

### Step 4: 理解（自分の言葉で説明する）

**目的**: コードを「読んだ」で終わらせず、「理解した」状態にする。

ここが最も重要なステップです。コードを読んだ後、**そのコードの処理内容を自分の言葉で要約** してみてください。

「このメソッドは○○を受け取って、○○をチェックして、○○を返す」のように、一文で説明できるか試してみます。説明できなければ、それはまだ理解できていないサインです。

たとえば、以下のようなコードを読んだとします。

```php
// app/Services/EnrollmentService.php
public function enroll(User $user, Course $course): Enrollment
{
    $existing = $course->enrollments()->where('user_id', $user->id)->first();
    if ($existing) {
        abort(409, 'すでに受講登録されています');
    }

    $enrollment = Enrollment::create([
        'user_id' => $user->id,
        'course_id' => $course->id,
        'enrolled_at' => now(),
    ]);

    event(new UserEnrolled($enrollment));

    return $enrollment;
}
```

自分の言葉で要約してみます。

```
enroll メソッド:
- ユーザーとコースを受け取る
- そのユーザーが既に受講登録済みでないかチェック（済みなら 409 エラー）
- Enrollment レコードを作成して DB に保存
- UserEnrolled イベントを発火（通知やログの処理がありそう）
- 作成した Enrollment を返す
```

言語化してみると、「`UserEnrolled` イベントの後に何が起きるんだろう?」のように、理解できていない部分が浮かび上がります。それが次に調べるべきポイントです。

> 🔑 「なんとなくわかった」は危険です。自分の言葉で説明できない部分こそ、理解が足りていない箇所です。

#### Claude Code の活用: Claude の説明を自分の言葉で言い換える

Claude にコードの説明を聞いた後、**そのまま受け入れるのではなく、自分の言葉で言い換えてみる** ステップを挟みましょう。

```
> （Claude の説明を読んだ後）
> 自分の理解を確認したい。この enroll メソッドは、
> 重複チェック → レコード作成 → イベント発火 の3段階で処理している、
> という理解で合ってる？何か見落としている処理はある？
```

Claude に自分の理解を検証してもらうことで、「読んだだけ」で終わらず「理解した」状態に近づけます。Anthropic の学習パターン研究では、AI にコード生成を丸投げした開発者の理解度は 40% 以下だったのに対し、概念的な質問をしながら自分でコードを理解した開発者は 65% 以上だったと報告されています（Anthropic「[How AI assistance impacts the formation of coding skills](https://www.anthropic.com/research/AI-assistance-coding-skills)」、2026年1月）。

#### Claude Code の活用: MCP でデータベースを確認する

コードだけでは「実際にどんなデータが入っているか」がわかりません。Part 2 で学んだ MCP（[2-3-4 MCP](../../part-02_Claude%20Codeの基礎/chapter-03_機能を使いこなす/2-3-4_MCP.md) で学習）を使えば、Claude Code からデータベースに直接クエリを実行できます。

「受講登録するとどんなレコードが作られるか」をコードで追った後、実際の `enrollments` テーブルの中身を確認すれば、コードの理解が正しいかを裏付けられます。コードの理解を「自分の言葉で説明する」だけでなく、**実データで検証する** ことで、理解の精度がさらに上がります。

> 💡 CourseHub は Sail 環境で MySQL を使っています。phpMyAdmin や `sail mysql` コマンドでもデータベースを確認できますが、MCP を使えば Claude Code との会話の中でシームレスにクエリを実行できる点がメリットです。

### Step 5: 増強（使える知識に変える）

**目的**: 理解した内容を長期記憶に定着させ、チャンキングの材料にする。

理解しただけで終わると、時間が経つにつれて記憶は薄れていきます。読んだコードに対して **何かひとつアウトプットする** ことで、長期記憶への定着が促進されます。

具体的なアウトプットの例:

- **コードにコメントを追加する**: 処理の意図や注意点を、未来の自分やチームメンバーのために書き残す
- **ドキュメントを更新する**: README や CLAUDE.md に、わかったことを追記する
- **テストを読む・書く**: 既存テストを読んでコードの期待動作を確認する。テストがなければ追加する
- **小さな変更を加えてみる**: 学んだパターンを使って、小さな機能追加やバリデーション追加に取り組む

> 💡 Step 5 を繰り返すことで、長期記憶にコードパターンが蓄積されます。チャンキング（複数の情報を「かたまり」として扱う力）は、この蓄積によって育ちます。増強を重ねるほどチャンクが増え、いつの間にかベテランの読み方に近づいていきます。

## Claude の説明を検証する

5ステップの中で Claude Code を活用する方法を紹介しましたが、Claude の説明は常に正確とは限りません。3-1-1 で学んだ「見極める力」をコードリーディングの段階から発揮する必要があります。

Claude の説明に含まれやすい3つの問題パターンを知っておくと、検証のポイントが明確になります。

### パターン 1: 省略

Claude は説明を簡潔にまとめようとするため、重要な処理を省略することがあります。

たとえば、「レッスン完了の処理を教えて」と聞いたときに、Claude が `LessonProgress` の更新までは説明するが、その後に実行される `checkCourseCompletion()` メソッド（全レッスン完了時に Event を発火する処理）に触れないことがあります。

**検証方法**: Claude の説明に「その後の処理」がないかを疑う。Controller メソッドの最後まで自分の目で読み、Claude が省略した処理がないか確認する。

### パターン 2: 誤り

Claude はコードの動作を誤って解釈することがあります。特に、条件分岐が複雑な箇所や、暗黙的な型変換が絡む箇所で起きやすいです。

たとえば、「この Controller はどんなバリデーションをしているか」という質問に対して、Claude が条件分岐を正しく読み取れず、「全てのケースでバリデーションされている」と説明するが、実際には特定の条件でバリデーションがスキップされている、ということがあり得ます。

**検証方法**: Claude の説明を読んだ後、該当するコードを自分で開いて、条件分岐のロジックを1行ずつ追う。Step 4（理解）で「自分の言葉で説明する」を実践していれば、Claude の説明との食い違いに気づきやすくなります。

### パターン 3: 古い情報

Claude のトレーニングデータには時点の制約があります。「Laravel ではこう書くのが一般的です」という説明が、プロジェクトが使っている Laravel 10 のバージョンでは異なる場合があります。

また、プロジェクト固有の問題として、「以前はこう実装されていたが、途中で変更された」コードに対して、Claude が変更前の実装を前提に説明することがあります。

**検証方法**: Claude の説明にバージョン依存の内容が含まれる場合は、公式ドキュメントで確認する。プロジェクト固有の実装は、`git log` や `git blame` でコードの変更履歴を確認する。

> 🔑 Claude の説明を検証する最も確実な方法は「自分でコードを読む」ことです。Claude に聞く → 説明を受け取る → 自分の言葉で言い換える → 該当コードを自分で確認する。この4ステップのサイクルを繰り返すことで、Claude の回答を効率的に活用しながら、確実に理解を深められます。

## コードリーディングは「参画時だけ」ではない

ここまで、プロジェクト参画時の全体像把握としてコードリーディングの方法論を説明しました。しかし、コードリーディングが必要になるのは参画時だけではありません。

**タスクに着手する前にも、関連するコードを読む**。これは、以降の Chapter で繰り返し実践するプロセスです。

- **バグ修正（[Chapter 3-3「バグを修正する」](../chapter-03_バグを修正する/3-3-0_Chapter%203の地図.md)）**: バグ報告を受け取ったら、まず関連するコード（Controller、Model、Policy、テスト）を読んで、現状の実装を理解する
- **機能開発（[Chapter 3-4「機能を開発する」](../chapter-04_機能を開発する/3-4-0_Chapter%204の地図.md)）**: 新しい機能を追加するとき、まず既存の設計パターンとコーディング規約を確認し、新しいコードがプロジェクト全体と調和するようにする
- **リファクタリング（[Chapter 3-5「コードを改善する」](../chapter-05_コードを改善する/3-5-0_Chapter%205の地図.md)）**: コードを改善するとき、改善対象のコードだけでなく、そのコードを使っている他のコードも読んで影響範囲を把握する

つまり、コードリーディングは「一回やって終わり」ではなく、実務タスクの最初のステップとして毎回行うものです。今この Chapter で学ぶ方法論は、Part 3 全体を通じて繰り返し使います。

```
                       ┌─────────────────────────────────────────────┐
  参画時:               │  各タスクの着手前:                            │
  プロジェクト全体を     │  タスクに関連するコードを読み、               │
  理解する              │  既存の設計パターンと規約を把握してから着手する │
                       └─────────────────────────────────────────────┘
                            ↑           ↑           ↑           ↑
  Chapter 3-2           Chapter 3-3  Chapter 3-4  Chapter 3-5  Chapter 3-6
  (今ここ)               バグ修正     機能開発     リファクタ    チーム共有
```

そして、各タスクでコードを読むたびに Step 5（増強）が発動します。バグを修正すれば「このパターンのバグはこう調査する」という知識が蓄積され、機能を開発すれば「このプロジェクトの設計パターン」が体に染み込みます。Part 3 を通じて CourseHub のコードを繰り返し読むこと自体が、チャンキングの材料を蓄積するプロセスなのです。

## ✨ まとめ

- コードリーディングが難しいのは能力不足ではなく、**長期記憶に蓄積されたパターンの量の差** である。チャンキングが効かないジュニアは認知負荷が高い状態でコードを読んでいる
- 認知負荷を下げるには、一度に処理する情報量を制限し、**5ステップ（探索→検索→転写→理解→増強）** で段階的に進める
- Claude Code は認知負荷を下げるパートナーとして各ステップで活用できる。ただし Claude の説明には**省略・誤り・古い情報** が含まれることがある。必ず自分でコードを読んで検証する「聞く→言い換える→確認する」のサイクルを回す
- コードリーディングは参画時の全体像把握だけでなく、**各タスク着手前にも行う**。Part 3 全体を通じて繰り返し実践するプロセス
