# 3-2-2 知らない技術への向き合い方

## 🎯 このセクションで学ぶこと

- 実務プロジェクトで未知の技術やパターンに出会ったときの対処フロー（理解→評価→判断→説明できる）を理解する
- 「今どこまで理解すべきか」の判断基準を持つ
- 知らないことへの心構えと、AI 時代の学び方の注意点を理解する
- 実務で出会う主要な Laravel パターンの全体像を把握する

まず「知らない技術だらけの環境は普通である」という前提を確認し、次に具体的な対処フローを学び、理解の深さの判断基準を身につけ、最後に AI を使った学び方の注意点と実務で出会う Laravel パターンの地図を確認します。

---

## 導入: 知らないことだらけは「普通」

3-2-1 ではコードリーディングの方法論を学びました。3ステップ（全体像→構造→フロー）に沿ってコードを読み進めていくと、必ずある瞬間が訪れます。

「これ、見たことない」

`app/Services/` というディレクトリ。`Event` と `Listener` という見慣れないクラス。Controller の中で呼び出されている `authorize()` メソッド。COACHTECH で学んだ MVC パターンの延長線上にあるものもあれば、まったく初めて見るものもあります。

この瞬間にどう反応するかが、その後の成長を大きく左右します。

「自分にはまだ早い」と感じて手が止まる人もいれば、「とりあえず動いているからそのままにしよう」と表面的な理解で先に進む人もいます。どちらも、実務ではうまくいきません。

実は、知らない技術だらけの環境に放り込まれるのは、ジュニアだけの経験ではありません。匿名キャリア SNS の Blind 社が 2018 年にテック企業の従業員1万人超を対象に実施した調査（「[58 Percent of Tech Workers Feel Like Impostors](https://www.teamblind.com/blog/58-percent-of-tech-workers-feel-like-impostors/)」）によると、開発者の **58%** がインポスター症候群（自分の能力が不十分だと感じる心理）を経験しています。これはシニアエンジニアも含めた数字です。

Redux の共同作成者で React コアチームのメンバーである Dan Abramov 氏は、「[Things I Don't Know as of 2018](https://overreacted.io/things-i-dont-know-as-of-2018/)」という記事で、自分が知らない技術の長いリストを公開しています。Unix コマンド、Docker、ネットワーキング、TypeScript の高度な型システムなど、世界トップクラスのエンジニアでさえ「知らないこと」は膨大にあるのです。

> 🔑 知らないこと自体は問題ではありません。問題になるのは、「知らない技術にどう向き合うか」のプロセスを持っていないことです。このセクションでは、そのプロセスを身につけます。

### 🧠 先輩エンジニアはこう考える

> 新しいプロジェクトに入るたびに「知らないこと」に出会う。何年やっていてもそうだ。最初のプロジェクトでは Service クラスを知らなかった。次のプロジェクトでは Event Sourcing を初めて見た。その次では GraphQL だった。
>
> 最初の頃は「こんなことも知らないのか」と思われるのが怖くて、一人で調べて時間を溶かしていた。でもあるとき先輩に「知らないことは聞けばいい。知ったふりをしてバグを埋め込むほうが怖い」と言われて、気が楽になった。
>
> 他のエンジニアが「超能力」に見えるのは、単にそのコードベースに慣れているだけということが多い。プロジェクト固有の知識は、そのプロジェクトに関わらなければ持っていなくて当然だ。

## 知らない技術への対処フロー

知らない技術に出会ったとき、闇雲に調べ始めると時間を浪費します。以下の4ステップで段階的に理解を深めていきます。

```
1. 理解する         2. 評価する          3. 判断する          4. 説明できる
───────────────   ───────────────    ───────────────    ───────────────
「これは何か？」   「なぜここで       「今の自分に       「これは○○で、
を Claude Code    使われているか？」  必要な理解度は？」  ○○のために
に質問する        を考える            を見極める         使われている」
                                                        と言える状態

Claude に聞く     ・コード上の        ・全体像の把握に    ・人に説明
 ↓               メリットを           必要なら深掘り      できるレベル
概要を掴む        確認する            ・今は概要で         が目標
 ↓               ・使わなかった       十分なら先に進む
公式ドキュメント   場合を想像する
を確認する
```

### ステップ 1: 理解する（「これは何か？」）

まず Claude Code に「これは何か」を質問します。

```
> app/Services/EnrollmentService.php を読んで。
> このクラスは何をしていて、なぜ Controller ではなくここに処理を書いているのか教えて
```

Claude は Service クラスの役割と、Controller から分離する理由（テストのしやすさ、再利用性、Controller の肥大化防止など）を説明してくれます。

ここで重要なのは、Claude の説明だけで満足しないことです。3-2-1 で学んだように、Claude の説明には省略や誤りが含まれることがあります。概要を掴んだら、公式ドキュメントやコミュニティのリソースで裏付けを取ります。

```
> /btw Laravel の Service クラスパターンについて、公式ドキュメントや
> コミュニティでの位置づけを教えて
```

`/btw` を使えば、コードリーディングの本筋を中断せずに補足情報を得られます。

> 💡 Service クラスのように Laravel 公式の機能ではないパターンもあります。その場合、公式ドキュメントには載っていませんが、Laravel コミュニティでは広く採用されています。Claude に「これは公式機能か、それともコミュニティのプラクティスか」を聞くと、技術の位置づけがわかりやすくなります。

### ステップ 2: 評価する（「なぜここで使われているか？」）

概要を掴んだら、「なぜこのプロジェクトでこのパターンが使われているか」を考えます。

たとえば、CourseHub の `EnrollmentService` の場合、受講登録の処理には「重複チェック」「コース公開状態の確認」「Enrollment レコードの作成」「全レッスンの LessonProgress レコードの一括作成」と複数のステップがあります。これを全て Controller に書くと、Controller が膨れ上がります。

評価のポイントは2つです。

- **コード上のメリットを確認する**: このパターンがあることで、コードの可読性やテストのしやすさがどう変わるか
- **使わなかった場合を想像する**: もしこのパターンを使わず、Controller に全て書いていたらどうなるか

「使わなかった場合」を想像すると、そのパターンの存在意義が腑に落ちます。CourseHub の `CoachCourseController@store` は実際に200行超の Fat Controller になっています。Service クラスに切り出した `EnrollmentService` と比較すると、「なぜ分離するのか」が具体的にわかります。

### ステップ 3: 判断する（「今どこまで理解すべきか？」）

「今の自分にどの程度の理解が必要か」を見極めます。このステップが最も重要であり、次のセクション「理解の深さを見極める」で詳しく解説します。

プロジェクト参画時の全体像把握の段階では、「Service クラスはビジネスロジックを Controller から分離するパターン」「Event/Listener はイベント駆動で処理を分離するパターン」という概要が理解できれば十分です。Service クラスの設計原則や Event のキューイングの仕組みまで深掘りする必要はありません。

一方、その Service クラスに関連するタスク（たとえば受講登録のバグ修正）を担当する場合は、`EnrollmentService` のコードを1行ずつ理解する必要があります。

> 📝 「今は概要だけ把握すれば十分」と判断することは、手を抜くことではありません。限られた時間の中で優先順位をつけて理解を進めることは、実務で必要なスキルです。必要になったタイミングで深掘りすればよいのです。

### ステップ 4: 説明できる（最終ゴール）

最終的なゴールは、「これは何で、なぜここで使われているか」を自分の言葉で説明できる状態です。

- 「`EnrollmentService` は、受講登録のビジネスロジックを Controller から分離した Service クラスで、受講済みチェック・コース公開確認・Enrollment 作成・LessonProgress 一括作成をまとめて処理している」
- 「`CourseCompleted` Event は全レッスン完了時に発火して、`UpdateEnrollmentStatus` Listener が受講ステータスを `completed` に更新している。将来的に完了証明書の発行やコーチへの通知などを追加しやすい設計」

このレベルで説明できれば、コードレビューで「このコードはなぜこうなっているのか」と聞かれても答えられます。[3-1-1 Claude Code で実務タスクを遂行する考え方](../chapter-01_実践の準備/3-1-1_Claude%20Codeで実務タスクを遂行する考え方.md) で学んだ「説明責任」は、AI が生成したコードだけでなく、既存コードの理解にも求められます。

> 🔑 4ステップ全てを一度に完了する必要はありません。ステップ 1（概要の理解）だけで先に進み、必要になったときにステップ 2〜4 に戻ることもあります。大事なのは、「知らない技術に出会ったとき何をすべきか」のプロセスを持っていることです。

## 理解の深さを見極める

対処フローのステップ 3「判断する」を掘り下げます。知らない技術に出会うたびに全てを深掘りしていたら、時間がいくらあっても足りません。逆に、全てを「なんとなく」で流していたら、タスクに着手したときに行き詰まります。

Itamar Turner-Trauring 氏は記事「[On learning new technologies: why breadth beats depth](https://codewithoutrules.com/2019/03/29/learn-new-technologies/)」で「5分ルール」を提唱しています。「新しい技術について5分学ぶだけで、5日かけて得られる理解の80%が手に入ることが多い」という考え方です。

この「5分で80%」は直感に反するかもしれませんが、理由は3つあります。

1. **陳腐化**: 技術の進化は速く、深く学んだ知識がすぐに古くなることがある
2. **関連性**: 学んだ技術が自分の担当範囲に関係しない場合、深い知識は活用されない
3. **API の変化**: ライブラリやフレームワークのバージョンアップで細かい使い方が変わる

ただし、これは「全てを浅く学べばよい」という意味ではありません。深掘りすべき場面と概要で十分な場面を見極める判断基準が必要です。

### 深掘りすべきとき

| 状況 | 理由 |
|---|---|
| 担当タスクに直結する | そのコードを修正・拡張するには、動作を正確に理解する必要がある |
| コアの技術スタックである | 毎日使う技術は、深い理解がないと繰り返しつまずく |
| セキュリティやパフォーマンスの重要パスにある | 浅い理解でコードを書くと、脆弱性やパフォーマンス問題を埋め込むリスクがある |
| 概要だけでは繰り返し混乱する | 何度も「これ何だっけ？」と戻ってしまうなら、一度深掘りした方が効率的 |

### 概要で十分なとき

| 状況 | 理由 |
|---|---|
| 今の担当範囲外である | 他のチームメンバーの責務。必要になったときに深掘りすればよい |
| 隣接領域で自分の責務ではない | インフラ設定、CI/CD パイプラインなど、概要を知っていれば十分 |
| 必要になったときに公式ドキュメントで補える | Laravel の機能であれば、公式ドキュメントが充実している |

### 実務での判断の流れ

```
知らない技術に出会った
        │
        ▼
  今の担当タスクに
  関係するか？ ──── No ──→ ステップ1（概要の理解）で十分。
        │                  メモだけ残して先に進む
       Yes
        │
        ▼
  そのコードを自分が
  修正・拡張するか？ ── No ──→ ステップ2（評価）まで。
        │                     なぜ使われているかを把握する
       Yes
        │
        ▼
  ステップ4（説明できる）まで深掘りする。
  コードを1行ずつ読み、動作を検証する
```

> 💡 Stuart Dreyfus と Hubert Dreyfus が 1980 年に提唱した Dreyfus モデル（スキル習得の5段階モデル。原論文: 「[A Five-Stage Model of the Mental Activities Involved in Directed Skill Acquisition](https://www.researchgate.net/publication/235125013_A_Five-Stage_Model_of_the_Mental_Activities_Involved_in_Directed_Skill_Acquisition)」）によると、スキルは **人単位ではなくスキル単位** で段階が異なります。あなたが Laravel の MVC パターンには詳しくても、Event/Listener パターンでは初心者であることは当然のことです。シニアエンジニアでも、特定の技術では初心者です。スキルごとに段階が違うことを受け入れると、「知らない」ことへのプレッシャーが軽くなります。

## AI 時代の「知らない技術」との付き合い方

Claude Code は「知らない技術」への対処を劇的に加速してくれます。以前なら公式ドキュメントを読み、ブログ記事を探し、Stack Overflow で類似の質問を検索していた作業が、Claude Code への一言の質問で始められます。

しかし、この便利さには落とし穴があります。

### 「理解せずに使う」リスク

Anthropic が 2026 年 1 月に発表した研究「[How AI assistance impacts the formation of coding skills](https://www.anthropic.com/research/AI-assistance-coding-skills)」は、52名のジュニア開発者を対象にした実験を行いました。全員が未経験の Python ライブラリを使ってコードを書き、その後に理解度テストを受けました。

結果は明確です。

- **AI にコード生成を丸投げした開発者**: 理解度 **40% 以下**
- **AI に概念的な質問をしながら自分でコードを書いた開発者**: 理解度 **65% 以上**

同じ AI ツールを使っていても、使い方によって理解度に大きな差が生まれます。

Addy Osmani 氏は記事「[The 70% Problem: Hard truths about AI-assisted coding](https://addyo.substack.com/p/the-70-problem-hard-truths-about)」で、こう指摘しています。「コードが理解なしに『出現』すると、依存が生まれる。AI に戻って問題を修正し続ける必要が生じ、自分で対処する専門性を育てられなくなる」

### 対処フローが「説明できる」で終わる理由

対処フローのステップ 4 が「説明できる」であるのは、理解の最終確認だからです。

「説明できる」とは、Claude の説明をそのまま繰り返すことではありません。自分の言葉で、自分の理解に基づいて説明できることです。この違いは大きく、Osmani 氏は「**説明できないコードは受け入れるな**」をルールとして提唱しています（[Avoiding Skill Atrophy in the Age of AI](https://addyo.substack.com/p/avoiding-skill-atrophy-in-the-age)）。

このルールは、AI が生成した新しいコードだけでなく、既存コードの理解にも当てはまります。「Claude がこう言っていたから」ではなく「自分で確認してこう理解した」と言えることが、AI 時代のエンジニアに求められる力です。

### 効果的な AI の使い方

Anthropic の研究で高い理解度を示した開発者たちには、共通するパターンがありました。

| 効果的な使い方 | 避けるべき使い方 |
|---|---|
| 概念を質問する（「これは何のためのパターン？」） | コード生成を丸投げする（「これを書いて」） |
| 生成されたコードを読んで理解する | 生成されたコードをそのまま使う |
| 「なぜこう書くのか」を追加で聞く | 動いたから OK とする |
| 自分でコードを書き、詰まったときに聞く | 最初から AI に書かせる |

知らない技術に出会ったときも同じです。Claude Code に「これは何か」「なぜこのパターンか」を聞き、説明を受けた後に自分でコードを読んで検証する。このサイクルが、理解を伴った学びにつながります。

> 📝 AI を使った学びで最も効果が高いのは、「AI に質問しながら自分で考える」パターンです。AI にコードを生成させるのではなく、AI を「いつでも質問できる先輩エンジニア」として活用することで、理解の深さが大きく変わります。

## 実務で出会う Laravel パターンの地図

最後に、実務の Laravel プロジェクトでよく見るパターンの全体像を「地図」として示します。全てを今すぐ理解する必要はありません。目的は、「こういうものがある」と知っておくことで、実際に遭遇したときの心理的負担を下げることです。

### よく出会うパターン

以下は、実務の Laravel プロジェクトで特によく使われるパターンを、出会いやすい順に整理したものです。上の2つ（Form Request、Policy）は COACHTECH で学んだパターンです。それより下は COACHTECH では扱っていませんが、実務プロジェクトでは頻繁に登場します。

| パターン | 何をするか | どんな課題を解決するか | こんな信号が出たら使う |
|---|---|---|---|
| **Form Request** | バリデーションロジックを専用クラスに分離する | Controller にバリデーションルールが散在して肥大化する | Controller に `$request->validate([...])` が何十行も書かれている |
| **Policy** | モデルに対する認可ロジックを専用クラスにまとめる | 「このユーザーはこの操作をしていいのか」の判定が Controller に散在する | `if ($user->role === 'admin')` のような条件分岐が複数の Controller にコピーされている |
| **Service クラス** | ビジネスロジックを Controller から分離する | Controller が肥大化し、テストしにくくなる | Controller のメソッドが100行を超え、複数の処理ステップを持っている |
| **Event / Listener** | 「何が起きたか」と「その後何をするか」を分離する | 副作用（メール送信、ログ記録、ステータス更新）が本処理のコードに混在する | メイン処理の後に「ついでに」行う処理が増えてきた |
| **Observer** | モデルのライフサイクルイベント（作成時、更新時、削除時など）にフックする | モデルの変更に伴う処理（スラグ生成、キャッシュクリア等）が散在する | 複数の場所で「○○モデルが保存されたら△△する」というコードが重複している |
| **Jobs / Queues** | 時間のかかる処理をバックグラウンドで非同期実行する | メール送信や CSV 出力がリクエストをブロックし、画面の応答が遅くなる | ユーザーが操作した後、数秒以上待たされる処理がある |

> 📝 Form Request と Policy は COACHTECH の教材で基本を学んでいます。実務ではこれらに加えて、Service クラス以降のパターンが当たり前のように使われています。CourseHub でも **Service クラス**（`EnrollmentService`）と **Event / Listener**（`CourseCompleted` Event + `UpdateEnrollmentStatus` Listener）が使われており、3-2-3 のハンズオンで実際にこれらのコードを読みます。

### パターンの関係性

これらのパターンは独立して存在するのではなく、組み合わせて使われることが多いです。典型的な Fat Controller のリファクタリングパスは、次のようになります。

```
Fat Controller（全ての処理が1つのメソッドに詰まっている）
    │
    ├── バリデーション    → Form Request に抽出
    ├── 認可チェック      → Policy に抽出
    ├── ビジネスロジック  → Service クラスに抽出
    ├── 副作用           → Event / Listener に分離
    └── 重い処理         → Jobs / Queues で非同期化
```

実際、CourseHub の [Chapter 3-5「コードを改善する」](../chapter-05_コードを改善する/3-5-0_Chapter%205の地図.md) では、Fat Controller から Form Request を抽出する実践を行います。今はこの全体像を頭の片隅に置いておくだけで十分です。

### 🧠 先輩エンジニアはこう考える

> この地図を全部暗記する必要はない。大事なのは「こういうパターンがあるらしい」と知っていることだ。
>
> 自分が Service クラスを初めて見たとき、「なぜ Controller に書かないんだろう」と疑問だった。でも Controller が200行を超えるコードを見た後で Service クラスの存在意義がわかった。Event/Listener も同じで、「なぜ直接呼び出さないんだろう」と思ったけれど、「後から処理を追加しやすくするため」という設計意図を知ったら納得できた。
>
> パターンの理解は「痛みを感じたとき」に最も深まる。Controller が長すぎて読めない、テストが書きにくい、処理を追加するたびに既存コードを壊す。こうした痛みを経験して初めて、「だからこのパターンがあるのか」と腑に落ちる。Claude Code があれば、「この痛み、どう解消する？」と聞いて即座にパターンを学べる。

## ✨ まとめ

- 知らない技術だらけの環境に入るのは、ジュニアもシニアも同じ。知らないこと自体は問題ではなく、「知らない技術にどう向き合うか」のプロセスを持つことが重要
- 対処フローは「理解→評価→判断→説明できる」の4ステップ。Claude Code に聞いて概要を掴み、公式ドキュメントで裏付け、プロジェクトでの使われ方を評価し、必要な深さまで理解する
- 理解の深さは「今の自分のタスクに関係するか」で判断する。全てを深掘りする必要はなく、概要で十分な場面も多い
- AI はこの対処を劇的に加速するが、「理解せずに使う」リスクもある。「説明できないコードは受け入れるな」を原則とする
- 実務でよく見る Laravel パターンの地図を持っておくと、出会ったときの心理的負担が下がる。全てを今理解する必要はない
