# 2-1-4 CLAUDE.md とプロジェクト作成

## 🎯 このセクションで学ぶこと

- CLAUDE.md の役割と、プロジェクトに合わせた指示の書き方を理解する
- `/init` コマンドでプロジェクトの土台を作る
- 自動メモリの仕組みを把握する
- 生成されたコンテンツを検証する「見極めチェック」の型を身につける

まず CLAUDE.md の役割と書き方を理解し、次に `/init` でプロジェクトの土台を作り、最後に生成物を検証する見極めチェックの型を実践します。

## 導入: 毎回同じことを説明していませんか

Claude Code を使っていて、こんな場面を想像してみてください。

新しいセッションを開始するたびに「このプロジェクトは Laravel で、認証には Fortify を使っていて、テストは PHPUnit で...」と毎回プロジェクトの背景を説明する。前のセッションで「テストは Pest ではなく PHPUnit を使ってください」と伝えたのに、次のセッションではまた Pest でテストを書いてくる。

Claude Code のセッションは独立しています。前のセッションで伝えた情報は、次のセッションには引き継がれません。つまり、プロジェクト固有の情報やルールを毎回伝え直す必要があります。これは非効率なだけでなく、伝え忘れによる品質のブレにもつながります。

この問題を解決するのが **CLAUDE.md** です。

### 🧠 先輩エンジニアはこう考える

> CLAUDE.md は、新しくチームに入ったメンバーに渡す「プロジェクトの歩き方ガイド」に近い。技術スタック、コーディング規約、テストの方針、やってはいけないことなど、「プロジェクトに参加したら最初に知っておくべきこと」を書いておく場所だ。自分がプロジェクトに途中参加したときに「これを読んでおいて」と渡されるドキュメントを想像すると、何を書けばいいかが見えてくる。

## CLAUDE.md とは

CLAUDE.md は、Claude Code に対するプロジェクト固有の指示書です。Claude Code はセッション開始時にこのファイルを自動で読み込み、その内容をすべてのやり取りの前提として使います。

CLAUDE.md に書かれた内容は、Claude Code の振る舞いに直接影響します。

- 「テストは PHPUnit で書く」と書けば、PHPUnit でテストを生成するようになります
- 「コミットメッセージは日本語で書く」と書けば、日本語のコミットメッセージを作成します
- 「`app/Models` 配下のファイルは変更しないで」と書けば、そのディレクトリのファイルを変更しなくなります

### CLAUDE.md の配置場所

CLAUDE.md は複数の場所に配置でき、それぞれスコープ（影響範囲）が異なります。

| 配置場所 | スコープ | 用途 |
|---|---|---|
| `~/.claude/CLAUDE.md` | ユーザー全体 | すべてのプロジェクトに共通の設定（言語設定、個人の好みなど） |
| `./CLAUDE.md` または `./.claude/CLAUDE.md` | プロジェクト | プロジェクト固有のルール（技術スタック、規約など） |
| サブディレクトリの `CLAUDE.md` | ディレクトリ限定 | 特定のディレクトリに固有のルール |

プロジェクトルートの CLAUDE.md が最も重要です。ここにプロジェクト全体に適用するルールを記載します。

> 📝 複数の CLAUDE.md が存在する場合、すべてが読み込まれます。ユーザーレベルの設定とプロジェクトレベルの設定が矛盾する場合は、プロジェクトレベルが優先されます。

### CLAUDE.md に書くべきこと

効果的な CLAUDE.md を書くためのガイドラインです。

**書くべきこと:**

- プロジェクトの技術スタックとバージョン（例: Laravel 11、PHP 8.3、MySQL 8.0）
- コーディング規約やスタイルガイド（例: PSR-12 準拠、Strict Types 使用）
- テストの方針（例: PHPUnit 使用、Feature テスト優先）
- ディレクトリ構成の説明（特殊な構成の場合）
- よく使うコマンド（例: `php artisan test`、`./vendor/bin/pint`）
- やってはいけないこと（例: `main` ブランチに直接コミットしない）

**書くべきでないこと:**

- 一般的な言語やフレームワークの説明（Claude Code はすでに知っています）
- 頻繁に変わる情報（チケット番号、進行中のタスクなど）
- 機密情報（API キー、パスワード、内部 URL など）

> 🔑 CLAUDE.md は **簡潔に保つ** ことが重要です。公式ドキュメントでは **1ファイルあたり 200 行以内** を目安としています。長すぎる CLAUDE.md は、毎回のやり取りで大量のトークンを消費し、コストに影響します。重要な指示に絞って書きましょう。

## `.claude/rules/` によるルールの分割

CLAUDE.md が長くなりそうな場合は、`.claude/rules/` ディレクトリにルールファイルを分割できます。

```
.claude/
├── CLAUDE.md          # 中核の指示（技術スタック、基本方針）
└── rules/
    ├── testing.md     # テストに関するルール
    ├── coding.md      # コーディング規約
    └── git.md         # Git 運用ルール
```

`.claude/rules/` 配下のファイルは、CLAUDE.md と同様にセッション開始時に自動で読み込まれます。

> 💡 ルールファイルには `paths:` フロントマターを使って、特定のファイルパスにのみ適用するルールを定義することもできます。たとえば、テストファイルを編集するときだけ適用されるルールを設定できます。今は「ルールを分割できる」ことだけ知っておけば十分です。

## `/init` でプロジェクトの CLAUDE.md を生成する

CLAUDE.md をゼロから書くのは大変です。Claude Code には、プロジェクトの構造を分析して CLAUDE.md のたたき台を自動生成する `/init` コマンドがあります。

```
/init
```

このコマンドを実行すると、Claude Code がプロジェクトのディレクトリ構成、設定ファイル、依存関係などを分析し、適切な CLAUDE.md を生成します。

> 📝 `/init` が生成する CLAUDE.md はあくまで「たたき台」です。プロジェクト固有のルールや方針は、生成後に手動で追加・修正する必要があります。

## 自動メモリ

CLAUDE.md はあなたが明示的に書くファイルですが、Claude Code には会話の中で学んだことを自動で記録する **自動メモリ** という機能もあります。

たとえば、セッション中に「コミットメッセージは Conventional Commits の形式で」と伝えると、Claude Code がそれを自動メモリに記録し、以降のセッションでも覚えている場合があります。

自動メモリは `~/.claude/projects/` 配下にプロジェクトごとに保存されます。

| 記憶の種類 | 仕組み | 管理方法 |
|---|---|---|
| CLAUDE.md | あなたが書く | 手動で編集 |
| 自動メモリ | Claude Code が学習 | `/memory` コマンドで確認・管理 |

> 💡 自動メモリはデフォルトで有効です。`/memory` コマンドで、現在記録されている内容の確認や、自動メモリの有効・無効の切り替えができます。

> 📝 自動メモリは便利ですが、CLAUDE.md の代わりにはなりません。プロジェクトの重要なルールや方針は、自動メモリに頼らず CLAUDE.md に明示的に書いてください。自動メモリは「補足的な記憶」として捉えましょう。

## 🏃 実践: `/init` で cc-practice の土台を作る

ここまで使ってきた cc-practice に CLAUDE.md を設定し、プロジェクトの土台を仕上げましょう。

### Step 1: `/init` を実行する

cc-practice で Claude Code を起動し、`/init` を実行します。

```bash
claude
```

```
/init
```

Claude Code がプロジェクトの構造を分析し、CLAUDE.md を生成します。cc-practice は Laravel プロジェクトなので、`composer.json`、`artisan`、`app/` ディレクトリなどから技術スタックを読み取り、かなり充実した内容が生成されるはずです。

### Step 2: 生成された CLAUDE.md を確認する

生成された CLAUDE.md の内容を確認しましょう。

```
> 生成された CLAUDE.md の内容を見せて
```

```bash

```

### Step 3: CLAUDE.md をカスタマイズする

生成された CLAUDE.md に、プロジェクト固有のルールを追加してもらいましょう。

```
> CLAUDE.md に以下のルールを追加して:
> - コミットメッセージは日本語で書く
> - コードのコメントは日本語で書く
> - テストは PHPUnit で書く（Pest は使わない）
> - 開発環境のコマンドは Sail 経由で実行する（例: ./vendor/bin/sail artisan test）
> - コーディング規約は PSR-12 に準拠する
```

Claude Code が CLAUDE.md を編集し、指定したルールを追加します。

> 📝 **なぜ `./vendor/bin/sail` とフルパスで書くのか**: [2-1-1 インストールと認証](2-1-1_インストールと認証.md) で `sail` エイリアスを設定しましたが、CLAUDE.md にはフルパス（`./vendor/bin/sail`）で書きます。Claude Code はコマンドを独自の Bash プロセスで実行するため、あなたのシェル設定（`~/.zshrc` のエイリアスや環境変数）が引き継がれません。CLAUDE.md には Claude Code が実際に実行するコマンドをそのまま書いてください。

### Step 4: CLAUDE.md が反映されていることを確認する

CLAUDE.md のルールが実際に反映されるか確認してみましょう。一度セッションをリセットします。

```
/clear
```

`/clear` は会話履歴をクリアして新しいセッションを開始するコマンドです。CLAUDE.md はセッション開始時に読み込まれるため、`/clear` の後も CLAUDE.md の内容は反映されています。

セッションをクリアした状態で、以下の指示を出してみましょう。[2-1-1 インストールと認証](2-1-1_インストールと認証.md) で welcome.blade.php のタイトルを変更しましたが、それを表示するコントローラーがまだありません。

```
> トップページ用の WelcomeController を作成して。/ ルートで welcome ビューを返すようにして
```

Claude Code がファイルを生成します。生成されたコードを確認しましょう。

- コメントが日本語で書かれているか（CLAUDE.md のルール）
- PSR-12 に準拠した書き方になっているか（クラス名、インデント、ブレースの位置など）
- `app/Http/Controllers/` ディレクトリに配置されているか
- `routes/web.php` の `/` ルートが、既存のクロージャから `WelcomeController` に変更されているか（Claude Code が既存ルートを置き換えるか、重複が残るかは環境によります。重複する場合は、既存のクロージャルートを削除してください）

> 💡 ルールが反映されていない場合もあります。CLAUDE.md の指示は「強い提案」であり、100% の保証ではありません。ただし、具体的に書くほど反映率は上がります。

## 🔍 見極めチェック

> 🧠 先輩エンジニアの思考: 「Claude Code が生成したコードを見て『動きそうだからOK』で済ませていないだろうか。実務では、自分が書いたコードと同じように、AI が生成したコードにも責任を持つ必要がある。生成されたコードを3つの観点で確認する習慣をここから始めよう。」

この教材では、Claude Code がコードやコンテンツを生成した場面で **見極めチェック** を実施します。見極めチェックは、生成物を **正しさ・品質・安全性** の3つの観点で検証する習慣です。

今回は、Step 3 と Step 4 で生成された CLAUDE.md とコントローラーを対象に確認しましょう。

**CLAUDE.md について:**

- [ ] **正しさ**: 追加を指示したルール（日本語コミットメッセージ、日本語コメント、PHPUnit、Sail、PSR-12）がすべて反映されているか
- [ ] **品質**: 記述が簡潔で明確か。曖昧な表現や矛盾する指示がないか。技術スタックの情報は正確か
- [ ] **安全性**: 機密情報（API キー、パスワード、内部 URL など）が含まれていないか

**生成されたコントローラーについて:**

- [ ] **正しさ**: PHP の文法として正しいか。名前空間（`App\Http\Controllers`）が正しく設定されているか
- [ ] **品質**: CLAUDE.md のルール通り、コメントが日本語で書かれているか。PSR-12 に準拠しているか（インデント、ブレースの位置など）
- [ ] **安全性**: 不要な外部通信やファイル操作のコードが含まれていないか

> 🔑 この Section では特に「CLAUDE.md に書いた指示が正しく反映されているか」に注目してください。Claude Code の出力をそのまま受け入れるのではなく、自分が意図した通りの結果になっているかを確認する習慣が、見極める力の第一歩です。

## ✨ まとめ

- CLAUDE.md はプロジェクト固有の指示書。セッション開始時に自動で読み込まれ、Claude Code の振る舞いに影響する
- 配置場所は3つ（ユーザー全体、プロジェクト、サブディレクトリ）。プロジェクトルートの CLAUDE.md が最も重要
- `/init` コマンドでたたき台を自動生成し、プロジェクト固有のルールを手動で追加していく
- `.claude/rules/` ディレクトリでルールを分割・整理できる
- 自動メモリは補足的な記憶。重要なルールは CLAUDE.md に明示的に書く
- 見極めチェック（正しさ・品質・安全性）は、Claude Code の生成物を検証する基本の型。ここから習慣にする
