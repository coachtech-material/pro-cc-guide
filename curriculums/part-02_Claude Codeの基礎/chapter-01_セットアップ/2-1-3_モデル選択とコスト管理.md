# 2-1-3 モデル選択とコスト管理

## 🎯 このセクションで学ぶこと

- Claude Code で利用できるモデルの種類と特性を把握する
- タスクに応じたモデル選択の判断基準を理解する
- 使用量の確認方法を知り、コストを意識した使い方を身につける

まず利用できるモデルの違いを把握し、次にモデルやエフォートレベルの切り替え方を学び、最後に使用量の確認とコスト管理の方法を身につけます。

---

## 導入: モデルを知らないと損をする

Claude Code は内部で AI モデルを呼び出して動作しています。このモデルは1種類ではありません。性能の高いモデル、応答が速いモデル、コストを抑えたモデルなど、複数のモデルが用意されています。

モデルの違いを知らないまま使っていると、簡単な作業に高性能モデルを使って使用量を浪費したり、逆に複雑な作業を低性能モデルで処理して品質の低い結果を得たりすることになります。「Part 1 で Max プランを契約したから使い放題」と思っていても、モデルの選択次第で使用量の消費速度は大きく変わります。

このセクションでは、利用可能なモデルの違いを理解し、タスクに応じた使い分けとコスト管理の方法を身につけます。

### 🧠 先輩エンジニアはこう考える

> モデル選択は「どのモデルが一番いいか」ではなく「このタスクにはどのモデルで十分か」という考え方が大事だと思う。一番高性能なモデルを常に使うのは、軽トラで済む引っ越しに大型トラックを出すようなもの。コードのフォーマット修正に Opus を使う必要はないし、アーキテクチャ設計の相談に Haiku を使うのは心もとない。タスクの複雑さに合わせてモデルを選ぶ感覚は、使っていくうちに身につく。

## 利用できるモデル

Claude Code で利用できる主要なモデルは3つです（2026年3月時点）。

| モデル | 特徴 | 向いているタスク |
|---|---|---|
| **Sonnet 4.6** | バランス型。速度と品質のバランスが良い | 日常的なコーディング、コードレビュー、テスト作成 |
| **Opus 4.6** | 最高性能。複雑な推論や大規模な変更に強い | アーキテクチャ設計、複雑なバグ修正、大規模リファクタリング |
| **Haiku 4.5** | 高速・軽量。シンプルなタスクを素早く処理 | 簡単な質問、コードのフォーマット、定型的な修正 |

### デフォルトモデル

契約しているプランによって、Claude Code 起動時のデフォルトモデルが異なります。

| プラン | デフォルトモデル |
|---|---|
| Max / Team Premium | Opus 4.6 |
| Pro / Team Standard | Sonnet 4.6 |

> 📝 Pro プランでも Opus 4.6 を選択することはできますが、使用量の消費が大きくなります。Pro プランの場合は、Sonnet 4.6 をメインに使い、複雑なタスクのときだけ Opus に切り替えるのが実用的です。

## モデルの切り替え方

モデルはセッション中にいつでも切り替えられます。

| 方法 | 操作 |
|---|---|
| スラッシュコマンド | `/model` と入力して選択 |
| キーボードショートカット | Option + P（macOS） / Alt + P（Windows / Linux） |
| 起動時に指定 | `claude --model sonnet` |
| 設定ファイルで指定 | `.claude/settings.json` の `model` フィールド |

`/model` コマンドを実行すると、利用可能なモデルの一覧が表示されます。モデル名の横にはスライダーが表示され、ここでエフォートレベル（後述）も同時に調整できます。

### モデルエイリアス

モデルを指定するとき、フルネームではなくエイリアス（短縮名）を使えます。

| エイリアス | 対応するモデル |
|---|---|
| `sonnet` | Sonnet 4.6 |
| `opus` | Opus 4.6 |
| `haiku` | Haiku 4.5 |
| `sonnet[1m]` | Sonnet 4.6（1M コンテキスト） |
| `opus[1m]` | Opus 4.6（1M コンテキスト） |

## エフォートレベル

モデルの選択とは別に、Claude Code がタスクにどれだけの「思考の深さ」を使うかを調整する **エフォートレベル** という設定があります。

| レベル | 説明 |
|---|---|
| **low** | 最小限の思考。シンプルな質問や定型作業向き |
| **medium** | 標準的な思考量 |
| **high** | 深い思考。複雑な問題に取り組む際に使用 |
| **max** | 最大限の思考。Opus 4.6 でのみ使用可能 |

デフォルトのエフォートレベルはプランによって異なります。Max / Team の Opus 4.6 では medium がデフォルトです。エフォートレベルが高いほど、Claude Code はより多くのトークンを「考える」ために使います。その分、応答に時間がかかり、使用量も増えます。

| 方法 | 操作 |
|---|---|
| スラッシュコマンド | `/effort low` のように指定 |
| `/model` のスライダー | `/model` 画面内で調整 |
| 起動時に指定 | `claude --effort medium` |

> 💡 エフォートレベルは「モデルの能力を下げる」のではなく「思考に使うリソースを調整する」イメージです。簡単なタスクにはエフォートを下げて高速に処理し、難しいタスクにはエフォートを上げて深く考えさせる、という使い分けが効果的です。

## Extended Context（拡張コンテキスト）

Claude Code のモデルは、通常のコンテキストウィンドウ（入力として受け取れるテキスト量の上限）に加えて、**1M（100万）トークン** のコンテキストウィンドウを利用できます。これは Extended Context（拡張コンテキスト）と呼ばれます。

大規模なコードベースを扱う場合や、長い会話の中で多くのファイルを参照する場合に有効です。目安として、数十ファイル以上のプロジェクトで広範囲のコードを一度に扱う場面（大規模リファクタリング、プロジェクト全体のアーキテクチャ分析など）で恩恵を感じやすくなります。cc-practice のような小規模プロジェクトでは通常のコンテキストで十分です。

| プラン | Opus 4.6 (1M) | Sonnet 4.6 (1M) |
|---|---|---|
| Max / Team / Enterprise | 含まれる | 追加使用量が必要 |
| Pro | 追加使用量が必要 | 追加使用量が必要 |

> 📝 「追加使用量が必要」とは、通常の使用量とは別に消費されるということです。Pro プランの場合、Extended Context を多用すると通常よりも早くレート制限に達する可能性があります。今は「大きなコンテキストが使える」という点だけ把握しておけば十分です。コンテキスト管理の詳細は Chapter 2-2 で学びます。

## Fast Mode

Opus 4.6 には、通常モードに加えて **Fast Mode** という高速モードがあります（2026年3月時点、リサーチプレビュー）。

Fast Mode は、同じ Opus 4.6 モデルを使いながら、出力速度が約 2.5 倍になります。モデルの品質は変わりませんが、コストが高くなります。

| 項目 | 通常の Opus 4.6 | Fast Mode |
|---|---|---|
| 品質 | 同じ | 同じ |
| 速度 | 標準 | 約 2.5 倍 |
| コスト | 標準 | 高い |

Fast Mode の切り替えは `/fast` コマンドで行います。

> ⚠️ Fast Mode は通常のサブスクリプション使用量には含まれず、追加使用量として課金されます。通常の Opus と比較して使用量の消費が大きくなるため、常時有効にすると Pro プランでは特にレート制限に達しやすくなります。Teams / Enterprise プランでは管理者による有効化が必要です。「速くなるなら常時使いたい」という気持ちになりますが、コストへの影響を理解してから使うようにしましょう。

> 💡 Fast Mode とエフォートレベルの違いを整理しておきましょう。Fast Mode は「同じ品質をより速く」、エフォートレベルを下げるのは「思考を浅くして速く」です。目的が異なるので、混同しないようにしてください。

## 使用量とコストの確認

### 使用量の確認方法

契約しているプランの種類によって、使用量の確認方法が異なります。

**サブスクリプション（Pro / Max）の場合:**

```
/stats
```

`/stats` コマンドで、現在のセッションのトークン使用量や、プランの使用状況を確認できます。

**API（従量課金）の場合:**

```
/cost
```

`/cost` コマンドで、セッション中に発生した費用を確認できます。

> 📝 この教材では Pro または Max プランの利用を前提としているため、`/stats` コマンドを使用します。

### 使用量を節約するコツ

Claude Code を効率的に使い、使用量を抑えるためのポイントをいくつか紹介します。詳細は Chapter 2-2 以降で深掘りしますが、最初から意識しておくと差が出ます。

- **タスクが切り替わったら `/clear` する**: 新しいタスクに移るとき、前のタスクの会話履歴をクリアすると、不要なコンテキストの蓄積を防げます。コンテキストが大きくなると、毎回のやり取りで送受信するトークン量が増え、使用量の消費が加速します
- **タスクに合ったモデルを選ぶ**: 前述の通り、シンプルなタスクには Sonnet や Haiku を使い、複雑なタスクに Opus を使う
- **具体的な指示を出す**: 曖昧な指示は Claude Code の試行錯誤を増やし、トークンを消費します。「このファイルのバリデーションを修正して」より「○○ Controller の store メソッドのバリデーションに、メールアドレスの形式チェックを追加して」の方が効率的です
- **エフォートレベルを調整する**: 簡単なタスクはエフォートを下げて処理する

> 💡 Anthropic の公式ドキュメント（[Costs](https://code.claude.com/docs/en/costs)）によると、API（従量課金）での平均的なコストは1人あたり1日約 $6、9割のユーザーが $12 以下です（2026年3月時点）。サブスクリプションの場合は使用量の上限内であれば追加費用はかかりませんが、「今日どれくらい使ったか」を把握する習慣は持っておきましょう。

### Step 1: 現在のモデルを確認する

Claude Code を起動します。

```bash
claude
```

`/model` と入力して、現在選択されているモデルを確認してください。契約プランに応じたデフォルトモデルが選択されているはずです。

### Step 2: モデルを切り替える

`/model` の画面で、別のモデルに切り替えてみましょう。Pro プランの方は Opus に、Max プランの方は Sonnet に切り替えてみてください。

Option + P（macOS）/ Alt + P（Windows / Linux）でもモデルを切り替えられます。好みの方法を試してみてください。

### Step 3: 同じ質問を異なるモデルで試す

モデルの違いを体感するために、cc-practice に関連する同じ質問を異なるモデルで聞いてみましょう。

```
> Laravel のサービスコンテナの仕組みを、初心者にもわかるように説明して
```

Sonnet と Opus でそれぞれ回答を見比べてみてください。回答の深さや構造化の丁寧さ、応答にかかる時間の違いを感じ取れるはずです。

### Step 4: エフォートレベルを試す

同じモデルでエフォートレベルの違いも体感してみましょう。

```
/effort low
```

```
> cc-practice の composer.json を読んで、どんなパッケージが入っているか教えて
```

次に、エフォートレベルを上げて同じ質問をします。

```
/effort high
```

```
> cc-practice の composer.json を読んで、どんなパッケージが入っているか教えて
```

low ではパッケージ名の一覧を簡潔に返し、high では各パッケージの役割や依存関係の解説まで踏み込んだ回答になる傾向があります。質問の複雑さに応じてエフォートを使い分ける感覚を掴んでおきましょう。

## ✨ まとめ

- Claude Code で利用できる主要モデルは Sonnet 4.6（バランス型）、Opus 4.6（最高性能）、Haiku 4.5（高速・軽量）の3つ
- デフォルトモデルはプランによって異なる（Max → Opus、Pro → Sonnet）。`/model` や Option + P（macOS）/ Alt + P で切り替えられる
- エフォートレベルで思考の深さを調整できる。タスクの複雑さに応じて使い分ける
- Fast Mode は Opus の出力を約 2.5 倍に高速化するが、追加コストがかかる
- `/stats`（サブスクリプション）または `/cost`（API）で使用量を確認できる。タスクに合ったモデル選択と `/clear` の活用で使用量を節約する
