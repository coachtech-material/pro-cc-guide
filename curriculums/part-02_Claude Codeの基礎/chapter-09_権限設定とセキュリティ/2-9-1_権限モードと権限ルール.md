# 2-9-1 権限モードと権限ルール

## 概要

ここまでの Chapter で、皆さんは Claude Code と一緒にテック記事プラットフォームを作り上げてきました。その過程で、Claude Code が「このコマンドを実行してよいですか？」「このファイルを編集してよいですか？」と聞いてくるたびに、「Yes」を押してきたはずです。

あの許可ダイアログ、何を許可しているか意識していましたか？

正直なところ、多くの人が「なんとなく Yes」を押していたと思います。開発がスムーズに進んでいるとき、いちいち立ち止まるのは面倒です。しかし実務では、許可すべき操作とそうでない操作を区別する必要があります。たとえば「テストを実行する」は気軽に許可できますが、「本番データベースに接続する」は慎重に判断すべきです。

このセクションでは、Claude Code の権限システムを体系的に理解します。許可ダイアログの仕組みを知り、権限モードと権限ルールを使ってプロジェクトに応じた設定ができるようになります。

## ツールと許可

### Claude Code が許可を求める仕組み

Claude Code は内部的に複数の「ツール」を使い分けて作業します。ファイルを読むツール、ファイルを編集するツール、シェルコマンドを実行するツール、Web にアクセスするツールなどです。

これらのツールは、操作のリスクに応じて3つのカテゴリに分かれています。

| カテゴリ | 例 | 許可が必要か | 「今後も許可」の範囲 |
|---|---|---|---|
| 読み取り専用 | ファイル読み取り、Grep | 不要 | — |
| Bash コマンド | シェルコマンドの実行 | 必要 | プロジェクトディレクトリ + コマンド単位で永続 |
| ファイル変更 | ファイルの編集・作成 | 必要 | セッション終了まで |

**読み取り専用** のツールは許可なしで実行されます。ファイルの内容を読んだり、コードベースを検索したりする操作はリスクが低いためです。

**Bash コマンド** は最も注意が必要です。シェルコマンドはファイルの削除、ネットワークアクセス、プロセスの起動など、あらゆる操作が可能です。許可を求められたとき「Yes, don't ask again」を選ぶと、そのプロジェクトでそのコマンドが永続的に許可されます。

**ファイル変更** は、セッション終了まで有効な許可です。Claude Code を再起動すると、再び許可を求められます。

### 許可ダイアログの読み方

Claude Code が許可を求めるとき、実行しようとしている操作の内容が表示されます。Bash コマンドの場合は実行するコマンドそのものが、ファイル編集の場合は変更内容の diff が表示されます。

許可の選択肢は以下の通りです。

- **Yes**: 今回だけ許可する
- **Yes, don't ask again**: 今後も同じ操作を許可する（Bash コマンドの場合はプロジェクト永続、ファイル変更の場合はセッション終了まで）
- **No**: 拒否する

「Yes, don't ask again」を選ぶと、許可ルールとして保存されます。保存された許可ルールは `/permissions` コマンドで確認できます。

```
> /permissions
```

ここまでの Chapter で「Yes, don't ask again」を選んだ操作が一覧に表示されるはずです。`sail artisan migrate` や `sail test` などが含まれているかもしれません。

## 権限モード

### 5つの権限モード

Claude Code には、許可の厳しさを切り替える **権限モード** があります。

| モード | 動作 |
|---|---|
| **default** | 標準動作。ツールの初回使用時に許可を求める |
| **acceptEdits** | ファイル編集を自動で許可する。Bash コマンドは引き続き確認される |
| **plan** | 分析のみ。ファイルの変更やコマンドの実行はできない |
| **dontAsk** | 事前に許可されたツール以外は自動で拒否する |
| **bypassPermissions** | 許可ダイアログをスキップする（コンテナ環境向け） |

日常的に使うのは **default** と **acceptEdits** の2つです。

**default** は最も安全なモードです。すべての変更操作で許可を求めるため、Claude Code が何をしているかを1つずつ確認できます。初めてのプロジェクトや、重要な変更を加えるときに適しています。

**acceptEdits** は、ファイル編集の許可ダイアログをスキップします。Claude Code がファイルを編集するたびに「Yes」を押す手間がなくなるため、開発のテンポが上がります。Bash コマンドは引き続き確認されるので、「ファイル編集は信頼するが、コマンド実行は確認したい」というバランスです。ここまでの Chapter で何度もファイル編集の許可を出してきた皆さんには、このモードが便利に感じるかもしれません。

**plan** は 2-4-2 で使った Plan Mode のベースになっているモードです。コードベースを読んで分析はできますが、変更を加えることはできません。「まず全体像を把握してから作業に入りたい」ときに使います。

**dontAsk** は、事前に許可ルール（次の節で解説）を設定しておき、それ以外の操作を自動で拒否します。CI/CD 環境やスクリプト実行など、人間が介在しない場面で使います。

**bypassPermissions** は、ほぼすべての許可ダイアログをスキップします。ただし、`.git`、`.claude`、`.vscode`、`.idea` ディレクトリへの書き込みは引き続き確認が入ります（リポジトリ状態やローカル設定の破損を防ぐため）。Docker コンテナや VM など、隔離された環境でのみ使ってください。通常の開発環境では使わないでください。

⚠️ `bypassPermissions` は隔離された環境（コンテナ、VM）専用です。通常の開発環境で使うと、Claude Code がファイルの削除やネットワークアクセスを確認なしで実行できてしまいます。

### 権限モードの切り替え

権限モードは2つの方法で切り替えられます。

**セッション中に切り替える**: 2-1-2 で紹介した `Shift+Tab` を押すと、権限モードの選択画面が表示されます。

**起動時に指定する**: `--permission-mode` フラグで指定します。

```bash
claude --permission-mode acceptEdits
```

試しに、今のセッションで `Shift+Tab` を押して権限モードの一覧を確認してみてください。現在の設定が `default` になっているはずです。確認したら、そのまま `default` のままにしておきましょう。

## 権限ルール

### allow / ask / deny ルール

権限モードがツール全体の許可レベルを決めるのに対し、**権限ルール** は個別のツールやコマンドに対して細かく制御します。

権限ルールには3種類あります。

| ルール | 動作 |
|---|---|
| **allow** | 許可ダイアログなしで実行を許可する |
| **ask** | 実行のたびに確認を求める |
| **deny** | 実行をブロックする |

ルールの評価順序は **deny → ask → allow** です。deny が最優先で、deny ルールに一致した操作は、他にどんな allow ルールがあってもブロックされます。

### ルール構文

権限ルールは `ツール名` または `ツール名(指定子)` の形式で記述します。

**ツール全体を対象にする場合**:

| ルール | 効果 |
|---|---|
| `Bash` | すべての Bash コマンドに一致 |
| `WebFetch` | すべての Web アクセスに一致 |
| `Edit` | すべてのファイル編集に一致 |

**指定子で細かく制御する場合**:

| ルール | 効果 |
|---|---|
| `Bash(npm run build)` | `npm run build` コマンドに完全一致 |
| `Bash(sail *)` | `sail` で始まるコマンドに一致 |
| `Bash(git push *)` | `git push` で始まるコマンドに一致 |
| `WebFetch(domain:example.com)` | example.com へのアクセスに一致 |
| `Read(./.env)` | `.env` ファイルの読み取りに一致 |
| `Edit(/src/**)` | `src/` 配下のファイル編集に一致 |

ワイルドカード `*` は Bash ルールで使えます。`Bash(sail *)` は `sail artisan migrate` や `sail test` など、`sail` で始まるすべてのコマンドに一致します。

💡 `Bash(ls *)` のようにスペース + `*` の場合、`ls -la` には一致しますが `lsof` には一致しません。スペースが単語の区切りとして機能するためです。一方、`Bash(ls*)` のようにスペースなしの場合は `ls -la` にも `lsof` にも一致します。

Read と Edit のルールは gitignore と同じパターン構文を使います。`*` は単一ディレクトリ内、`**` はディレクトリを再帰的に一致させます。

### 設定ファイルへの記述

権限ルールは `.claude/settings.json` に記述します。このファイルはすでに 2-6-2 で Hooks の設定に使いました。

テック記事プラットフォームに権限ルールを設定してみましょう。

```
> .claude/settings.json に以下の権限ルールを追加してください。既存の hooks 設定はそのまま残してください。
>
> allow:
> - sail で始まるコマンド
> - npm run で始まるコマンド
>
> deny:
> - git push で始まるコマンド（意図しない push を防ぐ）
> - .env ファイルの読み取り（機密情報の保護）
```

皆さんの環境では異なるコードが生成されます。以下は一例です。

```json
{
  "permissions": {
    "allow": [
      "Bash(sail *)",
      "Bash(npm run *)"
    ],
    "deny": [
      "Bash(git push *)",
      "Read(./.env)"
    ]
  },
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "..."
          }
        ]
      }
    ]
  }
}
```

この設定の効果を確認しましょう。

- `sail artisan migrate` や `sail test` は許可ダイアログなしで実行されます（allow ルール）
- `git push origin main` は実行がブロックされます（deny ルール）
- `.env` ファイルは Claude Code の Read ツールからの読み取りがブロックされます（deny ルール）

設定を反映したら、`/permissions` で確認してみてください。

```
> /permissions
```

追加したルールが一覧に表示されます。allow ルール、deny ルールがそれぞれ正しく登録されているか確認してください。

### チームでの共有

`.claude/settings.json` はリポジトリにコミットできます。つまり、権限ルールをチームで共有できるということです。

たとえば、チームで「本番環境への push を防ぐ」「機密ファイルへのアクセスを制限する」といったルールを `.claude/settings.json` に設定しておけば、全員が同じ安全基準で Claude Code を使えます。

一方、個人的な設定は `.claude/settings.local.json` に書きます。このファイルは gitignore 対象なので、リポジトリにはコミットされません。

| 設定ファイル | 適用範囲 | チームで共有 |
|---|---|---|
| `.claude/settings.json` | プロジェクト | はい（リポジトリにコミット） |
| `.claude/settings.local.json` | プロジェクト（個人） | いいえ（gitignore 対象） |
| `~/.claude/settings.json` | すべてのプロジェクト（個人） | いいえ |

ルールの優先順位は、マネージド設定 > コマンドライン引数 > `.claude/settings.local.json` > `.claude/settings.json` > `~/.claude/settings.json` です。より限定的なスコープの設定が優先されますが、**deny ルールはどのレベルで設定されても覆せません**。プロジェクト設定で deny されたツールは、ユーザー設定で allow しても使えません。

💡 企業環境では、管理者がマネージド設定（managed settings）で全社的なポリシーを強制できます。たとえば `bypassPermissions` モードの無効化や、特定のコマンドの禁止などです。実務で Claude Code を使う際は、所属企業のポリシーを確認してください。

## 見極めチェック

- [ ] **正しさ**: `/permissions` で表示されるルールが、設定ファイルに書いた内容と一致しているか。allow に `sail *` と `npm run *` が、deny に `git push *` と `.env` が含まれているか
- [ ] **品質**: 既存の hooks 設定が権限ルールの追加によって壊れていないか。`/hooks` で PostToolUse Hook が引き続き表示されるか
- [ ] **安全性**: deny ルールが意図通りに機能しているか。Claude Code に「.env ファイルの内容を教えて」と聞いたとき、読み取りがブロックされるか確認する。`git push` がブロックされるかも確認する

> このセクションでは特に「安全性」に注目してください。deny ルールに `.env` と `git push` を設定しましたが、deny ルールは Claude Code の組み込みツールに対してのみ有効です。たとえば `Read(./.env)` は Claude Code の Read ツールをブロックしますが、Bash で `cat .env` を実行すれば読めてしまいます。この制限を補うのが、次のセクションで学ぶサンドボックスです。

## まとめ

- Claude Code のツールは操作のリスクに応じて3カテゴリ（読み取り専用、Bash コマンド、ファイル変更）に分かれ、それぞれ異なる許可パターンを持ちます
- 権限モード（default、acceptEdits、plan、dontAsk、bypassPermissions）でツール全体の許可レベルを切り替えられます。日常的に使うのは default と acceptEdits です
- 権限ルール（allow / ask / deny）で個別のツールやコマンドを細かく制御できます。評価順序は deny → ask → allow で、deny が最優先です
- 権限ルールは `.claude/settings.json` に記述し、チームで共有できます。deny ルールは `.env` ファイルの保護や意図しない `git push` の防止に有効です
- ただし、deny ルールは Claude Code の組み込みツールに対してのみ有効です。Bash コマンドからのアクセスは防げません。この制限を補うのがサンドボックスです
