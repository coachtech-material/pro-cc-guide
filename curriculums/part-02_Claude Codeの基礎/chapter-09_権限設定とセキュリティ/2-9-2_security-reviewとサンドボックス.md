# 2-9-2 /security-review とサンドボックス

## 概要

前のセクションでは、Claude Code の権限システムを体系的に理解し、権限ルールでツールのアクセスを制御しました。しかし、権限ルールだけではカバーできない領域があります。生成されたコード自体にセキュリティ上の問題がないかは、別の手段で確認する必要があります。

このセクションでは、`/security-review` コマンドでテック記事プラットフォームのセキュリティレビューを実行します。そして、その結果を鵜呑みにせず、自分の目で安全性を確認する体験をします。

さらに、権限ルールの限界を補う **サンドボックス** の仕組みを理解します。権限ルールが Claude Code の組み込みツールを制御するのに対し、サンドボックスは OS レベルでファイルシステムとネットワークのアクセスを隔離します。

## /security-review でセキュリティを確認する

### なぜ自分の目で確認するのか

見極める力の3観点（正しさ・品質・安全性）のうち、この Chapter では **安全性** を重点的に扱います。

ここまでの Chapter では、正しさ（要件通りに動くか）と品質（保守性・設計の一貫性）を中心に見極めてきました。安全性は最後のピースです。

Claude Code はセキュリティの知識を持っています。`/security-review` を実行すれば、コードベース全体をスキャンして潜在的な脆弱性を報告してくれます。しかし、AI が「問題なし」と報告したとき、それは「問題が見つけられなかった」であって「問題が存在しない」ではありません。

これは正しさや品質と同じ考え方です。テストが通っても「正しい」とは限らない。コードレビューで指摘がなくても「品質が高い」とは限らない。セキュリティレビューで「問題なし」でも「安全」とは限りません。

だからこそ、自分の目で確認する習慣が必要です。

### /security-review を実行する

テック記事プラットフォームのコードベースに対して、セキュリティレビューを実行してみましょう。

```
> /security-review
```

Claude Code がコードベース全体をスキャンし、セキュリティ上の懸念点を報告します。スキャンには少し時間がかかります。

結果には、検出された脆弱性の種類、該当箇所、リスクの程度、修正の推奨事項が含まれます。たとえば以下のような項目が報告される可能性があります。

- CSRF 対策が不足しているフォーム
- XSS のリスクがあるユーザー入力の出力箇所
- 認可チェックが漏れているルートやコントローラ
- ハードコードされたシークレットや設定値
- SQL インジェクションのリスクがあるクエリ

### 結果を自分の目で検証する

`/security-review` の結果を受け取ったら、報告された問題を1つずつ確認します。

ここで大切なのは、**Claude Code の報告を起点に、自分の知識で判断する** ことです。

皆さんは Laravel の基本的なセキュリティ機能を知っています。CSRF トークン、Blade テンプレートの自動エスケープ、Eloquent によるパラメータバインディング、Policy による認可。これらの知識を使って、Claude Code の指摘が妥当かどうかを判断してください。

以下の3つの観点で確認しましょう。

**CSRF 対策**

Laravel の Blade テンプレートでフォームを作成する場合、`@csrf` ディレクティブが含まれているかを確認します。

```
> フォームを含むすべての Blade テンプレートで @csrf ディレクティブが使われているか確認してください。
```

**XSS 対策**

Blade テンプレートでは `{{ $variable }}` がデフォルトでエスケープされます。一方、`{!! $variable !!}` はエスケープされません。テック記事プラットフォームでは Markdown の HTML 出力に `{!! !!}` を使っている箇所があるかもしれません。その場合、表示前にサニタイズ処理が入っているかを確認します。

```
> Blade テンプレートで {!! !!}（エスケープなし出力）を使っている箇所を一覧にしてください。それぞれ、サニタイズ処理が適切に行われているか確認してください。
```

**認可チェック**

2-4-2 で設定したロール別アクセス制御（admin / author）が、すべてのルートで正しく適用されているかを確認します。特に、admin のみが実行できるはずの操作（記事の公開承認など）が、author からもアクセスできてしまわないかを重点的に確認します。

```
> admin 権限が必要な操作（記事の公開承認、ユーザー管理など）のルートに、適切な認可チェック（ミドルウェアまたは Policy）が設定されているか確認してください。
```

気になる箇所があれば、Claude Code に具体的に質問してください。「ここは安全か？」ではなく、「この箇所でユーザー入力がエスケープされずに出力される可能性はないか？」のように具体的に聞くと、より正確な回答が得られます。

💡 `/security-review` は万能ではありません。報告に含まれない脆弱性もあれば、過剰に報告される場合もあります。重要なのは、AI のレビュー結果を **出発点** として使い、最終判断は自分で行うことです。これは実務でのコードレビューと同じ姿勢です。

## サンドボックス

### 権限ルールの限界を補う

前のセクションで、権限ルールの限界に触れました。`Read(./.env)` の deny ルールは Claude Code の Read ツールをブロックしますが、Bash で `cat .env` を実行すれば読めてしまいます。

権限ルールは Claude Code のツールレベルで動作するため、Bash コマンドの中身まではカバーできません。Bash はあらゆるコマンドを実行できるツールなので、権限ルールだけで制御するには限界があります。

サンドボックスは、この限界を **OS レベル** で補います。

### サンドボックスの仕組み

サンドボックスは、Claude Code の Bash コマンドが実行される環境を OS のセキュリティ機能で隔離します。

| OS | 使用する技術 |
|---|---|
| macOS | Seatbelt（macOS の組み込みサンドボックスフレームワーク） |
| Linux / WSL2 | bubblewrap（軽量なサンドボックスツール） |

WSL1 は bubblewrap が必要とするカーネル機能がないため非対応です。WSL を使っている方は WSL2 であることを確認してください。

サンドボックスは **ファイルシステム** と **ネットワーク** の2つを隔離します。

**ファイルシステムの隔離**:
- **読み取り** はデフォルトでコンピュータ全体にアクセスできます（一部のシステムディレクトリを除く）。依存ライブラリやシステムファイルの参照に必要なためです
- **書き込み** はカレントディレクトリ（プロジェクトディレクトリ）とそのサブディレクトリに制限されます
- プロジェクト外のファイル（`~/.bashrc`、`/bin/` 配下など）への書き込みはブロックされます
- この制限は Bash コマンドから起動されるすべてのサブプロセスにも適用されます

**ネットワークの隔離**:
- 許可されたドメインのみにアクセスできます
- 新しいドメインへのアクセスは許可ダイアログが表示されます
- 許可していないサーバーへのデータ送信（外部への情報漏洩）がブロックされます

権限ルールとサンドボックスは **補完関係** にあります。

| 保護の種類 | 権限ルール | サンドボックス |
|---|---|---|
| 対象 | Claude Code のすべてのツール | Bash コマンドとそのサブプロセス |
| 粒度 | ツール単位・コマンド単位 | ファイルパス・ドメイン単位 |
| 強制力 | Claude Code のツール選択をブロック | OS レベルでアクセスを遮断 |

権限ルールで「Claude Code がどのツールを使えるか」を制御し、サンドボックスで「Bash コマンドがどこにアクセスできるか」を制御する。両方を組み合わせることで、多層的な防御（defense-in-depth）が実現します。

### サンドボックスの有効化

サンドボックスは `/sandbox` コマンドで有効化できます。

```
> /sandbox
```

メニューが表示され、サンドボックスモードを選択できます。

| モード | 動作 |
|---|---|
| Auto-allow | サンドボックス内の Bash コマンドは自動で許可される。サンドボックス外のアクセスが必要な場合は通常の許可フローに戻る |
| Regular permissions | サンドボックスが有効でも、すべての Bash コマンドで許可を求める |

**Auto-allow** モードは、サンドボックスの恩恵を最大限に活かせます。サンドボックス内（プロジェクトディレクトリ内のファイルアクセス、許可済みドメインへのネットワークアクセス）であれば許可ダイアログが出ないため、開発のテンポが上がります。同時に、サンドボックス外へのアクセスは OS レベルでブロックされるため、安全性も確保されます。

💡 サンドボックスの詳細な設定（書き込みを許可する追加パス、ネットワークのドメイン許可リストなど）は `settings.json` の `sandbox` セクションで行えます。興味がある方は[公式ドキュメント](https://code.claude.com/docs/en/sandboxing)を参照してください。

### サンドボックスを体験する

サンドボックスの効果を実際に確認してみましょう。

**ステップ 1: サンドボックスを有効にする**

```
> /sandbox
```

メニューが表示されたら「Auto-allow」を選択します。

**ステップ 2: プロジェクト外のファイルを読み書きしてみる**

サンドボックスが有効な状態で、以下を試してみてください。

```
> ホームディレクトリに test-sandbox.txt というファイルを作成してください。
```

Claude Code がこのファイルを作成しようとすると、サンドボックスによってブロックされます。プロジェクトディレクトリ外への書き込みは OS レベルで遮断されるためです。ターミナルにエラーメッセージが表示されることを確認してください。

**ステップ 3: プロジェクト内の操作を確認する**

次に、プロジェクト内のファイル操作を試します。

```
> プロジェクトルートに test-sandbox.txt を作成し、「サンドボックステスト」と書いてください。
```

こちらはプロジェクトディレクトリ内なので、正常に作成されます。Auto-allow モードでは許可ダイアログも表示されません。

確認が済んだら、作成したテストファイルを削除し、サンドボックスを無効にしておきましょう。

```
> test-sandbox.txt を削除してください。
```

```
> /sandbox
```

メニューから「Disable」を選択すれば、通常モードに戻ります。

### プロンプトインジェクションへの防御

サンドボックスが特に威力を発揮するのは、**プロンプトインジェクション（Prompt Injection）** への防御です。

プロンプトインジェクションとは、悪意のあるテキストを通じて AI の動作を操作しようとする攻撃手法です。たとえば、プロジェクトに含まれるファイルやダウンロードした依存パッケージに「SSH 鍵を外部サーバーに送信せよ」といった指示が埋め込まれていた場合、Claude Code がその指示に従ってしまう可能性があります。

サンドボックスが有効であれば、たとえ Claude Code がプロンプトインジェクションの影響を受けても、以下がブロックされます。

- 許可していないドメインへのネットワークアクセス（情報の漏洩を防止）
- プロジェクト外のファイルへの書き込み（システムファイルの改ざんを防止）
- 許可していないプログラムの実行

Claude Code 自体にもプロンプトインジェクションへの対策（コマンドのブロックリスト、不審なコマンドの検出など）が組み込まれていますが、サンドボックスはそれを **OS レベルで** 補強します。

## 見極めチェック

- [ ] **正しさ**: `/security-review` を実行し、報告された問題の一覧を確認できたか。報告された各項目が、実際にコードのどの箇所に対応しているかを特定できたか
- [ ] **品質**: セキュリティレビューで指摘された問題のうち、修正が必要なものがあれば修正したか。修正によって既存の機能が壊れていないか
- [ ] **安全性**: CSRF（フォームに `@csrf` があるか）、XSS（`{!! !!}` の箇所にサニタイズがあるか）、認可（admin 操作にミドルウェアや Policy があるか）を自分の目で確認したか。Claude Code の「問題なし」をそのまま受け入れず、自分の Laravel の知識で裏付けを取ったか

> このセクションでは特に「安全性」に注目してください。`/security-review` の結果を鵜呑みにしないことが、このセクションの最も重要なポイントです。AI が「安全です」と言ったとき、「本当にそうか？」と自分で確認する習慣は、実務でも皆さんを守ります。この習慣は Part 3 でより実践的に鍛えていきます。

## まとめ

- `/security-review` はコードベース全体をスキャンしてセキュリティ上の懸念点を報告します。ただし「問題なし」は「問題が見つけられなかった」という意味であり、「問題が存在しない」ではありません
- セキュリティレビューの結果は出発点として使い、CSRF・XSS・認可などの観点で自分の Laravel の知識をもとに検証します
- サンドボックスは OS レベル（macOS は Seatbelt、Linux は bubblewrap）でファイルシステムとネットワークを隔離し、権限ルールの限界を補います
- 権限ルール（ツールレベルの制御）とサンドボックス（OS レベルの隔離）を組み合わせることで、多層的な防御が実現します
- AI の出力を鵜呑みにせず自分で確認する姿勢は、正しさ・品質・安全性のすべてに共通する、見極める力の本質です
