# 2-6-1 テストの作成・実行とデバッグ

## 概要

ここまでのセクションでは、Claude Code が生成したコードをブラウザで開いて「動いているか」を目で確認してきました。ステータスバッジの色、遷移ボタンの表示、admin 承認画面の動作。目視で「OK」と判断してきたわけです。

しかし、目視確認には限界があります。次の機能を追加したとき、前に確認した機能が壊れていないかをもう一度ブラウザで確認しますか? 10画面を手動で確認しますか? 実務では、コードを変更するたびにすべての画面を手動でチェックする時間はありません。

テストを書けば、この確認を自動化できます。`sail test` の1コマンドで、すべての機能が期待通りに動いているかを数秒で検証できます。そしてこの「テストを書く」作業自体も、Claude Code に任せることができます。

このセクションでは、バリデーションを追加した上で Claude Code にテストを書かせ、テストの実行、テスト失敗時のデバッグまでを一気通貫で体験します。

## バリデーションの追加

### Form Request で入力検証を強化する

テストを書く前に、テスト対象となるロジックを増やしましょう。2-4-2 で状態遷移ルールを実装しましたが、記事の作成・更新時の入力バリデーションはまだ整備されていません。

Laravel では、バリデーションロジックをコントローラから分離するために **Form Request** を使います。Form Request を使うと、バリデーションルールと認可ロジックを1つのクラスにまとめて管理でき、コントローラをシンプルに保てます。

新しいセッションを起動しましょう。

```bash
claude -n "test-and-debug"
```

以下のプロンプトで、バリデーションの追加を指示します。

```
> 記事の作成・更新に Form Request を導入してください。
>
> バリデーションルール:
> - タイトル: 必須、最大255文字
> - 本文: 必須
> - カテゴリ: 必須、categories テーブルに存在すること
> - ステータス変更時: 既存の状態遷移ルール（draft→review は本文300字以上、review→published は admin のみ）をバリデーションに統合する
```

皆さんの環境では異なるコードが生成されます。以下は一例です。

```php
// app/Http/Requests/StoreArticleRequest.php
public function authorize(): bool
{
    return auth()->check();
}

public function rules(): array
{
    return [
        'title' => ['required', 'string', 'max:255'],
        'body_markdown' => ['required', 'string'],
        'category_id' => ['required', 'exists:categories,id'],
        'tags' => ['array'],
        'tags.*' => ['exists:tags,id'],
    ];
}
```

```php
// app/Http/Requests/UpdateArticleRequest.php
public function rules(): array
{
    $rules = [
        'title' => ['required', 'string', 'max:255'],
        'body_markdown' => ['required', 'string'],
        'category_id' => ['required', 'exists:categories,id'],
    ];

    if ($this->input('status') === 'review') {
        $rules['body_markdown'][] = 'min:300';
    }

    return $rules;
}
```

生成されたコードを確認しましょう。バリデーションルールがプロンプトの要件を満たしているかを見てください。特に、状態遷移ルール（draft→review は本文300字以上）がバリデーションに統合されているかが重要です。`authorize()` メソッドで認可チェックが行われているかも確認しましょう。

ブラウザで動作を確認します。記事の作成画面でタイトルを空にして送信し、バリデーションエラーが表示されればバリデーションが機能しています。

## テストの作成と実行

### Claude Code にテストを書かせる

バリデーションと状態遷移のロジックが揃いました。これらが正しく動くことを、テストで証明しましょう。

```
> 記事の公開ワークフローに関する Feature テストを作成してください。
>
> テスト対象:
> 1. 記事の作成バリデーション（タイトル必須、本文必須、カテゴリ存在チェック）
> 2. 状態遷移ルール（draft→review は本文300字以上、review→published は admin のみ）
> 3. 認可（author は自分の記事のみ編集可能、admin は全記事を操作可能）
>
> 正常系と異常系の両方をカバーしてください。
```

皆さんの環境では異なるコードが生成されます。以下は一例です。

```php
// tests/Feature/ArticleWorkflowTest.php
use App\Models\User;
use App\Models\Article;
use App\Models\Category;

public function test_article_requires_title(): void
{
    $user = User::factory()->create(['role' => 'author']);
    $category = Category::factory()->create();

    $response = $this->actingAs($user)->post(route('articles.store'), [
        'body_markdown' => 'テスト本文',
        'category_id' => $category->id,
    ]);

    $response->assertSessionHasErrors('title');
}

public function test_draft_to_review_requires_300_characters(): void
{
    $user = User::factory()->create(['role' => 'author']);
    $article = Article::factory()->create([
        'user_id' => $user->id,
        'status' => 'draft',
        'body_markdown' => '短い本文',
    ]);

    $response = $this->actingAs($user)->patch(
        route('articles.update', $article),
        ['status' => 'review', 'body_markdown' => '短い本文']
    );

    $response->assertSessionHasErrors('body_markdown');
}

public function test_only_admin_can_publish_article(): void
{
    $author = User::factory()->create(['role' => 'author']);
    $article = Article::factory()->create([
        'user_id' => $author->id,
        'status' => 'review',
    ]);

    $response = $this->actingAs($author)->patch(
        route('articles.update', $article),
        ['status' => 'published']
    );

    $response->assertForbidden();
}
```

### テストを実行する

テストを実行しましょう。Laravel Sail ではテストの実行に `sail test` を使います。

```
> sail test を実行してテスト結果を確認してください。
```

テストがすべて通れば、ターミナルに以下のような出力が表示されます。

```
Tests:    12 passed (28 assertions)
Duration: 3.42s
```

緑色の表示とともに全テストが pass していれば、バリデーションと状態遷移ルールがコードレベルで正しく実装されていることが証明されました。ブラウザを開く必要もなく、数秒で確認できます。

### Trust-then-Verify ギャップ

テストが全部通ったからといって、安心するのはまだ早いです。ここで **Trust-then-Verify ギャップ** について知っておきましょう。

Claude Code が書いたテストには、傾向的な偏りがあります。

- **正常系に偏りがち**: 「正しい入力で正しい結果が返る」テストは書くが、「不正な入力で適切にエラーになる」テストが不足しやすい
- **境界条件を見落としがち**: 本文がちょうど300字の場合（境界値）、0字の場合（空文字）、非常に長い場合（上限値）などのテストが抜けやすい
- **テストが実装に密結合しがち**: 実装の内部構造に依存したテストを書くことがあり、リファクタリングするとテストが壊れる

テストが通っている = 正しいとは限りません。Claude Code が書いたテストが「本当にテストすべきことをテストしているか」を確認するのは皆さんの仕事です。

生成されたテストファイルを読んで、以下を確認してください。

- 異常系のテストがあるか（バリデーションエラー、権限エラー）
- 境界条件のテストがあるか（本文がちょうど300字のケース）
- 正常系と異常系のバランスが取れているか

不足があれば、追加を指示しましょう。

```
> 以下の境界条件のテストを追加してください。
> - 本文がちょうど300字の場合、draft→review の遷移が成功すること
> - 本文が299字の場合、draft→review の遷移が失敗すること
> - タイトルがちょうど255文字の場合、記事の作成が成功すること
```

このように、Claude Code にテストを書かせた上で「テストの十分性」を自分の目で見極め、不足があれば追加を指示する。これが AI と協働するテストのワークフローです。

## テスト失敗時のデバッグ

### エラーメッセージを読む

テストが全部通る状態は安心ですが、実務ではテストが失敗することのほうが多いです。新機能を追加したら既存のテストが壊れた、バリデーションルールを変更したらテストが通らなくなった。こうした状況で、エラーメッセージを読んで原因を特定し、修正する力が求められます。

ここでは意図的にテストを失敗させて、デバッグの流れを体験しましょう。

バリデーションルールを変更して、テストが失敗する状況を作ります。

```
> 記事のタイトルのバリデーションに「10文字以上」の最小文字数制限を追加してください。テストは変更しないでください。
```

バリデーションルールだけを変更し、テストはそのままにします。テストが古いバリデーションルールを前提にしているため、失敗するはずです。

テストを実行しましょう。

```
> sail test を実行してください。
```

テストが失敗すると、以下のような出力が表示されます。

```
FAILED  Tests\Feature\ArticleWorkflowTest > test_article_creation_with_valid_data
Expected response status code [201] but received 422.
Failed asserting that 201 is identical to 422.

Tests:    1 failed, 11 passed (27 assertions)
Duration: 3.58s
```

このエラーメッセージを読み解きましょう。

- **FAILED**: どのテストが失敗したか（`test_article_creation_with_valid_data`）
- **Expected 201 but received 422**: 記事の作成が成功（201）するはずが、バリデーションエラー（422）になった
- **1 failed, 11 passed**: 12個中1個だけ失敗。影響範囲が限定的であることがわかる

ステータスコード 422 は Laravel がバリデーションエラーを返すときのコードです。つまり、テストデータのタイトルが新しいバリデーションルール（10文字以上）を満たしていないことが原因だとわかります。

### Claude Code にデバッグを任せる

エラーメッセージの内容が理解できたら、Claude Code に修正を依頼しましょう。エラーメッセージをそのまま伝えるのが最も効果的です。

```
> テストが失敗しました。タイトルに10文字以上の最小文字数制限を追加したので、テストデータのタイトルが短すぎてバリデーションエラーになっています。テストを新しいバリデーションルールに合わせて修正してください。
```

ここでのポイントは、エラーメッセージだけでなく **原因の仮説** も一緒に伝えていることです。「テストが失敗しました」だけよりも、「タイトルのバリデーションを変更したからテストデータが合わなくなった」と伝えるほうが、Claude Code は的確に修正できます。

修正後、もう一度テストを実行して全テストが通ることを確認しましょう。

```
> sail test を実行して、修正が正しいことを確認してください。
```

このように「テスト実行 → 失敗 → エラーを読む → 原因を特定 → 修正 → 再テスト」のサイクルは、実務で毎日使うワークフローです。Claude Code はこのサイクルのすべてのステップを支援できますが、エラーメッセージを読んで原因の見当をつけるのは皆さん自身の力です。

### `/debug` を使う

エラーの原因が自分ではわからないとき、Claude Code の `/debug` コマンドが役立ちます。

```
> /debug
```

`/debug` は、2-5-1 で紹介した Bundled Skills の1つです。セッションのデバッグログを読み取り、問題の切り分けと修正を支援します。実行すると、Claude Code がエラーの原因を調査し、修正案を提案します。エラーメッセージをコピーペーストしなくても、直近のコンテキストから問題を把握して対処してくれます。

💡 `/debug` は便利なショートカットですが、万能ではありません。エラーメッセージを自分で読み、原因の仮説を立てた上で Claude Code に伝えるスキルは、ツールが変わっても使い続けられる基礎力です。`/debug` に頼る前に、まず自分でエラーメッセージを読む習慣をつけてください。

## CLAUDE.md へのテスト方針追記

テストの書き方と実行方法が決まったので、CLAUDE.md に記録しておきましょう。Claude Code が今後テストを書くとき、ここに書かれた方針に従って生成するようになります。

```
> CLAUDE.md にテスト方針を追記してください。
>
> - テストフレームワーク: PHPUnit（Laravel 標準）
> - テスト実行コマンド: sail test
> - 単一ファイル実行: sail test --filter=クラス名
> - テスト方針: Feature テストを中心に、正常系・異常系・境界条件をカバーする
```

CLAUDE.md にテスト方針を記録しておくと、以降のセッションで Claude Code がテストを書くとき、この方針に沿ったテストを生成します。チームで開発する場合、メンバー全員が同じテスト方針で Claude Code を使えるようになります。

💡 ここまでの変更をコミットしておきましょう。コミットメッセージの例: 「Add form request validation and feature tests for article workflow」。

## 見極めチェック

- [ ] **正しさ**: テストが正常系（有効なデータでの作成成功）と異常系（バリデーションエラー、権限エラー）の両方をカバーしているか
- [ ] **正しさ**: 境界条件のテストがあるか。本文がちょうど300字の場合（遷移成功）と299字の場合（遷移失敗）をテストしているか
- [ ] **正しさ**: 状態遷移ルールのテストが 2-4-2 で定義したルール（draft→review: 本文300字以上、review→published: admin のみ）と一致しているか
- [ ] **品質**: Form Request が適切に作成され、バリデーションロジックがコントローラから分離されているか
- [ ] **品質**: テストメソッド名が何をテストしているか明確に伝えているか（例: `test_draft_to_review_requires_300_characters`）
- [ ] **安全性**: Form Request の `authorize()` メソッドで認証・認可チェックが行われているか。未認証ユーザーがバリデーションをバイパスできないか

> このセクションでは特に **正しさ** に注目してください。Claude Code が生成したテストが「テストすべきことをテストしているか」を見極める練習です。テストが通ることだけでなく、テストの網羅性（正常系・異常系・境界条件のバランス）を確認してください。不足があれば Claude Code に追加を依頼するか、自分でテストケースを書いてみてください。

## まとめ

- Claude Code にテストを書かせて `sail test` で実行し、コードの正しさをコマンド1つで検証できるようになりました。ブラウザでの目視確認を自動化する手段です
- Claude Code が書くテストには正常系への偏りや境界条件の見落としがあるため、テストの十分性を自分で見極め、不足があれば追加を指示する習慣が大切です
- テスト失敗時は、エラーメッセージを読んで原因の仮説を立て、Claude Code に伝えてデバッグする。「テスト → 失敗 → 修正 → 再テスト」のサイクルが実務の日常です
- CLAUDE.md にテスト方針を記録しておくと、以降のセッションで一貫したテストが生成されます

次のセクションでは、テスト（手動で実行する検証）をさらに進めて、Hooks による品質チェックの自動化を学びます。Claude Code がファイルを編集するたびにフォーマッタが自動で走る仕組みを設定します。
