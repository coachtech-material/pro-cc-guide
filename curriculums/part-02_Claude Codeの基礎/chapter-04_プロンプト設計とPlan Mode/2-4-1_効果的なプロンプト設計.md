# 2-4-1 効果的なプロンプト設計

## 概要

ここまでの Chapter では、Claude Code にかなり具体的な指示を出してきました。テーブル設計ではカラム名や型を細かく伝え、CRUD では「記事の一覧・作成・編集・削除」のように操作を列挙しました。これは序盤の学習として正しいアプローチです。Claude Code がどう動くかを把握するには、指示と生成コードの対応関係が明確な方がわかりやすいからです。

このセクションからは **中盤** に入ります。指示の抽象度を一段上げ、「認証機能を追加して」のようにビジネス要件レベルで伝え、Claude Code に実装判断を委ねていきます。技術レベルが上がるわけではありません。皆さんが COACHTECH で学んだ Laravel の知識はそのまま活きます。変わるのは **指示の出し方** と **生成されるコードの量** です。

ここでは、良い指示の条件、コンテキストの与え方、そして Claude Code との対話的な進め方を学びます。

## 良い指示の書き方

### 同じ機能でも、指示の質で結果が変わる

Claude Code に同じ機能を作らせても、指示の書き方次第で生成コードの品質は大きく変わります。次の2つの指示を比べてみてください。

**曖昧な指示:**

```
> ログイン機能を作って
```

**具体的な指示:**

```
> Fortify を使ったログイン機能を追加してください。
> メールアドレスとパスワードでログインし、ログイン後は記事一覧にリダイレクトします。
> 未認証ユーザーが記事の作成・編集ページにアクセスした場合はログインページにリダイレクトしてください。
```

曖昧な指示でも Claude Code はそれなりのコードを生成します。しかし、認証パッケージの選択、リダイレクト先、未認証時の挙動などをすべて Claude Code の判断に委ねることになります。結果として、皆さんの意図とは異なるコードが生成される可能性が高くなります。

具体的な指示では、技術選定（Fortify）、正常系の挙動（リダイレクト先）、異常系の挙動（未認証時の処理）を明示しています。Claude Code が判断すべき範囲が狭まり、意図に沿ったコードが生成されやすくなります。

### 良い指示の3つの条件

良い指示に共通する条件は次の3つです。

**1. スコープが明確であること**

何を作るのか、何を作らないのかがはっきりしている指示です。「ユーザー管理機能を作って」よりも、「ユーザーのプロフィール編集機能を作って。一覧・削除は不要です」の方が、Claude Code の判断のブレが小さくなります。

**2. 制約が伝わっていること**

使う技術、守るべきルール、既存の設計との整合性などの制約です。「テストを書いて」よりも、「PHPUnit で Feature テストを書いて。既存の `tests/Feature/ArticleTest.php` のスタイルに合わせてください」の方が、プロジェクトに馴染むコードが生成されます。

**3. 検証方法が含まれていること**

Claude Code が自分の出力を確認する手段を与えることは、指示の質を最も高める方法の一つです。

```
> バリデーションルールを追加してください。タイトルは必須で100文字以内、本文は必須です。
> 追加後、バリデーションのテストを書いて実行してください。
```

「テストを書いて実行してください」の一文で、Claude Code は自らの実装を検証します。テストが失敗すれば自分で修正を試みます。皆さんが手動でチェックする前に、一定の品質が担保されるわけです。

💡 スコープを広げるのは意図的に行いましょう。「このファイルで改善できることはありますか？」のように、あえて範囲を限定しない指示も探索には有効です。ただし、実装を依頼するときはスコープを絞った方が安定します。

### 練習: 曖昧な指示を書き換える

3つの条件を実感するために、1つ練習をしてみましょう。以下の曖昧な指示を、スコープ・制約・検証の3条件を満たすように書き換えてみてください。

**曖昧な指示:**

```
> 記事機能を改善して
```

この指示の問題点を考えてみてください。「改善」とは何を指すのか、どの範囲を対象にするのか、何をもって改善されたと判断するのか、すべてが曖昧です。

書き換えを考えたら、以下の例と比較してみてください。

**書き換え例:**

```
> 記事一覧ページの表示速度を改善してください。
> Eager Loading を使って N+1 クエリを解消し、ページネーションは15件ごとにしてください。
> 既存のテストが通ることを確認し、一覧表示のテストも追加してください。
```

この書き換えでは、スコープ（記事一覧ページの表示速度）、制約（Eager Loading、15件ページネーション）、検証（既存テスト＋新規テスト）がすべて含まれています。Claude Code が迷う余地がほとんどなくなりました。

### 既存のパターンを参照させる

プロジェクトにすでにあるコードのパターンを指し示すと、Claude Code はそのパターンに合わせたコードを生成します。

```
> 既存の ArticleController のスタイルに合わせて、CategoryController を作成してください。
```

「スタイルに合わせて」と伝えるだけで、Claude Code は ArticleController を読み、命名規則、エラーハンドリングの方法、レスポンスの形式などを踏襲します。新しいコントローラが既存のコードベースから浮かなくなります。

## コンテキストの与え方

良い指示を書くだけでなく、Claude Code に **必要な情報を過不足なく渡す** ことも重要です。Claude Code には、ファイル、画像、URL といった多様な情報を渡す方法があります。

### `@` でファイルを参照する

`@` に続けてファイルパスを入力すると、そのファイルの内容がプロンプトに含まれます。Tab キーで候補が補完されるので、パスを正確に覚えている必要はありません。

```
> @app/Models/Article.php のリレーション定義を確認して、不足しているものがあれば教えてください
```

Claude Code は指示を受けると自分でファイルを読みに行きますが、`@` で明示的に渡すことには利点があります。Claude Code がファイルを探す手間が省け、確実に正しいファイルを参照できます。特に似た名前のファイルが複数あるプロジェクトでは有効です。

ディレクトリを指定することもできます。ディレクトリを `@` で参照すると、ファイル一覧が表示されます（ファイルの中身は読み込まれません）。

```
> @app/Http/Controllers のコントローラ一覧を見て、命名規則を把握してください
```

### 画像を貼り付ける

Claude Code には画像を直接貼り付けることができます。デザインカンプ、エラー画面のスクリーンショット、データベースの ER 図などを渡すと、視覚的な情報をもとにコードを生成できます。

貼り付け方法は環境によって異なります。

- **macOS（iTerm2）**: `Ctrl+V` で貼り付けます（`Cmd+V` ではありません）
- **ドラッグ＆ドロップ**: 画像ファイルをターミナルにドラッグすることもできます
- **ファイルパス指定**: 「この画像を分析してください: /path/to/image.png」のように書くこともできます

⚠️ macOS では `Cmd+V` ではなくテキストとして貼り付けられてしまうことがあります。画像を貼り付けるときは `Ctrl+V` を使ってください。

### URL を渡す

外部のドキュメントや API リファレンスの URL を指示に含めると、Claude Code がその内容を取得して参照します。

```
> https://readouble.com/laravel/10.x/ja/fortify.html を参考に、Fortify を導入してください
```

頻繁に参照するドメインがある場合は、`/permissions` で許可リストに追加しておくと、毎回の承認が不要になります。

💡 URL、ファイル参照、パイプ入力のいずれを使うかは状況次第です。プロジェクト内のファイルは `@` で、外部ドキュメントは URL で、エラーログのような長い出力は `cat error.log | claude` のようにパイプで渡すのが自然です。

## 質問の活用

ここまでは「皆さんが Claude Code に指示を出す」方向の話でした。しかし、Claude Code との対話は一方通行ではありません。Claude Code に **質問させる** ことで、自分では気づかなかった検討事項を洗い出すこともできます。

### インタビュー型プロンプト

大きな機能を実装するとき、最初から完璧な指示を書くのは難しいものです。そんなときは、Claude Code にインタビューさせましょう。

```
> これから認証機能を実装します。実装に必要な情報を質問してください。
> 技術的な実装、UI/UX、エッジケース、トレードオフについて掘り下げてください。
> すべてカバーできたら、仕様を SPEC.md にまとめてください。
```

Claude Code は「認証方式はセッションベースですか、トークンベースですか？」「パスワードリセットは必要ですか？」「ログイン失敗時のロックアウトポリシーはありますか？」といった質問を次々と投げかけてきます。皆さんが答えていくことで、曖昧だった仕様が明確になっていきます。

このパターンは公式ドキュメントでも推奨されています。質問が終わったら仕様書（SPEC.md）を出力させ、新しいセッションでその仕様書をもとに実装を進めるのが効果的です。新しいセッションを使うのは、仕様策定の会話でコンテキストが埋まるのを避けるためです（2-3-2 で学んだコンテキスト管理の実践です）。

### `/btw` でサイド質問をする

作業中にふと疑問が浮かぶことがあります。「この Eloquent のメソッド、どういう意味だっけ？」「この設計、他のアプローチもある？」。しかし、本筋の作業を中断して質問すると、コンテキストが汚れます。

`/btw` コマンドは、この問題を解決します。

```
> /btw Eloquent の whereHas と with の違いは何ですか？
```

`/btw` で聞いた質問と回答は、会話の履歴に残りません。オーバーレイで表示され、Space キーや Enter キー、Escape キーで閉じるとそのまま消えます。本筋のコンテキストを汚さずに、ちょっとした疑問を解消できます。

`/btw` はサブエージェント（2-8 で学びます）と逆の特性を持っています。`/btw` は会話の全文脈を見られますがツール（ファイル読み取りやコマンド実行）は使えません。サブエージェントはツールを自由に使えますが、会話の文脈を持ちません。「文脈を見て答えるだけの質問」は `/btw`、「ファイルを調べて報告する調査」はサブエージェント、と使い分けます。

💡 `/btw` は Claude Code が作業中でも使えます。長い生成作業を待っている間に、関連する疑問を `/btw` で聞いておくと時間を有効に使えます。

## まとめ

- 良い指示には **スコープの明確さ**、**制約の伝達**、**検証方法の提供** の3つの条件があります
- `@` でファイルやディレクトリを参照し、画像の貼り付けや URL の提供で、Claude Code に必要なコンテキストを渡せます
- **インタビュー型プロンプト** で Claude Code に質問させると、自分では気づかなかった検討事項を洗い出せます
- `/btw` はコンテキストを汚さないサイド質問です。作業中の疑問解消に使いましょう

次のセクションでは、ここで学んだプロンプト設計の考え方を実践します。**Plan Mode** を使って認証機能の実装計画を立て、ビジネス要件レベルの指示で Claude Code に実装を任せます。生成されるコードの量が一気に増える、中盤最初のハンズオンです。
