# 2-4-2 Plan Mode で認証機能を実装する

## 概要

前のセクションで、良い指示の条件とコンテキストの与え方を学びました。このセクションでは、それを実践に移します。

テック記事プラットフォームに **認証機能**、**ロール別アクセス制御**、**記事ステータスの状態遷移** を一気に追加します。複数のモデル、ミドルウェア、ポリシー、マイグレーションにまたがる大きな変更です。こうした変更を場当たり的に進めると、途中で設計の矛盾に気づいて手戻りが発生します。

**Plan Mode** は、この「計画→実装」の分離を支援する機能です。Plan Mode で Claude Code に実装計画を立てさせ、計画を確認・修正してから実装に移ります。中盤最初のハンズオンとして、ビジネス要件レベルの指示で大きな機能を実装する体験を楽しんでください。

## Plan Mode

### なぜ計画を立ててから実装するのか

2-2-2 では、3つのモデルと CRUD を1つのプロンプトで一括生成しました。あのときは変更の範囲が比較的限られていたため、計画なしでも問題ありませんでした。

しかし、これから実装する認証・認可・状態遷移は事情が異なります。

- **認証**: Laravel Fortify のインストール、ログイン・登録画面、認証ミドルウェアの設定
- **認可**: User モデルへの role カラム追加、Policy の作成、ルートのアクセス制御
- **状態遷移**: Article モデルへの status カラム追加、遷移ルールの実装、遷移ログの記録

これらは互いに依存しています。認証がなければ認可は成り立たず、認可がなければ「admin だけが記事を公開できる」という状態遷移ルールも実装できません。場当たり的に進めると、「あ、先にこっちを作るべきだった」という手戻りが発生します。

Plan Mode を使えば、Claude Code に全体の実装計画を立てさせ、依存関係や実装順序を事前に確認できます。計画に問題があれば、コードを書く前に修正できます。

### Plan Mode の使い方

Plan Mode は `Shift+Tab` キーで切り替えます。

`Shift+Tab` を押すたびに、モードが次のように切り替わります。

1. **Normal Mode**（デフォルト）: 通常の動作。コードの読み書き、コマンドの実行ができます
2. **Auto-Accept Mode**（`accept edits on`）: ファイル編集の承認を自動的に行います
3. **Plan Mode**（`plan mode on`）: コードの読み取りと分析のみを行い、ファイルの変更やコマンドの実行はしません

画面左下のステータスバーに現在のモードが表示されます。`plan mode on` と表示されていれば Plan Mode です。

Plan Mode では Claude Code は「読むだけ」の状態になります。ファイルを読んでプロジェクトの構造を分析し、実装計画を作成しますが、コードを変更することはありません。計画の過程で、Claude Code が要件を確認する質問を投げかけてくることもあります。質問に答えることで、計画の精度が上がります。計画に納得してから Normal Mode に戻して実装を実行する、という流れです。

💡 Plan Mode 中に作成された計画は、`Ctrl+G` を押すとテキストエディタで開けます。計画を直接編集して、項目の追加や順序の変更ができます。

### 認証機能の計画を立てる

それでは実践です。テック記事プラットフォームのディレクトリで Claude Code を起動してください。

```bash
claude
```

まず `Shift+Tab` を2回押して Plan Mode に切り替えます。画面左下に `plan mode on` と表示されることを確認してください。

Plan Mode に切り替えたら、次のプロンプトを入力します。

```
> テック記事プラットフォームに認証・認可機能を追加する実装計画を作成してください。
> 要件は以下の通りです。
>
> 1. Laravel Fortify でメールアドレス・パスワード認証を実装する
> 2. User に role カラム（admin / author）を追加し、ロール別のアクセス制御を Policy で実装する
> 3. Article に status カラム（draft / review / published）を追加し、以下の状態遷移ルールを実装する
>    - draft → review: 本文が300字以上の場合のみ遷移可能
>    - review → published: admin のみ遷移可能
>    - published → draft: 遷移時に published_at をリセットする
> 4. 状態遷移のログを記録する
```

2-4-1 で学んだ「良い指示の条件」を意識してください。スコープ（認証・認可・状態遷移）が明確で、制約（Fortify、role の値、状態遷移ルール）が具体的です。一方で、実装の詳細（どのファイルに何を書くか）は Claude Code に委ねています。これが中盤の「ビジネス要件レベルの指示」です。

Claude Code はプロジェクトのコードを読み、実装計画を作成します。計画にはおよそ次のような内容が含まれるはずです（皆さんの環境では異なる計画が生成されます。以下は一例です）。

- Fortify のインストールと設定
- User マイグレーションへの role カラム追加
- 認証用の Blade ビュー作成
- ArticlePolicy の作成とロール別ルール定義
- Article マイグレーションへの status カラム追加
- 状態遷移ロジックの実装（モデルメソッドまたはサービスクラス）
- StatusChangeLog モデルとマイグレーションの作成
- ルーティングとミドルウェアの設定

### 計画を確認する

生成された計画を読み、以下の観点で確認してください。

**依存関係の順序は正しいか**: 認証 → 認可 → 状態遷移の順に実装されるべきです。認可は認証が前提であり、状態遷移の「admin のみ公開可能」というルールは認可が前提です。順序が逆転していたら、計画を修正しましょう。

**要件の漏れはないか**: 4つの要件がすべてカバーされているか確認します。特に、状態遷移の3つのルール（300字制限、admin 限定、published_at リセット）と遷移ログが計画に含まれているかを見てください。

**影響範囲は妥当か**: 既存のコードにどのような影響があるかを確認します。たとえば、認証を追加すると既存のルートにミドルウェアが適用される可能性があります。Seeder の更新が計画に含まれているかも確認しましょう。

計画に不足や問題があれば、Plan Mode のまま修正を依頼できます。

```
> Seeder の更新も計画に追加してください。admin ユーザーと author ユーザーをそれぞれ作成するようにします。
```

計画に納得したら、`Ctrl+G` でエディタに開いて全体を見渡すこともできます。問題なければ、次のステップに進みましょう。

## 実装の実行

### Normal Mode に切り替えて実装する

計画ができたら、`Shift+Tab` を押して Normal Mode に戻します。Plan Mode からは直接 Normal Mode に戻ります（Auto-Accept Mode を経由しません）。`plan mode on` の表示が消えたことを確認してください。

Normal Mode に戻したら、計画の実行を指示します。

```
> 計画に沿って実装を進めてください。
```

Claude Code は計画に基づいてコードを生成します。複数のファイルにわたる変更が次々と提案されるので、diff を確認しながら承認していきましょう。

皆さんの環境では異なるコードが生成されます。以下は生成されるコードの一例です。

**マイグレーション（role カラムの追加）:**

```php
// database/migrations/xxxx_add_role_to_users_table.php
Schema::table('users', function (Blueprint $table) {
    $table->enum('role', ['admin', 'author'])->default('author');
});
```

**Article モデル（状態遷移メソッド）:**

```php
// app/Models/Article.php
public function canTransitionTo(string $newStatus): bool
{
    return match ([$this->status, $newStatus]) {
        ['draft', 'review'] => mb_strlen($this->body_markdown) >= 300,
        ['review', 'published'] => auth()->user()?->role === 'admin',
        ['published', 'draft'] => true,
        default => false,
    };
}
```

**状態遷移ログモデル:**

```php
// app/Models/StatusChangeLog.php
class StatusChangeLog extends Model
{
    protected $fillable = ['article_id', 'user_id', 'from_status', 'to_status'];

    public function article()
    {
        return $this->belongsTo(Article::class);
    }

    public function user()
    {
        return $this->belongsTo(User::class);
    }
}
```

生成されるコードの量がこれまでの Chapter に比べて格段に多いことに気づくでしょう。これが中盤の特徴です。指示の抽象度が上がった分、Claude Code が判断する範囲が広がり、生成されるコードも増えます。

### マイグレーションを実行してブラウザで確認する

コードの生成が完了したら、マイグレーションを実行します。

```
> マイグレーションを実行してください
```

Claude Code が `sail artisan migrate` を実行します。完了したら、ブラウザでログイン画面にアクセスしてみましょう。URL は生成されたルート定義によりますが、多くの場合 `http://localhost/login` でアクセスできます。

ログイン画面が表示されたら、まず新規登録を行い、ログイン・ログアウトが正常に動作することを確認してください。

### tinker で状態遷移を確認する

状態遷移の UI は 2-5-3 で作成するため、現時点ではブラウザで遷移操作を試すことはできません。代わりに、Laravel の tinker を使ってバックエンドのロジックを確認しましょう。

```
> tinker で以下を確認してください。
> 1. 本文が300字未満の記事は draft から review に遷移できないこと
> 2. 本文が300字以上の記事は draft から review に遷移できること
> 3. author ユーザーでは review から published に遷移できないこと
```

Claude Code が tinker コマンドを実行し、各ルールの動作を確認します。期待通りの結果になっていれば、状態遷移ロジックは正しく実装されています。

💡 tinker はモデルやデータベースの動作を素早く確認するのに便利なツールです。UI を作る前にバックエンドのロジックを検証するワークフローは、実務でもよく使います。

## 権限モードの基本

### Claude Code の権限を意識する

ここまでの作業で、Claude Code にファイルの編集やコマンドの実行を許可してきました。diff が表示されるたびに `y` を押し、コマンドの実行確認にも `y` を押してきたはずです。

少し立ち止まって考えてみましょう。皆さんは今、テック記事プラットフォームに認証・認可の仕組みを実装しました。「誰が何をできるか」を明確に定義し、admin だけが記事を公開できるようにしました。

では、Claude Code に対してはどうでしょうか。Claude Code が提案するすべてのファイル変更、すべてのコマンド実行を「何となく y」で承認していませんでしたか？ アプリのユーザーには厳密な権限管理を求めながら、開発ツールに対しては無制限の権限を与えている。これは少し矛盾しています。

Claude Code にも **権限モード** があります。「何を許可し、何を制限するか」を自分で選べるのです。

### 3つの権限モード

先ほど Plan Mode への切り替えで使った `Shift+Tab` は、実は3つの **権限モード** を順に切り替えるキーでした。

| モード | 表示 | 動作 |
|---|---|---|
| Normal Mode（default） | なし | ファイル変更やコマンド実行のたびに承認を求めます |
| Auto-Accept Mode | `⏵⏵ accept edits on` | ファイル編集を自動承認します。コマンドの実行は引き続き確認されます |
| Plan Mode | `⏸ plan mode on` | 読み取りと分析のみ。ファイル変更もコマンド実行もしません |

普段の開発では Normal Mode を使い、diff を確認しながら承認するのが基本です。Claude Code の提案を一つ一つ確認できるので、意図しない変更を防げます。

Auto-Accept Mode は、Claude Code の出力を信頼してスピードを優先したいときに使います。たとえば、テストを書かせるときなど、生成結果を後からまとめて確認できる場面で便利です。

Plan Mode はこのセクションで使った通り、実装前の分析と計画に使います。

実は、Claude Code にはこの3つ以外にも権限モードがあります。また、特定のコマンドやファイルパスに対して細かく許可・拒否を設定する仕組みもあります。これらの詳細は 2-9 で体系的に学びます。今は「Claude Code にも権限管理がある」ことを意識しておいてください。

### 権限モードを切り替えてみる

`Shift+Tab` を押して、3つのモードを順に切り替えてみましょう。画面左下の表示が変わることを確認してください。

1. `Shift+Tab` → `⏵⏵ accept edits on`（Auto-Accept Mode）
2. `Shift+Tab` → `⏸ plan mode on`（Plan Mode）
3. `Shift+Tab` → 表示なし（Normal Mode に戻る）

最後に Normal Mode に戻しておいてください。

## 見極めチェック

- [ ] **正しさ**: ログイン・ログアウトがブラウザで正常に動作するか。tinker で状態遷移の3つのルール（300字制限、admin 限定、published_at リセット）が期待通りに動くか
- [ ] **品質**: Plan Mode で立てた計画と、実際に生成されたコードの構造は一致しているか。計画にあった項目が漏れていないか。計画になかった変更が含まれていないか。既存の ArticleController や Route 定義と一貫したスタイルで書かれているか
- [ ] **安全性**: 認証が必要なルートに `auth` ミドルウェアが適用されているか。未認証ユーザーが記事の作成・編集にアクセスできないか。admin 以外のユーザーが記事を公開（review → published）できないか

> このセクションでは特に **品質** に注目してください。Plan Mode の最大の利点は「計画と実装の整合性を検証できること」です。計画書に書かれた項目が実装にすべて反映されているか、逆に計画にない変更が紛れ込んでいないか。この確認は、チーム開発で設計レビューと実装レビューを行うのと同じプロセスです。

## まとめ

- **Plan Mode**（`Shift+Tab` で切り替え）は、コードを変更せずに分析と計画だけを行うモードです。計画は `Ctrl+G` でエディタに開いて直接編集できます
- 複数ファイルにまたがる大きな変更は、Plan Mode で計画を立ててから Normal Mode で実装すると、手戻りを防げます
- 計画と実装の **整合性** を確認することは、チーム開発の設計レビューと同じプロセスです。計画にある項目が漏れていないか、計画にない変更が紛れ込んでいないかを確認しましょう
- Claude Code にも **権限モード** があります。Normal Mode、Auto-Accept Mode、Plan Mode の3つを `Shift+Tab` で切り替えられます。詳細な権限設定は 2-9 で学びます

次の Chapter では、**Skills** で再利用可能なプロンプトを作成し、それを使いながらテック記事プラットフォームにモデルを追加していきます。このセクションで認証・認可・状態遷移のバックエンドを実装したので、次は Tag、Comment、Series といった機能を段階的に追加していきます。
