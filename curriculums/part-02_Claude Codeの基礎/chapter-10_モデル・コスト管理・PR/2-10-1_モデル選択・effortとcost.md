# 2-10-1 モデル選択・`/effort`・`/cost`

## 概要

ここまでの Chapter では、Claude Code のモデルを意識することなく開発を進めてきました。皆さんのサブスクリプションプランに応じたデフォルトモデルが自動的に使われていたからです。

しかし実務では、すべてのタスクに最高性能のモデルを使えばよいわけではありません。簡単なリネーム作業に高性能モデルを使えばコストの無駄ですし、複雑な設計タスクに軽量モデルを使えば品質が落ちます。タスクの性質に応じてモデルや推論レベルを切り替えることで、**速度・品質・コスト** のバランスを最適化できます。

このセクションでは、モデルの選択、推論レベルの調整、コストの確認方法を学びます。

## モデル選択

### なぜモデルを使い分けるのか

Claude のモデルファミリーには、性能と速度のトレードオフが異なる複数のモデルがあります。すべてのタスクに同じモデルを使うのは、すべてのネジをハンマーで打つようなものです。

実務での AI ツールの利用では、コストは常に意識すべき要素です。個人の学習では自分の財布との相談ですが、企業のプロジェクトではチーム全体のコストを管理する必要があります。「タスクに応じて適切なモデルを選べる」ことは、AI ツールを使いこなすエンジニアの基本スキルです。

### モデルエイリアス

Claude Code では、モデルを **エイリアス**（別名）で指定します。具体的なモデル名（`claude-sonnet-4-6` など）を覚える必要はありません。

| エイリアス | 特徴 | 向いているタスク |
|---|---|---|
| `sonnet` | バランス型。速度と品質の両立 | 日常的な開発タスク全般 |
| `opus` | 最高性能。複雑な推論に強い | 設計判断、大規模リファクタリング、難しいバグ修正 |
| `haiku` | 高速・低コスト | 単純な修正、フォーマット、定型的な作業 |

`default` エイリアスは、皆さんのサブスクリプションプランに応じたモデルが自動的に選ばれます（Max プランでは Opus、Pro プランでは Sonnet）。

💡 `opusplan` というエイリアスもあります。これは Plan Mode（2-4-2 で学びました）では Opus で計画を立て、実行時には Sonnet に自動で切り替わるハイブリッドモデルです。計画の質を保ちつつ、実行コストを抑えられます。

### モデルの切り替え方

モデルを切り替える方法は複数あります。

**セッション中に切り替える**:

```
> /model sonnet
```

`/model` を引数なしで実行すると、利用可能なモデルの一覧が表示され、矢印キーで選択できます。

**セッション起動時に指定する**:

```bash
claude --model opus
```

**環境変数で指定する**:

```bash
export ANTHROPIC_MODEL=sonnet
```

環境変数は `.zshrc` や `.bashrc` に追加しておけば、毎回指定する必要がなくなります。

### タスクに応じた使い分けを実践する

テック記事プラットフォームを使って、モデルの使い分けを試してみましょう。

まず、簡単なタスクを軽量モデルで実行します。

```bash
claude --model haiku
```

```
> テック記事プラットフォームのコードベースで、コメントが日本語で書かれている箇所があれば英語に統一してください。
```

Haiku は高速に動作するため、このような定型的な修正には最適です。変更内容を確認し、意図通りか確認してください。

次に、設計判断を伴うタスクを Opus で実行してみます。セッションを終了し、新しいセッションを起動します。

```bash
claude --model opus
```

```
> テック記事プラットフォームのパフォーマンスを改善するために、N+1 クエリの問題がないか調査してください。問題があれば修正方法を提案してください。
```

N+1 クエリの調査は、リレーションの構造を理解し、どのクエリが冗長かを判断する必要があるため、高性能モデルの方が精度の高い分析を行えます。

💡 モデルの使い分けに厳密なルールはありません。迷ったら Sonnet を使い、結果に満足できなければ Opus に切り替える。このフィードバックループで、自分なりの判断基準が育っていきます。

## 推論とコストの調整

### `/effort` で推論レベルを変える

モデルの選択とは別に、Claude Code がどれだけ深く考えるかを **推論レベル**（effort）で調整できます。

```
> /effort
```

メニューが表示され、以下のレベルから選択できます。

| レベル | 動作 | 向いている場面 |
|---|---|---|
| `low` | 最小限の推論。即答に近い | 単純な質問、フォーマット修正 |
| `medium` | 標準的な推論 | 通常の開発タスク |
| `high` | 深い推論。Extended Thinking が有効 | 複雑な設計判断、難しいデバッグ |
| `auto` | モデルのデフォルト値にリセット | 変更した推論レベルを元に戻したい場合 |

`max` レベルもありますが、Opus 4.6 でのみ使用可能で、セッション内でのみ設定できます。非常に高度な推論タスク向けです。

推論レベルを下げると、応答速度が上がりトークン消費が減ります。上げると、より深い分析と高品質な出力が得られますが、時間とコストがかかります。

💡 プロンプトに `ultrathink` というキーワードを含めると、そのターンだけ推論レベルが `high` に引き上げられます。普段は `medium` で作業しつつ、特定の質問だけ深く考えてほしいときに便利です。

### Fast Mode

Fast Mode は、Opus 4.6 の出力速度を約2.5倍に高速化する機能です。

```
> /fast
```

同じモデル、同じ品質で、応答が速くなります。ただし **Extra Usage**（サブスクリプション枠外の課金）が必要です。Fast Mode のトークン料金は通常の Opus より高く設定されています。

| 項目 | 通常の Opus 4.6 | Fast Mode |
|---|---|---|
| 出力速度 | 標準 | 約2.5倍 |
| 品質 | 同じ | 同じ |
| コスト | サブスクリプションに含まれる | Extra Usage（別料金） |

Fast Mode が有効な間は、プロンプトの横に `↯` アイコンが表示されます。再度 `/fast` で無効にできます。

> ⚠️ よくある誤解: Fast Mode は別のモデルに切り替わるわけではありません
>
> Fast Mode を有効にすると Opus 4.6 に自動で切り替わります。しかし、Fast Mode を無効にしても Opus のままです。元のモデル（たとえば Sonnet）に戻したい場合は、`/model sonnet` で明示的に切り替えてください。

### `/cost` と `/stats` でコストを確認する

AI ツールのコストを把握する習慣は、実務で非常に重要です。

**API / Console ユーザー** の場合:

```
> /cost
```

現在のセッションで使用したトークン数と、それに対応するコスト（概算）が表示されます。

**サブスクリプションユーザー**（Pro / Max）の場合:

```
> /stats
```

セッション内の使用パターンが表示されます。サブスクリプションの場合、トークンごとの課金ではなくレート制限で管理されるため、`/cost` ではなく `/stats` が参考になります。

### コストを抑えるテクニック

日常的に意識するだけでコストを大きく削減できるテクニックがあります。

- **タスクが変わったら `/clear`**: 無関係な過去のコンテキストを引きずると、不要なトークンを消費します。タスクが変わったらコンテキストをクリアしましょう（2-3-2 で学びました）
- **タスクに合ったモデルを選ぶ**: 簡単なタスクには Haiku、通常の開発には Sonnet、複雑な設計には Opus。これだけで大きな差が出ます
- **具体的なプロンプトを書く**: 曖昧な指示は Claude Code の試行錯誤を増やし、トークンを消費します。2-4-1 で学んだプロンプト設計が、ここでも活きます
- **CLAUDE.md を簡潔に保つ**: CLAUDE.md は毎回のリクエストに含まれます。不要な情報が多いと、それだけでトークンを消費します
- **早めに軌道修正する**: 方向が違うと感じたら、2-7-2 で学んだ `/rewind` や Escape キー（2回押し）で早めに戻りましょう。誤った方向に進み続けるほどコストがかさみます

💡 Anthropic の公式ドキュメント（2026年3月時点）によると、Claude Code の平均的なコストは1日あたり約6ドル、ユーザーの90%は1日12ドル以下です。チームでの利用では、Sonnet 4.6 を中心に使うことで月額100〜200ドル/人程度に収まります。料金体系は変更される可能性があるため、最新の情報は[公式ドキュメント](https://code.claude.com/docs/en/costs)を確認してください。

💡 Claude Code には Default / Explanatory / Learning の3つの Output Styles があります。`/config` で切り替えられます。特に Learning モードは Claude Code が `TODO(human)` を残して皆さん自身にコードを書かせるスタイルで、見極める力の実践と親和性が高いです。詳しくは[公式ドキュメント](https://code.claude.com/docs/en/output-styles)を参照してください。

## 見極めチェック

- [ ] **正しさ**: Haiku で実行した修正が意図通りの箇所だけを変更しているか。Opus で実行した N+1 クエリの調査結果が妥当か（指摘されたクエリが実際に N+1 問題を含んでいるか）
- [ ] **品質**: タスクの複雑さに応じて適切なモデルを選択できたか。推論レベル（`/effort`）の設定がタスクに合っているか
- [ ] **安全性**: この Section では該当なし

> この Section ではモデルの使い分けを実践しました。正しさの確認として、異なるモデルで実行した結果が期待通りかを比較してみてください。特に Haiku の出力は、Sonnet や Opus と比べて精度が落ちる場面があります。「速くて安いが、品質は十分か」を自分の目で判断する習慣が、コスト最適化の第一歩です。

## まとめ

- Claude Code はタスクに応じてモデルを切り替えられます。Haiku（高速・低コスト）、Sonnet（バランス）、Opus（最高性能）の3つのエイリアスを使い分けましょう
- `/effort` で推論レベルを調整できます。簡単なタスクは `low`、複雑な設計は `high` にすることで、速度とコストを最適化できます
- `/cost`（API ユーザー）や `/stats`（サブスクリプションユーザー）でコストを確認する習慣をつけましょう。タスクが変わったら `/clear`、具体的なプロンプト、適切なモデル選択がコスト削減の基本です
- Output Styles（`/config` で設定）の Learning モードは、`TODO(human)` で皆さん自身にコードを書かせるスタイルです
