# 2-10-2 PR 作成と振り返り

## 概要

テック記事プラットフォームは10個のモデルを持つ、本格的な Web アプリケーションに成長しました。ここまでの開発で、Claude Code の主要な機能をひと通り体験してきました。

実務では、コードを書いたらそれで終わりではありません。変更を PR（Pull Request）にまとめ、チームメンバーにレビューしてもらい、フィードバックを反映して、はじめてコードがメインブランチにマージされます。

このセクションでは、Claude Code を使って PR を作成し、レビュー対応のワークフローを体験します。そして Part 2 全体を振り返り、Part 3 への準備をします。

## PR の作成

### Claude Code で PR を作成する

まず、PR を作成するためのブランチを用意します。これまでの変更がすべてコミットされていることを確認してください（2-7-1 で Git 操作を学びました）。

```bash
claude -n "create-pr"
```

```
> 現在の変更をもとに、テック記事プラットフォームの機能追加をまとめた PR を作成してください。
> PR のタイトルと説明は、レビュー担当者が変更内容を理解しやすいように書いてください。
```

Claude Code は変更内容を分析し、`gh pr create` コマンドで PR を作成します。PR のタイトル、説明文、変更の要約を自動で生成します。

PR の作成プロセスを観察してください。Claude Code は以下のようなステップを踏みます。

1. `git diff` で変更内容を分析する
2. 変更の意図を読み取り、PR のタイトルと説明文を生成する
3. `gh pr create` で PR を作成する

作成された PR を GitHub で確認してみてください。タイトルは簡潔で、説明文には変更の概要が記載されているはずです。

### Attribution（帰属表示）

Claude Code で作成したコミットや PR には、**attribution**（帰属表示）が自動的に付与されます。

コミットメッセージには以下のような行が追加されます。

```
Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

PR の説明文にも「Generated with Claude Code」といった表記が含まれます。

これは「このコードは AI と協働して書いた」ことを透明にするための仕組みです。実務では、AI を使ったことを隠す必要はありません。むしろ、チームメンバーが「この PR は AI で生成されたコードを含む」と分かることで、レビューの観点が明確になります。

attribution の内容はプロジェクトの `.claude/settings.json` や `~/.claude/settings.json` の `attribution` セクションでカスタマイズできます。

```json
{
  "attribution": {
    "commit": "Generated with Claude Code",
    "pr": "Generated with Claude Code"
  }
}
```

空文字列を設定すれば attribution を非表示にすることもできますが、チームのポリシーに従って判断してください。

### `/pr-comments` でレビュー指摘を取得する

PR を作成すると、チームメンバーからレビューコメントが付くことがあります。Claude Code はそのレビューコメントを取り込んで、修正に活かすことができます。

```
> /pr-comments
```

このコマンドは、現在のブランチに紐づく PR のレビューコメントを取得します。Claude Code がコメントの内容を読み取り、「この指摘に対応するにはどう修正すればよいか」を提案してくれます。

実際にレビューコメントが付いていない場合でも、このコマンドの存在を覚えておいてください。実務では以下のようなワークフローになります。

1. Claude Code で PR を作成する
2. チームメンバーがレビューコメントを付ける
3. `/pr-comments` でコメントを取得する
4. Claude Code に修正を依頼する
5. 修正をコミットし、PR を更新する

### `--from-pr` でセッションを再開する

PR を作成した後、時間をおいてレビュー対応をする場面はよくあります。`--from-pr` オプションを使えば、PR に関連する過去のセッションを再開できます。

```bash
claude --from-pr 1
```

PR 番号を指定すると、その PR を作成したセッションの文脈を引き継いだ状態で再開します。「あの PR の修正をしたいけど、何を変えたか覚えていない」という場面で、コンテキストをゼロから構築し直す必要がなくなります。

## GitHub Actions と Code Review

Claude Code には、GitHub と連携した自動化機能もあります。ここでは概要を紹介します。

**GitHub Actions 連携**: CI/CD パイプラインの中で Claude Code を実行できます。たとえば、PR が作成されたときに自動でコードレビューを実行する、テストの失敗を自動で修正するといったワークフローが構築できます。

**Code Review**: PR に対して `@claude review` とコメントすると、Claude Code が自動でコードレビューを行い、差分に対してインラインコメントを付けます。指摘には重要度（バグ、nit、既存の問題）のタグが付き、レビュー担当者の負担を軽減します。

これらの機能は **チーム開発** で真価を発揮します。詳細は Part 3 で実践する予定です。今の段階では「Claude Code は PR の作成だけでなく、レビューの自動化まで支援してくれる」ことを覚えておいてください。

### よくある落とし穴と対処法

Part 2 を通じて、いくつかの「やりがちな失敗」にも触れてきました。ここで改めて整理しておきます。

**過度な修正指示に注意する**

Claude Code に修正を依頼して、結果が期待と違う。もう一度指示を変えて修正を依頼する。それでもうまくいかない。この繰り返しは、コンテキストを汚し、結果をどんどん悪化させます。

目安として、**同じ修正を2回指示してもうまくいかない場合は、`/clear` でコンテキストをクリアしてやり直す** ことを検討してください。新しいコンテキストで、より具体的な指示を出す方が効率的です。

**タスク分割と介入のタイミング**

Claude Code に大きなタスクを丸投げすると、途中で方向がずれても気づきにくくなります。2-4-1 で学んだタスク分割を習慣にしましょう。大きなタスクを小さなステップに分け、各ステップの出力を確認してから次に進む。介入のタイミングを自分で持つことが、品質を保つ鍵です。

### まだ見ていない景色

Part 2 では Claude Code の主要な機能を一通り学びましたが、まだ紹介していない機能もあります。

**設定の5レベル階層**

2-6-2（Hooks）や 2-9-1（権限ルール）で `settings.json` に触れましたが、Claude Code の設定には実は5つのレベルがあります。

| 優先度 | レベル | 場所 | 影響範囲 |
|---|---|---|---|
| 1（最高） | Managed | OS の管理ポリシー | マシン上のすべてのユーザー |
| 2 | コマンドライン引数 | `--model` 等 | そのセッションのみ |
| 3 | Project（ローカル） | `.claude/settings.local.json` | 自分だけ、このプロジェクトのみ |
| 4 | Project（共有） | `.claude/settings.json` | このプロジェクトの全メンバー |
| 5（最低） | User | `~/.claude/settings.json` | 自分のすべてのプロジェクト |

優先度の高いレベルが低いレベルの設定を上書きします。たとえば、User 設定で `model: opus` を指定しても、Project（共有）で `model: sonnet` が指定されていればそちらが優先されます。チーム開発では、共有設定（Project）で全員の基本ルールを揃え、ローカル設定で個人の好みを調整する使い方が一般的です。Part 3 のチーム開発で、この階層を活用します。

**Auto Memory**

Claude Code には、セッション中に学んだことを自動的にメモとして保存する **Auto Memory** 機能があります。保存されたメモは次回のセッション開始時に自動で読み込まれ、プロジェクトの文脈が引き継がれます。

```
> /memory
```

このコマンドで、保存されたメモの閲覧や Auto Memory の有効/無効を切り替えられます。

Auto Memory は CLAUDE.md を補完する仕組みです。CLAUDE.md が「皆さんが意図的に書くプロジェクトの指示」であるのに対し、Auto Memory は「Claude Code が会話の中から自動的に記録するメモ」です。どちらもセッション開始時に読み込まれ、プロジェクトの文脈を形成します。

## 見極めチェック

- [ ] **正しさ**: PR の差分を確認し、意図した変更だけが含まれているか。テストが通ること、未コミットの変更が残っていないことを確認したか
- [ ] **品質**: PR のタイトルと説明文がレビュー担当者にとって分かりやすいか。コミット履歴が論理的にまとまっているか。attribution が適切に設定されているか
- [ ] **安全性**: PR の差分に `.env` や秘密鍵などの機密情報が含まれていないか。デバッグ用のコード（`dd()` や `dump()` など）が残っていないか

> このセクションでは特に「正しさ」と「品質」に注目してください。PR を第三者の視点で読み返してみましょう。自分が書いたコードを他の人が読むとき、「この変更は何をしているのか」「なぜこの変更が必要なのか」が伝わるかどうか。これは実務のコードレビューと同じ視点です。

## まとめ

- Claude Code は `gh pr create` で PR を作成し、タイトルと説明文を自動で生成します。attribution（`Co-Authored-By`）が自動的に付与され、AI との協働を透明にします
- `/pr-comments` でレビューコメントを取得し、Claude Code に修正を依頼できます。`--from-pr` で PR に関連するセッションを再開すれば、コンテキストを引き継いで作業を再開できます
- 同じ修正を2回指示してもうまくいかない場合は、`/clear` でやり直す判断が大切です。タスク分割と介入のタイミングを自分で持つことが、品質を保つ鍵です
- Claude Code の設定には5つのレベルがあり、チーム開発では共有設定とローカル設定を使い分けます。Auto Memory は会話から学んだことを自動保存し、次回セッションに引き継ぎます
