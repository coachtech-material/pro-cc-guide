# 2-11-2 Notification 機能を実装する

## 概要

前のセクションで、知らない技術に向き合うための4つの検証ステップを学びました。

1. Claude に設計判断の理由を質問する
2. 公式ドキュメントで裏取りする
3. テストを書いて動作を確認する
4. 理解できない部分を明確にする

このセクションでは、このステップを実際に使います。題材は **Laravel の Notifications**（通知機能）です。

Laravel Notifications は、メール・データベース・Slack など複数のチャネルに通知を送信するための仕組みです。COACHTECH のカリキュラムでは詳しく扱っていない可能性があるため、「知らない技術」の実践に適しています。もし皆さんが Notifications を既に知っているなら、「知っている技術をこのステップで検証するとどうなるか」という視点で進めてみてください。検証プロセスの練習にはなります。

テック記事プラットフォームに以下の通知機能を追加します。

- 自分の記事にコメントが付いたとき、著者に通知する
- 自分の記事がいいねされたとき、著者に通知する
- 自分の記事が公開承認されたとき（admin が review → published にしたとき）、著者に通知する
- 通知の種類ごとに ON/OFF を設定できる
- 未読通知の一覧を表示し、既読にできる

## Notification の実装と検証

### Claude Code に実装を依頼する

まず、通知機能の実装を Claude Code に依頼します。

```
> テック記事プラットフォームに通知機能を追加してください。
> - 自分の記事にコメントが付いたとき、著者に通知
> - 自分の記事がいいねされたとき、著者に通知
> - 記事が公開承認されたとき、著者に通知
> - 通知の種類ごとにON/OFF設定できる（ユーザーごと）
> - 未読通知の一覧表示と既読機能
> - 通知チャネルはデータベースのみで構いません
```

Claude Code がコードを生成します。皆さんの環境では異なるコードが生成されます。以下は一例です。

複数のファイルが生成されるはずです。Notification クラス、通知設定のモデルやマイグレーション、コントローラー、ビューなど。ファイルの承認を求められたら、すべて `y` で承認してください。

ここからが本番です。普段なら生成されたコードを読んで見極めチェックに進みますが、今回は **まだ判断できない** はずです。`$user->notify(new CommentNotification($comment))` というコードを見ても、`notify` メソッドがどこから来て何をしているのか、すぐには分からないかもしれません。

2-11-1 で学んだ検証ステップを順番に適用していきましょう。

### ステップ 1: 設計判断の理由を質問する

生成されたコードについて、Claude に「なぜそうしたのか」を質問します。

```
> 今作成した通知機能について質問です。
> 1. なぜ Notification クラスを個別に作るのですか？1つのクラスで全種類の通知を扱ってはいけないのですか？
> 2. toDatabase メソッドで返している配列の構造は何に使われますか？
> 3. 通知設定を別テーブルにしている理由は何ですか？User テーブルにカラムを追加するのとどう違いますか？
```

Claude の回答を読んでみてください。おそらく以下のような説明が返ってくるはずです。

- Notification クラスを分けるのは **単一責任の原則** に従うため。通知ごとにメッセージや通知先チャネルが異なるため、1つのクラスに詰め込むと条件分岐が増えて保守しにくくなる
- `toDatabase` メソッドの返り値は、`notifications` テーブルの `data` カラムに JSON として保存される。通知を表示するときにこの JSON を読み取って「○○さんがコメントしました」のようなメッセージを組み立てる
- 通知設定を別テーブルにするのは、設定項目が増えたときに User テーブルにカラムを追加し続けなくて済むため

これで、コードの大まかな構造と意図が分かりました。ただし、これは Claude の説明です。次のステップで裏取りします。

### ステップ 2: 公式ドキュメントで裏取りする

```
> Laravel の Notifications の公式ドキュメントの URL を教えてください。
```

Claude が教えてくれた URL（`https://laravel.com/docs/10.x/notifications` など）をブラウザで開いてください。

公式ドキュメントで以下の点を確認してみましょう。

- **`Notifiable` トレイト**: User モデルに `use Notifiable;` が付いていることを確認する。このトレイトが `notify` メソッドを提供している
- **`toDatabase` メソッド**: 公式ドキュメントに「Database Notifications」のセクションがある。`notifications` テーブルのスキーマと `data` カラムの使い方が説明されている
- **`via` メソッド**: 通知をどのチャネル（mail, database, slack 等）に送るかを決めるメソッド。今回は `['database']` のみを返しているはず

いきなり公式ドキュメントを読むと情報量が多くて大変ですが、Claude の説明で大枠を掴んでいるので、「あ、ここに書いてある」と見つけやすいはずです。これが「質問してから裏取り」の効率的なところです。

もし Claude の説明と公式ドキュメントの間に食い違いがあれば、公式ドキュメントを優先してください。その上で Claude に「公式ドキュメントにはこう書いてありましたが、実装と異なるのはなぜですか？」と質問すれば、バージョン差異や設計判断の理由が明確になります。

### ステップ 3: テストを書いて動作を確認する

概念を理解したら、テストで動作を確認します。

```
> 通知機能のテストを書いてください。以下を確認したいです。
> - コメントが付いたとき、記事の著者に通知が作成されること
> - いいねされたとき、記事の著者に通知が作成されること
> - 記事が公開承認されたとき、著者に通知が作成されること
> - 自分自身の操作では通知が作成されないこと（自分の記事に自分がコメントしても通知しない）
> - 通知設定がOFFの場合、通知が作成されないこと
> - 未読通知を既読にできること
```

テストが生成されたら、実行します。

```bash
sail artisan test --filter=Notification
```

テストが通れば、「自分が定義した振る舞い」は正しく実装されていることが確認できます。テストが失敗した場合は、Claude Code に修正を依頼してください。

ここで気づいてほしいのは、テスト自体は Laravel の PHPUnit の仕組みで書かれているということです。`assertDatabaseHas`、`actingAs`、Factory。すべて皆さんが知っている技術です。**知らない技術で実装されたコードを、知っている技術で検証できる**。これが、テストによる検証の強みです。

### ステップ 4: 理解できない部分を明確にする

ここまでのステップを経て、自分の理解度を振り返ってみてください。

**理解できたこと** の例:
- Notification クラスの役割と `toDatabase` の仕組み
- `Notifiable` トレイトが `notify` メソッドを提供すること
- `notifications` テーブルに JSON で通知データが保存される仕組み

**まだ理解できていないかもしれないこと** の例:
- `notify` メソッドの内部で何が起きているか（キューイングの仕組み）
- 通知のチャネルをメールや Slack に切り替えたときの動作
- 大量の通知が同時に発生したときのパフォーマンス

「まだ理解できていない部分」があること自体は問題ありません。テストで動作は確認できています。今の時点では「データベース通知は動いている。メール通知やキューイングは今後必要になったら調べる」という判断ができれば十分です。

### 動作確認

テストに加えて、ブラウザでも動作を確認しておきましょう。

1. 著者としてログインし、記事を作成する
2. 別のユーザー（admin）でログインし、その記事にコメントする
3. 最初の著者で再度ログインし、通知が表示されていることを確認する
4. 通知を既読にできることを確認する
5. 通知設定で特定の通知を OFF にし、その通知が作成されないことを確認する


## 見極めチェック

- [ ] **正しさ**: コメント・いいね・公開承認のそれぞれで通知が作成されるか。自分自身の操作では通知が作成されないか。通知設定の ON/OFF が正しく反映されるか。未読/既読の切り替えが動作するか
- [ ] **品質**: Notification クラスが通知の種類ごとに分かれているか。通知設定が拡張しやすい設計になっているか（新しい通知種類を追加するときに既存コードの修正が少ないか）。`notifications` テーブルのマイグレーションが Laravel の標準に従っているか
- [ ] **安全性**: 他のユーザーの通知を閲覧・操作できないようになっているか（認可の確認）。通知の表示でユーザー入力をエスケープしているか（XSS 対策）

> このセクションでは特に「正しさ」に注目してください。知らない技術で実装されたコードの「正しさ」を、自分がまだ完全に理解していない段階でどう検証するか。テストが通ること、ブラウザで動作すること、公式ドキュメントと実装が整合していること。この3点で「正しさ」を確認するプロセスを体験してください。

## まとめ

- Laravel の Notifications という「知らない（かもしれない）技術」を、Claude Code で実装し、4つの検証ステップを実践しました
- ステップ 1（質問）で大枠を掴み、ステップ 2（裏取り）で公式ドキュメントと照合し、ステップ 3（テスト）で動作を確認し、ステップ 4（明確化）で自分の理解度を把握しました
- テストは「知っている技術」（PHPUnit）で書けるため、知らない技術の実装を検証する強力な手段になります
- 次のセクションでは、さらに馴染みの薄い領域（外部 API 連携）で同じステップを繰り返し、このアプローチを定着させます
