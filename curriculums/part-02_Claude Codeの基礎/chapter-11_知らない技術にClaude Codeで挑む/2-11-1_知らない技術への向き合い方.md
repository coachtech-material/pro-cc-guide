# 2-11-1 知らない技術への向き合い方

## 概要

Part 2 の本編（2-1〜2-10）を通じて、皆さんは COACHTECH で学んだ Laravel の知識を土台に、Claude Code で高速にアプリを構築する体験を積んできました。Eloquent のリレーション、認証・認可、テスト。生成されたコードを読めば「正しいか」を判断できました。それは、皆さんが **知っている技術** だったからです。

しかし実務では、自分が経験していない技術を使う場面が必ず来ます。プロジェクトで採用されているキューイング、WebSocket、外部 API 連携、サーバーレス。COACHTECH のカリキュラムではカバーしきれない技術は山ほどあります。

Claude Code はそのような場面でもコードを生成してくれます。しかし、ここで問題が生じます。**生成されたコードを読んでも、正しいかどうか判断できない**。これまで磨いてきた「見極める力」の使い方が変わるのです。

このセクションでは、知らない技術に Claude Code で向き合うときの心構えと、具体的な検証方法を学びます。

> 💡 この Chapter は応用編です。スキップしても Part 3 には影響しません。2-10 までの本編で学んだ知識で Part 3 に進めます。

## 知っている技術と知らない技術の違い

### 検証方法が変わる

Part 2 を振り返ってください。たとえば 2-4-2 で認証機能を実装したとき、生成されたコードに `auth()->check()` や `Policy` が含まれていれば、「これは Laravel の認証・認可の仕組みだ」と分かりました。コードの正しさを **自分の知識で判断** できたのです。

では、もし Claude Code が Laravel の Broadcasting（WebSocket を使ったリアルタイム通信）のコードを生成したらどうでしょう。`event(new ArticlePublished($article))` や `Echo.channel('articles')` といったコードが出てきたとき、「これは正しい実装なのか？」と聞かれても、経験がなければ答えられません。

ここに、知っている技術と知らない技術の本質的な違いがあります。

**知っている技術の場合**:
1. コードを読む
2. 自分の知識と照らし合わせる
3. 正しいか判断する

**知らない技術の場合**:
1. コードを読む
2. ……判断できない
3. ?

ステップ 2 で止まります。知識がないから比較対象がない。ここで2つの落とし穴が待っています。

### 2つの落とし穴

**落とし穴 1: 思考の放棄**

「よく分からないけど、Claude が書いたんだから大丈夫だろう」。これは 2-1-3 で体験したバイブコーディングと同じ構造です。知っている技術なら「コードを読まない」ことが問題でしたが、知らない技術では「理解しないまま受け入れる」ことが問題になります。

**落とし穴 2: 萎縮**

「自分には判断できないから、この技術は使えない」。これも実務では困ります。知らない技術を避け続けることはできません。チームの技術選定に「自分が知らないから」という理由で反対するわけにもいきません。

どちらの落とし穴にも落ちない第三の道があります。**「分からないから聞いて、理解してから受け入れる」** という方法です。

## AI の使い方で結果が変わる

この「第三の道」が本当に有効なのか。それを裏付ける研究があります。

2025年1月、Anthropic（Claude の開発元）が52人のプロの Python 開発者を対象にした実験の結果を発表しました（以下のデータは2025年1月公開時点のものです）。開発者たちに「Trio」という馴染みのない非同期ライブラリを使ってタスクを完成させてもらい、その後に理解度テストを実施しました。AI を使えるグループと使えないグループに分け、さらに AI を使ったグループの中でも **使い方のパターン** を分析しました。

結果は明確でした。AI を使った開発者の中でも、**使い方によって理解度に大きな差** が出たのです。

**理解度が低かったパターン（テストスコア 24〜39%）**:
- **コード生成を丸投げ**: 「このコードを書いて」と依頼し、出力をそのまま使う
- **デバッグも AI 任せ**: エラーが出たら自分で原因を考えず、AI にそのまま投げる

**理解度が高かったパターン（テストスコア 65〜86%）**:
- **概念を質問する**: 「この設計はなぜこうなっているのか」「この関数の役割は何か」と聞く
- **コード生成後に説明を求める**: 生成されたコードを受け取った後、「なぜこの実装にしたのか」と理由を聞く

注目すべきは、理解度が高かったグループも AI を使っていたことです。AI を使うかどうかではなく、**どう使うか** が理解度を決めました。

この研究が示しているのは、皆さんがこの教材で実践してきたことの延長線上にある考え方です。Part 2 では「見極めチェック」で生成コードを検証する習慣を身につけました。知らない技術でもこの習慣を手放す必要はありません。ただ、検証の前に **「理解する」というステップを挟む** のです。

> 💡 この研究の詳細は Anthropic の公式ブログ「[How AI assistance impacts the formation of coding skills](https://www.anthropic.com/research/AI-assistance-coding-skills)」で読めます。AI との向き合い方を考える上で、一読の価値があります。

## 検証のステップ

知らない技術の生成コードを検証するための、具体的なステップを整理します。

### ステップ 1: Claude に設計判断の理由を質問する

コードを生成してもらったら、まず **「なぜそうしたのか」** を聞きます。

```
> この実装でクラスを分けている理由は何ですか？1つのクラスにまとめてはいけないのですか？
```

```
> このメソッドが配列を返している設計上の理由を教えてください。この配列はどこで使われますか？
```

ポイントは、コードの内容ではなく **設計の意図** を聞くことです。「このコードは何をしていますか」という質問は表面的な説明を返すだけですが、「なぜこうしたのですか」という質問は、その技術の設計思想や背景にある考え方を引き出します。

### ステップ 2: 公式ドキュメントで裏取りする

Claude の説明を聞いたら、次は **公式ドキュメントで裏取り** します。Claude は正確な情報を提供しますが、それでも裏取りは習慣にしてください。理由は2つあります。

1つ目は、Claude の知識には時点がある（最新の仕様変更を反映していない場合がある）こと。2つ目は、公式ドキュメントを読むこと自体が理解を深めるプロセスになること。Claude の説明で大枠を掴んでから公式ドキュメントを読むと、いきなりドキュメントを読むよりもはるかに効率的に理解できます。

```
> Laravel の Notifications の公式ドキュメントの URL を教えてください。
```

Claude が教えてくれた URL をブラウザで開き、実装と照らし合わせてみてください。

### ステップ 3: テストを書いて動作を確認する

概念を理解したら、**テストで動作を確認** します。テストは「自分が理解した通りに動くか」を検証する最も確実な方法です。

```
> 通知が正しく送信されることを確認するテストを書いてください。通知の種類ごとの ON/OFF 設定が反映されることもテストに含めてください。
```

テストが通れば、少なくとも「自分が定義した振る舞い」は満たしています。テストが失敗すれば、自分の理解か実装のどちらかに問題があることが分かります。

2-6-1 で学んだテストの知識は、知らない技術でも同じように使えます。テスト自体は Laravel の PHPUnit の仕組みなので、皆さんの知識の範囲内です。「知らない技術で実装されたコードを、知っている技術（テスト）で検証する」。これが、知らない技術への有効なアプローチです。

### ステップ 4: 理解できない部分を明確にする

ステップ 1〜3 を経ても、すべてを理解できるとは限りません。ここで大切なのは、**何が分からないかを明確にする** ことです。

「なんとなく分からない」と「この部分の動作原理が分からないが、テストで動作は確認できている」では、リスクの質が全然違います。後者であれば、動作は保証されているので、理解が追いつくまでそのまま進めるという判断もできます。

すべてを完璧に理解してからでないと先に進めない、という姿勢では実務は回りません。**「何が分かっていて、何が分かっていないか」を自分で把握していること** が重要です。

## 思考の放棄との違い

ここで改めて確認しておきます。「分からないけどテストが通るから使う」は、思考の放棄ではないのか？

違いは明確です。

**思考の放棄**: 分からない → そのまま受け入れる（検証なし）

**このアプローチ**: 分からない → 質問して概念を理解する → 公式ドキュメントで裏取りする → テストで動作を確認する → 残りの「分からない」を明確にする → 判断して受け入れる

プロセスを経ているかどうかの違いです。全部を理解していなくても、**理解しようとした上で、自分の判断で採用している** のであれば、それは思考の放棄ではありません。

Thoughtworks のエンジニアは、AI を使った開発について「AI があるときこそ、もっと考える必要がある（We need to think *more*, not less）」と述べています。これは知らない技術を扱うときに特に当てはまります。Claude Code が生成してくれるからこそ、「なぜそうなっているのか」を考え、理解するプロセスが重要になるのです。

## まとめ

- 知っている技術では「コードを読んで判断する」検証ができますが、知らない技術では「理解する」ステップを先に挟む必要があります
- AI の使い方で結果は大きく変わります。コード生成を丸投げするのではなく、「なぜそうしたのか」を質問し、概念を理解してから検証するアプローチが有効です
- 検証の4ステップ（設計理由を質問する → 公式ドキュメントで裏取り → テストで動作確認 → 分からない部分を明確にする）を次のセクションで実践します
- すべてを完璧に理解する必要はありません。「何が分かっていて、何が分かっていないか」を把握した上で判断することが、思考の放棄との違いです
