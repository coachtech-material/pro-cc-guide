# 2-11-3 外部 API 連携で検証をさらに実践する

## 概要

> 💡 このセクションは任意です。Claude API の利用には API 料金が発生します。スキップしても Part 3 に影響しません。

前のセクションでは、Laravel Notifications を題材に4つの検証ステップを実践しました。このセクションでは、さらに馴染みの薄い領域である **外部 API 連携** で同じステップを繰り返します。

題材は、テック記事プラットフォームに **AI 記事アシスタント** を追加することです。Claude API を使って、記事の下書きに対するフィードバック（文章の改善提案、技術的な誤りの指摘、読みやすさのアドバイス）を取得する機能を実装します。

外部 API 連携には、これまで触れていない技術要素が複数含まれます。

- **API キーの管理**: 環境変数による秘密情報の管理
- **Service クラスパターン**: API 呼び出しをコントローラーから分離する設計
- **HTTP クライアント**: 外部 API へのリクエスト送信
- **API モックテスト**: テスト時に実際の API を呼ばずにテストする方法
- **エラーハンドリング**: ネットワークエラーやレート制限への対応

「知らない技術」が1つではなく複数重なる場面です。2-11-2 と同じ検証ステップがここでも使えることを確認しましょう。

## 外部 API 連携の実装

### Claude API のセットアップ

まず、Claude API を利用するための API キーを取得します。Anthropic のコンソール（`console.anthropic.com`）にアクセスし、API キーを作成してください。

取得した API キーを `.env` ファイルに追加します。

```
ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxxxxx
```

> ⚠️ よくあるエラー: API キーの取り扱い
>
> API キーは **絶対にコードに直接書かない** でください。`.env` ファイルに記載し、`config()` 経由でアクセスします。`.env` は `.gitignore` に含まれているため、Git にコミットされません。もし誤ってコミットしてしまった場合は、すぐに API キーを無効化（revoke）してください。
>
> **原因**: API キーが公開リポジトリに含まれると、第三者に悪用され、意図しない課金が発生する可能性があります。
>
> **対処法**: API キーは環境変数で管理する。`.env.example` にはキー名のみ記載し、値は空にしておく。

### Claude Code に実装を依頼する

Claude Code には `/claude-api` という Bundled Skill があります。このスキルを呼び出すと、Claude API のリファレンスが自動で読み込まれ、正確な API 実装を生成しやすくなります。

```
> /claude-api
```

スキルが読み込まれたら、実装を依頼します。

```
> テック記事プラットフォームに AI 記事アシスタント機能を追加してください。
> - 記事の下書き画面に「AIレビューを受ける」ボタンを追加
> - ボタンを押すと、記事の本文を Claude API に送信し、フィードバックを取得
> - フィードバックは「文章の改善提案」「技術的な正確性」「読みやすさ」の3観点で表示
> - API 呼び出しは Service クラスに分離してください
> - .env の ANTHROPIC_API_KEY を使用してください
> - API エラー時のハンドリングも実装してください
```

生成されたコードをすべて承認してください。ここから、検証ステップを適用します。

### ステップ 1: 設計判断の理由を質問する

外部 API 連携では、聞くべきことが多くなります。

```
> 今作成した AI アシスタント機能について質問です。
> 1. なぜ API 呼び出しを Service クラスに分離したのですか？コントローラーに直接書くのと何が違いますか？
> 2. HTTP クライアントに Laravel の Http ファサードを使っている理由は何ですか？PHP の curl や Guzzle を直接使うのとどう違いますか？
> 3. API のエラーハンドリングで retry やタイムアウトを設定している理由は何ですか？
> 4. テストで Http::fake() を使っている理由は何ですか？実際の API を呼んではいけないのですか？
```

Claude の回答を読んでみてください。ここでは複数の技術要素が絡み合っています。

- **Service クラスパターン**: コントローラーの責務を「HTTP リクエストの受付とレスポンスの返却」に限定し、ビジネスロジック（API 呼び出し）を分離する。テストしやすさ、再利用性、可読性が向上する
- **Laravel Http ファサード**: Laravel が提供する HTTP クライアントのラッパー。`Http::fake()` によるテスト時のモックが容易。Guzzle をベースにしている
- **retry / タイムアウト**: 外部 API はネットワークの状態や相手サーバーの負荷で失敗することがある。一時的な障害に対するリトライと、応答がない場合のタイムアウトは外部 API 連携の基本
- **Http::fake()**: テスト時に実際の API を呼ぶと、テストの結果が外部要因（ネットワーク状態、API の応答内容）に依存してしまう。`Http::fake()` でレスポンスを固定することで、テストの再現性を確保する

### ステップ 2: 公式ドキュメントで裏取りする

外部 API 連携では、確認すべきドキュメントが複数あります。

```
> 以下の公式ドキュメントの URL を教えてください。
> - Laravel の HTTP Client
> - Laravel の HTTP Client のテスト（faking）
> - Anthropic の Claude API リファレンス
```

それぞれのドキュメントを開いて、実装と照らし合わせてみてください。

特に確認したいのは以下の点です。

- **HTTP Client**: `Http::post()` の引数とレスポンスの取り方が公式ドキュメントと一致しているか
- **Http::fake()**: テストでのモック方法が公式の推奨に従っているか
- **Claude API**: リクエストのフォーマット（`model`、`messages`、`max_tokens` 等）が最新の仕様と一致しているか

### ステップ 3: テストを書いて動作を確認する

```
> AI アシスタント機能のテストを確認させてください。以下がカバーされていますか？
> - API からフィードバックを正常に取得できること
> - API キーが設定されていない場合にエラーメッセージが表示されること
> - API がタイムアウトした場合に適切なエラーハンドリングが行われること
> - API がエラーレスポンスを返した場合の処理
> - 下書き状態の記事でのみ AI レビュー機能が利用できること
> 不足しているテストがあれば追加してください。
```

テストを実行します。

```bash
sail artisan test --filter=AiAssistant
```

外部 API 連携のテストでは `Http::fake()` によるモックが使われています。これは 2-11-2 のデータベース通知とは違うテスト手法です。実際の API を呼ばないことで、テストの速度と再現性を確保しています。

### ステップ 4: 理解できない部分を明確にする

外部 API 連携は技術要素が多いため、「分かったこと」と「分からないこと」を意識的に整理してみてください。

**理解できたこと** の例:
- Service クラスに API 呼び出しを分離する意図
- `Http::fake()` でテスト時にモックする仕組み
- API キーを `.env` で管理する理由

**まだ理解できていないかもしれないこと** の例:
- `Http` ファサードの内部実装（Guzzle との関係）
- API のレート制限に引っかかった場合の本番環境での対処法
- キューを使った非同期 API 呼び出しのパターン

2-11-2 と同じように、テストで動作は確認できています。「Service クラスパターンの意図は分かった。Http ファサードの内部実装はまだ追えていないが、テストで動作は確認できている。本番環境のレート制限対策は、実際にその問題に直面したときに調べる」。このように、理解度を段階的に把握できていれば十分です。

### 動作確認

ブラウザで動作を確認します。

1. 著者としてログインし、下書き状態の記事を作成する
2. 記事の編集画面で「AI レビューを受ける」ボタンを押す
3. フィードバックが3つの観点で表示されることを確認する
4. 公開済みの記事では AI レビューボタンが表示されない（または無効化されている）ことを確認する

💡 ここまでの変更をコミットしておきましょう。

```
> /commit
```

## 検証ステップの振り返り

2-11-2（Notification）と 2-11-3（外部 API 連携）の2つの実践を通じて、同じ4ステップを異なる技術領域で使いました。

振り返ってみると、ステップ自体はシンプルです。

1. **「なぜ？」と聞く**: コードの意図を理解する
2. **公式ドキュメントを読む**: Claude の説明を裏取りする
3. **テストで確かめる**: 動作を検証する
4. **分からない部分を明確にする**: 判断の根拠を持つ

このステップは、知らない技術に限らず使えます。知っている技術でも、Claude Code が予想外の実装をしたとき、「なぜそうしたのか」を聞くことでより良い実装に気づくことがあります。

大切なのは、このステップを **習慣にする** ことです。Part 3 では既存プロジェクトの実務タスクに取り組みますが、そこでも馴染みのないコードベースやライブラリに出会うはずです。そのとき、このステップが自然に使えるかどうかが、皆さんの「見極める力」の幅を広げます。

## 見極めチェック

- [ ] **正しさ**: AI レビュー機能がフィードバックを正常に返すか。Claude API のリクエストフォーマット（model, messages, max_tokens）が公式仕様と一致しているか。下書き状態の記事でのみ機能が利用できるか
- [ ] **品質**: API 呼び出しが Service クラスに分離されているか。タイムアウトやリトライが設定されているか。テストで `Http::fake()` を使い、外部依存なく実行できるか
- [ ] **安全性**: API キーがコードにハードコードされていないか（`.env` 経由で取得しているか）。API キーが `.gitignore` 対象の `.env` にのみ記載されているか。ユーザー入力をそのまま API に送信していないか（プロンプトインジェクション対策）

> このセクションでは特に「正しさ」に注目してください。外部 API 連携では、自分の知識だけでは正しさを判断しにくい部分が多くあります。Claude に質問し、公式ドキュメントで確認し、テストで動作を検証する。この3段階の検証プロセスを経て「正しい」と判断できたか、振り返ってみてください。

## まとめ

- 外部 API 連携（Claude API）という馴染みの薄い領域で、2-11-2 と同じ検証ステップを実践しました。Service クラスパターン、HTTP クライアント、API モックテストなど、複数の「知らない技術」が重なる場面でもステップは有効です
- `/claude-api` Bundled Skill を使うと、Claude Code に Claude API のリファレンスが読み込まれ、正確な実装を生成しやすくなります
- API キーの管理は安全性の観点で特に重要です。`.env` での管理、`.gitignore` への記載、コードへのハードコード禁止を徹底してください
- 知らない技術への向き合い方（質問する → 裏取りする → テストする → 分からない部分を明確にする）は、技術領域を問わず使える汎用的なアプローチです。Part 3 の実務タスクでも、この習慣を活かしてください
