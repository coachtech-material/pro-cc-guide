# 2-3-2 Skills

## 🎯 このセクションで学ぶこと

- Skills の仕組みと、定型作業を効率化する意義を理解する
- Skill の構造（SKILL.md、frontmatter）と配置場所を把握する
- 自分のプロジェクトに Skill を1つ作成する

まず Skills の仕組みと定型作業を効率化する意義を理解し、次に Skill の構造と配置場所を学び、最後に cc-practice で Skill を作成・使用・改善します。

---

## 導入: 毎回同じ指示を書いていませんか

Claude Code を使い始めると、似たような指示を繰り返し書いていることに気づきます。たとえば、コードレビューを頼むたびに「セキュリティの観点も含めて、OWASP Top 10 を意識してレビューして。変更点ごとに問題があれば指摘して」と書く。テストを書くたびに「既存のテストファイルのパターンに合わせて、正常系と異常系の両方をカバーして」と書く。

これは非効率です。同じ指示を何度も書くと、書き忘れや表記揺れが起きます。「前回はセキュリティの観点を含めたのに、今回は忘れた」「正常系と異常系と書いたり、正常ケースと異常ケースと書いたり」といった不一致が生じます。

**Skills** は、こうした繰り返しの指示を **再利用可能なカスタムコマンド** として定義する仕組みです。一度作っておけば、スラッシュコマンド（`/skill-name`）で呼び出すだけで、毎回同じ品質の指示を Claude Code に渡せます。

> 📝 **`/clear` などのビルトインコマンドとの違い**: これまでに登場した `/clear` や `/compact`、`/help` なども同じ `/` で始まる形式です。これらは Claude Code 本体に組み込まれた **ビルトインコマンド** で、固定のロジックが直接実行されます。一方 Skills は SKILL.md に書いたプロンプトを Claude Code が読み込んで動かす **プロンプトベース** の仕組みです。なお以前は Skills に相当する機能が「カスタムコマンド」（`.claude/commands/` 配下の `.md` ファイル）として提供されていましたが、現在は Skills に統合されています。呼び出し方は同じ `/` ですが、「本体の機能」と「自分で定義した手順書」の2種類があると理解してください。

### 🧠 先輩エンジニアはこう考える

> Skills は「自分だけの作業マニュアル」をコードとして管理するようなものだ。実務では、レビューのチェックリストやデプロイの手順書をドキュメントにまとめることが多い。Skills もそれと同じ発想で、「自分がいつもやっている作業の手順」を Skill として定義しておけば、新しいプロジェクトでもすぐに同じ品質で作業できる。しかも、Git で管理できるからチームで共有もできる。

## Skills とは

Skills は、Claude Code の振る舞いを拡張するための仕組みです。実態は **SKILL.md** というマークダウンファイルで、Claude Code への指示（プロンプト）を事前に定義しておきます。

Skills には2つの呼び出し方があります。

### 手動呼び出し（スラッシュコマンド）

入力欄で `/` に続けて Skill 名を入力すると、その Skill が呼び出されます。

```
> /review
```

Tab キーで補完もできます。`/` と入力すると利用可能な Skills の一覧が表示されるので、選択するだけで呼び出せます。

### 自動呼び出し

Skill の `description` フィールド（後述）に記載された条件に一致する会話が発生すると、Claude Code が自動的にその Skill を読み込みます。たとえば、「コードレビューをして」という会話で自動的にレビュー用の Skill が読み込まれる、という動作です。

> 💡 自動呼び出しを無効にしたい場合は、frontmatter に `disable-model-invocation: true` を設定します。

## Skill の構造

Skill は、**フォルダの中に SKILL.md ファイルを配置する** という構造です。

```
.claude/skills/
└── review/          ← Skill 名（フォルダ名）
    └── SKILL.md     ← Skill の定義ファイル
```

SKILL.md は、YAML frontmatter とマークダウン本文で構成されます。

```markdown
---
name: review
description: コードレビューを実行する。セキュリティ、パフォーマンス、可読性の観点でチェックする
---

# コードレビュー

以下の観点で変更されたコードをレビューしてください。

## チェック観点

1. **セキュリティ**: SQLインジェクション、XSS、CSRF の脆弱性がないか
2. **パフォーマンス**: N+1 クエリ、不要なループ、キャッシュの考慮
3. **可読性**: 命名規則、関数の責務分離、コメントの適切さ

## 出力形式

- ファイルごとに問題点を列挙する
- 各問題に「重要度（高/中/低）」を付ける
- 修正案を提示する
```

### frontmatter の主要フィールド

| フィールド | 必須 | 説明 |
|---|---|---|
| `name` | はい | Skill の識別名。スラッシュコマンドで使う名前（小文字、ハイフン区切り） |
| `description` | はい | Skill の用途説明。Claude Code が自動呼び出しの判断に使うため、具体的に書く |
| `disable-model-invocation` | いいえ | `true` にすると自動呼び出しを無効化。手動呼び出しのみになる |
| `user-invocable` | いいえ | `false` にするとスラッシュコマンドのメニューに表示しない |
| `model` | いいえ | この Skill の実行に使うモデルを指定（例: `sonnet`） |
| `allowed-tools` | いいえ | この Skill が使えるツールを制限する |

> 📝 `description` は非常に重要です。自動呼び出しの精度は `description` の書き方に大きく左右されます。「いつ使うべきか」を具体的に書くことで、意図しないタイミングでの呼び出しを防げます。

## Skill の配置場所

Skills は配置場所によってスコープ（適用範囲）が異なります。

| 配置場所 | スコープ | 共有 |
|---|---|---|
| `.claude/skills/<skill-name>/SKILL.md` | プロジェクト | Git 経由でチーム共有可能 |
| `~/.claude/skills/<skill-name>/SKILL.md` | ユーザー全体 | 自分の全プロジェクトで使える |

**プロジェクトスコープ**（`.claude/skills/`）は、そのプロジェクト固有の Skill を定義するのに使います。プロジェクトのコーディング規約に沿ったレビュー Skill や、プロジェクト特有のデプロイ手順を定義する場合です。Git で管理できるため、チームメンバーと共有できます。

**ユーザースコープ**（`~/.claude/skills/`）は、プロジェクトを問わず使いたい Skill を定義するのに使います。自分の開発スタイルに合わせた汎用的な Skill を置くのに適しています。

> 💡 同名の Skill が複数のスコープにある場合、優先順位は Enterprise > Personal（ユーザー） > Project（プロジェクト） > Plugin の順です。Personal（`~/.claude/skills/`）が Project（`.claude/skills/`）より優先される点に注意してください。

## Skill で使える便利な機能

### 引数の受け渡し

スラッシュコマンドに引数を渡すと、Skill 内で `$ARGUMENTS` として参照できます。

```
> /review セキュリティに特に注意して
```

SKILL.md 内で以下のように使います。

```markdown
---
name: review
description: コードレビューを実行する
---

変更されたコードをレビューしてください。

追加の指示: $ARGUMENTS
```

`$ARGUMENTS` の部分が「セキュリティに特に注意して」に置き換わります。

### 動的コンテキストの注入

Skill の中でシェルコマンドを実行し、その結果を Claude Code に渡すこともできます。`` !`<command>` `` 構文を使います。

```markdown
---
name: status
description: プロジェクトの現在の状態を報告する
---

以下の情報をもとにプロジェクトの状態を報告してください。

## Git の状態
!`git status --short`

## 最近のコミット
!`git log --oneline -5`
```

Skill が呼び出されたとき、バッククォート内のコマンドが実行され、その結果が Claude Code に渡されます。

## ビルトイン Skills

Claude Code にはあらかじめ組み込まれた Skills があります。

| Skill | 説明 |
|---|---|
| `/batch` | 大規模な変更を並列で処理する |
| `/claude-api` | Claude API のリファレンスを読み込む |
| `/debug` | Claude Code セッションのトラブルシューティング |
| `/loop` | プロンプトを定期的に実行する |
| `/simplify` | 変更したコードの品質を確認する |

これらは Claude Code をインストールした時点で使えます。`/` と入力して一覧を確認してみてください。

## 🏃 実践: Skill を作成・使用・改善する

cc-practice に、Laravel のコードを解説する Skill を作成し、使って、改善するサイクルを体験しましょう。

### Step 1: Skill を作成する

Claude Code を起動し、以下のように指示します。

```
> .claude/skills/explain/ ディレクトリに SKILL.md を作成して。内容は以下の通り:
> - name: explain
> - description: 指定されたファイルのコードを初心者向けに解説する
> - 本文: 「$ARGUMENTS で指定されたファイルを読み、初心者にもわかるように日本語で解説してください。Laravel の規約やデザインパターンに関連する部分は特に丁寧に説明してください。各メソッドの役割を説明し、重要なポイントには💡を付けてください。」
```

Claude Code が `.claude/skills/explain/SKILL.md` を作成します。

> 📝 あなたの環境では異なるファイル内容が生成される場合がありますが、frontmatter に `name` と `description` が含まれ、本文に指示が書かれていれば問題ありません。

### Step 2: Skill を使ってみる

作成した Skill を、2-3-1 で作成した DailyReportController に対して使ってみましょう。

```
/explain app/Http/Controllers/DailyReportController.php
```

Claude Code が `explain` Skill を読み込み、DailyReportController の内容を初心者向けに解説してくれます。スラッシュコマンド1つで、毎回同じ品質の解説が得られるようになりました。

### Step 3: Skill を改善する

出力を確認して、改善の余地がないか考えてみましょう。たとえば「バリデーションの流れも説明してほしい」「関連するファイル（Model、FormRequest）も合わせて解説してほしい」といった改善点が見つかるかもしれません。

```
> .claude/skills/explain/SKILL.md を改善して。
> 対象ファイルが Controller の場合は、関連する Model や FormRequest も読んで、全体の流れを解説するようにして。
> また、解説の最後に「🔍 確認ポイント」として、このコードをレビューするときに注目すべき点を3つ挙げるようにして
```

改善された Skill で再度試してみましょう。

```
/explain app/Http/Controllers/DailyReportController.php
```

1回目と比べて、解説の質が向上していることを確認してください。

### Step 4: Skill の一覧と成果物を確認する

まず `/` と入力して Tab キーを押し、利用可能な Skills の一覧を表示します。先ほど作成した `explain` が一覧に含まれていることを確認してください。

Claude Code に直接聞く方法もあります。

```
> 利用可能な Skill を一覧で教えて
```

`explain` を含む Skill 一覧が返ってきます。以下の方法でファイル本体も確認しておきましょう。

```bash
! cat .claude/skills/explain/SKILL.md
```

Skill のファイルが cc-practice に残っていることを確認しましょう。この Skill は今後の学習でも使えます。

> 💡 **Skill 作成を支援するプラグインもあります**: Anthropic が公開している `skill-creator` プラグインを導入すると、`/skill-creator:skill-creator` で Skill の新規作成・改善・評価を対話的に進められます。本教材ではまず SKILL.md を手書きする基本の流れを押さえますが、Skill を本格的に使い込む段階ではこのプラグインを試してみると効率が上がります。プラグインの導入方法は 2-3-6 で扱います。

## 🔍 見極めチェック

> 🧠 先輩エンジニアの思考: 「Skill は一度作ったら何度も使う。だからこそ、最初の設計が重要だ。description が曖昧だと意図しないタイミングで自動呼び出しされるし、指示内容が不十分だと毎回期待外れの結果になる。」

- [ ] **正しさ**: frontmatter の `name` と `description` が正しく記述されているか。スラッシュコマンドで呼び出せるか
- [ ] **品質**: `description` が具体的か。自動呼び出しの判定に使えるレベルの記述になっているか
- [ ] **安全性**: `description` が曖昧すぎないか。曖昧な description は、意図しない場面で Skill が自動呼び出しされる原因になる（例: 「コードを見る」のような汎用的な description だと、関係ないタスクでも呼び出される可能性がある）

> 🔑 この Section では特に「description の具体性」に注目してください。

## ✨ まとめ

- **Skills** は繰り返しの指示を再利用可能なカスタムコマンドとして定義する仕組み
- SKILL.md ファイルに frontmatter（name、description）と本文（指示内容）を記述する
- 配置場所によってスコープが決まる（プロジェクト: `.claude/skills/`、ユーザー: `~/.claude/skills/`）
- スラッシュコマンド（`/skill-name`）での手動呼び出しと、会話に基づく自動呼び出しがある
- `$ARGUMENTS` で引数を受け取り、`` !`command` `` で動的コンテキストを注入できる

---

次のセクションでは、「ファイルを保存するたびに手動でフォーマッターを走らせる」といった面倒を自動化する **Hooks** を学びます。ツールの実行に連動して自動的に処理を走らせる仕組みです。
