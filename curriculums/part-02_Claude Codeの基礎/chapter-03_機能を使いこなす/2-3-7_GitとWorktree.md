# 2-3-7 Git と Worktree

## 🎯 このセクションで学ぶこと

- Claude Code の Git 連携機能と、チェックポイントによるセーフティネットを理解する
- Rewind 機能で変更を安全に巻き戻す方法を覚える
- Worktree による並列開発の仕組みを理解する

まず Claude Code の Git 連携機能とチェックポイントの仕組みを理解し、次に Worktree による並列開発の方法を学び、最後に cc-practice で Rewind と Worktree を体験します。

---

## 導入: Claude Code に「やっぱり元に戻して」と言いたくなったら

Claude Code でコードを生成していると、「この変更はやっぱり違った」「さっきの状態に戻したい」と思う場面が必ず出てきます。

手動でコードを書いているなら、Ctrl + Z で取り消したり、`git stash` で一時退避したりできます。しかし Claude Code は、1つの指示で複数のファイルを同時に変更することがあります。「どのファイルのどの部分を元に戻せばいいのか」を自分で把握するのは困難です。

ここで Claude Code の **Git 連携** と **チェックポイント** が力を発揮します。Claude Code は Git と深く統合されており、変更の追跡、巻き戻し、ブランチ操作を会話の中で自然に行えます。さらに、自動的に変更のスナップショットを記録するチェックポイント機能により、いつでも安全に過去の状態に戻れます。

### 🧠 先輩エンジニアはこう考える

> Claude Code を使い始めた頃、「大丈夫かな」と不安になりながらコード生成を頼んでいた時期がある。特に既存のコードを変更するとき、「壊れたらどうしよう」と。でもチェックポイントの仕組みを知ってからは、安心して「とりあえず試してみて」と言えるようになった。失敗しても一瞬で元に戻せるとわかっていれば、思い切った変更も試せる。これは Git を使うようになって「いつでも戻せるから安心」と感じるのと同じ感覚だ。

## Claude Code の Git 連携

Claude Code は Git の操作を会話の中で自然に行えます。

### 基本的な Git 操作

Claude Code に依頼できる Git 操作の例です。

```
> 変更をコミットして。メッセージは「ユーザー管理機能を追加」で
```

```
> 新しいブランチ feature/user-auth を作成して、そのブランチに切り替えて
```

```
> 今の変更の差分を見せて
```

```
> この変更で PR を作成して
```

Claude Code はこれらの指示を受けて、`git add`、`git commit`、`git checkout`、`git diff` などのコマンドを実行します。コマンドを1つずつ覚えていなくても、自然言語で Git 操作ができます。

### GitHub CLI（gh）との連携

PR の作成や Issue の操作には、**GitHub CLI（`gh`）** というコマンドラインツールが使われます。Claude Code は `gh` コマンドを通じて GitHub とやり取りします。

```
> この変更で PR を作成して
```

と指示すると、Claude Code は内部で `gh pr create` を実行します。次のセクション（2-3-8 GitHub Actions）でも `gh` を使うため、まだインストールしていない場合はここで準備しておきましょう。

**インストール:**

```bash
# macOS
brew install gh

# Windows
winget install GitHub.cli
```

**認証:**

```bash
gh auth login
```

ブラウザが開くので、GitHub アカウントでログインして認証を許可します。

```bash
# 認証の確認
gh auth status
```

> 📝 `gh` は GitHub が公式に提供する CLI ツールです。Pro生 はブラウザで GitHub を使っていると思いますが、`gh` を使うとターミナルから PR の作成、Issue の操作、リポジトリの管理ができます。Claude Code と組み合わせると、ブラウザを開かずに GitHub のワークフローが完結します。

### Git の状態を自動で把握

Chapter 2-2 で学んだように、Claude Code はセッション開始時に Git の状態を自動的に把握しています。

- 現在のブランチ
- コミットされていない変更
- 直近のコミット履歴

これにより、「今の変更をコミットして」と指示するだけで、Claude Code は何が変更されているかを理解し、適切にステージングとコミットを行えます。

## チェックポイントと Rewind

Git 連携に加えて、Claude Code 独自の変更追跡機能が **チェックポイント** と **Rewind** です。これらの仕組みは 2-2-2「コンテキストとセッション管理」で詳しく解説しました。ここでは実践での使い方に焦点を当てます。

おさらいすると、チェックポイントはプロンプトを送信するたびに自動作成されるスナップショットで、`Esc` を2回押す（または `/rewind`）で任意の時点にコードと会話を巻き戻せます。Bash コマンドによる変更や外部からの変更は追跡されない点、Git の代替ではない点に注意してください（詳細は 2-2-2 参照）。

## Worktree: 並列開発

実務では、「バグ修正をしながら新機能の開発も進めたい」「2つの異なるアプローチを並行して試したい」という場面があります。

### 通常の Git ではなぜ困るのか

普段の Git 運用を思い出してください。1つのリポジトリには **1つの作業ディレクトリ** しかありません。`feature/auth` ブランチで認証機能を開発しているときに、緊急のバグ修正が入ったら、以下のような手順が必要です。

```
① 作業中の変更を退避（git stash）
        │
        ▼
② バグ修正用のブランチに切り替え（git checkout hotfix/login）
        │
        ▼
③ バグを修正してコミット
        │
        ▼
④ 元のブランチに戻る（git checkout feature/auth）
        │
        ▼
⑤ 退避した変更を復元（git stash pop）
```

ブランチを切り替えるたびに、作業ディレクトリの中身が丸ごと入れ替わります。変更を退避し忘れたり、復元時にコンフリクトが起きたりするリスクがあります。

### Git worktree という仕組み

Git には、この問題を解決する **worktree** という仕組みがあります。1つのリポジトリから **複数の作業ディレクトリ** を作成できる機能です。

```
通常の Git                     Git worktree を使った場合
┌────────────────────┐         ┌────────────────────┐
│ cc-practice/       │         │ cc-practice/       │
│ └── main ブランチ  │         │ └── feature/auth   │
│                    │         └────────────────────┘
│ ブランチを切り替え │          ↑ メインの作業ディレクトリ
│ るとファイルが     │
│ 丸ごと入れ替わる   │         ┌────────────────────┐
│                    │         │ cc-practice-fix/   │
└────────────────────┘         │ └── hotfix/login   │
                               └────────────────────┘
 ↑ 1つの作業ディレクトリ        ↑ 2つ目の作業ディレクトリ
   を使い回す                     が同時に存在する
```

各作業ディレクトリは独立しています。片方でファイルを変更しても、もう片方には影響しません。しかし **Git の履歴やリモート接続は共有** されているので、リポジトリを丸ごとコピーするのとは違います。

> 📝 `git worktree` は Git 2.5 以降で利用できる標準機能ですが、あまり知られていません。普段の開発で使う必要はありませんが、Claude Code と組み合わせると非常に便利になります。

### Claude Code と Worktree

Claude Code はこの Git worktree の仕組みを、`--worktree` フラグで簡単に使えるようにしています。`git worktree add` コマンドを覚える必要はありません。

```
┌─ Claude Code の Worktree ──────────────────────────────┐
│                                                         │
│  cc-practice/                                           │
│  ├── main ブランチで作業中                              │
│  │                                                      │
│  └── .claude/worktrees/                                 │
│      ├── feature-auth/        ← Worktree①              │
│      │   worktree-feature-auth ブランチ                 │
│      │   （認証機能を開発中）                            │
│      │                                                  │
│      └── bugfix-login/        ← Worktree②              │
│          worktree-bugfix-login ブランチ                  │
│          （ログインバグを修正中）                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

各 Worktree は以下の特性を持ちます。

- **独立したファイル**: 他の Worktree やメインリポジトリの変更に影響しない
- **独立したブランチ**: 自動的に専用ブランチが作成される
- **共有された履歴**: 同じリポジトリの Git 履歴とリモート接続を共有する

### Worktree の作成

`--worktree` フラグを付けて Claude Code を起動します。

```bash
# 名前を指定して Worktree を作成
claude --worktree feature-auth

# 名前を省略するとランダムな名前が付く
claude --worktree
```

セッション中に Worktree を作成することもできます。

```
> 新しい Worktree で認証機能の開発を始めて
```

### Worktree のクリーンアップ

Worktree のセッションを終了すると、Claude Code がクリーンアップを提案します。

- **変更なし**: Worktree とブランチが自動的に削除される
- **変更あり**: 保持するか削除するかを選択できる
  - **保持**: ディレクトリとブランチがそのまま残る。後で `claude --worktree <name>` で再開できる
  - **削除**: Worktree ディレクトリとブランチが削除される。コミットされていない変更は失われる

> 💡 `.claude/worktrees/` を `.gitignore` に追加しておくと、Worktree の内容が Git の未追跡ファイルとして表示されるのを防げます。

### Sub-agent と Worktree の組み合わせ

前のセクションで学んだ Sub-agents は、Worktree と組み合わせることもできます。Sub-agent の frontmatter に `isolation: worktree` を指定すると、Sub-agent が独自の Worktree 内で作業します。

```markdown
---
name: experimental
description: 実験的な変更を試す
isolation: worktree
---

指示された変更を試してください。メインのコードに影響を与えません。
```

この Sub-agent が変更を加えると、その変更は Worktree 内に閉じ込められます。メインのリポジトリには影響しません。変更がなければ Worktree は自動的にクリーンアップされます。

## 🏃 実践: チェックポイントと Rewind、Worktree を体験する

cc-practice で、チェックポイントからの Rewind と Worktree による並列作業を体験してみましょう。

### Step 1: 日報一覧ページのデザインを変更する

Claude Code を起動します。2-3-1 の Plan Mode 実践で日報の Web CRUD 画面を作成しました。その一覧ページ（`resources/views/daily-reports/index.blade.php`）のデザインを変更してみましょう。

```
> resources/views/daily-reports/index.blade.php のデザインを変更して。テーブルに背景色のストライプ（奇数行・偶数行で色を変える）を追加して、ステータスに応じてバッジの色を変えて（下書き: グレー、提出済: 緑）
```

### Step 2: 続けてレイアウトを変更する

続けてさらにデザインを変更します。

```
> daily-reports/index.blade.php のレイアウトをカード形式に変更して。各日報を枠で囲み、日付を大きく表示するデザインにして
```

### Step 3: Rewind で巻き戻す

ここで「やっぱりカード形式よりテーブル形式の方が見やすかった」とします。

`Esc` キーを2回押して Rewind メニューを開きます。

<!-- TODO: 画像追加 - Rewind メニューの画面 -->

チェックポイントの一覧が表示されます。Step 1 の直後（ストライプを追加した時点）のチェックポイントを選択し、「**Restore code and conversation**」を選んでください。

コードと会話の両方が、カード形式に変更する前の状態に巻き戻されます。ブラウザで `http://localhost/daily-reports` を再読み込みして、ストライプのテーブル形式に戻っていることを確認してください。

このように、チェックポイントを使えば「やり直し」のコストがほぼゼロになります。「とりあえず試してみて、ダメなら戻す」というサイクルを気軽に回せるのが、チェックポイントの最大の価値です。

### Step 4: ここまでの変更をコミットする

Worktree を体験する前に、ここまでの変更をコミットしておきましょう。

```
> ここまでの変更をコミットして。メッセージは「日報一覧ページのデザインを改善」で
```

### Step 5: Worktree で実験的な変更を試す

続けて、Worktree による並列作業を体験します。Claude Code を一度終了し、`--worktree` フラグ付きで起動します。

```bash
claude --worktree experiment
```

新しい Worktree が作成され、`worktree-experiment` ブランチで作業が始まります。

```
> daily-reports/index.blade.php にステータスで絞り込むフィルター機能を追加して。「すべて」「下書きのみ」「提出済のみ」を選べるようにして
```

ファイルが変更されたら、Claude Code を終了します（Ctrl + D）。Worktree のクリーンアップについて質問されるので、「保持」を選択してください。

### Step 6: Worktree の分離を確認する

元のディレクトリで、ファイルが影響を受けていないことを確認します。

```bash
cat resources/views/daily-reports/index.blade.php
```

フィルター機能は追加されていません。Worktree 内での変更はメインのリポジトリに影響していないことが確認できます。実験的な変更を安全に試せるのが Worktree の価値です。

## 🔍 見極めチェック

> 🧠 先輩エンジニアの思考: 「チェックポイントと Git コミットの使い分けが重要だ。チェックポイントはセッション内の安全ネットで、30日で消える。本当に残したい変更は必ず Git でコミットする。」

- [ ] **正しさ**: Rewind 後のファイル内容が期待通りに巻き戻っているか
- [ ] **品質**: 該当なし（Claude Code の機能の体験であり、生成コードの品質は問わない）
- [ ] **安全性**: 重要な変更をチェックポイントだけに頼っていないか。永続化が必要な変更は Git でコミットしているか

> 🔑 この Section では特に「チェックポイントと Git コミットの使い分け」に注目してください。

## ✨ まとめ

- Claude Code は Git と深く統合されており、自然言語でコミット、ブランチ操作、PR 作成ができる
- **チェックポイント** はプロンプトごとに自動作成されるスナップショットで、手動操作は不要
- **Rewind**（`Esc` × 2 または `/rewind`）で任意の時点に巻き戻せる。コードと会話の両方、またはどちらか一方を選択できる
- Bash コマンドや外部からの変更はチェックポイントに記録されない点に注意
- **Worktree** は独立した作業ディレクトリで、複数の Claude Code セッションを並行して動かせる
- `claude --worktree <name>` で Worktree を作成し、終了時にクリーンアップを選択する
