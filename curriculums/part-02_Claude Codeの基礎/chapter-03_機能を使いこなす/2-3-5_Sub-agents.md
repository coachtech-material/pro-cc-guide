# 2-3-5 Sub-agents

## 🎯 このセクションで学ぶこと

- Sub-agents の仕組みと、コンテキスト分離の意義を理解する
- ビルトイン Sub-agents（Explore、Plan、General-purpose）の役割を把握する
- カスタム Sub-agent を作成し、特定のタスクを委譲する

まず Sub-agents のコンテキスト分離の仕組みを理解し、次にビルトイン Sub-agents とカスタム Sub-agent の定義方法を学び、最後に cc-practice で品質チェック用の Sub-agent を作成して使います。

---

## 導入: コンテキストを食い潰す問題、再び

Chapter 2-2 で学んだ「無限探索」の問題を覚えていますか。「このコードベースを調査して」と指示すると、Claude Code が大量のファイルを読み込み続け、コンテキストウィンドウを圧迫してしまう問題です。

この問題への対処として「調査の範囲を限定する」ことを学びましたが、もう1つ強力な方法があります。**調査そのものを別のコンテキストで実行させる** ことです。

**Sub-agents** は、メインの会話とは独立したコンテキストウィンドウを持つ「分身」です。Sub-agent にタスクを委譲すると、そのタスクは Sub-agent 自身のコンテキスト内で処理されます。結果だけがメインの会話に返ってくるため、メインのコンテキストを消費しません。

### 🧠 先輩エンジニアはこう考える

> Sub-agents は「調査を後輩に任せる」のと同じだ。自分がメインの実装に集中しつつ、「あのライブラリの使い方を調べておいて」「テスト結果をまとめておいて」と並行して調査を進められる。ポイントは、調査の過程（途中で読んだファイルの内容など）は自分の頭（コンテキスト）を使わず、結論だけもらえること。大きなコードベースで作業するときほど、この「コンテキストの節約」が効いてくる。

## Sub-agents とは

Sub-agents は、Claude Code のメインセッションから独立して動作する **専門的な AI アシスタント** です。

```
┌─ メインセッション ─────────────────────────────────────┐
│                                                         │
│  あなた ◀──▶ Claude Code（メインコンテキスト）          │
│                      │                                  │
│                タスクを委譲                              │
│                      │                                  │
│        ┌─────────────┼─────────────┐                    │
│        ▼             ▼             ▼                    │
│  ┌───────────┐ ┌───────────┐ ┌───────────┐              │
│  │ Sub-agent │ │ Sub-agent │ │ Sub-agent │              │
│  │ (Explore) │ │ (Plan)    │ │(独自定義) │              │
│  │ 独自の    │ │ 独自の    │ │ 独自の    │              │
│  │コンテキスト│ │コンテキスト│ │コンテキスト│              │
│  └─────┬─────┘ └─────┬─────┘ └─────┬─────┘              │
│        └─────────────┼─────────────┘                    │
│                結果だけ返る                              │
│                      ▼                                  │
│            メインコンテキストに反映                      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

各 Sub-agent は以下の特性を持ちます。

| 特性 | 説明 |
|---|---|
| **独立したコンテキスト** | メインセッションとは別のコンテキストウィンドウで動作する |
| **カスタムシステムプロンプト** | Sub-agent ごとに異なる指示を設定できる |
| **ツールの制限** | 使えるツールを制限できる（読み取り専用にする等） |
| **モデルの選択** | Sub-agent ごとに異なるモデルを使える（コスト最適化） |

> 📝 Sub-agents は別の Sub-agent を起動できますが、ネストの深さには制限があります。無限に連鎖することはありません。

## ビルトイン Sub-agents

Claude Code には3つの Sub-agent があらかじめ組み込まれています。

### Explore

コードベースの探索に特化した **高速・読み取り専用** の Sub-agent です。

| 項目 | 内容 |
|---|---|
| モデル | Haiku（高速・低コスト） |
| ツール | 読み取り専用（Read、Glob、Grep など） |
| 用途 | ファイルの検索、コードの調査、構造の把握 |

Claude Code が「コードベースを調査する必要がある」と判断すると、自動的に Explore Sub-agent を起動して調査を委譲します。Haiku モデルを使うため高速かつ低コストで、メインのコンテキストを消費しません。

### Plan

前のセクション（2-3-1 Plan Mode）で学んだ Plan Mode で使われる調査用の Sub-agent です。

| 項目 | 内容 |
|---|---|
| モデル | メインセッションと同じ |
| ツール | 読み取り専用 |
| 用途 | 計画を立てるためのコードベース調査 |

Plan Mode に切り替えて指示を出すと、Plan Sub-agent がコードベースを調査し、その結果をもとに計画を提示します。

### General-purpose

複雑なマルチステップタスクを処理できる **汎用** Sub-agent です。

| 項目 | 内容 |
|---|---|
| モデル | メインセッションと同じ |
| ツール | すべて利用可能 |
| 用途 | ファイルの編集を含む複雑なタスク |

Explore とは異なり、ファイルの編集やコマンドの実行もできます。Claude Code が「この作業はメインコンテキストを圧迫しそうだ」と判断したときに使われます。

## カスタム Sub-agent を作る

ビルトインの Sub-agent に加えて、**自分で Sub-agent を定義** することもできます。

### カスタム Sub-agent の用途

- **テスト実行の専門家**: テストの実行と結果の分析に特化した Sub-agent
- **セキュリティ監査**: セキュリティの観点でコードをレビューする Sub-agent
- **ドキュメント生成**: コードからドキュメントを生成する Sub-agent
- **特定の MCP サーバーを使う作業**: Playwright を使ったブラウザテストなど

### 作成方法

カスタム Sub-agent は、マークダウンファイルとして定義します。

**配置場所:**

| 場所 | スコープ |
|---|---|
| `.claude/agents/<name>.md` | プロジェクト（チーム共有可能） |
| `~/.claude/agents/<name>.md` | ユーザー全体 |

**ファイルの構造:**

```markdown
---
name: test-runner
description: テストの実行と結果分析を行う。テストの実行を依頼されたときに使用する
model: sonnet
tools:
  - Bash
  - Read
  - Glob
  - Grep
---

あなたはテスト実行の専門家です。

## 作業手順

1. テストファイルの構造を確認する
2. テストを実行する
3. 失敗したテストがあれば原因を分析する
4. 結果をわかりやすくまとめて報告する

## 報告形式

- 実行したテストの総数
- 成功 / 失敗の内訳
- 失敗したテストの詳細（テスト名、エラーメッセージ、推定原因）
```

YAML frontmatter の主要フィールドは以下の通りです。

| フィールド | 必須 | 説明 |
|---|---|---|
| `name` | はい | Sub-agent の識別名 |
| `description` | はい | どんなタスクで使うか。Claude Code が自動委譲の判断に使う |
| `model` | いいえ | 使用するモデル（`sonnet`、`opus`、`haiku`） |
| `tools` | いいえ | 使用可能なツールのリスト |
| `disallowedTools` | いいえ | 使用を禁止するツールのリスト |
| `mcpServers` | いいえ | この Sub-agent だけが使う MCP サーバー |
| `maxTurns` | いいえ | 最大ターン数（暴走防止） |

### MCP サーバーのスコープ

前のセクションで学んだ MCP サーバーを、**特定の Sub-agent だけに割り当てる** ことができます。

```markdown
---
name: browser-tester
description: Playwright でブラウザテストを実行する
mcpServers:
  - playwright:
      type: stdio
      command: npx
      args: ["-y", "@anthropic-ai/mcp-playwright"]
---

ブラウザテストを実行してください。
```

この設定により、Playwright MCP サーバーは `browser-tester` Sub-agent からのみアクセスできます。メインセッションには MCP ツールが追加されないため、コンテキストの消費を抑えられます。

## フォアグラウンドとバックグラウンド

Sub-agents は2つの実行モードで動作します。

| モード | 動作 | 用途 |
|---|---|---|
| **フォアグラウンド** | Sub-agent の完了を待ってからメインが再開 | 結果を使って次の作業を進める場合 |
| **バックグラウンド** | Sub-agent とメインが並行して動作 | 独立した調査や処理を同時に進める場合 |

Claude Code は状況に応じてモードを自動選択しますが、プロンプトで明示的に指定することもできます。

```
> テストの実行はバックグラウンドで進めて、その間にドキュメントの更新をやろう
```

## 🏃 実践: カスタム Sub-agent を定義して使う

cc-practice に、Laravel コードの品質チェックを行うカスタム Sub-agent を作成してみましょう。

### Step 1: Sub-agent を作成する

Claude Code を起動し、以下のように指示します。

```
> .claude/agents/quality-check.md を作成して。
> name は quality-check、description は「Laravel コードの品質をチェックする。PSR-12 準拠、メソッドの長さ、N+1 問題の有無を確認する」。
> model は haiku にして、使えるツールは Read、Glob、Grep だけに制限して。
> 本文には、チェック観点（PSR-12 準拠、メソッドの長さ30行以内、N+1 問題）と報告形式（問題の箇所、理由、改善案）を記載して
```

Claude Code が `.claude/agents/quality-check.md` を作成します。

### Step 2: コンテキスト使用量を確認する（実行前）

Sub-agent のコンテキスト分離効果を確認するため、現在のコンテキスト使用量を記録しておきましょう。

```
/context
```

表示された使用量を覚えておいてください。

### Step 3: Sub-agent を使ってみる

2-3-1 で作成した DailyReportController に対して品質チェック Sub-agent を実行します。

```
> quality-check で app/Http/Controllers/DailyReportController.php の品質をチェックして
```

Claude Code が `quality-check` Sub-agent を起動し、コードの品質分析を行います。Sub-agent は独自のコンテキストで動作するため、メインのコンテキストを圧迫しません。分析結果だけがメインの会話に返ってきます。

### Step 4: コンテキストの違いを確認する

Sub-agent の実行後、再度コンテキストの使用量を確認します。

```
/context
```

Step 2 と比較して、メインのコンテキスト使用量がほとんど増えていないことが確認できるはずです。Sub-agent がファイルを読み込んで分析した処理は、すべて Sub-agent のコンテキスト内で完結しています。

> 💡 もし Sub-agent を使わず「DailyReportController の品質をチェックして」と直接指示した場合、Claude Code はメインのコンテキスト内でファイルを読み込み分析するため、コンテキストの消費が大きくなります。大きなコードベースの調査には Sub-agent が特に有効です。

### Step 5: 成果物を確認する

```bash
! cat .claude/agents/quality-check.md
```

Sub-agent の定義ファイルが cc-practice に残っていることを確認しましょう。

## 🔍 見極めチェック

> 🧠 先輩エンジニアの思考: 「Sub-agent の定義は、結局のところ『誰にどんな仕事を任せるか』の設計だ。description が曖昧だと意図しないタイミングで起動されるし、ツール制限が甘いと安全性が下がる。」

- [ ] **正しさ**: frontmatter の `name`、`description`、`tools` が正しく記述されているか。Sub-agent が期待通りのツールだけを使っているか
- [ ] **品質**: `description` が具体的で、自動委譲の判定に使えるレベルか。`model` がタスクの複雑さに見合っているか（単純な調査に Opus は過剰）
- [ ] **安全性**: 読み取り専用で十分なタスクに書き込み権限を与えていないか。`maxTurns` で暴走を防止しているか

> 🔑 この Section では特に「ツール制限の適切さ」に注目してください。

## ✨ まとめ

- **Sub-agents** はメインセッションとは独立したコンテキストで動作する「分身」
- ビルトイン Sub-agents: Explore（高速調査）、Plan（計画策定）、General-purpose（汎用）
- カスタム Sub-agent は `.claude/agents/<name>.md` にマークダウンで定義する
- `tools` フィールドでツールを制限し、`mcpServers` で MCP サーバーをスコープできる
- フォアグラウンド（完了待ち）とバックグラウンド（並行実行）の2つの実行モードがある
- コンテキストの分離により、メインセッションのコンテキスト消費を抑えられる
