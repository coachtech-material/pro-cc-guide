# 2-2-2 コンテキストとセッション管理

## 🎯 このセクションで学ぶこと

- コンテキストウィンドウの仕組みと制約を理解する
- コンテキストを管理するためのコマンドと戦略を知る
- セッションの継続・分岐・チェックポイントの使い方を把握する

まずコンテキストウィンドウの仕組みと制約を理解し、次にコンテキストとセッションの管理方法を学び、最後に実際のプロジェクトで管理コマンドを体験します。

## 導入: 長く使っていると精度が落ちる？

前のセクションで、Claude Code がエージェントループによって自律的に動作する仕組みを学びました。では、そのループが動作する「場」にはどんな制約があるのでしょうか。

Claude Code を使っていて、こんな経験はないでしょうか。

最初は的確なコードを生成してくれていたのに、セッションが長くなるにつれて、指示を忘れたり、同じファイルを何度も読んだり、明らかに質が下がったように感じる。あるいは、CLAUDE.md に書いたルールが無視されるようになる。

これは気のせいではありません。Claude Code の性能は **コンテキストウィンドウ** の使用状況に直接影響されます。コンテキストの管理は、Claude Code を使いこなすうえで最も重要なスキルです。

### 🧠 先輩エンジニアはこう考える

> コンテキスト管理は、最初は意識しなくても困らないが、タスクが複雑になるほど効いてくるスキルだ。自分も最初は「なんか精度落ちたな」と思いながらそのまま使い続けていたが、`/clear` を挟むだけで劇的に改善することに気づいてからは、タスクの切れ目で意識的にリセットするようになった。メモリやスペックではなくコンテキストという概念がボトルネックになるのは、AI ツール特有の感覚だと思う。ここを理解しているかどうかで、使い方がまったく変わる。

## コンテキストウィンドウとは

コンテキストウィンドウは、Claude Code が1回のやり取りで参照できる情報の総量です。「Claude Code の作業机」とイメージしてください。机の上に載せられる書類の量には限りがあります。

コンテキストウィンドウには、以下のすべてが含まれます。

- **会話履歴**: あなたの指示と Claude Code の応答のすべて
- **ファイルの内容**: Claude Code が読み取ったファイルの中身
- **コマンドの実行結果**: テスト結果、ビルドログなど
- **CLAUDE.md**: セッション開始時に読み込まれるプロジェクト指示
- **自動メモリ**: MEMORY.md の先頭200行
- **システム指示**: Claude Code 自身の動作に必要な内部情報
- **スキルの定義**: 読み込まれた Skills の内容
- **MCP ツールの定義**: 接続している MCP サーバーのツール定義

つまり、セッション中に Claude Code が「見たもの」「やったこと」「受け取った結果」はすべてコンテキストウィンドウの中に蓄積されていきます。

### コンテキストが埋まるとどうなるか

コンテキストウィンドウが埋まっていくと、以下の問題が発生します。

**性能の低下**

コンテキストが大きくなるほど、Claude の応答精度が下がる傾向があります。これは Claude Code 固有の問題ではなく、LLM（大規模言語モデル）全般の特性です。大量の情報の中から関連する部分を見つけ出す負荷が増えるため、指示の見落としやミスが起きやすくなります。

**自動コンパクション**

コンテキストの上限に近づくと、Claude Code は自動的にコンパクション（要約による圧縮）を実行します。古いツールの出力から順にクリアし、必要に応じて会話全体を要約します。あなたのメッセージや重要なコードは保持されますが、会話の序盤に出した詳細な指示が失われることがあります。

```
コンテキストウィンドウの使用量
│
│  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓░░░░░  余裕あり: 快適に動作
│  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  上限付近: 自動コンパクション発動
│  ▓▓▓▓▓▓▓▓▓▓▓▓░░░░░░░░░░░  コンパクション後: 要約で圧縮
│
```

> ⚠️ **よくある誤解**: 「コンテキストが大きい方が多くの情報を持てて有利」と思うかもしれませんが、実際は逆です。関係のない情報が混ざるほど、Claude Code は重要な指示を見落としやすくなります。**コンテキストは小さく保つのが原則です。**

## コンテキストを管理する

コンテキストを効果的に管理するためのコマンドと戦略を紹介します。

### `/clear`: コンテキストを完全にリセットする

最も基本的で、最も効果的な方法です。

```
/clear
```

`/clear` は会話履歴をすべてクリアし、新しいセッションを開始します。CLAUDE.md と自動メモリは新しいセッションでも再度読み込まれるため、プロジェクトの基本設定は引き継がれます。

**使うタイミング:**

- タスクが完了して、別のタスクに取りかかるとき
- 同じセッション内で何度も修正を繰り返して、会話が散らかったとき
- Claude Code の応答精度が明らかに落ちてきたとき

> 🔑 **タスクの切れ目で `/clear` を挟む** のが、コンテキスト管理の最も重要な習慣です。無関係なタスクの履歴が残っていると、以降のすべてのやり取りでトークンを無駄に消費し、精度も下がります。

### `/compact`: コンテキストを要約して圧縮する

セッションを続けたいが、コンテキストが膨らんできた場合に使います。

```
/compact
```

Claude Code が会話履歴を要約し、重要な情報を保持しながらコンテキストを圧縮します。何を保持するかを指示することもできます。

```
/compact API の変更内容に焦点を当てて
```

> 📝 自動コンパクションが発動する前に、自分のタイミングで `/compact` を実行する方が、何を残すかをコントロールしやすくなります。

### `/context`: コンテキストの使用状況を確認する

```
/context
```

現在のコンテキストウィンドウで何がどれだけの容量を使っているかを確認できます。MCP サーバーのツール定義が想像以上に大きいことに気づく場合もあります。

### `/btw`: コンテキストを消費せずに質問する

```
/btw このメソッドの引数は何だっけ？
```

`/btw` は、メインの会話履歴に残らないサイド質問です。ちょっとした確認をしたいが、コンテキストを汚したくないときに使います。回答はオーバーレイで表示され、`Space`、`Enter`、`Esc` のいずれかで閉じます。

`/btw` の特徴は以下の通りです。

- 現在の会話の内容を参照して回答するが、会話履歴には残らない
- Claude Code が処理中でも使える（メインの作業を中断しない）
- ファイルの読み取りやコマンドの実行はできない（今のコンテキスト内の情報だけで回答する）
- フォローアップの質問はできない（1問1答）

### `Ctrl+S`: 入力中のプロンプトを一時退避する

長いプロンプトを書いている途中で、「先にこっちを確認したい」と思うことがあります。そんなとき、入力中のテキストをコピーして別の場所に退避して......という手間をかける必要はありません。

`Ctrl+S` を押すと、入力中のプロンプトが **スタッシュ**（一時退避）され、入力欄がクリアされます。

```
[長いプロンプトを入力中...]
                            ← ここで Ctrl+S を押す
[入力欄がクリアされる]
                            ← 別のプロンプトを送信して作業する
[作業が終わり、入力欄が空の状態]
                            ← スタッシュしたプロンプトが自動的に復元される
```

Git の `git stash` コマンドに似た考え方です。作業中の変更を一時退避して別の作業をし、戻ってきたら復元する。Claude Code のスタッシュは、これをプロンプト入力に対して行います。

スタッシュの特徴は以下の通りです。

- スタッシュできるのは **1つだけ**（新しくスタッシュすると、前のスタッシュは上書きされる）
- 別のプロンプトを送信した後、入力欄が空になると **自動的に復元される**
- セッション内でのみ有効（セッションを終了するとスタッシュは消える）

> 💡 `/btw` とスタッシュは、どちらも「作業中の割り込み」に対処する機能ですが、用途が異なります。`/btw` は「ちょっとした質問をコンテキストに残さず聞きたい」とき、スタッシュは「今書いているプロンプトを退避して別の作業をしたい」ときに使います。

### Sub-agents: 調査をコンテキストの外に出す

Claude Code にコードベースの調査を依頼すると、大量のファイルを読み込むため、コンテキストが一気に膨らみます。Sub-agents を使うと、調査を別のコンテキストウィンドウで実行し、要約だけをメインのコンテキストに返すことができます。

プロンプトに「Sub-agents を使って」と含めるだけで、Claude Code が自動的にサブエージェントに調査を委譲します。

```
> Sub-agents を使って、認証周りの実装を調査して
```

詳細な調査はサブエージェントに任せ、メインのセッションには「現在の認証方式は Fortify を使用しており...」のような要約だけが返ってきます。メインのコンテキストはほとんど消費されません。

Sub-agents は Chapter 2-3 で詳しく学びます。ここでは「コンテキストを分離する手段がある」ことだけ覚えておいてください。

### CLAUDE.md に持続的な指示を書く

会話の序盤で出した指示は、自動コンパクション時に要約されて失われる可能性があります。**すべてのセッションで守ってほしいルールは、会話で伝えるのではなく CLAUDE.md に書いてください。** CLAUDE.md の内容はコンパクション後も保持されます。

> 💡 コンパクション時に保持する内容をカスタマイズすることもできます。CLAUDE.md に `# Compact Instructions` というセクションを作り、「変更したファイルの一覧とテストコマンドを必ず保持すること」のように書いておくと、要約時にその情報が優先的に残ります。

## セッションを管理する

エージェントループが動作する「場」はセッションです。セッションの管理方法を知っておくと、作業の継続性が上がります。

### セッションの独立性

各セッションは独立しています。新しいセッションを開始すると、前のセッションの会話履歴は引き継がれません。Claude Code がセッション間で持続的に参照できるのは、CLAUDE.md と自動メモリだけです。

### セッションを再開する: `--continue` と `--resume`

「昨日の続きをやりたい」というとき、前のセッションを再開できます。

```bash
claude --continue
```

直前のセッションを再開します。会話履歴がすべて復元され、中断したところから作業を続けられます。

```bash
claude --resume
```

過去のセッション一覧から選んで再開します。複数のタスクを並行して進めているときに便利です。

> 📝 セッション再開時、セッション中に許可した権限（ファイル変更の許可など）はリセットされます。再度許可が必要になります。

### セッションに名前をつける: `/rename`

セッションに名前をつけておくと、後で再開するときに見つけやすくなります。

```
/rename 認証バグ修正
```

`--resume` で一覧表示したとき、名前がついたセッションはすぐに識別できます。

### セッションを分岐する

「今の時点から別のアプローチを試したい」というとき、セッションを分岐できます。

セッション中であれば、`/branch` または `/fork` コマンドで会話を分岐できます。

```
/branch
```

起動時に分岐する場合は、`--fork-session` フラグを使います。

```bash
claude --continue --fork-session
```

いずれの方法でも、元のセッションの会話履歴をコピーした新しいセッションが作成されます。元のセッションには影響しません。Git のブランチのように、同じ出発点から異なるアプローチを試す使い方ができます。

## チェックポイントで変更を巻き戻す

Claude Code はファイルを編集する前に、自動的にその時点のファイル状態をスナップショットとして保存します。これが **チェックポイント** です。

### チェックポイントの仕組み

- あなたがプロンプトを送信するたびに、新しいチェックポイントが作成される
- チェックポイントはセッションをまたいで保持される（再開したセッションでも利用可能）
- 30日後に自動的にクリーンアップされる（設定で変更可能）

### リワインド: 過去の状態に戻る

`Esc` キーを2回押す（`Esc` + `Esc`）か、`/rewind` コマンドを実行すると、リワインドメニューが表示されます。セッション中の各プロンプトが一覧表示され、戻りたいポイントを選択できます。

リワインドでは、以下の操作を選べます。

| 操作 | 内容 |
|---|---|
| **Restore code and conversation** | コードと会話の両方をその時点に戻す |
| **Restore conversation** | 会話だけ戻す（コードは現在のまま） |
| **Restore code** | コードだけ戻す（会話は現在のまま） |
| **Summarize from here** | 選択したポイント以降の会話を要約して圧縮する |
| **Never mind** | 何もせずにメニューを閉じる |

> 💡 「Summarize from here」は `/compact` の部分版です。会話の前半は詳細なまま残し、後半だけを圧縮したいときに使います。たとえば、長いデバッグセッションの序盤の指示は保持しつつ、試行錯誤の部分だけ圧縮する、といった使い方ができます。

### チェックポイントの限界

チェックポイントが追跡するのは、**Claude Code のファイル編集ツールによる変更のみ** です。以下の変更は追跡されません。

- Bash コマンドによるファイル操作（`rm`、`mv`、`cp` など）
- あなたが手動で行ったファイル変更
- 他のセッションからの変更

```
追跡される変更          追跡されない変更
─────────────          ──────────────
Edit ツール             rm, mv, cp
Write ツール            手動でのファイル編集
                        他セッションからの変更
```

> ⚠️ チェックポイントは Git の代わりにはなりません。チェックポイントはセッション内の「ローカルな Undo」、Git は「永続的な履歴」として使い分けてください。また、チェックポイントが巻き戻せるのはファイル変更のみです。データベースの変更、API への送信、デプロイなど、外部システムに影響を与える操作は巻き戻せません。Claude Code が外部に影響するコマンドの実行前に確認を求めるのはこのためです。重要な変更を行う前には、Git でコミットしておくことをお勧めします。

## コンテキスト管理の戦略まとめ

ここまで学んだ内容を、実践的な戦略としてまとめます。

**基本ルール: タスクの切れ目で `/clear`**

最も重要な習慣です。1つのタスクが終わったら、次のタスクに入る前にコンテキストをリセットしましょう。

**セッション中のコンテキスト管理**

| 状況 | 対処法 |
|---|---|
| タスク完了、次のタスクに移る | `/clear` |
| セッションは続けたいが、コンテキストが膨らんできた | `/compact` |
| 2回以上修正を繰り返しても改善しない | `/clear` してより具体的なプロンプトで再開 |
| ちょっとした確認がしたい | `/btw` |
| 入力中のプロンプトを退避して別の作業をしたい | `Ctrl+S`（スタッシュ） |
| 大量のファイル調査が必要 | Sub-agents に委譲 |
| コンテキストの使用状況を知りたい | `/context` |

**セッション管理**

| 状況 | 対処法 |
|---|---|
| 昨日の続きをやりたい | `claude --continue` |
| 特定のセッションに戻りたい | `claude --resume` |
| 別のアプローチを試したい | `claude --continue --fork-session` |
| セッションを後で見つけやすくしたい | `/rename` |
| 変更を巻き戻したい | `Esc` + `Esc` または `/rewind` |

## 🏃 実践: コンテキストとセッションを管理する

ここまで学んだコマンドを cc-practice で一通り体験しましょう。「こういうときにこのコマンドを使う」という感覚を掴むことが目的です。

### Step 1: コンテキストの使用状況を確認する

cc-practice で Claude Code を起動します。

```bash
claude
```

まず、現在のコンテキストの状態を確認しましょう。

```
/context
```

セッション開始直後の状態が表示されます。CLAUDE.md やシステム指示が最初から一定量のコンテキストを占めていることがわかるはずです。この「起動時点での消費量」を把握しておくことは、後のセッション設計に役立ちます。

### Step 2: タスクを実行してコンテキストの変化を見る

コンテキストがどう変化するか観察するために、少し作業をしましょう。

```
> app/Models/DailyReport.php のコードを読んで、改善点があれば教えて
```

Claude Code がファイルを読み込んで分析した後、再度コンテキストを確認します。

```
/context
```

Step 1 と比較して、使用量が増えていることを確認してください。ファイルの読み取り結果や会話履歴が蓄積されたためです。

### Step 3: `/compact` でコンテキストを圧縮する

コンテキストが膨らんできた場合の対処法を体験します。

```
/compact DailyReport モデルの改善点に焦点を当てて
```

Claude Code が会話履歴を要約し、コンテキストを圧縮します。圧縮後に再度 `/context` で確認すると、使用量が減っていることがわかります。

> 📝 `/compact` に引数を渡すと、要約時に何を重視するかを指示できます。引数なしで実行すると、Claude Code が自動で要約の重点を判断します。

### Step 4: `Ctrl+S` でプロンプトをスタッシュする

「作業中に別のことを確認したくなる」場面を体験しましょう。

まず、長めのプロンプトを入力します。**まだ送信しないでください。**

```
> app/Models/DailyReport.php にバリデーションルールを追加して。title は必須で最大255文字、content は必須で最低10文字、reported_at は必須で日付形式
```

ここで「そういえば、既存のバリデーションがあるか先に確認したい」と思ったとします。`Ctrl+S` を押してください。

入力欄がクリアされ、今書いていたプロンプトがスタッシュされます。代わりに別の質問を送信しましょう。

```
> このプロジェクトで Form Request は使われている？
```

Claude Code が回答したら、入力欄が空の状態になっています。ここで、先ほどスタッシュしたプロンプト（バリデーションルールの追加指示）が **自動的に復元される** ことを確認してください。

復元されたプロンプトはそのまま送信しても、編集してから送信しても構いません。ここでは確認だけで十分なので、`Ctrl+U` で入力欄をクリアして次に進みましょう。

> 📝 スタッシュは1つしか保持できません。複数の長いプロンプトを退避したい場合は、`Ctrl+G` で外部エディタを開いてテキストファイルに保存する方法もあります。

### Step 5: `/clear` でリセットする

次のタスクに移ることを想定して、コンテキストを完全にリセットします。

```
/clear
```

新しいセッションが始まります。前の会話履歴は消えますが、2-1-4 で設定した CLAUDE.md のルールは引き継がれています。確認してみましょう。

```
> このプロジェクトのテストフレームワークは何？
```

CLAUDE.md に書いた「PHPUnit」を回答してくれれば、`/clear` 後も CLAUDE.md が読み込まれていることの確認になります。

### Step 6: セッションに名前をつけて終了する

このセッションに名前をつけてから終了しましょう。

```
/rename コンテキスト管理の練習
```

`Ctrl + D` で Claude Code を終了します。

### Step 7: セッションを再開する

終了したセッションを再開してみましょう。

```bash
claude --continue
```

直前のセッション「コンテキスト管理の練習」が再開され、先ほどの会話の続きから作業できます。`/clear` 後の内容（Step 4 以降）が復元されていることを確認してください。

確認できたら、`Ctrl + D` で終了して構いません。

> 💡 `--resume` を使うと、過去のセッション一覧から選んで再開できます。複数のタスクを並行して進めるときに便利です。

## ✨ まとめ

- **コンテキストウィンドウ** は Claude Code が参照できる情報の総量。会話履歴、ファイル内容、コマンド結果、CLAUDE.md などすべてが含まれる
- コンテキストが埋まると性能が低下し、自動コンパクションが発動する。**コンテキストは小さく保つのが原則**
- `/clear` がコンテキスト管理の基本。タスクの切れ目で必ず使う
- `/compact` で手動圧縮、`/btw` でコンテキストを消費しない質問、`Ctrl+S` でプロンプトの一時退避、Sub-agents でコンテキスト分離ができる
- 持続的な指示は会話ではなく CLAUDE.md に書く
- セッションは `--continue`、`--resume` で再開、`--fork-session` で分岐できる
- チェックポイントでファイル変更を巻き戻せるが、Git の代わりにはならない
