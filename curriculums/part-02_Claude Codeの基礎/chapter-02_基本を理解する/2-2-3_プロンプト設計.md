# 2-2-3 プロンプト設計

## 🎯 このセクションで学ぶこと

- Claude Code に的確な指示を出すための4つの原則を理解する
- よくある失敗パターンとその回避方法を知る
- 「丸投げ」と「設計されたプロンプト」の違いを体感する

まず Claude Code のプロンプトが ChatGPT とどう違うかを理解し、次に4つの設計原則とよくある失敗パターンを学び、最後にプロンプトの質による結果の違いを体感します。

---

## 導入: ChatGPT に聞くのと何が違うのか

前のセクションで、コンテキスト管理の重要性を学びました。コンテキストを効率的に使うためには、Claude Code への指示の出し方そのものを見直す必要があります。

ChatGPT で「ログイン機能を作って」と聞くと、汎用的なコードスニペットが返ってきます。それをコピーして自分のプロジェクトに合わせて調整する、という使い方です。

Claude Code では、同じ「ログイン機能を作って」という指示でも、まったく異なることが起きます。Claude Code はプロジェクトのファイルを読み、既存の認証設定を確認し、テストを書き、実行し、問題があれば修正する。つまり、**あなたのプロジェクトの文脈の中で** 作業を行います。

この違いは、プロンプトの書き方にも影響します。ChatGPT では「なるべく詳しく要件を伝える」ことが重要でしたが、Claude Code では「何をしてほしいかを伝えつつ、Claude Code が自分で調べられる余地を残す」というバランスが求められます。

このセクションでは、Claude Code の特性を活かした効果的なプロンプトの設計原則を学びます。

### 🧠 先輩エンジニアはこう考える

> プロンプト設計というと難しく聞こえるが、要は「仕事の依頼の仕方」だ。後輩に仕事を頼むとき、「あのバグ直しといて」だけでは情報が足りないし、逆に1行ずつ「このファイルを開いて、この行を変えて」と指示するのは非効率だ。「ログイン後にセッションが切れるバグがある。`src/auth/` あたりが怪しい。まず再現するテストを書いてから修正してほしい」くらいの粒度が、相手が自分で考えて動ける適切なレベル。Claude Code への指示もまったく同じ感覚でいい。

## 原則1: 具体的なコンテキストを与える

Claude Code はプロジェクト全体にアクセスできますが、あなたの頭の中にある情報までは読めません。タスクの背景、対象のファイル、制約条件を明示することで、Claude Code は的確に動けます。

### 曖昧なプロンプト vs 具体的なプロンプト

以下の2つを比較してみてください。

**曖昧なプロンプト:**

```
> テストを追加して
```

この指示では、Claude Code はどのファイルに、何のテストを、どのテストフレームワークで書くべきか、すべて自分で判断しなければなりません。結果として、大量のファイルを読み込んでコンテキストを消費し、的外れなテストを書く可能性があります。

**具体的なプロンプト:**

```
> app/Services/PaymentService.php にテストを追加して。
> ユーザーが退会済みの場合にエラーを返すケースをカバーしたい。
> モックは使わず、PHPUnit の Feature テストで書いて。
```

ファイル、テストケース、テストフレームワーク、テストの種類が明確です。Claude Code は迷わずに作業を開始できます。

### 具体性のレベル

すべてのプロンプトで最大限の具体性が必要なわけではありません。タスクの複雑さに応じて、適切なレベルを選びましょう。

| タスクの種類 | プロンプト例 | 具体性のレベル |
|---|---|---|
| 簡単な修正 | `UserController のバリデーションにメールの形式チェックを追加して` | 低（何をするかだけで十分） |
| 機能追加 | `ユーザープロフィール画面にアバター画像のアップロード機能を追加して。保存先は storage/app/public/avatars。画像はリサイズして 200x200 にする` | 中（何を + 制約） |
| 複雑な修正 | `ログイン後にセッションが切れる報告がある。src/auth/ あたりが怪しい。特に token refresh の処理を確認して。まず再現するテストを書いてから修正して` | 高（症状 + 場所 + 方針） |

### `@` でファイルを直接参照する

プロンプト中で `@` に続けてファイルパスを入力すると、Claude Code はそのファイルを事前に読み込んでから応答します。「`src/auth/` を見て」と書くよりも確実です。

```
> @app/Models/User.php のリレーションを確認して、足りないものがあれば追加して
```

### その他のコンテキスト提供方法

`@` 以外にも、Claude Code にコンテキストを渡す方法があります。

| 方法 | 用途 |
|---|---|
| `@ファイルパス` | ファイルを事前に読み込ませる |
| 画像の貼り付け（`Ctrl+V`） | UI のスクリーンショットやデザインカンプを渡す |
| URL の提示 | ドキュメントや API リファレンスを参照させる |
| パイプ（`cat error.log | claude`） | ログやデータを直接渡す |

状況に応じて最適な方法を選びましょう。

## 原則2: 検証手段を渡す

前のセクションで学んだエージェントループを思い出してください。収集 → 実行 → **検証** の3フェーズです。検証フェーズで Claude Code が自分の仕事をチェックできるかどうかが、最終的な品質を大きく左右します。

### 検証なしの場合

```
> メールアドレスのバリデーション関数を作って
```

Claude Code はそれらしい関数を生成しますが、正しいかどうかを確認する手段がありません。あなたが目視で確認するしかなく、エッジケースの見落としが発生しやすくなります。

### 検証ありの場合

```
> メールアドレスのバリデーション関数を作って。
> テストケース: user@example.com → true、invalid → false、user@.com → false
> テストを書いて、実行して確認して。
```

Claude Code はテストを書き、実行し、失敗すれば自分で修正します。エージェントループの検証フェーズが効果的に機能し、**あなたが確認する前に自己修正が完了** します。

### 検証手段の例

テストだけが検証手段ではありません。タスクに応じた検証手段を渡しましょう。

| タスクの種類 | 検証手段の例 |
|---|---|
| ロジックの実装 | テストケースと期待する結果 |
| API エンドポイント | リクエスト例とレスポンス例 |
| バグ修正 | 再現手順とエラーメッセージ |
| リファクタリング | 既存テストがすべて通ること |
| コマンドラインツール | 実行例と期待する出力 |

> 🔑 **検証手段を渡すことは、Claude Code の精度を上げる最も効果のある方法です。** プロンプトに悩んだら、まず「Claude Code が自分の仕事をどうやってチェックできるか」を考えてください。

## 原則3: 探索と実装を分離する

複雑なタスクでは、いきなりコードを書かせるのではなく、まず調査・計画を行い、それから実装に入る、という2段階のアプローチが有効です。

### なぜ分離するのか

Claude Code にいきなり「OAuth を追加して」と指示すると、Claude Code は自分の判断で実装を始めます。しかし、その方向性があなたの意図と異なっていた場合、大量のコードが生成された後に軌道修正が必要になります。コンテキストも消費され、手戻りのコストが大きくなります。

### Plan Mode を使った2段階アプローチ

Claude Code には **Plan Mode** があります。`Shift+Tab` を2回押すと Plan Mode に切り替わり、Claude Code は読み取り専用のツールだけを使って調査・計画を行います。ファイルの編集やコマンドの実行は行いません。

**Step 1: Plan Mode で調査・計画**

```
> (Plan Mode) src/auth/ を読んで、現在のセッション管理とログインの仕組みを理解して。
> そのうえで、Google OAuth を追加する計画を立てて。
```

Claude Code が計画を提示します。「どのファイルを変更するか」「セッションフローはどうなるか」が明らかになります。

> 💡 `Ctrl+G` を押すと、Claude Code が提示した計画をテキストエディタで開いて直接編集できます。計画の一部を修正したり、追加の制約を書き込んだりしてから、Claude Code に実装を進めさせることができます。

**Step 2: 計画を確認してから実装**

計画に納得したら、`Shift+Tab` で通常モードに戻し、実装を指示します。

```
> (通常モード) さっきの計画に沿って OAuth を実装して。
> callback ハンドラーのテストも書いて、テストを実行して確認して。
```

この2段階のアプローチにより、「間違った方向に大量のコードを書いてしまう」リスクを大幅に減らせます。

> 💡 Plan Mode はすべてのタスクに必要なわけではありません。「タイポを修正して」「ログ行を追加して」のような簡単なタスクでは、そのまま実装させた方が効率的です。Plan Mode が有効なのは、アプローチが不確実なとき、変更が複数ファイルにまたがるとき、馴染みのないコードを修正するときです。

## 原則4: 委譲する、指図しない

Claude Code はエージェントです。ファイルの開き方もコマンドの実行順序も、自分で判断できます。あなたの役割は **何を達成してほしいか** を伝えることであり、**どう実現するか** の詳細を指示することではありません。

### 指図するプロンプト（非効率）

```
> まず app/Models/User.php を開いて。
> 次に orders リレーションを確認して。
> それから app/Http/Controllers/OrderController.php を開いて。
> index メソッドを探して。
> そのメソッドに Eloquent の eager loading を追加して。
```

5つの指示を出していますが、Claude Code なら「OrderController の一覧取得で N+1 問題が起きている。eager loading を追加して修正して」の一文で十分です。

### 委譲するプロンプト（効率的）

```
> OrderController の一覧取得で N+1 問題が起きている。eager loading で修正して。
```

Claude Code は自分で User モデルのリレーションを確認し、OrderController のコードを読み、適切な箇所に eager loading を追加します。あなたが1つずつ指示するよりも速く、しかもコンテキストの消費も少なくなります。

### 委譲のバランス

とはいえ、すべてを Claude Code に任せるわけではありません。以下のバランスを意識してください。

```
あなたが伝えるべきこと          Claude Code に任せること
───────────────────          ──────────────────
何を達成したいか               どのファイルを読むか
制約条件（フレームワーク、規約） どのコマンドを実行するか
検証の基準（テスト、期待出力）   どの順序で作業するか
対象の範囲（ファイル、ディレクトリ） 具体的なコードの書き方
```

## 思考の深さを制御する

4つの原則に加えて、もう一つ知っておくと便利なテクニックがあります。Claude Code の「思考の深さ」をプロンプトで制御する方法です。

Claude Code は内部で「Extended thinking」という仕組みを持っており、回答を出す前に段階的に思考を深めます。この思考にどれだけリソースを割くかは、通常は Claude Code が自動で判断しますが、プロンプトに特定のキーワードを含めることで明示的に制御できます。

### 思考キーワード

プロンプトのどこかに特定のキーワードを書くと、その回だけ思考の深さを制御できます。4段階のキーワードがあり、段階が上がるほど深い推論を行います。

| キーワード | 思考の深さ |
|---|---|
| `think` | 標準的な思考 |
| `think hard` | より深い思考 |
| `megathink` | さらに深い思考 |
| `ultrathink` | 最大限の思考 |

```
> ultrathink このプロジェクトの認証フローを分析して、
> セキュリティ上の問題点があれば洗い出して
```

複雑な設計判断、セキュリティ分析、バグの根本原因の特定など、深い推論が求められるタスクで効果的です。簡単なタスクには `think`、難しいタスクには `ultrathink` というように使い分けましょう。

### 思考の深さの設定方法

`ultrathink` は1回のプロンプトだけに効く一時的な方法ですが、セッション全体や常時の設定も可能です。

| 方法 | 効果範囲 | 使い方 |
|---|---|---|
| `ultrathink` キーワード | その回のプロンプトだけ | プロンプト中に `ultrathink` と書く |
| `/effort` コマンド | セッション中 | `/effort` で low / medium / high を切り替える |
| `Option+T`（Mac）/ `Alt+T` | セッション中 | thinking モードのオン・オフを切り替える |
| `/config` | 常時 | `alwaysThinkingEnabled` を設定 |

> 💡 思考を深くするほど回答の精度は上がりますが、その分トークンを消費し、応答時間も長くなります。簡単なタスク（タイポ修正、ログ追加など）では不要で、複雑なタスク（設計判断、セキュリティ分析、難しいバグの原因特定など）で使い分けるのが効果的です。

## よくある失敗パターン

最後に、Claude Code の利用者がよく陥る失敗パターンと、その対処法を紹介します。これらを知っておくことで、同じ落とし穴を回避できます。

### キッチンシンクセッション

1つのセッションで無関係なタスクを次々と処理してしまうパターンです。バグ修正の後にドキュメント更新、その後に別の機能追加と、コンテキストが関係のない情報で埋まっていきます。

**対処法:** タスクの切れ目で `/clear` を使いましょう。

### 過修正ループ

Claude Code の出力が期待と違い、修正を依頼し、また違い、また修正を依頼する、という繰り返しです。2〜3回修正しても改善しない場合、会話の文脈自体が問題になっています。失敗した試行錯誤がコンテキストに蓄積され、Claude Code が混乱しています。

**対処法:** 2回修正しても改善しなければ、`/clear` して最初からやり直しましょう。このとき、失敗から学んだことを反映したより具体的なプロンプトで再開します。「きれいなコンテキストでより良いプロンプトを書く」方が、「汚れたコンテキストで修正を重ねる」よりほぼ常に良い結果を生みます。

### 無限探索

「このコードベースを改善して」「調査して」のようにスコープを絞らない指示を出すと、Claude Code は大量のファイルを読み込み続けます。コンテキストが一気に埋まり、肝心の作業を行う余裕がなくなります。

**対処法:** 調査の範囲を明確に限定するか、Sub-agents に委譲して、メインのコンテキストへの影響を抑えましょう。

### 信頼してから検証の空白

Claude Code が生成したコードがもっともらしく見えたので、そのまま採用する。しかし、エッジケースが考慮されていなかったり、既存の処理と矛盾していたりする。

**対処法:** 常に検証手段を渡しましょう（原則2）。テスト、スクリプト、期待出力など、Claude Code が自分の仕事をチェックできる手段を用意してください。検証できないものは、出荷しないのが鉄則です。

> 🔑 これらの失敗パターンのほとんどは、**コンテキスト管理**（前のセクション）と **プロンプトの具体性**（このセクション）に帰結します。この2つを意識するだけで、Claude Code の利用効率は大きく改善します。

## 🏃 実践: プロンプトの質で結果が変わることを体感する

ここまで学んだ4つの原則を、cc-practice で実際に試してみましょう。同じ目的のタスクを「曖昧なプロンプト」と「設計されたプロンプト」の2パターンで実行し、結果を比較します。

### Step 1: 曖昧なプロンプトで試す

cc-practice で Claude Code を起動し、曖昧な指示を出してみましょう。

```bash
claude
```

```
> バリデーションを追加して
```

Claude Code はどのモデル・コントローラーにどんなバリデーションを追加するか、すべて自分で判断しなければなりません。結果を確認し、以下の点を観察してください。

- どのファイルが対象になったか
- どんなバリデーションルールが追加されたか
- 作業にどれくらい時間がかかったか（ファイルの読み込み量も含めて）

結果を覚えておいて、一度セッションをリセットします。

```
/clear
```

### Step 2: 設計されたプロンプトで試す

今度は、4つの原則を適用したプロンプトを試します。

```
> @app/Models/DailyReport.php の date、title、content、status カラムに対するバリデーションを追加して。
> 具体的には:
> - date: 必須、日付形式
> - title: 必須、文字列、最大50文字
> - content: 必須、文字列、10文字以上
> - status: 必須、draft または submitted のいずれか
> バリデーションは FormRequest クラスとして app/Http/Requests/StoreDailyReportRequest.php に作成して。
> テストも書いて、Sail で実行して確認して。
```

このプロンプトには4つの原則が含まれています。

- **原則1（具体的なコンテキスト）**: `@` でファイルを指定、バリデーションルールを明示
- **原則2（検証手段）**: テストを書いて実行するよう指示
- **原則3（探索と実装の分離）**: 今回は対象が明確なので分離不要と判断
- **原則4（委譲）**: 何を達成するかを伝え、FormRequest の具体的な実装方法は Claude Code に任せている

### Step 3: 結果を比較する

Step 1 と Step 2 は `/clear` を挟んで別々のセッションで実行しているため、厳密な比較はできません。しかし、設計されたプロンプトの方がより的確な結果を得やすいことを体感できたはずです。それぞれの体験を振り返って、以下の観点で違いを確認してみましょう。

- **的確さ**: Step 1（曖昧）では、どのファイルにどんなバリデーションが追加されましたか? 意図した DailyReport の 4 カラムに対するバリデーションになっていましたか? Step 2（具体的）ではどうでしたか?
- **効率**: Step 1 では、Claude Code が対象ファイルを特定するために余計なファイルを読んだり、試行錯誤していませんでしたか? Step 2 では、`@` でファイルを指定したことで迷いなく作業が進んだはずです
- **検証**: Step 2 ではテストの作成と実行まで指示に含めました。Claude Code が自分の仕事を自己検証するサイクルが回ったはずです。Step 1 ではテストは書かれましたか?

### 🧠 先輩エンジニアはこう考える

> プロンプトを丁寧に書くのが面倒に感じるかもしれないが、実は曖昧なプロンプトの方がトータルのコストが高い。修正の往復、コンテキストの消費、手動での確認作業、すべてを含めると、最初に30秒かけてプロンプトを設計する方がはるかに効率的だ。

### Step 4: 成果物を確認する

最終的に cc-practice に以下が追加されていることを確認しましょう。

```bash
! ls app/Http/Requests/
! ls tests/Feature/
```

- `StoreDailyReportRequest.php`（FormRequest クラス）
- テストファイル（名前は Claude Code が判断）

## 🔍 見極めチェック

> 🧠 先輩エンジニアの思考: 「FormRequest のバリデーションルールは、意図通りの内容になっているか? テストは実際のリクエストをシミュレートしているか? 生成されたからOKではなく、自分の目で確認する。」

- [ ] **正しさ**: `StoreDailyReportRequest` のバリデーションルールが指示通りか（date: required, date / title: required, string, max:50 / content: required, string, min:10 / status: required, in:draft,submitted）。テストが通っているか
- [ ] **品質**: FormRequest クラスが Laravel の規約に沿っているか（`authorize()` メソッドが `true` を返すか、`rules()` メソッドの戻り値が適切か）。コメントが日本語か（2-1-4 で CLAUDE.md に設定したルール。プロンプトに書かなくても CLAUDE.md 経由で反映される）
- [ ] **安全性**: バリデーションルールに漏れがないか（content に HTML タグを許可していないか、status に想定外の値を受け入れていないか）
