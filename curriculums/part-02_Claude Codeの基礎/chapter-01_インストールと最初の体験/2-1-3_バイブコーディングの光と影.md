# 2-1-3 バイブコーディングの光と影

## 概要

前のセクションで、Claude Code に一言指示を出すだけで動くアプリが生まれる体験をしました。このセクションでは、その威力をさらに押し進めて「もっと大きなものを一気に作る」ことに挑戦します。そして、その体験を通じて「バイブコーディング」という概念と、その光と影について考えます。

Claude Code は強力なツールです。しかし、強力だからこそ使い方を誤ると危険です。このセクションは、次の Chapter 以降で学ぶ「見極める力」がなぜ必要なのか、その動機付けとなる内容です。解説と実践を織り交ぜて進めます。

## バイブコーディングを体験する

前のセクションで作った Markdown エディタは、見た目も機能もしっかりしたアプリでした。では、もっと大きなものを一気に作ったらどうなるでしょう？

Claude Code をインタラクティブモードで起動してください（`~/claude-test` ディレクトリで）。

```bash
cd ~/claude-test
claude
```

以下のプロンプトを入力してください。一言です。

```
> このMarkdownエディタを、フル機能のブログアプリに拡張してください。記事一覧ページ、記事投稿フォーム、タグによる絞り込み、記事の検索機能を追加して、データはJSONファイルに保存するようにしてください。PHPの組み込みサーバーで動くようにしてください
```

Claude Code が動き始めます。今度はこれまでと規模が違います。ファイルが次々に生成され、diff が何度も表示されます。すべて `y` で承認してください。

数分後、すべての承認が終わったら、サーバーを起動して動作を確認しましょう。

```
> サーバーを起動して
```

ブラウザで `http://localhost:8000` を開いてみてください。

記事の一覧ページ、投稿フォーム、タグ絞り込み、検索機能。一言の指示から、複数のファイルにわたるブログアプリが生まれました。実際に記事を投稿し、検索やタグでの絞り込みを試してみてください。

ここで少し立ち止まって、自分に問いかけてみてください。

- 今生成されたコードを、**1行でも読みましたか？**
- データはどうやって保存されていますか？ JSON ファイルのパスは？
- 入力値のバリデーションはありますか？
- XSS 対策は施されていますか？
- 同時に2人がアクセスしたらデータは壊れませんか？

おそらく、ほとんどの質問に答えられないはずです。diff が表示されるたびに `y` を押しましたが、中身を確認していたでしょうか。

これが**バイブコーディング**です。

## バイブコーディングとは何か

バイブコーディング（vibe coding）は、OpenAI の共同創業者 Andrej Karpathy が 2025 年 2 月に提唱した概念です。Karpathy 自身の言葉を借りると、「コードの存在を忘れ、雰囲気（vibes）に完全に身を委ねる」コーディングスタイルです。

具体的には、AI に指示を出し、生成されたコードを読まずにすべて受け入れ、エラーが出たらエラーメッセージをコピペして修正を依頼し、動けば OK とするやり方です。Karpathy 自身も「使い捨ての週末プロジェクト向け」と限定していましたが、この言葉はたちまち広がり、[Collins Dictionary の 2025 年 Word of the Year](https://www.collinsdictionary.com/us/woty) に選出されるほどの注目を集めました。

ここで大事な区別があります。Django の共同作成者 Simon Willison はこう述べています。

> AI にコードを書かせ、それをレビューし、テストし、他者に説明できるなら、それはバイブコーディングではない。それはソフトウェア開発だ。

つまり、**AI にコードを書かせること自体はリスクではありません**。書かれたコードを読まず、理解せず、検証せずに受け入れることがバイブコーディングであり、リスクなのです。

先ほど皆さんがやったことを振り返ると、diff が表示されるたびに `y` を押してすべてを受け入れました。生成されたコードの中身は確認していません。これはまさにバイブコーディングです。

## なぜ危険なのか

「動いているなら問題ないのでは？」と思うかもしれません。しかし、バイブコーディングで作られたコードには、**動いているように見えるだけで深刻な問題が潜んでいる** ケースが多数報告されています。

### セキュリティの脆弱性

[Veracode の調査](https://www.veracode.com/blog/genai-code-security-report/)（2025年）によると、AI が生成したコードの **45% が OWASP Top-10 の脆弱性** を含んでいます。具体的には以下のような問題です。

- **SQL インジェクション**: ユーザー入力がそのまま SQL に埋め込まれ、データベースが操作される
- **XSS（クロスサイトスクリプティング）**: 入力値のサニタイズが欠落し、悪意あるスクリプトが実行される
- **認証・認可の欠如**: 管理者ページが認証なしでアクセスできる状態で公開される
- **秘密情報のハードコード**: API キーやパスワードがコード内に直接書かれる

コード品質の面でも、[CodeRabbit による 470 件の GitHub PR の分析](https://www.coderabbit.ai/blog/state-of-ai-vs-human-code-generation-report)（2025年12月）では、AI が共著したコードは人間のコードと比較してロジックエラーが約 1.7 倍、セキュリティ脆弱性が 2.74 倍多いという結果が出ています。

先ほど皆さんが作ったブログアプリにも、これらの脆弱性が含まれている可能性があります。COACHTECH で学んだバリデーションや認証の知識があれば、「入力値をそのまま表示していないか」「アクセス制御はあるか」をコードを読んで判断できます。しかし、バイブコーディングではその判断を放棄しています。

### 実際に起きたインシデント

バイブコーディングが原因と報告された事例をいくつか紹介します。

- あるスタートアップ（Enrichlead）は「100% AI がコードを書いた」と公言してサービスを公開しましたが、有料機能への未認証アクセスやデータ改ざんが可能な脆弱性が発覚し、サービス停止に追い込まれました
- ある SaaS 企業（SaaStr CEO の報告）では、AI エージェントがテスト結果を偽造し、最終的に**本番データベースを全削除** する事態が発生しました
- あるプラットフォーム（Lovable）では、AI で構築された 1,645 アプリのうち **170 アプリにユーザーの個人情報への未認証アクセスを許す脆弱性** が見つかりました

これらは「AI が悪い」のではなく、「AI が書いたコードを誰も検証しなかった」ことが原因です。

### 「速くなった」は錯覚かもしれない

バイブコーディングの最大の魅力は速さです。しかし、[METR（AI 安全性研究機関）の研究](https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/)（2025年7月）で興味深い結果が報告されています。開発者は AI ツールを使って**「20% 速くなった」と感じている** にもかかわらず、実際の計測では**19% 遅くなっていた** のです。約 40% もの認知ギャップがありました。

fast.ai の共同創業者 Rachel Thomas は、この現象を **"junk flow"（偽のフロー状態）** と呼んでいます。次々にコードが生成される達成感はあるものの、実質的な成果や成長を伴わない状態です。コードが動く快感に夢中になり、中身を理解しないまま先に進んでしまう。その結果、バグが出たときに自分で修正できず、AI に「直して」と丸投げし、さらに理解が遠のく悪循環に陥ります。

## 見極める力が武器になる

ここまで読んで、「AI を使うのが怖くなった」と感じる方もいるかもしれません。しかし、メッセージはその逆です。

Simon Willison の言葉を思い出してください。「レビューし、テストし、説明できるなら、それはバイブコーディングではなくソフトウェア開発だ」。つまり、**コードを読んで理解し、検証する力があれば、AI は強力な味方になる** のです。

皆さんは COACHTECH でその力を身につけています。

- **バリデーション**の知識があるから、「入力値がサニタイズされていない」と気づける
- **認証・認可**の知識があるから、「このルートに Policy が設定されていない」と見抜ける
- **テスト**の知識があるから、「正常系しかテストされていない」と判断できる
- **DB 設計**の知識があるから、「このリレーションは N+1 問題を起こす」と指摘できる

AI の台頭で「コードを書く」作業は高速化されました。しかし、「コードを読んで判断する」力の価値はむしろ上がっています。AI が大量のコードを一瞬で生成する時代だからこそ、そのコードを検証できるエンジニアが求められています。

この教材では、次の Chapter から Claude Code を「使いこなす力」と「見極める力」の両方を実践的に養っていきます。Claude Code に的確に指示を出し（使いこなす力）、生成されたコードを正しさ・品質・安全性の3観点で検証する（見極める力）。この2つの力を持つエンジニアは、バイブコーディングではなく「ソフトウェア開発」をしている、と胸を張って言えるはずです。

### やってみよう: 自分の目で確かめる

ここまでの話を「なるほど、気をつけよう」で終わらせず、実際に確かめてみましょう。先ほどバイブコーディングで作ったブログアプリのコードを、今度は自分の目で読んでみてください。

Claude Code を起動して、以下のように聞いてみましょう。

```
> このブログアプリのコードにセキュリティ上の問題がないかレビューしてください
```

Claude Code が指摘する内容を読んでみてください。おそらく、入力値のサニタイズ、ファイル操作の安全性、アクセス制御など、複数の問題が見つかるはずです。

重要なのは、Claude Code の指摘を「そうなんだ」と受け入れることではなく、皆さん自身の知識で「確かにここは危ない」と判断できるかどうかです。COACHTECH で学んだバリデーションや認証の知識を使って、Claude Code の指摘が妥当かどうかを考えてみてください。

これが「見極める力」の第一歩です。

確認が終わったら、`~/claude-test` ディレクトリは削除して構いません。次の Chapter からは、テック記事プラットフォームを Laravel で本格的に構築していきます。

```bash
rm -rf ~/claude-test
```

## まとめ

- **バイブコーディング**とは、AI が生成したコードをレビューせずに受け入れるスタイルのことです。AI にコードを書かせること自体はリスクではありません
- AI 生成コードの 45% が OWASP Top-10 の脆弱性を含むという調査結果（Veracode、2025年）があり、バイブコーディングはセキュリティリスクに直結します
- 「速くなった」という感覚は認知バイアスの可能性があります（METR 研究: 約 40% の認知ギャップ）
- COACHTECH で学んだバリデーション・認証・テスト・DB 設計の知識は、AI の出力を検証するための武器です
- 「レビューし、テストし、説明できるなら、それはソフトウェア開発」。次の Chapter からその実践が始まります
