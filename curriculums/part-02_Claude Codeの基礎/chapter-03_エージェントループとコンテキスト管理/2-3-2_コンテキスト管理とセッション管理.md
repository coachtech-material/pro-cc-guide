# 2-3-2 コンテキスト管理とセッション管理

## 概要

前のセクションで、Claude Code がエージェントループを回しながらタスクを遂行する仕組みを学びました。このセクションでは、そのエージェントループを支える **コンテキスト** と、作業を中断・再開するための **セッション** を扱います。解説しながら、実際に `/context` や `/compact` を試していきます。

実務では、1つの機能を1回のセッションで完成させることはむしろ稀です。数時間にわたる作業でコンテキストが膨らんだり、翌日に作業を再開したり、別の機能に切り替えたりする場面が日常的に発生します。コンテキストとセッションを適切に管理することで、長時間の作業でも Claude Code の応答品質を維持し、作業の文脈を失わずに開発を進められます。

## コンテキストウィンドウ

### コンテキストとは何か

Claude Code が1回のセッションで扱える情報量には上限があります。この上限を **コンテキストウィンドウ** と呼びます。

コンテキストウィンドウには、以下の情報が入っています。

- **CLAUDE.md** の内容
- `@` で添付したファイルの内容
- 会話の履歴（皆さんのプロンプトと Claude Code の応答）
- Claude Code がツールで読み取ったファイルの内容やコマンドの実行結果
- システムの指示（Claude Code が内部で使う情報）

セッションが長くなるほど、会話履歴やファイルの読み取り結果が蓄積され、コンテキストウィンドウが埋まっていきます。

### コンテキストが溢れるとどうなるか

コンテキストウィンドウが上限に近づくと、Claude Code の応答品質が低下します。セッションの初期に伝えた指示を忘れたり、すでに読んだファイルの内容を正しく思い出せなくなったりします。

たとえば、長時間のセッションで「さっき修正した ArticleController の検索ロジックを使って」と指示しても、そのコードがコンテキストから押し出されていると、Claude Code は的外れな応答を返すことがあります。

これは Claude Code の不具合ではなく、コンテキストウィンドウの構造的な制約です。対処法を知っていれば、品質の低下を防げます。

### `/context` で使用量を確認する

現在のコンテキスト使用量は `/context` コマンドで確認できます。

```
> /context
```

カラーグリッドが表示され、コンテキストウィンドウのどれだけが使われているかが視覚的にわかります。MCP サーバーやスキルの定義など、会話以外で消費されている領域も確認できます。

「なんとなく応答が不安定だな」と感じたら、まず `/context` で状況を確認する習慣をつけましょう。

### `/compact` でコンテキストを圧縮する

コンテキストが溢れそうになったら、`/compact` コマンドで会話を圧縮できます。

```
> /compact
```

`/compact` は会話履歴を要約し、詳細なツール出力を削除して、コンテキストの使用量を減らします。皆さんのプロンプトや重要なコードの断片は保持されますが、セッション初期の詳細な指示は失われることがあります。

特定の内容を保持したい場合は、フォーカスパラメータを指定できます。

```
> /compact 認証機能の変更に集中して
```

こうすると、認証に関連する情報を優先的に残し、それ以外を圧縮します。

💡 コンテキストが上限に近づくと、Claude Code は **自動コンパクション** を実行します。手動で `/compact` を使わなくても、ある程度は自動で管理されます。ただし、自動コンパクションはタイミングを選べないため、重要な指示が失われることがあります。品質が気になり始めたら、早めに手動で `/compact` を実行するのがおすすめです。

### `/clear` で会話をリセットする

`/compact` では対処しきれないほどコンテキストが汚れている場合や、まったく別のタスクに切り替えたい場合は、`/clear` で会話をリセットします。

```
> /clear
```

`/clear` を実行すると、会話履歴がすべて消え、新しいセッションが始まります。CLAUDE.md は新しいセッションでも自動的に読み込まれるので、プロジェクトの基本情報は保持されます。ファイルの変更もそのまま残ります。

### `/compact` と `/clear` の使い分け

| 状況 | コマンド | 理由 |
|---|---|---|
| コンテキストが膨らんできたが、同じタスクを続けたい | `/compact` | 会話の要約を残しつつ空きを作る |
| 別のタスクに切り替えたい | `/clear` | 前のタスクのコンテキストは不要 |
| Claude Code が同じ問題を繰り返している | `/clear` | コンテキストが根本的に混乱している |
| 翌日に作業を再開したい | セッション継続（後述） | `/clear` より文脈を保てる |

⚠️ よくある問題: CLAUDE.md の過剰記述

CLAUDE.md にプロジェクトの情報を詳しく書くほど Claude Code の出力は良くなりますが、書きすぎるとコンテキストを圧迫します。CLAUDE.md だけで大量のコンテキストを消費すると、実際の作業に使える領域が減り、かえって指示が無視される原因になります。

CLAUDE.md にはプロジェクトのルールや制約など「毎回のセッションで必要な情報」を書き、一時的な作業メモや詳細な仕様書は含めないようにしましょう。

💡 CLAUDE.md に `## Compact Instructions` というセクションを追加すると、コンパクション時に保持する内容を制御できます。「Sail コマンド体系は常に保持する」「テスト方針は常に保持する」のように、失われてほしくない情報をここに書いておくと、`/compact` や自動コンパクション後も確実に残ります。

## セッション管理

### セッションとは何か

Claude Code での1回の会話を **セッション** と呼びます。`claude` コマンドで起動してから `exit` で終了するまでが1つのセッションです。

セッションは独立しています。新しいセッションを始めると、前のセッションの会話履歴は引き継がれません。ただし、CLAUDE.md の内容はセッションを越えて毎回読み込まれるため、プロジェクトの基本情報は常に共有されます。

重要なのは、セッションの会話データはローカルに保存されているということです。ターミナルを閉じてもセッションは消えません。後から再開したり、別のセッションに切り替えたりできます。

### セッション継続: `claude -c`

2-2-1 で紹介した `claude -c` を、ここで詳しく説明します。

```bash
claude -c
```

`claude -c`（`--continue`）は、現在のディレクトリで最後に使っていたセッションを再開します。翌日に作業を再開するとき、最も頻繁に使うコマンドです。

たとえば、テック記事プラットフォームのプロジェクトディレクトリで作業していたセッションを翌日再開するには、同じディレクトリで `claude -c` を実行するだけです。

```bash
cd tech-article-platform
claude -c
```

会話履歴がすべて復元され、前日の作業の続きから始められます。

### セッション再開: `claude -r`

`claude -c` は「直前のセッション」を再開しますが、`claude -r`（`--resume`）はセッション一覧から選んで再開できます。

```bash
claude -r
```

引数なしで実行すると、過去のセッション一覧が表示され、再開したいセッションを選択できます。セッション名や ID を指定して直接再開することもできます。

```bash
claude -r "auth-refactor"
```

複数の機能を並行して開発しているときに、ブランチや機能ごとにセッションを切り替えるのに便利です。

### セッション命名: `--name` と `/rename`

セッションが増えてくると、どのセッションがどの作業だったか分からなくなります。セッションに名前をつけておくと、後から探しやすくなります。

起動時に名前をつけるには `--name`（`-n`）フラグを使います。

```bash
claude -n "tag-feature"
```

セッションの途中で名前をつけたり変更したりするには `/rename` コマンドを使います。

```
> /rename tag-feature
```

名前をつけたセッションは、`claude -r "tag-feature"` のように名前で再開できます。

💡 機能単位でセッションを命名する習慣をつけましょう。「tag-feature」「auth-refactor」「comment-nested」のように、何の作業をしているかが一目でわかる名前が理想です。

### セッション分岐: `/branch`

作業の途中で「別のアプローチも試してみたい」と思うことがあります。でも、今の会話の流れを壊したくない。そんなときは `/branch` コマンド（エイリアス: `/fork`）でセッションを分岐できます。

```
> /branch
```

`/branch` は現在のセッションのコピーを作り、新しいセッション ID を割り当てます。元のセッションはそのまま残るので、別のアプローチを試した後に元のセッションに戻ることもできます。

Git のブランチと似た考え方です。本流を壊さずに、実験的な変更を試せます。

### コンテキストとセッションの使い分け

ここまで学んだ機能を整理します。「長い作業を効率よく進める」ための両輪として、状況に応じて使い分けてください。

| やりたいこと | 使う機能 | 理由 |
|---|---|---|
| コンテキストが溢れてきた | `/compact` | 会話を要約して空きを作り、同じタスクを続けられる |
| 別のタスクに切り替えたい | `/clear` | 新しいセッションでまっさらに始める |
| 翌日に作業を再開したい | `claude -c` | 前日のセッションを会話履歴ごと再開する |
| 別の機能のセッションに切り替えたい | `claude -r` | セッション一覧から選んで再開する |
| 別ブランチの作業に切り替えたい | `claude -r` | ブランチごとにセッションを分けて管理する |
| 別のアプローチを試したい | `/branch` | セッションを分岐して実験する |

実務では、これらを組み合わせて使います。朝出社したら `claude -c` で昨日の続きを再開し、コンテキストが溢れてきたら `/compact` で圧縮し、別の機能に取りかかるときは `/clear` で新しいセッションを始め、`--name` で名前をつけておく。この流れが自然にできるようになると、長時間の開発でも Claude Code の品質を維持できます。

実際に試してみましょう。今起動しているセッションで `/context` を実行して、現在のコンテキスト使用量を確認してみてください。

```
> /context
```

次に、`/compact` を実行して、コンテキストがどれだけ圧縮されるか確認してみてください。

```
> /compact
```

もう一度 `/context` を実行すると、使用量が減っていることがわかります。この「確認 → 圧縮 → 再確認」のサイクルを、作業中に定期的に行うと効果的です。

## まとめ

- **コンテキストウィンドウ** は Claude Code が1回のセッションで扱える情報量の上限です。溢れると応答品質が低下します
- `/context` で使用量を確認し、`/compact` で圧縮します。特定の内容を残したい場合はフォーカスパラメータを指定します
- `/clear` は会話を完全にリセットします。別のタスクに切り替えるときや、コンテキストが根本的に混乱しているときに使います
- **セッション** はローカルに保存されます。`claude -c` で直前のセッションを再開、`claude -r` で過去のセッションを選んで再開できます
- `--name` でセッションに名前をつけておくと、後から探しやすくなります。`/branch` でセッションを分岐し、別のアプローチを試すこともできます
