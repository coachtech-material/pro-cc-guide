# 2-7-2 Checkpointing と `/rewind`

## 概要

2-7-1 で Git によるバージョン管理を整備しました。Git はコードの長期的な履歴を管理するための仕組みです。しかし、開発中のすべてのタイミングでコミットするわけではありません。

たとえば、Claude Code に「認証システムを JWT に変更して」と大きな指示を出したとします。Claude Code は複数のファイルを一気に変更しますが、結果がうまくいかなかったとき、どうしますか? `git checkout .` で変更を全部捨てますか? でも、一部の変更は残したいかもしれません。

Claude Code には **Checkpointing**（チェックポイント）という仕組みがあります。ファイルの編集を自動的にスナップショットとして記録し、`/rewind` コマンドで任意の時点に巻き戻せます。Git が「保存した履歴」だとすれば、チェックポイントは「元に戻す（undo）」です。

このセクションでは、チェックポイントの仕組みを理解し、`/rewind` で安全に巻き戻す方法を実践します。

## チェックポイントの仕組み

### 自動スナップショット

チェックポイントは、皆さんが意識しなくても自動的に動いています。Claude Code でファイル編集ツール（Edit や Write）が実行されるたびに、変更前の状態が記録されます。

具体的には、**皆さんがプロンプトを送信するたびに**新しいチェックポイントが作成されます。つまり「この指示を出す前の状態」がすべて保存されているということです。

チェックポイントの特徴をまとめます。

- 皆さんのプロンプトごとに自動作成される
- セッションをまたいで保持される（`--continue` で再開した会話でも利用可能）
- 30日後にセッションとともに自動削除される（日数は設定で変更可能）

### 追跡される変更と追跡されない変更

チェックポイントが追跡するのは、**Claude Code のファイル編集ツールによる変更だけ** です。

追跡される変更:
- Claude Code が Edit ツールや Write ツールで編集したファイル

追跡されない変更:
- Bash コマンドによるファイル操作（`rm`、`mv`、`cp` など）
- 皆さんが手動で行ったファイルの編集
- 他の Claude Code セッションからの変更（ただし、現在のセッションと同じファイルを変更した場合は例外的に追跡されることがあります）

⚠️ よくあるエラー: `/rewind` で巻き戻したのにファイルが元に戻らない

```
ファイルを巻き戻しましたが、一部のファイルが元に戻っていません。
```

**原因**: Claude Code が Bash コマンド（`sail artisan` や `rm` など）で変更したファイルはチェックポイントの追跡対象外です

**対処法**: Bash コマンドで変更されたファイルは Git で管理しましょう。重要な変更の前にコミットしておけば、`git checkout` で復元できます

### Git との使い分け

チェックポイントと Git は、役割が異なります。

- **チェックポイント**: セッション中の「ローカル undo」。試行錯誤のために使う。30日で消える
- **Git**: コードの「永続的な履歴」。チームとの共有やリリース管理のために使う

チェックポイントは Git を置き換えるものではなく、**補完する** 仕組みです。大きな変更を試す前にコミットしておき、セッション中の細かな試行錯誤はチェックポイントで巻き戻す。この二段構えが安全な開発スタイルです。

## `/rewind` による巻き戻し

### `/rewind` コマンドを使ってみる

実際にチェックポイントを使ってみましょう。まず、巻き戻す対象を作るために、意図的に大きな変更を加えます。

```bash
claude -n "checkpoint-practice"
```

```
> 記事一覧ページのデザインを大幅に変更してください。カード型レイアウトからテーブル型レイアウトに変更し、各記事にサムネイル画像欄を追加してください。
```

Claude Code が複数のファイルを変更します。変更が完了したら、`/diff` で確認してみてください。多くのファイルが変更されているはずです。

ここで「やっぱりカード型レイアウトのほうがよかった」と思ったとします。`/rewind` で巻き戻しましょう。

```
> /rewind
```

セッション中のプロンプト一覧がスクロール可能なリストで表示されます。巻き戻したい時点のプロンプトを選択すると、5つのアクションから選べます。

1. **Restore code and conversation**: コードと会話の両方をその時点に戻す
2. **Restore conversation**: 会話だけ戻す（コードはそのまま）
3. **Restore code**: コードだけ戻す（会話はそのまま）
4. **Summarize from here**: 選択した時点以降の会話を要約して圧縮する（コードは変更しない）
5. **Never mind**: 何もせずリストに戻る

今回は「カード型レイアウトに戻したい」ので、**Restore code and conversation** を選びます。先ほどの「記事一覧ページのデザインを大幅に変更してください」の一つ前のプロンプトを選択してください。

コードと会話がその時点に巻き戻され、テーブル型レイアウトへの変更がなかったことになります。

どのアクションを選んでも、選択したメッセージのプロンプトが入力欄に復元されます。そのまま再送信することも、内容を編集して別のアプローチで再指示することもできます。

### `Esc+Esc` ショートカット

`/rewind` と同じ操作を、キーボードショートカットでも行えます。`Esc` キーを2回連続で押すと、巻き戻しメニューが開きます。

Claude Code の実行中に「あ、これは違う方向に進んでいる」と感じたら、まず `Esc` で処理を中断し、もう一度 `Esc` を押して巻き戻しメニューを開く。この流れを覚えておくと、素早く軌道修正できます。

### 5つのアクションの使い分け

5つのアクションは、状況に応じて使い分けます。

**Restore code and conversation**（コードと会話を両方戻す）

最もよく使うアクションです。「この指示は失敗だった、なかったことにしたい」という場合に使います。先ほどのレイアウト変更のように、試したけど気に入らなかった場合はこれを選びます。

**Restore conversation**（会話だけ戻す）

コードは今の状態を維持したまま、会話のコンテキストだけ巻き戻したい場合です。たとえば「会話が長くなりすぎて Claude Code の応答が不正確になってきた」ときに、会話だけ巻き戻して新鮮な状態から続けられます。

**Restore code**（コードだけ戻す）

会話の流れは維持しつつ、コードの変更だけ取り消したい場合です。「Claude Code の説明は参考になったが、生成されたコードは使いたくない」というケースで役立ちます。

**Summarize from here**（要約して圧縮する）

コンテキストウィンドウの節約に使います。長いデバッグセッションの途中から要約し、前半の指示は詳細に残しつつ後半を圧縮できます。2-3-2 で学んだ `/compact` の部分適用版です。コードは一切変更しません。要約時にオプションで指示テキストを入力すると、要約のフォーカスを調整できます。また、元のメッセージはセッションのトランスクリプトに保持されるため、Claude Code が必要に応じて詳細を参照することもできます。

💡 「大きな変更を試す前にコミット、セッション中の試行錯誤はチェックポイント」を習慣にしましょう。この二段構えがあれば、Claude Code に大胆な指示を出しても安心です。失敗を恐れずに試せることが、Claude Code を最大限に活用するコツです。

## まとめ

- チェックポイントは Claude Code のファイル編集を自動的にスナップショットとして記録する仕組みです。プロンプトごとに作成され、セッションをまたいで保持されます
- `/rewind` コマンドまたは `Esc+Esc` ショートカットで巻き戻しメニューを開き、5つのアクション（コードと会話の復元、会話のみ復元、コードのみ復元、要約、キャンセル）から選べます
- Bash コマンドによる変更や手動編集はチェックポイントの対象外です。重要な変更の前には Git でコミットしておきましょう
- 「Git で永続的な履歴を管理、チェックポイントでセッション中の試行錯誤を管理」の二段構えで、安全に開発できます
