# 2-7-3 Worktree で並列作業する

## 概要

ここまでの Chapter では、1つの機能を1つずつ順番に実装してきました。「タグ機能を追加して」→ 完了 →「コメント機能を追加して」→ 完了。この進め方はわかりやすいですが、実務では複数の機能を同時に開発することがあります。

たとえば、「いいね機能」と「ブックマーク機能」を同時に開発したいとします。しかし、同じディレクトリで2つの Claude Code セッションを走らせると、一方が変更したファイルをもう一方が上書きしてしまう危険があります。

Git Worktree は、同一リポジトリの**別の作業ディレクトリ**を作成する機能です。Claude Code は `--worktree`（`-w`）フラグでこの機能をサポートしており、それぞれの Worktree で独立したセッションを走らせることで、変更が互いに干渉しない並列開発が可能になります。

このセクションでは、2つの機能を Worktree で並列に開発し、マージして動作確認するまでを体験します。

## Worktree の仕組み

### Git Worktree とは

通常、Git リポジトリには1つの作業ディレクトリしかありません。ブランチを切り替えるには `git checkout` が必要で、切り替えるたびにファイルが書き変わります。

Git Worktree は、同一リポジトリに**複数の作業ディレクトリ**を作ります。各 Worktree は独自のブランチを持ち、ファイルも独立しています。しかし、コミット履歴やリモートの設定は共有しています。

イメージとしては「同じプロジェクトのコピーを別フォルダに作るが、Git の履歴は1つ」です。

### `--worktree` フラグ

Claude Code では `--worktree`（短縮形 `-w`）フラグを使って Worktree を作成し、その中でセッションを開始します。

```bash
claude --worktree feature-likes
```

このコマンドを実行すると、以下のことが起きます。

1. `<リポジトリ>/.claude/worktrees/feature-likes/` にプロジェクトのコピーが作成される
2. デフォルトのリモートブランチ（通常 `main`）から分岐して、`worktree-feature-likes` という名前のブランチが自動作成される
3. その Worktree 内で Claude Code のセッションが開始される

名前を省略すると、ランダムな名前（例: `bright-running-fox`）が自動生成されます。

```bash
claude --worktree
```

### Worktree のクリーンアップ

Worktree のセッションを終了すると、Claude Code は状態に応じた処理を行います。

- **変更なし**: Worktree とブランチが自動削除される
- **変更やコミットがある場合**: 保持するか削除するか確認される。保持すれば後から再開でき、削除すれば未コミットの変更と Worktree ブランチ上のコミットがすべて破棄される

💡 `.claude/worktrees/` を `.gitignore` に追加しておくと、Worktree の内容がメインリポジトリの未追跡ファイルとして表示されなくなります。

## 並列実装

### 2つの機能を同時に開発する

いよいよ、2つの機能を Worktree で並列に開発します。ここからは「終盤」の指示スタイルに移行します。これまでのセクションでは1つずつ具体的に指示してきましたが、ここでは**まとめて要件を伝え、Claude Code に実装判断を委ねます**。

実装する2つの機能:

- **Like（いいね）+ トレンド記事一覧**: 記事にいいねできる機能と、いいね数と新着度を組み合わせたトレンドランキング
- **Bookmark（ブックマーク）+ コレクション管理**: 記事をブックマークし、フォルダに分類して管理する機能

### Worktree A: Like + トレンド記事一覧

ターミナルを開き、プロジェクトのルートディレクトリで以下を実行します。

```bash
claude --worktree feature-likes
```

Worktree 内でセッションが起動したら、まとめて指示を出します。

```
> 記事に「いいね」できる機能を追加してください。
>
> - Like モデルを作成し、User と Article の多対多リレーションとする
> - 記事詳細ページにいいねボタンを設置し、いいね済みかどうかを表示する
> - いいねの追加・取り消しができる
> - トップページにトレンド記事一覧を表示する。スコアは「いいね数 × 新着度（投稿日からの経過日数で減衰）」で計算する
> - Seeder にいいねデータを追加する
```

Claude Code が Like モデル、マイグレーション、コントローラ、ルート、ビュー、トレンド計算ロジック、Seeder を一括で生成します。生成が完了するまで待ちましょう。

### Worktree B: Bookmark + コレクション管理

**別のターミナルウィンドウ**を開きます（Worktree A のセッションはそのまま走らせておきます）。同じくプロジェクトのルートディレクトリで以下を実行します。

```bash
claude --worktree feature-bookmarks
```

```
> 記事をブックマークしてコレクション（フォルダ）で管理できる機能を追加してください。
>
> - Bookmark モデルを作成し、User と Article の多対多リレーションとする
> - コレクション（フォルダ）を作成でき、ブックマークをコレクションに分類できる
> - コレクション内のブックマークは並び順を管理できる
> - 記事詳細ページにブックマークボタンを設置する
> - マイページにブックマーク一覧とコレクション一覧を表示する
> - Seeder にブックマークとコレクションのデータを追加する
```

2つの Worktree が同時に動いています。それぞれが独立したブランチで独立したファイルを編集しているため、互いに干渉しません。

⚠️ よくあるエラー: Sail コンテナの競合

```
Error: Sail container is already running
```

**原因**: 2つの Worktree が同じ Docker コンテナ（Sail）を使おうとしている

**対処法**: Worktree ごとに別の Sail コンテナを起動する必要があります。一方の Worktree で `APP_PORT` を変更するか、Claude Code にデータベースのマイグレーションとシーディングだけを行わせ、ブラウザ確認はマージ後にメインの作業ディレクトリで行う方法もあります。最もシンプルなのは、両方の Worktree の作業が完了してからマージし、メインのディレクトリで `sail up` して動作確認することです

### 両方をマージする

2つの Worktree の作業が完了したら、マージします。メインのプロジェクトディレクトリで新しいセッションを起動します。

```bash
claude -n "merge-features"
```

```
> feature-likes と feature-bookmarks の2つの Worktree ブランチをマージしてください。
> コンフリクトがあれば解決してください。
```

Claude Code は `git merge` を実行し、コンフリクトがあれば自動的に解決を試みます。Like と Bookmark は独立した機能なので、大きなコンフリクトは起きにくいですが、ルートファイル（`routes/web.php`）やシーダー（`DatabaseSeeder.php`）で競合する可能性があります。

皆さんの環境では異なるコードが生成されます。以下はマージ後のルートファイルの一例です。

```php
// routes/web.php（マージ後）
// Like 機能のルート
Route::post('/articles/{article}/like', [LikeController::class, 'toggle'])
    ->middleware('auth');

// Bookmark 機能のルート
Route::post('/articles/{article}/bookmark', [BookmarkController::class, 'toggle'])
    ->middleware('auth');
Route::resource('collections', CollectionController::class)
    ->middleware('auth');
```

マージが完了したら、データベースを更新してブラウザで確認します。

```
> マイグレーションとシーディングを実行して、ブラウザで動作確認できる状態にしてください。
```

```bash
sail artisan migrate:fresh --seed
```

ブラウザで以下を確認しましょう。

- トップページにトレンド記事一覧が表示されているか
- 記事詳細ページにいいねボタンとブックマークボタンがあるか
- いいねの追加・取り消しができるか
- ブックマーク、コレクションの作成・分類ができるか

動作確認ができたら、マージ後の変更をコミットします。

```
> マージ後の変更をコミットしてください。
```

## 見極めチェック

- [ ] **正しさ**: Like モデルと Bookmark モデルが正しく作成され、User と Article へのリレーションが機能しているか。いいねの追加・取り消し、ブックマークの分類・並び替えが要件通りに動作するか
- [ ] **品質**: 並列実装された2つの機能が互いに干渉していないか。マージ後に既存機能（記事一覧、認証、状態遷移など）が壊れていないか。ルートやコントローラの命名規則が既存コードと一貫しているか
- [ ] **安全性**: いいねやブックマークの操作に認証（`auth` ミドルウェア）が適用されているか。他のユーザーのブックマークやコレクションを操作できないよう認可が実装されているか

> このセクションでは特に「品質」に注目してください。並列開発ではそれぞれの実装が独立に進むため、マージ後に初めて問題が発覚します。ルートの重複、モデルのリレーション名の衝突、ビューのレイアウト崩れなど、個別には問題なくてもマージすると壊れるケースに注意しましょう。`/diff` でマージ後の差分を確認し、2-6-1 で作成したテストを `sail test` で実行して既存機能が壊れていないことを確認してください。問題が見つかったら `/rewind` で戻してやり直せます。

## まとめ

- Git Worktree は同一リポジトリに複数の作業ディレクトリを作る仕組みです。Claude Code は `--worktree`（`-w`）フラグで Worktree の作成とセッション開始を一度に行えます
- Worktree ごとに独立したブランチとファイルを持つため、複数の Claude Code セッションが互いに干渉せず並列開発できます
- セッション終了時、変更がなければ Worktree は自動削除され、変更があれば保持・削除を選択できます
- 並列実装後のマージでは、コンフリクトの解決だけでなく、既存機能への影響もテストと目視で確認しましょう

次の Chapter では、MCP（Model Context Protocol）で外部ツールに接続し、Sub-agents で探索を委譲する方法を学びます。Claude Code の「つながる力」を拡張していきます。
