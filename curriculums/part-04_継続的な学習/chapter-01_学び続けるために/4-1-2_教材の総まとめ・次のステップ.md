# 4-1-2 教材の総まとめ・次のステップ

## 🎯 このセクションで学ぶこと

- 教材全体で養った3つの能力を振り返り、自分の成長を確認する
- Claude Code との協働で「変わるもの」と「変わらないもの」を理解する
- ジュニアからミドル相当のバリューを出すための具体的な振る舞いを知る

まず教材全体を3つの能力の視点で振り返り、次に Claude Code との協働の本質を整理し、最後にここから先の成長の道筋を描きます。

## 導入: ここまでの道のりを振り返る

Part 1 からここまで、あなたは Claude Code の「使い方」だけでなく「付き合い方」を学んできました。

Part 1 では「なぜ今 Claude Code を学ぶのか」を理解し、3つの能力という学習の軸を得ました。Part 2 では個々の機能を手を動かしながら習得しました。Part 3 では CourseHub という実務に近い環境で、バグ修正から機能開発、リファクタリング、チーム共有まで一気通貫のタスクを遂行しました。そして Part 4 のここまでで、学び続けるためのサイクルを手に入れました。

この教材を通じて得たものを、3つの能力の視点で整理しましょう。

### 🧠 先輩エンジニアはこう考える

> 教材を一通り終えたとき、「結局どこまでできるようになったんだろう」と不安になることがある。でも、振り返ってみると、最初の頃とは明らかに違う判断ができるようになっている。Claude Code に何を任せて何を自分で判断するか、その線引きが自然とできるようになっていること自体が、成長の証拠だと思う。

## 3つの能力の振り返り

### 使いこなす力

Claude Code の機能、設定、制約を把握し、目的に応じて的確に活用する力です。

**Part 2 で基礎を固めた**

Part 2 では、Claude Code の機能を一つずつ学びました。

| Chapter | 学んだこと |
|---|---|
| Chapter 2-1 セットアップ | インストール、権限設定、CLAUDE.md、セッション管理 |
| Chapter 2-2 基本を理解する | エージェントループ、コンテキストウィンドウ、プロンプト設計 |
| Chapter 2-3 機能を使いこなす | Plan Mode、Skills、Hooks、MCP、Sub-agents、Plugins、Git/Worktree、GitHub Actions |

この段階では「機能を知っている」状態です。各機能を個別に体験し、何ができるかを把握しました。

**Part 3 で実務に応用した**

Part 3 では、Part 2 で学んだ機能を組み合わせて実務タスクを遂行しました。

| 実務タスク | 組み合わせた機能 |
|---|---|
| コードリーディング | Sub-agents による並列探索、MCP でデータベース確認、`/btw` で補足質問 |
| バグ修正 | プロンプト設計（症状→再現→期待値）、`/test` で検証、`/rewind` で巻き戻し |
| 機能開発 | Plan Mode で計画、`--continue` で途中再開、Hooks で自動フォーマット |
| リファクタリング | Worktree で並列作業、`/cost` でトークン管理 |
| チーム共有 | `/commit`・`/diff` で Git 操作、GitHub Actions で自動レビュー、Skills でワークフロー化 |

「機能を知っている」から「タスクに応じて機能を選択できる」への転換が、使いこなす力の成長です。

### 見極める力

AI が生成したコードを、正しさ・品質・安全性の3つの観点で検証し、責任を持って採用する力です。

Part 3 では、Chapter ごとに見極めの重点を変えながら、この力を鍛えました。

| Chapter | 見極めの重点 | 鍛えたこと |
|---|---|---|
| 3-3 バグ修正 | **安全性** | 認可の欠落、権限チェック漏れの検出 |
| 3-4 機能開発 | **品質** | 既存の設計パターンとの整合性、コーディング規約の遵守 |
| 3-5 リファクタリング | **正しさ** | 振る舞いを変えずに構造を改善する判断 |
| 3-6 チーム共有 | **品質** | 説明責任を意識した PR、環境設定の適切な分離 |

繰り返しの見極めチェックを通じて、3つの観点が自然と頭に浮かぶようになったはずです。これは、チームメンバーのコードをレビューするときにも、自分が書いたコードをセルフレビューするときにも使える力です。

> 💡 見極める力は、Claude Code を使わない場面でも活きます。他の AI ツールを試す場合、スタックオーバーフローからコードを拾う場合、あるいはチームメンバーの PR をレビューする場合にも、正しさ・品質・安全性の3観点は判断の軸になります。

### 学び続ける力

ツールや技術の進化を追い、自ら試し、実務に適応し続ける力です。

Part 3 では、知らない技術（Service クラス、Event/Listener など）に遭遇したとき、Claude Code を活用して理解する体験をしました。4-1-1 では、情報収集→検証→適用判断のサイクルを学びました。

この力は、他の2つの能力を支える土台です。Claude Code の機能が増えれば「使いこなす力」のアップデートが必要になり、AI の出力精度が向上すれば「見極める力」の基準も変わります。学び続ける力があれば、他の2つの能力は常に最新の状態に保たれます。

```
使いこなす力              見極める力
        │                       │
        └───────────┬───────────┘
                    │
              学び続ける力
            が土台として支える
                    │
                    ↓
      変化に適応し続けるエンジニア
```

## Claude Code との協働で変わるもの、変わらないもの

Claude Code を使いこなせるようになったあなたに、一つ整理しておきたいことがあります。Claude Code との協働によって「変わるもの」と「変わらないもの」の区別です。

### 変わるもの

**開発速度が変わる**

定型的なコードの生成、テストの作成、リファクタリングの実行が大幅に速くなります。Part 3 で体験した通り、数時間かかるタスクが数十分で完了することもあります。

**探索の範囲が変わる**

大規模なコードベースの理解、知らないライブラリの調査、複数ファイルにまたがる影響範囲の分析。これまでは経験豊富なシニアエンジニアにしかできなかった広い範囲の探索が、Claude Code を活用すれば可能になります。

**学習の速度が変わる**

Part 3 で Service クラスや Event/Listener に初めて出会ったとき、Claude Code に質問しながら理解を深めたことを思い出してください。新しい技術を学ぶ速度が圧倒的に上がります。「まず動くものを見て理解する」というアプローチが取れるようになります。

### 変わらないもの

**コードへの責任は変わらない**

Claude Code が書いたコードであっても、それをプロジェクトに採用した責任はあなたにあります。「AI が書いたのでわかりません」は通用しません。3-6-1 で学んだ説明責任の考え方は、AI ツールを使う限りずっとついてまわるものです。

**設計判断は変わらない**

「このテーブルを正規化すべきか」「この処理を同期で行うか非同期で行うか」「この機能をどのレイヤーに配置するか」。こうしたアーキテクチャレベルの判断は、ビジネス要件や運用環境、チームの技術力を総合的に考慮して行うものであり、AI に委任できる領域ではありません。Claude Code は選択肢を提示してくれますが、最終的に選ぶのはあなたです。

**説明責任は変わらない**

PR のレビューで「なぜこの実装にしたのか」を問われたとき、「Claude Code がそう提案したから」では不十分です。実装の意図とトレードオフを自分の言葉で説明できることが、チームで働くエンジニアの前提です。

> 🔑 「変わるもの」は Claude Code が与えてくれる**レバレッジ** （てこ）です。「変わらないもの」はあなた自身が持ち続ける **責任** です。レバレッジを最大限に活かしながら、責任を果たし続けること。これが Claude Code との協働の本質です。

### 🧠 先輩エンジニアはこう考える

> AI ツールが進化しても「コードの責任は書いた人にある」という原則は変わらない。変わったのは「書いた人」の定義。Claude Code にコードを生成させ、それを採用したなら、あなたが「書いた人」です。レビュアーに聞かれて答えられないコードを出すのは、AI を使っていようがいまいが同じこと。むしろ、AI ツールを使うことで「なぜこのコードなのか」を説明する力がより重要になっている。

## 70-30 モデルの先にあるもの

3-1-1 で紹介した 70-30 モデルを振り返りましょう。Addy Osmani 氏の「[The 70% Problem: Hard Truths about AI-Assisted Coding](https://addyo.substack.com/p/the-70-problem-hard-truths-about)」が提唱する、AI がタスクの約 70% を処理し、人間が残りの 30%（設計判断、エッジケース、セキュリティ、説明責任）を担う、という考え方です。

Part 3 を通じて、あなたはこの 30% の判断を繰り返し実践してきました。ここで、この 30% をもう少し深く掘り下げます。

### ジュニアの 30% とミドルの 30% は何が違うか

同じ「30% の判断」でも、ジュニアとミドルでは判断の深さが異なります。

| 判断の観点 | ジュニアの 30% | ミドルの 30% |
|---|---|---|
| 正しさ | 要件を満たしているか確認する | 要件の曖昧さを指摘し、確認してから実装する |
| 品質 | 既存の規約に従う | 規約の妥当性を評価し、改善を提案する |
| 安全性 | チェックリストに沿って確認する | 潜在的なリスクを自ら想定する |
| 設計 | 既存パターンに合わせる | パターンの適用が適切かを判断し、代替案を検討する |
| 説明 | 何をしたかを報告する | なぜそうしたかを、トレードオフを含めて説明する |

この教材を終えた時点で、あなたはジュニアの 30% を確実に遂行できる状態になっています。ここからミドルの 30% に近づくためには、実務での経験を積みながら、以下の3つの振る舞いを意識していく必要があります。

### ミドル相当のバリューを出す3つの振る舞い

**1. 「なぜ」を問い続ける**

Claude Code が提案した実装をそのまま採用するのではなく、「なぜこのアプローチなのか」「他の選択肢はないか」を問いかけます。

```
> この実装でユーザー認証を行っていますが、他にどのようなアプローチがありますか？
> それぞれのメリット・デメリットを教えてください。
> このプロジェクトの規模・要件を考慮した場合、どれが最適ですか？
```

選択肢を知ったうえで「このプロジェクトではこれが最適だから採用した」と説明できる。これがミドル相当の判断です。

**2. 影響範囲を自分で考える**

Claude Code は変更の直接的な影響範囲は分析してくれますが、間接的な影響（パフォーマンスへの影響、他チームの機能との関連、将来の拡張性）まで考慮するのは人間の役割です。

変更を加えるとき、以下を自問します。

- この変更は、他のエンドポイントのレスポンスタイムに影響しないか
- この設計は、半年後に想定される機能追加に対応できるか
- この変更によって、他のチームメンバーのタスクに影響が出ないか

**3. ドキュメントとして残す**

コードの変更だけでなく、判断の経緯を残します。PR の説明文、コミットメッセージ、CLAUDE.md の更新。「なぜこの判断をしたか」が記録されていれば、将来の自分やチームメンバーが同じ判断に悩む時間を節約できます。

Part 3 で Skills として定義したワークフロー、CLAUDE.md に追記した規約、PR の説明文に書いた変更の意図。これらはすべて「判断を残す」行為です。この習慣を実務でも続けてください。

## 具体と抽象の行き来を振り返る

1-1-1 で、AI との協働の本質は「具体と抽象の行き来」であると学びました。ゴールを構造化して AI に伝え、AI が生成した具体的なコードを検証し、次のサイクルに進む。この教材を通じて、あなたはこのサイクルを何度も実践してきました。

ここで、Part 3 の各タスクを振り返ってみましょう。

```
Part 3 で実践した「具体と抽象の行き来」

Chapter 3-2 コードリーディング:
  具体（コード）→ 抽象（全体像・業務フローの理解）
  「個々のファイルを読み、プロジェクト全体の構造を頭の中で組み立てた」

Chapter 3-3 バグ修正:
  具体（エラー報告）→ 抽象（原因仮説の構造化）→ 具体（修正・検証）
  「症状から原因を推測し、調査範囲を絞り、修正が他に影響しないか検証した」

Chapter 3-4 機能開発:
  抽象（要件）→ 分解（タスク分割）→ 具体（実装・テスト）
  「要件を機能単位に分解し、既存パターンに合わせて実装し、検証した」

Chapter 3-5 リファクタリング:
  具体（問題のあるコード）→ 抽象（改善方針）→ 具体（改善後のコード + テスト）
  「Fat Controller という具体から責務分離という抽象を導き、具体のコードに落とした」

Chapter 3-6 チーム共有:
  具体（変更内容）→ 抽象（判断の意図・方針）→ 具体（PR・ドキュメント）
  「コードの変更を、なぜそうしたかという判断と合わせてチームに伝えた」
```

どの Chapter でも、あなたは具体と抽象の間を往復していました。バグ修正では具体的なエラーから抽象的な原因仮説を立て、機能開発では抽象的な要件を具体的なタスクに分解しました。このサイクルを意識的に回せるようになったことが、教材を通じた最大の成長です。

### この力が実務で活きる場面

具体と抽象の行き来は、Claude Code を使う場面だけでなく、エンジニアとしてのあらゆる活動の土台になります。

| 場面 | 具体 → 抽象 | 抽象 → 具体 |
|---|---|---|
| 要件定義 | クライアントの個別要望を整理して、機能要件にまとめる | 機能要件を画面・API・DB 設計に落とし込む |
| コードレビュー | 差分の1行ずつを読み、設計意図や品質基準に照らす | 抽象的な「改善すべき」を具体的な修正案として伝える |
| 障害対応 | ログやエラーの具体から、影響範囲と原因を構造化する | 対応方針を具体的な手順に分解して実行する |
| 技術選定 | 候補ライブラリの具体的な特性を比較して、評価軸を抽出する | 評価基準に基づき、プロジェクトに最適な選択を決める |

1-1-1 では「AI の進化で指示の抽象度が上がり続ける」と述べました。この傾向は今後も続きます。AI がより高い抽象度の指示を理解できるようになるほど、あなたの「分解する力」と「検証する力」の価値は高まり続けます。なぜなら、抽象度が上がるほど AI の解釈の自由度が増し、意図通りの結果を得るための構造化がより重要になるからです。

### 🧠 先輩エンジニアはこう考える

> 教材を一通り終えて振り返ると、結局やっていたのは「大きな問題を小さく分けて、一つずつ確認する」ということだった。バグ修正もリファクタリングも機能開発も、根っこは同じ。これは AI を使うかどうかに関係なく、エンジニアの基本動作だと思う。
>
> ただ、AI を使うようになって、この基本動作の「精度」が問われるようになった。自分で書いていた頃は多少曖昧でも手を動かしながら調整できた。でも AI に任せるなら、分解の粒度が甘いとすぐに結果に出る。だから AI との協働は、自分の思考の精度を鍛えるトレーニングでもある。この力は AI がなくなっても、チームが変わっても、技術が変わっても、ずっと使い続けられる。

## ここから先の道筋

この教材を終えたあなたには、いくつかの成長の方向があります。

### 実務で経験を積む

最も重要なのは、実際のプロジェクトで Claude Code を使い続けることです。教材で学んだことは「型」であり、型を自分のものにするには反復が必要です。

実務で意識するポイントをまとめます。

- **毎回の PR で見極めチェックを行う**: 正しさ・品質・安全性の3観点を習慣にする
- **CLAUDE.md を育てる**: プロジェクトの進行に合わせて更新し続ける
- **失敗から学ぶ**: Claude Code の出力をそのまま採用して問題が起きた場合、「どの段階で気づけたか」を振り返る

### Claude Code の活用を深める

次のセクション（4-1-3）で紹介する発展機能は、使いこなす力をさらに高めるための選択肢です。すべてを使う必要はありません。自分の開発スタイルに合うものを見つけて、4-1-1 で学んだサイクルに沿って検証・導入していきましょう。

### AI ツール全般への応用

この教材で学んだ考え方は、Claude Code に限らず、あらゆる AI コーディングツールに応用できます。

- **使いこなす力**: どのツールにも固有の機能・設定・制約がある。それを把握するアプローチは同じ
- **見極める力**: 正しさ・品質・安全性の3観点は、どの AI が生成したコードにも適用できる
- **学び続ける力**: 情報収集→検証→適用判断のサイクルは、あらゆる技術の学習に使える

将来、新しい AI ツールが登場したとき、この教材で身につけた「考え方」が最大の資産になります。

### 🧠 先輩エンジニアはこう考える

> ジュニアからミドルへの成長は、ある日突然起きるものじゃない。「今日の PR で、なぜこの実装にしたか説明できた」「今日のレビューで、セキュリティの懸念を先に指摘できた」。そういう小さな判断の積み重ねが、気づいたときにはミドル相当の動き方になっている。Claude Code はその成長を加速させてくれるツールだけど、成長するのはあなた自身。この教材で学んだ「型」を、実務で毎日使い続けてほしい。

## ✨ まとめ

- **使いこなす力**: Part 2 で機能を学び、Part 3 でタスクに応じた機能の選択ができるようになった
- **見極める力**: 正しさ・品質・安全性の3観点で AI 生成コードを検証する習慣が身についた
- **学び続ける力**: 情報収集→検証→適用判断のサイクルと、知らない技術への対処フローを手に入れた
- Claude Code との協働で**変わるもの** （速度・範囲・学習効率）と **変わらないもの** （コードへの責任・設計判断・説明責任）を区別する
- 教材全体を通じて実践してきたのは**具体と抽象の行き来** であり、この力は AI の進化に関係なくエンジニアとしての土台になる
- ジュニアからミドルへの成長は、「なぜを問い続ける」「影響範囲を自分で考える」「判断をドキュメントとして残す」の3つの振る舞いで加速する
- この教材で学んだ考え方は Claude Code に限らず、あらゆる AI ツール・技術の学習に応用できる
