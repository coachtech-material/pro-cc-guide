# outline の作成・更新・検証

outline に関する操作を一貫して行う。3つのモードを持ち、単独でも組み合わせても使用できる。

**入力**: $ARGUMENTS

## モード判定

入力からモードとスコープを判定する。明示的なモード指定がない場合は、入力の意図から推測する。

### モード

| モード | トリガーとなる表現 | 内容 |
|---|---|---|
| **再構成** | 「追加」「削除」「並べ替え」「統合」「分割」「番号変更」「再構成」 | Part/Chapter の構成変更 + カスケード更新 |
| **検証** | 「ブラッシュアップ」「MECE」「整理」「分解を確認」「検証」 | MECE 構造の検証・改善 + CLAUDE.md コンセプト照合 |
| **照合** | 「公式ドキュメント」「最新化」「照合」 | 公式ドキュメントとの差分検出・反映 |

複合指示の場合は複数モードを順に実行する（例: 「再構成してから照合して」→ 再構成 → 照合）。
モードが不明な場合はユーザーに確認する。

### スコープ

| 指定形式 | 対象範囲 | 例 |
|---|---|---|
| `Part番号` | その Part 全体 | `2` → Part 2 |
| `Part番号-Chapter番号` | その Chapter | `2-3` → Chapter 2-3 |
| `Part番号-Chapter番号-Section番号` | その Section | `2-3-1` → Section 2-3-1 |
| 省略 | ユーザーの指示内容から推測、または確認 | |

**階層レベル**（検証モードのみ、任意）: `Chapter` / `Section` / `Subsection` を指定すると、検証対象の階層を絞れる。

**例**:
- `/outline 再構成 2` → Part 2 を再構成
- `/outline 検証 2-3 Subsection` → Chapter 2-3 の Subsection レベルを検証
- `/outline 照合 3` → Part 3 を公式ドキュメントと照合

## 再構成モード

Part/Chapter/Section の追加・削除・並べ替え・統合・分割を行い、影響する全ファイルを一括更新する。

### ステップ 1: 変更計画の策定

ユーザーの指示に基づき、以下を提示して承認を得る:

- **変更内容の概要**: 何をどう変えるか
- **影響範囲の一覧**:
  - outline ファイル（`outline/part-XX.md`）
  - 依存関係マップ（`outline/CLAUDE.md`）
  - root `CLAUDE.md`（教材構成テーブル、フォルダ構造）
  - Skills（`.claude/skills/review-chapter/SKILL.md` の初出マップ、`.claude/skills/write-chapter/SKILL.md` の Part 参照）
  - コンテンツファイル（`curriculums/` 配下で Part 構成を参照している Section）
- **番号変更マッピング**（該当する場合）: 旧番号 → 新番号

> 💡 Part レベルの変更（統合・分割・番号変更）は影響範囲が大きい。計画段階で漏れがないよう、Grep で旧 Part 名・旧番号の参照箇所を事前に洗い出す。

### ステップ 2: outline の更新

承認された計画に基づき、outline ファイルを更新する。操作の規模に応じて Write（完全書き直し）または Edit（部分更新）を使い分ける。

1. **`outline/part-XX.md`**: Chapter/Section の追加・削除・移動・番号振り直し
2. **`outline/CLAUDE.md`**: 依存関係マップの更新
3. **前後の Chapter の調整**: 「扱わないこと」や接続文の修正

**Part レベルの変更時の追加操作**:
- Part ファイルの作成・削除・リネーム
- MECE 設計方針テーブルの更新

**番号がずれる場合**: 上書きを防ぐため、**番号が大きい方から順に**リネームする。

### ステップ 3: カスケード更新

outline 以外の影響ファイルを更新する。

| 対象ファイル | 更新内容 |
|---|---|
| `CLAUDE.md`（root） | 教材構成テーブル、フォルダ構造の Part 範囲 |
| `.claude/skills/review-chapter/SKILL.md` | 初出マップ（Section 番号・復習箇所の更新） |
| `.claude/skills/write-chapter/SKILL.md` | Part 番号参照（テンプレート、記述パターンの Part 参照） |
| `curriculums/` 配下のコンテンツファイル | Part 構成を説明している Section（例: 1-1-3）の書き直し |

> Chapter レベルの変更で影響が outline 内に閉じる場合、このステップは初出マップの更新のみで済む。

### ステップ 4: 整合性検証

変更後の整合性を確認する:

1. **番号の整合**: outline 内の Chapter 番号・Section 番号に抜け・重複がないか
2. **相互参照**: Grep で旧番号・旧 Part 名の残留を検索し、漏れを修正
3. **依存関係マップ**: `outline/CLAUDE.md` の依存関係マップが outline の Chapter 構成と一致しているか
4. **初出マップ**: `.claude/skills/review-chapter/SKILL.md` の初出マップが outline の Subsection と整合しているか

問題があれば修正し、結果を報告する。

## 検証モード

outline の構造（MECE 分解・順序）を設計原則に基づいて検証・改善する。CLAUDE.md コンセプトの分解漏れも検出する。

### 設計原則

検証の基準となる原則。outline は以下のゴール構造を持つ:

```
CLAUDE.md 目標（ゴール）: 3つの能力（使いこなす力・見極める力・学び続ける力）
└── Part ゴール: 能力を具体化（part-XX.md の → ゴール）
    └── Chapter ゴール: Part ゴールを分解（part-XX.md の **ゴール**）
        └── Section ゴール: Chapter ゴールを分解（part-XX.md の各 Section の ゴール）
            └── Subsection ゴール: Section ゴールを分解（part-XX.md の各 Subsection の — 以降）
                └── Sub-subsection: Subsection 内の具体項目（part-XX.md の各 Subsection 配下の箇条書き）
```

各レベルのゴールは上位のゴールを MECE に分解したものであること。

| # | 原則 | 内容 |
|---|---|---|
| 1 | **階層的 MECE 分解** | 各レベルは上位を MECE に分解する。Subsection のゴールを合わせると Section ゴールになること |
| 2 | **抽象度の一貫性** | 同じ親の下の兄弟要素は同じ抽象度であること |
| 3 | **抽象→具体** | 上位ほど抽象（能力・状態）、下位ほど具体（機能・操作） |
| 4 | **各レベル 2〜4 個** | 5 個以上ならグルーピング不足。1 個ならそのレベルが不要 |
| 5 | **学習の流れ** | 理解（何/なぜ）→ 活用（どう使う）→ 実践（ハンズオン）の順 |
| 6 | **分解軸の選択** | Subsection は Section ゴールを**単一の軸**で分解する。軸は Section の性質から自然に導く。兄弟が同じ軸上にあること（軸の混在禁止） |

**補足ルール**:
- ハンズオンは MECE の分解対象外。混合パターンでは常に末尾に配置する
- 手順パターンはフェーズ（時系列）分解を優先する
- Sub-subsection にはゴールを書かない（タイトルから自明）

**分解軸の例**（例示であり網羅ではない）:

| 軸 | 適する Section | 例 |
|---|---|---|
| 能力の側面 | 「○○できる」系 | 概念の理解 / 管理手法 |
| 機能・カテゴリ | 「○○を使い分ける」系 | セッション管理 / 情報確認 / 開発支援 |
| フェーズ | 手順パターン | インストール → 確認 → 診断 |
| 場面・状況 | ベストプラクティス系 | 個人での実践 / チームでの運用 |

### Phase 1: 構造検証

#### 1-1. 現状分析

`outline/part-XX.md` を Read し、対象スコープの全レベルで上記の設計原則への適合を確認する:

**Level 1: Part → Chapter**
- Chapter ゴールを合わせると Part ゴールの MECE 分解になっているか？

**Level 2: Chapter → Section**
- Section ゴールを合わせると Chapter ゴールの MECE 分解になっているか？

**Level 3: Section → Subsection**
- Subsection ゴールを合わせると Section ゴールの MECE 分解になっているか？
- Subsection 数は 2〜4 個か？（ハンズオンは MECE の分解対象外）

**Level 4: Subsection → Sub-subsection**
- Sub-subsection を合わせると Subsection ゴールの MECE 分解になっているか？
- Sub-subsection 数は 2〜4 個か？

**全レベル共通の観点**:
- **原則 2（抽象度の一貫性）**: 兄弟要素が同じ抽象度か？
- **原則 6（分解軸の選択）**: 単一の軸で分解されているか？軸が混在していないか？

#### 1-2. 再構成案の作成

**Level 1〜2（Chapter・Section レベル）の問題**:
検出と報告を行い、対応方針をユーザーと相談する。Chapter ゴールや Section ゴールの文言修正はインラインで対応できるが、構造的な変更が必要な場合はユーザーの判断を仰ぐ。

**Level 3〜4（Subsection・Sub-subsection レベル）の問題**:
以下の形式で再構成案を作成する:

```
Section X-X-X: 現状 → 提案
  Before: [フラットなリスト N 個]
  After: [Subsection A — ゴール / Subsection B — ゴール]
  分解軸: [選択した軸]
  理由: [なぜこの軸が適切か]
```

#### 1-3. 順序の検証

再構成後の outline 全体を対象に、全レベルの順序を検証する:

- **原則 5（学習の流れ）**: 理解（何/なぜ）→ 活用（どう使う）→ 実践（ハンズオン）の順になっているか？
- **原則 3（抽象→具体）**: 上位ほど抽象、下位ほど具体の流れになっているか？
- **依存関係**: 前提が先に配置されているか？
- **Section タイトルとの整合**: 問いかけ型のタイトルの場合、最初の Subsection がその問いに答えているか？

#### 1-4. ユーザー承認と適用

再構成案と順序変更案を一覧で提示し、承認を得て `outline/part-XX.md` に適用する。

### Phase 2: CLAUDE.md コンセプト照合

outline はピラミッドストラクチャーである。CLAUDE.md の抽象的なコンセプト（頂点）を、Part → Chapter → Section → Subsection → Sub-subsection と各階層で MECE に分解し、下位ほど具体化する。この Phase では、トップダウンの分解に漏れ・ダブり・曖昧さがないかを検証する。

```
CLAUDE.md 背景・目的 / 目標（頂点: 抽象）
└── Part ゴール（学習フェーズによる具体化 — 1つの能力が複数 Part にまたがりうる）
    └── Chapter ゴール（MECE 分解）
        └── Section ゴール（MECE 分解）
            └── Subsection ゴール（MECE 分解）
                └── Sub-subsection（MECE 分解 — ゴールはタイトルから自明）
```

#### 2-1. 頂点の要素抽出

CLAUDE.md から、対象 Part に関連する上位コンセプトを抽出する:

- **背景・目的（コンセプト）**: 教材の哲学・動機づけ
- **目標（ゴール）**: 3つの能力とその下位要素

#### 2-2. 階層ごとの分解検証

各要素について、ピラミッドの頂点から末端まで分解の連鎖を追跡する。各階層で以下を確認する:

| 判定 | 基準 |
|---|---|
| **OK** | 上位の要素が適切な階層で MECE に分解されている |
| **不十分** | 分解先はあるが、表現が曖昧で上位の意図が正確に伝わらない |
| **欠落** | 上位の要素が下位に分解されていない（ピラミッドの断絶） |

#### 2-3. レポートと修正

「不十分」「欠落」の要素を一覧で提示し、修正案を示す。ユーザーの承認を得て適用する。

## 照合モード

outline の内容を公式ドキュメントと照合し、差分を検出・反映する。

### Phase 1: 照合

#### 1-1. URL 抽出とエージェント起動

対象の outline から公式ドキュメント URL を抽出し、**Chapter ごとにサブエージェントを並列起動**する。各エージェントに以下を指示する:

- 担当 Chapter の公式ドキュメント URL を全て WebFetch する
- outline の Subsection / Sub-subsection と公式ドキュメントの内容を照合する
- 以下の3観点で差分を報告する:
  1. **公式ドキュメントにあるが outline に無い**機能・コマンド・設定
  2. **outline にあるが公式ドキュメントと異なる**（名称変更・廃止・誤り）
  3. **outline の記述が不十分**な箇所（重要な詳細の欠落）

#### 1-2. 結果の統合と分類

全エージェントの結果を統合し、以下に分類する:

| 分類 | 基準 | 例 |
|---|---|---|
| **HIGH** | 誤り・廃止・リネーム。outline が不正確 | コマンド名の変更、存在しない設定項目 |
| **MEDIUM** | 欠落・追加が望ましい。outline は不正確ではないが不十分 | 新機能の追加、詳細の補完 |

#### 1-3. レポート提示

```
## HIGH: 誤り・廃止（修正必須）
| # | 箇所 | 問題 | 修正内容 |
...

## MEDIUM: 追加が望ましい
| # | 箇所 | 内容 |
...
```

### Phase 2: 修正適用

1. **HIGH の適用**: ユーザー承認を得て全て適用
2. **MEDIUM の適用**: HIGH 完了後、一括適用してよいかユーザーに確認
3. **完了報告**: 適用した修正の一覧を報告
