# Chapter のレビュー

問題点は自動修正せず、すべて報告のみとする。

**入力**: $ARGUMENTS

**解釈ルール**:
- 入力を `Part番号-Chapter番号` として解釈する（例: `2-3` → Part 2 の Chapter 3）
- Chapter ディレクトリパス: `curriculums/part-XX_タイトル/chapter-XX_タイトル/`（タイトルは `outline/part-XX.md` から取得）

## 1. 準備

1. `outline/part-XX.md` を Read し、Part ゴール・Chapter ゴール・Section 構成（ゴール・Subsection）を確認する
2. `outline/CLAUDE.md`（自動ロード）の依存関係マップで依存パターン（線形/非線形）を確認する。線形の場合は `outline/part-XX.md` のアプリ進行表から具体的な依存内容を把握する
3. 該当 Chapter ディレクトリ内のすべての Section ファイルを Read する
4. 前後の Chapter の Section ファイルを Read する（未執筆の場合は `outline/part-XX.md` の設計情報で代替し、接続文の妥当性を確認する）
5. `curriculums/` 配下に既存 Chapter が2つ以上あるか確認する（ステップ3の横断整合性チェックの実行判定に使用）

## 2. Chapter 単体レビュー

以下の5観点で検証する。

### 構成原則の準拠

`.claude/skills/write-chapter/SKILL.md` の Section ファイルの構成原則を Read し、実際の Section ファイルが準拠しているかを確認する:
- 固定要素（`# タイトル`, `## 概要`, `## 公式ドキュメント`, `## まとめ`）が揃っているか
- `## まとめ` が箇条書き形式か
- `---`（水平線）が使われていないか
- タイトル形式（`# X-X-X タイトル`、コロンなし）が正しいか
- 混合パターンの Section で独立した `## ハンズオン` セクションが作られていないか（解説一体型であること）
- 混合パターンの `## 概要` に、解説と実践が一体であることが読者に伝わる記述があるか
- 見極めチェックが3観点（正しさ・品質・安全性）すべてに具体的なチェック項目を MECE に列挙しているか

### 設計との整合

`outline/part-XX.md` の設計と実際の内容を照合する:
- ゴール構造（Part → Chapter → Section → Subsection）が MECE に分解されているか
- Section 構成の過不足はないか
- 各 Section のゴールが達成される内容になっているか
- ハンズオン方針が反映されているか

### 正確性

公式ドキュメントと照合し、内容が正確かを確認する:
- 単独実行時: `outline/part-XX.md` の参考文献 URL を WebFetch で取得し照合する
- `/write-chapter` から呼び出された場合: ステップ4で照合済みのため構造的チェックに集中する

コード例・コマンドがある場合:
- コードブロックの言語指定、コマンドの実在性、前提条件の説明有無

概念パターンの Section では:
- 公式ドキュメントの説明・用語と教材の記述が一致しているか
- 機能や環境の列挙に漏れ・誤りがないか
- 年号・時系列の事実関係が正確か

### 一貫性

既存 Chapter との一貫性を確認する:
- 本 Skill の初出マップと照合し、初出・復習の区別が正しいか
- Chapter 内に登場するコマンド/機能が初出マップに漏れなく記載されているか
- 機能の学習マップや一覧表がある場合、初出マップおよび outline の設計と矛盾がないか
- 前後の Chapter との接続が自然か（前提概念の言及、先取りの有無）
- `outline/CLAUDE.md` の依存パターンおよび `outline/part-XX.md` のアプリ進行表との一致

### CLAUDE.md 準拠

CLAUDE.md に定義されたルールへの準拠を確認する:
- 文体: 敬体（です・ます調）で統一されているか
- 絵文字: ⚠️、💡 の2種のみ使用しているか（👀 は廃止。見極めチェックは `## 見極めチェック` として配置）
- 用語:
  - WSL / MCP: 初出時に正式名称が記載されているか
  - Pro生: Chapter の初出 Section で定義し、以降「皆さん」を使っているか
  - Claude Code の機能名: 英語のまま使用しているか（Hooks, Skills, Sub-agents 等）
- 違反箇所は正しい表記とともに報告する

## 3. 横断整合性チェック

> `curriculums/` 配下に既存 Chapter が2つ以上ある場合のみ実行する。1つ以下の場合はスキップする。

教材を通しで読んだときに自然かを検証する。Chapter 数が多い場合は Agent ツールでサブエージェントを起動して分析させる。

### 依存関係マップの構造検証

`outline/CLAUDE.md` の依存パターンと `outline/part-XX.md` のアプリ進行表の整合性を検証: 循環依存、ノードの存在確認、アプリ進行表の入力状態と前 Section の出力状態の一致、初出マップとの整合。

### ハンズオンの整合性

最も重点的にチェックする。

1. **前提の連続性**: 前の Chapter の完了状態を正しく前提としているか。`outline/CLAUDE.md` の依存パターンおよび `outline/part-XX.md` のアプリ進行表と実際の内容の一致。暗黙の依存がないか
2. **アプリ進行との整合**: ハンズオンプロジェクトがある Part では、`outline/part-XX.md` の「Section ごとのアプリ進行」テーブルと照合し、入力状態・追加内容・出力状態が正しいか
3. **サンプルの一貫性**: Chapter をまたいで使うサンプルの構成・命名が一貫しているか
4. **難易度の進行**: 自然に難易度が上がっているか。飛躍や逆行がないか
5. **操作手順の完全性**: 「前の Chapter で作成した〇〇を使って」が実際に存在するか

### 概念・用語の一貫性

1. **用語の定義の一貫性**: 同じ概念が異なる言葉で説明されていないか
2. **初出/復習の正確性**: 初出マップと実際の内容の一致
3. **説明の深さの整合**: 後の Chapter で初出より浅い説明になっていないか
4. **相互参照の正確性**: 「Chapter X で学んだように」の参照先に該当内容が存在するか

### 文体・トーンの統一性

1. **説明密度のばらつき**: Chapter による説明の詳しさの差
2. **読者への語りかけ方**: フォーマル/カジュアルの統一
3. **実務コンテキストの密度**: 特定 Chapter への偏り
4. **注意書き・補足のスタイル**: `> ⚠️ 注意:` 等の統一

## 4. 出力

出力は以下の3パートで構成する。読み手がレビュー結果を受け取った後、すぐに修正作業に移れることを最優先とする。

### パート1: サマリー（最初に出力）

レビュー全体の概況を3行以内で報告する。

```
**Chapter 2-1 レビュー結果**: 指摘 X 件（🔴 X / 🟡 X / 🔵 X）
修正すればリリース可能 | 設計変更が必要 | 大幅な書き直しが必要
```

### パート2: 指摘詳細

問題のない観点は「**観点名: 問題なし**」と1行で済ませ、指摘事項がある観点のみテーブルで報告する。各指摘に重要度と通し番号を付与する。重要度は絵文字プレフィックスで視認性を確保する。

| 表記 | 重要度 | 基準 |
|---|---|---|
| 🔴 | 高 | 技術的誤り（コード例の間違い、公式ドキュメントとの矛盾等）。ハンズオンが動かなくなる不整合 |
| 🟡 | 中 | 設計との不整合、一貫性の問題。読者が混乱しうる |
| 🔵 | 低 | 表記揺れ、軽微なスタイルの問題。文体の軽微なばらつき |

横断整合性チェックを実行した場合は、ハンズオンの整合性・概念/用語の一貫性・文体/トーンの統一性・依存関係マップの整合性も報告する。

### パート3: 修正アクションリスト（最後に出力）

指摘詳細から修正すべき項目を抽出し、重要度の高い順に並べた修正アクションリストを出力する。ユーザーが「これを修正して」と言えばそのまま作業に移れる粒度にする。

各アクションには以下を含める:
- **対象ファイル**: Section ファイル名
- **対象箇所**: 該当する見出しまたは行の内容（特定できる程度に）
- **現状**: 今どうなっているか（短く）
- **修正内容**: 具体的に何をどう変えるか

```
## 修正アクションリスト

重要度順に並べています。「修正して」と言っていただければ対応します。

| # | 重要度 | ファイル | 箇所 | 修正内容 |
|---|---|---|---|---|
| 1 | 🔴 | 2-1-1 | `## 認証` のコマンド表 | `/status` → `claude auth status` に修正（公式ドキュメントに `/status` の記載なし） |
| 2 | 🟡 | 2-1-1 | `## 概要` | 引用文 `> このセクションでは、手順に沿って進めます。` を追加 |
| 3 | 🔵 | 2-1-1 | `### 💡 クラウドプロバイダ…` | 見出しから 💡 を除去し本文に移動 |
```
