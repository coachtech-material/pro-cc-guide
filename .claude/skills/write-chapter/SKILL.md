# Chapter の執筆ワークフロー

**入力**: $ARGUMENTS

**解釈ルール**:
- 入力を `Part番号-Chapter番号` として解釈する（例: `2-3` → Part 2 の Chapter 3）
- Chapter 番号以降のテキストは追加の執筆指示として反映する
- Chapter ディレクトリパス: `curriculums/part-XX_タイトル/chapter-XX_タイトル/`（タイトルは `outline/part-XX.md` から取得）

以下のステップは判断の指針であり、チェックリストではない。準備と検証は確実に行い、執筆は自由にやる。

## 1. 準備

必要な情報をすべて集める。

1. **設計読み込み**: `outline/part-XX.md` を Read し、対象 Chapter の設計全体を把握する（ゴール、記述パターン、スコープ外、Subsection、執筆メモ、公式ドキュメント URL、見極めの重点観点など）。ハンズオンプロジェクトがある Part ではアプリ進行表から入力状態・出力状態も把握する
2. **依存関係確認**: `outline/CLAUDE.md`（自動ロード）の依存関係マップで依存パターンを確認する。線形の場合はアプリ進行表で前 Chapter との接続を検証する。前提 Chapter が未執筆の場合はユーザーに警告する
3. **情報取得**: `outline/part-XX.md` 内の公式ドキュメント URL を WebFetch する。Claude Code の機能・コマンドの正確性の拠り所になるため省略しない。業界動向・調査データ・専門家の見解を扱う Section では、WebSearch / WebFetch で外部ソースも調査する
4. **既存 Chapter の分析**: `curriculums/` 配下に既存 Section ファイルがある場合、サブエージェントで分析する（ハンズオンの連続性、概念・用語の初出と一貫性、前 Chapter からの接続）

## 2. 内面化

CLAUDE.md の哲学（WHO / WHY / WHAT）と準備で集めた情報を踏まえ、「この Chapter は Pro生 にとって何か」を考える。CLAUDE.md より: 「この哲学と方向性を踏まえた上で、題材の選択・構成のアレンジ・外部調査の実施など、執筆上の判断は好きにやってよい。臨機応変に自分の強みを活かし、Pro生に届く最高の教材を作ること。」

- この Chapter は Pro生 にとって**どういう体験であるべきか**（何に驚き、何ができるようになり、何が変わるか）
- outline の設計で**その体験が実現できるか**。できない場合、題材の変更・追加調査・構成のアレンジで何を変えるか

ここでの判断が執筆の質を決める。形式的に通過せず、本気で考える。

## 3. 方針合わせ

内面化の判断結果をユーザーに伝え、方向性を揃える。このステップで執筆のビジョンを共有し、ユーザーの視点を取り込むことで、書き直しのリスクを減らし、より良い教材になる。

伝える内容:
- **体験設計**: この Chapter で Pro生 にどういう体験を届けるか（内面化の核）
- **構成判断**: 各 Section の見出し構成を提案する。outline のスコープを変更する場合は理由を含める
- **不明点**: 判断に迷う点や確認したい点があれば質問する

ユーザーの反応を受けてから執筆に入る。ここは対話の場であり、承認ゲートではない。ユーザーが「いいね、書いて」と言えばそのまま進み、「ここはこうしたい」と言えば取り込む。大事なのは、自分が何を考えてどう書こうとしているかを伝えること。

## 4. 執筆

CLAUDE.md の哲学を踏まえ、`.claude/rules/writing-style.md` の表記ルールに沿って書く。

**Section ファイルの固定要素**（1 Section = 1 ファイル）:

- `# X-X-X タイトル`（コロンなし）
- `## 概要`
- 本文
- `## 公式ドキュメント`（URL がない場合のみ省略可）
- `## まとめ`（箇条書き + 次への接続）
- `---`（水平線）は使用しない

**記述パターン**（outline の `記述パターン:` が示す性質）:

- **手順**: How 主体。順を追って作業する
- **概念**: Why → What。手を動かす内容は含めない
- **混合**: Why → What → How の流れで解説と実践を一体的に書く。独立した `## ハンズオン` は作らない。コード生成を伴う場合は Section 末尾に `## 見極めチェック` を配置する

記述パターンは構造の目安であり、テンプレートの強制ではない。

- Section 間に内容の依存がない場合は並列に執筆してよい

## 5. 検証・完了

1. **公式ドキュメント照合**: サブエージェントに公式ドキュメントを再取得させ、コード例・コマンド例・機能説明との差異を報告させる
2. **レビュー**: `/review-chapter [Part番号-Chapter番号]` を実行する
3. **ユーザー確認**: レビュー結果を報告する。軽微な修正は報告と同時に修正してよい。設計変更を伴う修正はユーザー確認を取る
4. **依存関係マップ更新**: `outline/CLAUDE.md` の依存関係マップを検証し、差分があれば反映する
