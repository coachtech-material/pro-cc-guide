# 執筆ルール

## 1. 文体

### トーン

学習者の隣にいる**先輩エンジニア**として書く。Claude Code との協働を通じて**驚き**と**面白さ**を届ける。

- **本文**: 落ち着いた敬体で丁寧に解説する。親しみやすいが、あくまで「教材」の語り口。過度に個人的な語りかけ（「率直に言うと」「正直に言えば」等）は避ける
- **🧠 先輩エンジニアはこう考える**: ここに実体験ベースの語り（「現場ではこうだった」「自分もこれで失敗した」）を集中させる。本文との緩急でメリハリを出す

### 用語・表記

- 敬体（です・ます調）で統一
- 読者への呼びかけは「あなた」
- ダッシュ記号（`——`、`—`、`–`）は使わない。句読点や括弧で表現する
- Claude Code の機能名は公式ドキュメント（ https://code.claude.com/docs/en/ ）の表記に従い英語のまま
- コード例・コマンド例は公式ドキュメントと照合する。範囲外の内容は WebSearch / WebFetch で裏付ける
- 時間経過で変わる情報（料金・バージョン等）は最新情報を取得し「○○時点」と日付を明記
- 外部ソースを参照する場合は出典を明記する（人名 + 記事タイトル or URL）。統計データは調査元と時期を示す。例: 「Veracode の 2025 年調査によると」「Addy Osmani 氏の記事 [The 70% Problem](URL) によると」
- Markdown の `**text**` は閉じ `**` 直後にスペースなしで日本語が続くと適用されない。`**○○** の場合` のようにスペースを入れる

| 表記 | 備考 |
|---|---|
| WSL | Windows Subsystem for Linux。初出時に正式名称を記載 |
| MCP | Model Context Protocol。初出時に正式名称を記載 |
| セクション | 教材本文ではカタカナ表記。内部ドキュメントでは英語 |

### 絵文字

| 絵文字 | 用途 |
|---|---|
| 🎯 | 学習目標（Section 冒頭） |
| ✨ | まとめ（Section 末尾） |
| 💡 | TIP・補足・実務知見 |
| ⚠️ | 注意事項・よくあるエラー |
| 📝 | ノート・定義・補足説明 |
| 🔑 | 重要概念・キーポイント |
| 🏃 | ハンズオン手順・実践 |
| 🧠 | 先輩エンジニアはこう考える |
| 🔍 | 見極めチェック |
| 📌 | 操作前の確認事項（ハンズオン・実践の前提条件） |
| 📖 | この Chapter の進め方 |

## 2. 構造

### Section の種類

| 種類 | 内容 | 学習者向け表記（学び方列） |
|---|---|---|
| **概念** | 機能の意義・仕組み・使い方を解説。コード生成なし | 📖 座学 |
| **ハンズオン** | 概念で学んだ機能を実践。冒頭に概念 Section への逆リンク | 🏃 実践 |
| **混合** | 概念を学びながらすぐに手を動かす。セットアップや機能の初回体験など | 🏃 実践 |

各 Section の種類は `outline/part-XX.md` で指定する。X-X-0（Chapter 地図）や Chapter 内の Section 一覧テーブルでは、執筆都合の「種類」ではなく学習者向けの「**学び方**」列（📖 座学 / 🏃 実践）で表記する。読者向けは「Claude Code を実際に触るかどうか」の 2 区分に集約する（ハンズオンと混合はどちらも 🏃 実践）。

### 見出しの設計

Section 内の見出しは **Why → What → How**（抽象→具体）の流れで組む。

- **Why**: なぜ必要か（課題・動機）
- **What**: 何か、何ができるか（定義・機能・概念）
- **How**: どう使う / どう判断するか（使い方・判断基準）

### テンプレート

**概念 Section**（Chapter 地図（X-X-0）がある場合、Chapter 全体像はそちらに記載されるため、X-X-1 は `# X-X-X セクションタイトル` → `🎯 このセクションで学ぶこと` から始める。Chapter 地図がない場合のみ、最初の Section に Chapter 全体像（概要・セクション一覧・進め方）を含める。2つ目以降は常に `# X-X-X セクションタイトル` から始める）:

## 3. コンテンツ

### コードの見せ方

| 種類 | 言語指定 |
|---|---|
| ターミナルコマンド | `bash` |
| Claude Code への入力（プロンプト） | なし（コードブロック内で行頭に `>` を付ける） |
| 生成コード・ファイル内容 | 適切な言語（`php`, `json` 等） |
| コマンド実行結果 | なし |

- 生成コードには「あなたの環境では異なるコードが生成されます」と前置き（初回は丁寧に、以降は簡潔に）
- コードブロック上部にファイルパスをコメントで明示
- diff 表示は使わない（テキストで説明し変更後のコードを示す）
- 複雑なコマンドは分解して1つずつ解説する
- プロンプト例・コード例・Section 間の接続は要件レベル（「認証機能を追加して」）で書き、特定のファイル名・コード構造に依存しない

### 解説の深さ

- 前提知識（CLAUDE.md のペルソナ参照）は基本説明を省略する。ただし Claude Code と組み合わせたときの注意点（例: Sail コマンド体系）は丁寧に説明する
- Claude Code 固有の概念は初出時に丁寧に解説し、2回目以降は初出 Section への参照を示す
- 同じ概念を別 Chapter で繰り返さない
- 導入は「これがないと何が困るか」（課題）から入り、前のセクションへの参照と次への予告で接続する
- 図の使い分け:
  - **Mermaid（```` ```mermaid ````）**: フローチャート、アーキテクチャ図、循環プロセスなど、ノードと矢印で構成される図。GitHub がネイティブレンダリングする
  - **ASCII（`→` `↓` `│` `├` `└` 等）**: ディレクトリツリー、比較表、2-3 行の簡易フローなど、テキストのままの方が直感的な図
- 生成コードの重要箇所（なぜそのコードが生成されたか、何に注意すべきか）は解説する。逐語解説は不要だが、理解すべきポイントは明示する
- **つまずきポイントを先回りする**: エラーが起きやすい箇所、混同しやすい概念、設定ミスしやすい手順は、読者が遭遇する前に ⚠️ で説明する
- **「今は理解しなくてOK」を明記する**: 先の Chapter で詳しく扱う内容や、高度な概念に触れるときは「今は概要だけ把握すれば十分です」と心理的負担を下げる
- 初回リリースでは画像を含めず `<!-- TODO: 画像追加 - ○○の画面 -->` でプレースホルダーを残し、画面の状態をテキストで描写する
- 1 Section あたり**最低 4,000 文字**。上限は設けない。説明を省略・簡略化せず、一切の省略なく詳細に丁寧に書く
