# 表記ルール

## 1. 文体・用語

- 文体は敬体（です・ます調）で統一する
- ダッシュ記号（`——`、`—`、`–`）を文中の接続・言い換え・挿入に使わない（例: 「ジュニアエンジニア——「Pro生」の皆さん」）。句読点や括弧で表現する
- Claude Code の機能名・コマンド・フラグの説明は公式ドキュメント（https://code.claude.com/docs/en/）を正確性の拠り所とする。コード例・コマンド例は公式ドキュメントと照合する。業界動向・外部の調査データ・専門家の見解など、公式ドキュメントの範囲外の内容は WebSearch / WebFetch で外部ソースを調査して裏付ける
- 料金・バージョン番号・統計データなど時間経過で変わる情報を記載する場合、執筆時点の最新情報を取得して反映する。「○○時点の情報です」と日付を明記し、過去の情報をそのまま転記しない
- Claude Code の機能名は公式ドキュメントの表記に従い、英語のまま使用する
- Markdown の太字（`**text**`）は、閉じ `**` の直後に半角スペースなしで日本語テキストが続くと太字が適用されない（CommonMark の仕様）。`**Console（API）**の場合` ではなく `**Console（API）** の場合` のようにスペースを入れる。特に全角括弧 `）` の直後の `**` で発生しやすい

| 表記 | 備考 |
|---|---|
| WSL | Windows Subsystem for Linux。初出時に正式名称を記載 |
| MCP | Model Context Protocol。初出時に正式名称を記載 |
| Pro生 | 教材本文では各 Part の最初の Section で「Pro生」と定義し、以降は「皆さん」を使う |
| セクション | 教材本文では Section をカタカナ「セクション」で表記する。outline やスキル定義など内部ドキュメントでは英語表記のまま |

## 2. コードブロックの表記

Claude Code の教材では、ターミナルコマンド・Claude Code への入力・生成コードが頻繁に登場する。言語指定で種類を区別する。

| 種類 | 言語指定 | 用途 |
|---|---|---|
| ターミナルコマンド | `bash` | シェルで実行するコマンド（`claude`, `sail artisan` 等） |
| Claude Code への入力（プロンプト） | なし（言語指定なし）。行頭に `>` を付ける | Claude Code のインタラクティブモードで入力する指示。スラッシュコマンド（`/init`, `/commit` 等）も同じ形式 |
| 生成コード・ファイル内容 | 適切な言語（`php`, `json`, `yaml` 等） | Claude Code が生成したコード、設定ファイルの内容 |
| コマンド実行結果 | なし（言語指定なし） | コマンドの出力、ターミナル表示 |

例:

ターミナルコマンド:
````
```bash
claude
```
````

Claude Code への入力（1行）:
````
```
> Userモデルにroleカラムを追加してください。enum型でadminとauthorの2値です。
```
````

Claude Code への入力（複数行）:
````
```
> 以下の3つのモデルを作成してください。
> 1. User（role: admin / author）
> 2. Article（title, body_markdown, status）
> 3. Category（name, slug）
```
````

スラッシュコマンド:
````
```
> /init
```
````

生成コード:
````
```php
// database/migrations/xxxx_add_role_to_users_table.php
Schema::table('users', function (Blueprint $table) {
    $table->enum('role', ['admin', 'author'])->default('author');
});
```
````

- 生成コードには「皆さんの環境では異なるコードが生成されます。以下は一例です」と前置きする。Chapter の最初のコード例で丁寧に説明し、以降は簡潔に
- 生成コードのコードブロック上部にファイルパスをコメントで明示する
- diff 表示は使用しない（変更内容はテキストで説明し、変更後のコードをコードブロックで示す）
- プロンプト例・コード例・見極めチェック・Section 間の接続はすべて要件レベル（「認証機能を追加して」）で書き、特定のファイル名・メソッド名・コード構造に依存しない。前の Section の生成結果を参照する場合は「既存の実装に合わせて」と要件レベルで指示し、Claude Code に既存コードを読ませて判断させる

## 3. 定型フォーマット

### 絵文字

⚠️（注意事項・よくあるエラー）、💡（TIP・補足・実務知見）の2種を使用する。

### エラーフォーマット

エラーが起きやすい箇所は以下のフォーマットで先回りして説明する:

> ⚠️ よくあるエラー: [エラーの概要]
>
> ```
> [エラーメッセージ例]
> ```
>
> **原因**: [なぜ発生するか]
>
> **対処法**: [具体的な解決手順]

### 見極めチェック

コード生成を伴う混合パターンの Section 末尾に配置する。3観点（正しさ・品質・安全性）すべてにチェック項目を MECE に列挙し、該当しない観点は「この Section では該当なし」と明記する。重点観点は blockquote で補足する。

```markdown
## 見極めチェック

- [ ] **正しさ**: [この Section で生成されたコードが要件を満たすかの具体項目]
- [ ] **品質**: [保守性・パフォーマンス・設計の一貫性の具体項目]
- [ ] **安全性**: [脆弱性・機密情報の具体項目]

> この Section では特に「○○」に注目してください。[重点観点の補足説明]
```

## 4. 解説の深さ

### 前提知識の扱い

- Pro生が COACHTECH で学んでいる技術（Laravel, Git, Docker 等）は前提知識として扱い、基本的な説明は省略する。ただし Claude Code と組み合わせたときの注意点（例: Sail コマンド体系）は丁寧に説明する
- Claude Code 固有の概念・操作は初出時に丁寧に解説する。2回目以降は簡潔に触れ、初出 Section への参照を示す

### 画面の描写

初回リリースでは画像を含めず `<!-- TODO: 画像追加 - ○○の画面 -->` でプレースホルダーを残す。画像がない分、以下で補完する:

- 画面の状態・ターミナル出力・UI の見え方をテキストで詳細に描写する
- コマンドの実行結果は、正常系の出力例を必ずコードブロックで示す
- 操作手順は「何をするか」だけでなく「画面にどう表示されるか」「何が変わるか」まで書く
