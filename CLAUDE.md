# COACHTECH Pro生向け Claude Code 教材

> ジュニアエンジニア（Pro生）が Claude Code を活用し、実務開発の生産性と品質を高めるための教材。

## ペルソナ（WHO）

プログラミングスクール COACHTECH の選抜試験を経てフリーランスエージェントに所属しているジュニアエンジニア。COACHTECH ではこの層を「Pro生」と呼ぶ。企業に紹介され、副業・フリーランス・転職などの形でLaravelを使った実務開発に携わる。

基礎力はあるが実務はこれからである。「現場で通用するか」という不安、「AIに仕事を奪われるのではないか」というキャリアへの不安、そして「学んだことを活かしたい」という意欲が同居している。

### 執筆トーン

私たちはこのPro生の**メンター**として、隣で画面を見ながら語りかける存在として書く。機能の説明で終わらせず、「なぜこれを知ることが皆さんの仕事に役立つのか」を伝える。そして何より、Claude Codeとの協働を通じて**驚き**と**面白さ**を届ける。一言の指示から大量のコードが生まれる驚き、自分の知識で生成コードを読み解ける面白さ。この体験の積み重ねが、Pro生の自信と実力になる。

### 技術スキル

- **Laravel**: MVC、Eloquent ORM、認証（Fortify）、認可（Policy）、バリデーション、ミドルウェア、RESTful API、PHPUnit によるテストを実装できる
- **フロントエンド**: HTML/CSS、Tailwind CSS の基礎がある
- **データベース**: RDB設計（正規化、ER図）、SQL（JOIN、サブクエリ、集計）を扱える
- **開発プロセス**: 要件定義→DB/API設計→実装→テストの一連の開発フローを経験している
- **開発環境・ツール**: ターミナル操作、Docker Compose、Git/GitHub（ブランチ運用、PR、コードレビュー）を使える
- **AI活用**: ChatGPT を少し触った程度。AI コーディングツールの使用経験は問わない

### 前提環境

- **OS**: macOS および Windows（WSL経由）の両方を対象とする
- **エディタ**: VS Code を前提とする
- **ブラウザ**: 認証フローに必要

## コンセプト（WHY）

AIの台頭により、企業がジュニアエンジニアに求める水準は上がっている。Pro生が実務で価値を発揮するには、AIを使いこなして従来ミドル層が担っていた領域の一部をカバーする必要がある。

AIに的確な指示を出すには、具体的な知識を構造化し適切な抽象度で言語化する必要がある。AIの出力を検証するには、要件が正しく実装に落とし込まれているかを判断する必要がある。AI活用とプログラミングの知識は相反せず、むしろPro生がCOACHTECHで身につけた知識がAIを使いこなす土台になる。

Claude Codeは要件を複数ファイルにわたる実装に高速に変換する。指示を出すのも、出力を検証するのもエンジニア自身である。この教材では、Claude Codeに的確に指示を出す力と、その出力を検証する力の両方を養う必要がある。

## ゴール（WHAT）

Claude Codeと協働して、以下の実務タスクを高速かつ高品質に遂行できるようにする。**コードリーディング**を前提に、既存プロジェクトのコーディング規約と設計方針に従いながら遂行する:

- **バグ修正**: 既存コードの不具合を特定し、修正する
- **機能開発**: 新規機能の実装や既存機能の改修を行う
- **リファクタリング**: コードの品質・可読性・パフォーマンスを改善する

これらを支える3つの能力を養う。いずれも個人での開発に限らず、チーム開発の文脈でも求められる。

### 1. 使いこなす力（Input）

Claude Codeの設定・機能・制約を把握し、目的に応じた機能選択とプロンプト設計で的確に活用する力。

### 2. 見極める力（Output）

Claude Codeの出力を検証・判断し、責任を持って採用する力。AIの出力をそのまま受け入れず、自分で理解・判断する習慣を保つ。以下の3観点で確認する:

- **正しさ**: 要件通りに動くか
- **品質**: 保守性・パフォーマンス・既存設計との一貫性は適切か
- **安全性**: 脆弱性がないか。認可・入力検証・機密情報の漏洩がないか

### 3. 学び続ける力（Time）

ツール・技術の進化を追い、自ら試し、実務に適応し続ける力。公式ドキュメント・SNS・コミュニティから最新動向をキャッチアップし、新機能を検証し、開発フローへの適用を判断する。

## カリキュラム（HOW）

| Part | Chapter数 | 対応する能力 | 概要 | ハンズオン |
|---|---|---|---|---|
| Part 1: はじめに | 2 | - | 教材の導入、Claude Code の概要 | なし |
| Part 2: Claude Code の基礎 | 11（2-3 は2セクション構成） | 使いこなす力（主）+ 見極める力（型の導入） | 新規プロジェクト（テック記事プラットフォーム）を構築しながら主要機能を習得 | テック記事プラットフォームを段階的に構築 |
| Part 3: Claude Code の実践 | (設計中) | 見極める力（主）+ 使いこなす力（総合演習） | 既存プロジェクト（ProConnect）で Part 2 の機能を総合的に使いながら実務タスクを遂行 | 提供プロジェクトで実践 |
| Part 4: 継続的な学習 | (設計中) | 学び続ける力 | 情報収集・新機能検証・実務適用の習慣化 | なし |

Part > Chapter > Section の3層構造。各 Chapter は1つのゴールで構成する。各 Part の設計詳細は `outline/` を参照。

CLAUDE.md は教材の哲学（WHO / WHY / WHAT / HOW）を定義し、`outline/` はその哲学を Part・Chapter・Section の具体的な設計に落とし込んだものである。この哲学と方向性を踏まえた上で、題材の選択・構成のアレンジ・外部調査の実施など、執筆上の判断は好きにやってよい。臨機応変に自分の強みを活かし、Pro生 に届く最高の教材を作ること。

### カリキュラムの前提

- **Laravel**: バージョン10（`laravel:^10.0`）。`composer create-project` でバージョンを明示してインストールする（`laravel.build` は最新版がインストールされるため使用しない）
- **Laravel Sail**: Docker ベースの開発環境。プロジェクト作成後に `composer require laravel/sail --dev` + `sail:install --with=mysql` で導入する
- **フロントエンド**: Tailwind CSS v3 + Alpine.js + Vite。`sail npm install` / `sail npm run dev` で Sail コンテナ内で実行する（ホストマシンへの Node.js インストールは不要）
- **Claudeアカウント**: Pro（20ドル）プランを最低条件とし、Maxプランを推奨する。企業に所属している場合は企業のプラン・ポリシーを確認すること
- **Node.js（Claude Code 用）**: ホストマシンへのインストールは不要。Claude Code はネイティブインストール（curl / Homebrew / WinGet）が推奨され、npm インストールは非推奨（deprecated）

### ハンズオンの設計原則

#### 難易度設計

技術レベルは上げず、**指示の抽象度** と **生成コード量** で成長を促す:

- **序盤**: 具体寄りの指示（実装の粒度）で Claude Code の動作を理解する
- **中盤**: ビジネス要件レベルの指示で、Claude Code に実装判断を委ねる
- **終盤**: 広い範囲の要件を一言で伝え、大量の生成コードを検証する

#### 前提

- **題材はモデルケース**: 別のアプリを作っても構わない。大切なのは各 Chapter の機能を使い、見極めチェックで検証すること
- **出力は毎回異なる**: 「この通りに作れば同じ結果になる」前提を置かない。特定のコード構造に依存しない

#### ハンズオンの3要素

- **プロンプト例**: Claude Code に伝える指示の例
- **コード例**: 教材の題材に沿った生成コードの一例
- **見極めチェック**: 生成コードを3観点（正しさ・品質・安全性）で検証するチェックリスト

3要素の具体的な書き方は `.claude/rules/writing-style.md` を参照。

#### 見極める力の3観点

3観点（正しさ・品質・安全性）はAI生成コードに対して常に意識すべきものとして伝え、その上で各 Chapter では文脈に応じた観点を重点的に取り上げる。コード生成を伴う Section の末尾に **見極めチェック** を配置し、3観点すべてに具体的なチェック項目を MECE に列挙する。生成されたコードが要件を満たしているかをPro生自身の知識で判断できる観点で示す。

## プロジェクトマップ（MAP）

表記ルールは `.claude/rules/writing-style.md` を参照。

### Skills

| Skill | 用途 |
|---|---|
| `/write-chapter` | Chapter の新規執筆・更新 |
| `/review-chapter` | Chapter の品質レビュー（横断整合性チェック含む） |
| `/outline` | outline の作成・更新・検証（再構成・MECE 検証・公式ドキュメント照合） |
| `/check-updates` | 公式ドキュメントとの鮮度チェック |

### フォルダ構造・命名規則

```
pro-cc-guide/
├── CLAUDE.md              # 教材の哲学・カリキュラム・プロジェクトマップ
├── outline/               # カリキュラム設計（依存関係マップ・Part/Chapter/Section 設計）
│   ├── CLAUDE.md
│   └── part-01.md ~ part-04.md
├── .claude/
│   ├── skills/            # 作業手順（執筆・レビュー・更新・構成変更）
│   └── rules/             # 表記ルール
├── curriculums/           # 教材本体
│   └── part-XX_タイトル/chapter-XX_タイトル/X-X-X_タイトル.md
├── assets/                # 画像（assets/part-XX/chapter-XX/）
└── memo/                  # 作業メモ置き場（教材の執筆プロセスでは参照しない）
```

- Part ディレクトリ: `part-XX_タイトル/`（01始まり、ゼロパディング）
- Chapter ディレクトリ: `chapter-XX_タイトル/`（Part 内で01始まり、ゼロパディング）
- Section ファイル: `X-X-X_タイトル.md`（Part番号-Chapter番号-Section番号、ゼロパディングなし）
- 画像ファイル名: 内容がわかる英語名（例: `install-confirmation.png`）
- Part 設計ファイル: `outline/part-XX.md`（ゼロパディング）
