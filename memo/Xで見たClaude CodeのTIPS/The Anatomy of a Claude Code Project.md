「The Anatomy of a Claude Code Project」
（Claude Codeプロジェクトの「解剖学」＝体全体の仕組み）

必要なのはたった4つの柱：
Why：このシステムは何のために存在する？（目的）
Map：ファイルやモジュールはどこにある？（地図・構造）
Rules：何をやっていい／ダメ？（ルール・禁止事項）
Workflows：仕事はどう進める？（繰り返し手順）

これを永続的に実装するための実践的なフォルダ＆ファイル構成がこれです↓

1. CLAUDE.md（プロジェクトの「脳みそ」・北極星ファイル）
ルート直下に置く。絶対に短く・明確に！
入れるのは厳選3つだけ：

Purpose（なぜこのプロジェクトがあるのか）
Repo map（フォルダ全体の構造・どこに何があるか）
Rules + commands（コーディング規約、禁止ディレクトリ、よく使うコマンド）

長くなるとClaudeが大事なシグナルを見逃す → 「明確さ > ボリューム」 が命。

2. .claude/skills/（再利用可能な「専門スキル」集）
毎回同じ長い指示をプロンプトで打つのを卒業！

よく使うワークフローをフォルダ化してスキル化
（例）：
code-review/ → SKILL.md（レビュー基準リスト）
refactor/ → SKILL.md（リファクタ手順）
release/ → SKILL.md（リリースフロー）

Claudeが自動で読み込んで即「専門モード」に切り替わる。
チーム共有しても出力が超安定！

3. .claude/hooks/（自動で守ってくれる「ガードレール」）

Claudeはセッションごとに忘れちゃうけど、hooksは忘れない。

必ず自動実行したいルールをここに設定（例）：
編集後に自動フォーマット実行
コア変更後にテスト自動トリガー
auth/billing/migrationsなどの機密フォルダを触らせないブロック
これでAIが「ただの賢いツール」→「信頼できるエンジニアリングパートナー」に進化。

4. docs/（「真実の図書館」・詰め込み禁止のコンテキスト）

プロンプトに全部情報を詰め込まない！
代わりにClaudeに「自分で探させる」設計に：

architecture.md（全体アーキテクチャ概要）
decisions/（ADR：アーキテクチャ決定記録）
runbooks/（運用手順書）

Claudeは「真実がどこにあるか」さえ知っていれば、必要なときにナビゲートしてくれる。

5. 危険ゾーン専用：ローカルCLAUDE.mdミスりやすい複雑な部分

（例：auth、persistence、infraなど）のフォルダ直下にCLAUDE.mdを置く。

その場所で作業するときだけ自動読み込み → Claudeが「ここはヤバい！」と正確に警戒。

致命的ミスが劇的に減る！最後に一番大事な気づきプロンプトは「一時的な魔法」

構造は「永続的な基盤」この「解剖学」を一度整えるだけで：
新しいチャットを開いてもClaudeが即「プロジェクトの住人」になる

毎回ゼロから説明不要
チーム全体の生産性が爆上がり
